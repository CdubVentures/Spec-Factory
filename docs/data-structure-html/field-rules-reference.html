Sub-consumers
How to read this doc. Each field path expands to show every sub-consumer that reads it — one chip per sub-consumer. Chip labels follow the PARENT.SUBNAME pattern (e.g. IDX.NEEDSET, LLM.KF, REV.GRID) so two rev chips on the same row means two distinct rev-stage consumers (e.g. REV.COMPONENT + REV.GRID), not a duplicate. Chip color encodes the parent group; the full key encodes the specific stage. A field with multiple sub-consumer chips has a real dependency on each — editing the field value in Field Studio affects every one. The GUI Key Navigation breadcrumb (e.g. Contract (Type, Shape, Unit) › Data Type) mirrors the layout of the Field Studio editor, so a path's name here and the path a GUI author clicks through match one-to-one.

Parent Groups (Pipeline Owners)

IDX.*
Indexing Lab — search planning, query composition, field-need scoring. Reads happen during needset construction and per-tier query generation in src/features/indexing/. Chips: IDX.NEEDSET, IDX.SEARCH
ENG.*
Field Rules Engine — runtime validation, normalization, enum resolution, component matching, evidence gating. Lives under src/engine/. Chips: ENG.VALIDATE, ENG.NORMALIZE, ENG.LIST, ENG.ENUM, ENG.GATE, ENG.COMPONENT
VAL.*
Publish Pipeline Validation — deterministic validators and publish-gates that run just before publish. Type, shape, unit, range, list, rounding, enum, format checks in src/features/publisher/validation/, plus the unk-block publish gate (shouldBlockUnkPublish.js). Chips: VAL.TYPE, VAL.SHAPE, VAL.UNIT, VAL.RANGE, VAL.LIST, VAL.ROUNDING, VAL.ENUM, VAL.FORMAT, VAL.PUBLISH_GATE
PUB.*
Publisher Pipeline — set-union merges and final evidence gate at publish time. Lives under src/features/publisher/publish/. Chips: PUB.UNION, PUB.GATE
SEED.*
Seed Pipeline — compile-time seeding of SpecDb field meta, enum lists, and component identities from the compiled field_rules artifacts. Chips: SEED.SCHEMA, SEED.COMPONENT
REV.*
Component Review — review grid, component review helpers, and review-surface override services. Four distinct sub-consumers. A field may have multiple rev chips if it flows through several review surfaces. Chips: REV.COMPONENT, REV.GRID, REV.METADATA, REV.OVERRIDE
COMP.*
Component System — component review panel column mapping. Lives alongside the review surface. Chips: COMP.REVIEW
FLAG.*
Review Flags — reserved for field-level flag consumers. No live consumer at the moment; retained for future use.
LLM.*
LLM Finder — per-key LLM finder surface: prompt composition (LLM.KF), tier-based model routing (LLM.ROUTE), per-key attempt budget (LLM.BUDGET), and passenger bundling (LLM.BUNDLE). Lives in src/features/key/keyLlmAdapter.js (prompts), keyFinder.js (routing), keyBudgetCalc.js (budget), keyBundler.js + keyPassengerBuilder.js (bundling). Chips: LLM.KF, LLM.ROUTE, LLM.BUDGET, LLM.BUNDLE
