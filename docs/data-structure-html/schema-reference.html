Manual audit metadata
Rebuild, source edit, strict badges, and lifecycle notes are maintained as audited judgments in this generator and must be updated when behavior changes.

Badge meaning: rebuild yes/partial/no answers whether a deleted SQLite file can be reconstructed from the current durable source. source edit yes/partial/no answers whether add/remove edits in that durable source are honored on a fresh rebuild.

Interpretation: rebuild yes + source edit yes means a fresh deleted-DB rebuild is expected to reflect the authoritative durable source set for that table. It is not a blanket claim that every runtime path, existing-DB reconcile path, write-back path, or non-authoritative file is perfect. strict yes is the closest thing to “perfect” in this audit, including add/remove edits in the authoritative durable source set being reflected on that fresh rebuild.

Badge What it means What it does not mean
rebuild yes A deleted DB can be rebuilt for that table from the current authoritative durable source set without depending on hidden SQL-only state. Not a promise that existing-DB reconcile, runtime write-back, or every upstream file path is perfect.
rebuild partial Some rebuild path exists, but it has a known loss, omission, authority split, or dependency that prevents a full deleted-DB guarantee. Not safe to treat as lossless.
rebuild no No current durable source can fully restore that table on a deleted-DB rebuild. Not recoverable unless a new durability surface is added.
system System-managed operational metadata, not a user durability surface. Not a user-editable source-of-truth claim.
strict yes No known caveat remains for the deleted-DB rebuild contract or authoritative source-edit contract for that table. Add/remove edits in the authoritative durable source set are expected to be reflected on a fresh rebuild. Still assumes valid source payloads and the documented authoritative source set; it is not a promise about unrelated runtime behavior.
strict no At least one known caveat still applies, even if the table is rebuildable or source-editable in a narrower sense. Not the same as unrebuildable; it means “works, but not clean enough to be caveat-free.”
strict n/a The strict user-durability concept does not apply, usually because the table is system-managed metadata. Not a missing audit.
strict ext-dep Rebuild logic is correct and seeds from JSON, but the table references external artifacts (files on disk, S3 objects) that must survive independently for rows to be fully usable. Not a code-level rebuild defect; the SQL+JSON contract is honored. The caveat is an operational dependency on external storage.
source edit yes Direct add/remove edits in the authoritative durable source set are expected to show up on a fresh deleted-DB rebuild. Not a promise that runtime SQL edits write back out, or that non-authoritative mirrors also control the row.
source edit partial Some direct source edits propagate, but delete handling, multi-source authority, generated artifacts, or another caveat prevents a full guarantee. Not safe to assume arbitrary edits will always land cleanly.
source edit no Direct edits in the cited durable source are not currently a reliable way to drive that table on rebuild. Not an editable rebuild surface.
source edit n/a No user-editable durable source applies to that table. Not a missing audit; the concept does not apply there.
json Table is backed by a durable JSON source and can be rebuilt from it on a deleted-DB rebuild. Not a promise that every column round-trips perfectly; partial-rebuild caveats may still apply.
db-only Table lives only in SQL with no durable JSON backing. Data is lost on a deleted-DB rebuild. Not necessarily a bug; some tables (telemetry, audit trails) are intentionally ephemeral.
deferred Schema and some frontend/backend wiring exist, but the feature is not scoped or fully implemented yet. Not dead code; it is pre-wired for a planned feature.
system System-managed operational metadata, auto-recreated by the runtime. Not a user-editable persistence surface.
✓ CQRS compliant: table meets the Dual-State Architecture Mandate. Either JSON-backed with verified dual-write (edits propagate both ways and rebuild is correct), or intentionally ephemeral per the db-only carve-out (telemetry, runtime queues, audit trails). Not a guarantee of zero bugs; it means the architectural contract is satisfied.
✗ Not CQRS compliant: table has a known gap — deferred scope with missing write-back paths, or a durability concern that has not been addressed. Not necessarily broken; deferred tables are pre-wired for planned features.
