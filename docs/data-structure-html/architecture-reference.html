System Overview

High-level architecture showing how the GUI, API server, feature modules, pipeline, and storage layers connect. SQLite is the single source of truth for all mutable state. JSON files serve as immutable config or rebuildable checkpoint safety nets.

Settings Registry (SSOT)

146 runtime entries O(1) scaling

The RUNTIME_SETTINGS_REGISTRY in src/shared/settingsRegistry.js is the single source of truth for all runtime configuration. Each entry is a plain object defining key, type, default, constraints, and UI metadata. Adding one entry auto-derives: config manifest, TypeScript types, UI rendering, validation, clamping, and API payloads. The file also exports BOOTSTRAP_ENV_REGISTRY (3 entries: startup-only env vars) and UI_SETTINGS_REGISTRY (4 entries: auto-save booleans).

What the Registry Drives

Config Manifest

The runtime config object is built by iterating the registry. Each entry's configKey + envKey + default produces the resolved value. No manual wiring.

TypeScript Types

GUI frontend types (runtimeSettingsManifestTypes.ts) are derived from registry keys. The settingsPropagationContract.ts enforces alignment.

UI Rendering

The settings page reads uiCategory, uiSection, uiGroup, uiHero, uiTip, and disabledBy to auto-generate the full settings UI.

Validation + Clamping

Type, min/max, allowed, tokenClamped, and clampModelKey drive server-side validation. Invalid values are clamped or rejected before persistence.

Store Module Registry

20 modules

Each store module owns a set of SQL tables and provides the read/write API for its domain. No other module may directly query another store's tables. Store modules are initialized at database open time and exposed through the SpecDb facade. Key Architectural Principles

The foundational design decisions that shape every part of Spec Factory.

SQL is SSOT for All Mutable State

SQLite owns every piece of mutable domain data. Products, components, reviews, evidence, run metadata, billing, telemetry, source intelligence — all in SQL. No JSON file is the authoritative source for mutable state.

JSON is Config or Safety Net

JSON files are either immutable config (category authority, component DBs) or checkpoint safety nets (run.json, product.json) that can be rebuilt from SQL. The dual-write pattern maintains JSON files for recovery without treating them as truth.

Event-Driven Runtime

Pipeline phases emit structured events via the Logger. Events flow through the Bridge into bridge_events tables. The GUI consumes events via WebSocket for live updates and via pre-built panels for historical runs.

Pre-Built Panels at Finalize

At run finalization, 11 runtimeOps panel builders process the event stream once and cache the results in run_summary. Historical runs serve these pre-built panels directly — zero re-aggregation, instant load.

O(1) Feature Scaling via Registry

The RUNTIME_SETTINGS_REGISTRY (146 entries) drives config, TypeScript types, UI rendering, and validation from a single definition. Adding a new setting requires modifying exactly one file — everything else auto-derives.

Dual-Write Transition

SQL is the primary store. File writes are maintained for recovery and backward compatibility. user-settings.json has a SQL primary (app.sqlite settings table) with JSON as fallback. product.json and run.json are rebuilt from SQL data.
