System Overview

The IndexLab pipeline uses a 3-tier search system to discover product information. Each run generates search queries across 3 tiers, executes them, crawls the results, and records cooldowns. The next run reads those cooldowns and shifts budget to the next tier.

Run N starts → Read query_cooldowns from SQLite → deriveSeedStatus() // which seeds are on cooldown? → computeGroupQueryCount() // how many times has each group been searched? → isGroupSearchWorthy() // coverage, unresolved count, max repeats → computeTierAllocation() // budget split: T1 → T2 → T3 → buildSearchProfile() // generate actual query strings per tier → Execute queries (Serper API) → recordQuery() writes cooldown to SQLite // REAL-TIME, not end-of-run → Crawl discovered URLs (Playwright) → Run N+1 reads updated cooldowns

Key Phase Details

Phase 01: NeedSet (where tier decisions happen)

File: src/features/indexing/pipeline/needSet/searchPlanningContext.js

This is where the tier system makes its decisions. It runs 4 steps:

  1. deriveSeedStatus() — Reads queryExecutionHistory from SQLite. For each seed type (specs, brand, sources), checks if cooldown_until > now. If yes → is_needed: false.
  2. computeGroupQueryCount() — Counts how many active-cooldown group queries exist per group_key. Only queries with cooldown_until > now are counted.
  3. isGroupSearchWorthy() — A group is "worthy" of a broad search if:
    • Coverage ratio < 80% (needsetGroupSearchCoverageThreshold)
    • Unresolved fields ≥ 3 (needsetGroupSearchMinUnresolved)
    • Group query count < 3 (needsetGroupSearchMaxRepeats)
    • Phase is not "hold"
  4. computeTierAllocation() — Splits the query budget across tiers in priority order:
    • T1 seeds get first dibs (needed seeds consume budget)
    • T2 groups fill remaining budget (worthy groups sorted by productivity)
    • T3 keys fill whatever's left (individual field queries from non-worthy groups)

Phase 03: Search Profile (where queries are generated)

File: src/features/indexing/pipeline/searchProfile/queryBuilder.js

Iterates through tierHierarchyOrder and generates queries per tier until budget is consumed:

for (const tierId of effectiveTierOrder) {
  if (tierRows.length >= maxQueryCap) break;  // budget exhausted
  if (tierId === seedTierOrder[0]) → buildTier1Queries(...)
  if (tierId === 'group_searches') → buildTier2Queries(...)
  if (tierId === 'key_searches')   → buildTier3Queries(...)
}

Phase 06: Search Execution (where cooldowns are written)

File: src/features/indexing/pipeline/searchExecution/executeSearchQueries.js

After each query executes, frontierDb.recordQuery() writes to SQLite immediately — not at end-of-run:

// Line 208-218 — called after EACH query completes
frontierDb?.recordQuery?.({
  productId, query, provider: seedSourceHost,
  tier, group_key, normalized_key, hint_source,
  results: providerResults,
});
// → upsertQueryCooldown() sets cooldown_until = now + 30 days

This means: even if a run is killed mid-execution, cooldowns for completed queries are already persisted.

The 3-Tier Search Model

TIER 1 Seed Queries — Cast the Wide Net

Broad queries targeting known authority sites and product specification pages.

3 Sub-Types (in tierHierarchyOrder priority):

brand_seeds{product} {official_domain}, {product} {support_domain}Brand resolver provides domains
spec_seeds{product} specificationsAlways 1 query, template-driven
source_seeds{product} {source_host}One per source host (rtings.com, techpowerup.com, etc.)

Cooldown:

deriveSeedStatus() checks cooldown_until > now per seed. Provider field stores the source_host (e.g., "rtings.com") for source seeds, empty string for spec/brand seeds. This is how per-source cooldowns are tracked.

Budget Impact:

With 22 source hosts, T1 can consume the entire budget (10) for the first 2-3 runs. Seeds cool down in batches of ~10 per run.

TIER 2 Group Searches — Targeted Field Clusters

One broad query per "search-worthy" field group, ordered by productivity_score.

Query Format:

{brand} {model} {variant} {group_label} {group_description_long}

Example: Razer Viper V3 Pro sensor performance dpi polling rate lift distance

Worthiness Gate (isGroupSearchWorthy):

Coverage ratio< 80%needsetGroupSearchCoverageThreshold
Unresolved fields≥ 3needsetGroupSearchMinUnresolved
Prior searches< 3needsetGroupSearchMaxRepeats
Phase≠ "hold"Budget-aware phase assignment

Exhaustion:

computeGroupQueryCount() counts active-cooldown rows where tier = 'group_search' AND group_key matches AND cooldown_until > now. After 3 searches, the group is exhausted → keys take over.

TIER 3 Key Searches — Individual Field Queries

Targeted queries for specific unresolved fields from non-worthy groups. Uses progressive enrichment.

Progressive Enrichment (controlled by keySearchEnrichmentOrder):

repeat_count=0{product} {normalized_key}Bare query
repeat_count=1+ {aliases}Add alternative names
repeat_count=2+ {domain_hint}Add untried source site
repeat_count=3++ {content_type}Add doc type (manual_pdf, etc.)

Each enrichment level is cumulative. The normalized_key_queue is sorted by availability → difficulty → repeat count → need score.

Budget Allocation Flow

Total budget = searchProfileQueryCap (default: 10)
│
├─ T1: Seeds (brand → spec → sources in tierHierarchyOrder)
│  Consume: min(needed_seeds, remaining_budget)
│  Remaining = budget - T1_count
│
├─ T2: Groups (sorted by productivity_score DESC)
│  Consume: min(worthy_groups, remaining_budget)
│  Remaining = budget - T1_count - T2_count
│
└─ T3: Keys (from non-worthy groups, sorted by availability)
   Consume: min(available_keys, remaining_budget)

Cooldown System

How Cooldowns Work

Both seed and group cooldowns use the same pattern:

// When a query executes (real-time, not end-of-run):
cooldown_until = new Date(Date.now() + queryCooldownDays * 86400000).toISOString()
→ upsertQueryCooldown({ query_hash, product_id, cooldown_until, ... })

// When next run starts:
queryExecutionHistory = buildQueryExecutionHistory(productId)
→ SELECT * FROM query_cooldowns WHERE product_id = ? ORDER BY last_executed_at DESC

// Seed check (deriveSeedStatus):
if (latest.cooldown_until > now) → is_needed: false  // on cooldown, skip

// Group check (computeGroupQueryCount):
count queries WHERE tier='group_search' AND group_key=? AND cooldown_until > now
if (count >= 3) → group exhausted, skip

Rebuild Safety: Cooldowns Persist via product.json

Query cooldowns are snapshotted into product.json at end-of-run via buildProductCheckpoint(). On DB rebuild, seedProductCheckpoint() re-seeds them into query_cooldowns SQL table.

If spec.sqlite is deleted and rebuilt from checkpoints:

  • url_crawl_ledger is rebuilt from product.json sources ✓
  • query_cooldowns is rebuilt from product.json query_cooldowns array ✓

Product.json is the durable SSOT — never pruned (unlike run.json which was pruned after 7 days, now kept forever).

JSON product.json (.workspace/products/) — PERSISTENT

Contains: identity, sources (URL crawl history), fields, provenance, query_cooldowns. Updated at end-of-run.

Query cooldowns are snapshotted from SQL at end-of-run and re-seeded on DB rebuild.

JSON run.json (.workspace/runs/) — PERSISTENT

Contains: run metadata, fetch plan, counters, sources crawled, search profile, NeedSet output.

Written at end-of-run. Used for GUI replay and audit trail.

Data Flow Diagram

Pipeline Start → specDb.buildQueryExecutionHistory(productId) // SQL SELECT * FROM query_cooldowns WHERE product_id = ?
→ deriveSeedStatus(queryHistory, identity, config, sourceHosts) // For each seed: cooldown_until > now? → is_needed: false
→ buildSearchPlanningContext({needSetOutput, queryHistory, ...}) // Groups fields, checks worthiness, allocates budget
→ buildSearchProfile({seedStatus, focusGroups, ...}) // Generates query strings: T1 seeds → T2 groups → T3 keys
→ executeSearchQueries() // Runs queries. After EACH query:
    → crawlLedgerAdapter.recordQuery({...}) // SQL UPSERT INTO query_cooldowns (cooldown_until = now + 30 days)
    // provider = source_host for seeds, search_engine for groups/keys
→ Crawl URLs with Playwright
→ Write run.json + update product.json // end-of-run checkpoint (no cooldown data)
