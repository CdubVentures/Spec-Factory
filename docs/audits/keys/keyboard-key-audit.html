Purpose — what this doc is and what it isn't

This is a review artifact. Each row's guidance cell is the exact text that will land in field_overrides.<key>.ai_assist.reasoning_note inside category_authority/keyboard/_control_plane/field_studio_map.json. The compiler flows that string into every keyFinder LLM call for that key as the {{PRIMARY_FIELD_GUIDANCE}} slot.

What a reasoning_note is for: field-specific pitfalls and disambiguations the generic prompt can't express — vendor-to-canonical name maps, null-vs-No rules, narrow traps where a reasonable LLM would mis-classify, derivation rules from sibling fields. It is the only author-editable per-key hook into the prompt.

What a reasoning_note is NOT for: anything already supplied by the generic prompt (listed below). Repeating those wastes tokens and dilutes the signal.

What the generic keyFinder prompt already supplies — do not restate in guidance

The per-call prompt (KEY_FINDER_DEFAULT_TEMPLATE in src/features/key/keyLlmAdapter.js) auto-injects the following for every primary + passenger key. A per-key reasoning_note that repeats these is pure noise.

PRIMARY_FIELD_CONTRACT — type, shape, unit, rounding, enum values (first 24), enum policy, variance policy, aliases
PRIMARY_SEARCH_HINTS — domain_hints + query_terms from field_overrides.search_hints
PRIMARY_CROSS_FIELD_CONSTRAINTS — cross-field ops (lte/gte/requires_when_value, etc.)
PRIMARY_COMPONENT_KEYS — relation pointer (parent / subfield_of)
PRODUCT_COMPONENTS — resolved component identity + subfields (e.g. switch_name → actuation_force/feel/lifespan auto-pulled)
KNOWN_PRODUCT_FIELDS — all already-published scalar/list values on this product
EVIDENCE_CONTRACT + EVIDENCE_VERIFICATION — tier definitions + mandatory URL fetch
SOURCE_TIER_STRATEGY — PRIMARY / INDEPENDENT / RETAILER / COMMUNITY tier rubric
VALUE_CONFIDENCE_GUIDANCE — 0-100 epistemic confidence rubric
UNK_POLICY — "honest unk beats low-confidence guess"
IDENTITY_INTRO + IDENTITY_WARNING — tiered sibling-model disambiguation
PREVIOUS_DISCOVERY — prior run URLs and queries for the same (product, field_key)

Rule: Guidance gets written only when a field has a pitfall the contract + components + source-tier rules can't express. "Check the mfg page", "don't hallucinate", "in mm", "derive from switch_name", or lists of enum values never belong here. No brand-default shortcuts (warranty years, polling defaults by vendor, software-suite maps) — those rot. No convenience defaults (None / Standard / Tray Mount / Silent as a fallback when evidence is absent) — UNK policy owns that.

Component DB & Enum Cleanup

Every enum becomes a facet filter on the site. Long enums and overlaps bloat the filter UI and confuse finders.

switch_name · known_values out of sync with component DB

Issue. known_values.switch_name has 15 entries; component_db/switches.json has 44. Products using e.g. Cherry MX Black, Gateron Lekker L60, Kailh Box Brown, Razer Orange, Logitech GX Red miss the known-values hint in the contract.
Fix. Generate known_values.switch_name from component_db/switches.json at compile time. Policy stays open_prefer_known.

wireless_technology vs bluetooth_version · duplicative

Issue. wireless_technology enum contains "Bluetooth 5.0 / 5.1 / 5.2 / 5.3"; bluetooth_version covers exactly the same axis at finer grain.
Fix. Reduce wireless_technology to 2.4 GHz / Bluetooth / Both, or retire it — connection already distinguishes radio types and bluetooth_version carries the BT number.

case_material · ABS/Plastic duplication, "CNC Aluminum"

Issue. Enum contains both ABS and Plastic — plastic keyboard cases are nearly always ABS. CNC Aluminum is a manufacturing method, not a material.
Fix. Collapse to: Aluminum, Plastic, Polycarbonate, FR4, Acrylic, Wood, Steel. Drop CNC Aluminum → Aluminum. Collapse ABS → Plastic.

stabilizer_type vs stabilizer_mount · overlap

Issue. stabilizer_type: Costar / PCB-mounted / Plate-mounted / Screw-in / Pre-lubed Screw-in. stabilizer_mount: Clip-in PCB / Screw-in PCB / Plate-mount / Plate-mount TPU. Same axis split across two fields.
Fix. Keep stabilizer_mount with enum Plate-mount clip-in, Plate-mount screw-in, PCB clip-in, PCB screw-in, Costar, None/Other. Retire stabilizer_type.

gasket_mount vs mounting_style · derived redundancy

Issue. mounting_style has Gasket Mount. gasket_mount repeats the same signal as Yes/No/Double Gasket.
Fix. Derive gasket_mount from mounting_style or retire it. Double-gasket distinction lives in a component-db flag, not a filter.

anti_ghosting · duplicative enum values

Issue. Values Full anti-ghosting / No / Yes. "Full anti-ghosting" and "Yes" are the same thing.
Fix. Make binary Yes/No. Or retire — key_rollover (NKRO/6KRO/10KRO) subsumes it.

socd_cleaning · mixes mode with toggle

Issue. Enum Last input priority / Neutral / No / Snap Tap / Yes mixes on/off state with mode name.
Fix. Split: socd_cleaning = Yes / No. socd_mode = Last-input / Neutral / Snap Tap / Other (only when Yes).

colors closed enum · 76 values

Issue. 76 closed tokens. Owned by CEF/variantGenerator (authoring is automatic) but the filter UI surface is massive.
Fix. Keep raw tokens (CEF needs precision for variant matching). At the filter layer, expose a hierarchical roll-up (chromatic family → shade). No change to tokens themselves.

plate_material · POM casing

Issue. Both plate_material and keycap_material use POM — fine — but normalize casing across all material enums.
Fix. Normalize acronyms: ABS, PBT, POM, PC, FR4. No value changes, just the normalize rule.

rgb_zones · derivable from per_key_rgb

Issue. When per_key_rgb=Yes, zones = Per-key. Two facets that collapse together.
Fix. Derive in the validator: per_key_rgb=Yes → rgb_zones=Per-key. Filter UI hides zones when per-key is selected.

included_accessories · keep closed at 8

Issue. Niche bundle items (travel pouch, coiled cable in a special color) don't fit the 8-value enum.
Fix. Keep enum closed at 8. If not on the list, drop it. Authored 8-item enum covers 95%+ of real bundles; filter surface stays clean.
