Monitor key priority matrix

Audit of all 113 existing monitor field keys + 11 proposed additions. v3 applies the peer audit in monitor_extraction_guidance_audit.md — 17 substantive rewrites + 24 minor rewrites + factual corrections.
Current priority values come from category_authority/monitor/_generated/field_rules.json. Proposed priority + the Extraction-guidance column land in ai_assist.reasoning_note, which the keyFinder prompt renders as PRIMARY_FIELD_GUIDANCE.

What this matrix is for — and what it is NOT

Purpose of the Extraction-guidance column: to produce a short, additive per-field note that lands in the keyFinder LLM prompt alongside the auto-generated contract. The goal is to close field-specific knowledge gaps (derivation formulas, terminology traps, measurement-method notes, version gotchas) that the contract, enum, and aliases cannot express on their own.

The generic keyFinder prompt already does the following — per-key guidance must NOT restate any of it:

  • Field contract (type, shape, unit, rounding, list rules, enum values, variance policy, aliases) — buildFieldContractBlock in keyLlmAdapter.js
  • Search hints (domain_hints + query_terms from field_overrides)
  • Cross-field constraints + component relation pointer + product-level resolved components (auto-pulled sibling subfields when a component is resolved)
  • Known-published product fields (dedup excludes current primary + passengers + components)
  • Evidence contract: minimum evidence refs, 5-tier classification (tier1 mfr / tier2 independent / tier3 retailer / tier4 community / tier5 aggregator), URL-verification mandate (fetch-it-live), evidence-kind tagging (direct_quote / structured_metadata / lab_measurement / etc.)
  • Source-tier strategy (PRIMARY → INDEPENDENT → RETAILER → COMMUNITY fall-through)
  • Value confidence rubric (0-100 scale, tier-independent)
  • UNK policy: honest "unk" + unknown_reason beats low-confidence guess
  • Identity warning + sibling exclusion (easy / medium / hard tiered by family-model count)
  • Previous discovery log (URLs + queries already tried for this field on this product)

So a reasoning_note should only contain one or more of:

  • Derivation formulas (e.g., PPI = √(hPx² + vPx²) / diagonal_inches)
  • Field-specific pitfalls (e.g., "dtype is integer — don't write 'per-pixel'")
  • Non-obvious terminology (e.g., "Nano-texture is Apple-proprietary")
  • Unit / methodology ambiguity (e.g., "RTINGS uniformity is % std dev, not the 0-10 rating")
  • Version / revision traps (e.g., "HDMI 2.1 FRL is UP TO 48Gbps; version label alone doesn't prove 48Gbps")
  • Cross-field logic not captured in cross_field_constraints (e.g., "Derive from adaptive_sync — No only when adaptive_sync='None'")

Result: 72 of 124 rows carry an additive note; the rest rely on the generic prompt alone. Median note length is short (<200 chars) so prompt bloat stays bounded.

How to read the matrix

  • Each existing key has two columns per axis (required_level / availability / difficulty): current on the left, proposed on the right.
  • Any proposed cell that differs from current is outlined in blue; the whole row is tinted blue if anything on the row changed.
  • Proposed new keys (not yet in field_rules.json) are tinted orange and tagged ★NEW. User must approve before adding.
  • Extraction guidance is the proposed ai_assist.reasoning_note value — it lands in PRIMARY_FIELD_GUIDANCE at prompt-build time. Where it says "(generic prompt suffices)" the contract + enum + aliases already communicate everything.
  • Required level is binary (mandatory / non_mandatory) per feedback_field_axis_simplification.md.

Top-level calls (v3)

  • Promote to mandatory: aspect_ratio, curved, panel_bit_depth, pixel_density, adaptive_sync, variable_refresh_rate, brightness_sdr, contrast_ratio, hdr_support, displayport_ports, hdmi_ports, vesa_mount.
  • Downgrade easy→hard: gradient_handling, text_clarity, flicker_frequency, response_time_60hz, response_time_120hz — lab measurements, not spec-sheet data.
  • Fix availability claim: panel_type / resolution / refresh_rate / weight_without_stand / viewing angles / native_colors / osd_type / color tagged sometimes but always published; flipping to always stops wasting tier-3 budget.
  • Enum cleanups: overdrive dtype=boolean but enum = 10 OSD level-names. refresh_rate includes 600/720/1040 Hz with no shipping products. pivot_rotation mixes boolean with direction. audio_output vs headphone_jack and cable_type vs cables_included are redundant. local_dimming_zones is integer — guidance must not propose writing strings like 'per-pixel' / 'n/a' (fixed in v3).
  • Proposed new keys (11): bezel_width_top/side_mm, stand_included, power_supply_type, refresh_rate_dual_mode_max, ambient_light_sensor, burn_in_warranty_years, nvidia_reflex_analyzer, gsync_pulsar, srgb_emulation_mode, color_certifications.
  • v3 factual corrections (per monitor_extraction_guidance_audit.md): G-SYNC Pulsar uses MediaTek integrated scalers (not the retired dedicated G-Sync module); HDMI 2.1 FRL is UP TO 48Gbps (can be 24/32/40/48) and HDMI 2.1 features are optional — version label alone doesn't prove anything; DP 2.1 has UHBR10/13.5/20 sub-tiers; RTINGS black / gray-50 uniformity is % std dev (not raw number and not the 0-10 rating); Dolby Vision is not available on any desktop monitor (TVs only) — do not require it for Xbox compat; PS5 supports 1440p output but cannot combine 1440p + VRR; local_dimming_zones integer field must not receive string values.
  • v2 factual corrections (still applicable): Nano-texture is Apple-proprietary; IPS Black (LG+Dell, ~2000:1 contrast) is distinct from Nano IPS (LG, wider gamut); curve-radius direction: lower R = tighter.
