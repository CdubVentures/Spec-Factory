Intent of this document

What this doc produces

The Extraction guidance column in every table below is ready-to-paste content for one slot in the keyFinder LLM prompt: field_rule.ai_assist.reasoning_note. When a keyFinder call runs for a given (product, field_key), that string is rendered into the prompt as {{PRIMARY_FIELD_GUIDANCE}} (or as a per-passenger guidance block for bundled keys). Nothing else in this doc ships to the LLM — the priority triples (required_level / availability / difficulty) drive the tier router and budget calculator; the Components & Enums Audit section drives a separate config cleanup.

What reasoning_note is FOR

The keyFinder default template renders ~13 other slots automatically (identity warning, field contract with type/shape/unit/enum/aliases, search hints, cross-field constraints, component inventory, evidence contract, source tier strategy, confidence rubric, unk policy, previous discovery, return JSON shape). Those slots handle most of what an LLM needs. reasoning_note fills the gap between those generic slots and the field's specific semantics.

Use it for:

  • Visual / photographic cues the LLM must apply when the field is decided from a product image (thumb_rest shelf vs side-button housing, hump apex position, front_flare direction).
  • Semantic disambiguation between adjacent enum values (palm vs claw grip triggers; symmetrical shape vs ambidextrous form factor).
  • Field-specific gotchas that repeat across mice (sensor_brand follows the extracted sensor name, not the upstream fab; scroll-click ≠ middle button).
  • Rebrand / alias rules tied to extraction behavior (Razer Optical switch → switch_brand=razer, not the upstream OEM).
  • "Don't confuse with" anchors when two fields share surface vocabulary (cable_type=paracord ≠ the paracord field; sensor_latency ≠ click_latency).
  • Interpretation rules for ambiguous fields (nvidia_reflex conflates two features; document the project's rule without forcing a schema fix).

What reasoning_note is NOT for

Anything already rendered by another template slot. If the generic prompt covers it, the guidance cell MUST NOT repeat it — doing so creates conflicting or stale instructions that drift away from the template.

  • Enum allowed values / aliases — rendered by PRIMARY_FIELD_CONTRACT.
  • Type, shape, unit, rounding, list-dedupe, list-sort rules — rendered by PRIMARY_FIELD_CONTRACT.
  • Preferred source domains / search query terms — rendered by PRIMARY_SEARCH_HINTS.
  • Source tier preference (manufacturer vs review vs retailer) — rendered by SOURCE_TIER_STRATEGY.
  • Evidence structure, min-ref count, URL verification — rendered by EVIDENCE_CONTRACT + EVIDENCE_VERIFICATION.
  • "Return unk when uncertain" policy — rendered by UNK_POLICY.
  • 0–100 confidence rubric — rendered by VALUE_CONFIDENCE_GUIDANCE.
  • Cross-field constraints (must be ≤X, required when Y) — rendered by PRIMARY_CROSS_FIELD_CONSTRAINTS from the rule's cross_field_constraints array.
  • Already-resolved component values on this product — rendered by PRODUCT_COMPONENTS (sensor → ips/acceleration/flawless etc. flow through automatically when sensor is resolved). Do not hardcode chip specs in guidance.
  • Identity sibling-confusion warnings — rendered by IDENTITY_WARNING.
  • Output JSON envelope — rendered by RETURN_JSON_SHAPE.
  • Priority / difficulty rationale — that belongs in the priority block, not in guidance.
  • Schema decisions under debate — flag them via the AMBIGUOUS badge on the row; don't write the debate into the LLM's prompt.

Quick test before adding to a guidance cell

  1. Is the concept covered by a slot above? → Delete it.
  2. Is it the enum value list, aliases, unit, or rounding? → Belongs in the field contract. Delete.
  3. Is it a preferred domain or search term? → Belongs in search_hints. Delete.
  4. Is it a "required when X = Y" rule? → Belongs in cross_field_constraints. Delete.
  5. Is it a chip- or component-specific spec that will flow through PRODUCT_COMPONENTS? → Keep only the interpretation rule; let the component DB carry the values.
  6. Is it a visual cue, semantic gotcha, rebrand rule, or "don't confuse with X" anchor? → Keep.

Components & Enums Audit

material should not be a component

  • component_db_sources.material has property_mappings: [] and a 3-row table — identical to a closed enum. Drop it from components and convert to enum [plastic, metal, carbon fiber, magnesium].

Enum pollution (known_values.json)

  • coating has "A", "help" (garbage). Propose [matte, glossy, soft-touch, textured, rubber].
  • switch_type polluted with ~20 switch model strings + junk ("eqwe", "yay"). Lock to 4 values: [mechanical, optical, optical-mechanical, inductive].
  • feet_material has "l". Propose [PTFE, PTFE (Virgin-Grade), PTFE (Black-Dyed), Glass, Ceramic].
  • connectivity has "1", "2.5". Real values: [2.4GHz Dongle, 2.4GHz WIFI Dongle, 2.4GHz RF Dongle, n/a].
  • colors has 75 values — collapse to ~16 base colors to limit filter-facet explosion.
  • lighting 17 values — overlaps with rgb boolean; pick one owner.
  • hump mixes 3-value scheme and legacy 2-axis notation (back - front, back mid); pick one.
