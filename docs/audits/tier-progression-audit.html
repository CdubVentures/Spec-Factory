What This Proves

Each run opens a real SQLite database, reads query_cooldowns via buildQueryExecutionHistory(), runs the real production planning pipeline (buildSearchPlanningContext + buildSearchProfile), then writes the generated queries back to the DB via crawlLedgerAdapter.recordQuery().

The next run reads those cooldowns back and the tier system reacts:

  • Run 1: No history in DB → seeds (T1) fire, consuming budget
  • Runs 2-3: Seeds on cooldown → groups (T2) get full budget
  • Runs 4-9: Groups exhausted (3x max) → keys (T3) take over
  • Run 10: Cooldowns force-expired (simulating 30+ days) → full cycle resets

Persistence Proof

All cooldowns are persisted in SQLite table query_cooldowns with columns: query_hash, product_id, tier, group_key, cooldown_until, attempt_count, result_count, last_executed_at.

deriveSeedStatus() now checks cooldown_until > now — identical pattern to computeGroupQueryCount() which has always worked correctly for group cooldowns.

The old broken logic checked status === 'scrape_complete' and new_fields_closed >= 1, but the DB never stored those fields. Seeds were always is_needed: true, starving T2/T3 of budget.
