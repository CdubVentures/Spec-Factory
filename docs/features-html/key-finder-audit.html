Key Finder Audit Handoff

How to generate, review, and apply per-key Field Studio guidance so Spec Factory can feed the EG-HBS site with benchmark-grade product data

Purpose This document is the pick-up point for the Key Finder Audit work. A new human or LLM agent should be able to read this page, understand the goal, generate or review the audit reports, produce clean Field Studio change files, and continue the loop without needing the original conversation.
Current north star Spec Factory should become the single source of truth for the EG-HBS consumer site. Each category needs a benchmark/example set that represents the depth expected from publish-grade product pages. Key Finder output should eventually match that target because the field contracts, enum lists, component mappings, search routing, evidence gates, and per-key guidance are good enough.
1 What This Audit Is
Definition Key Finder Audit

The Key Finder Audit is a generated self-contained review package for a product category. It tells a reviewer everything Key Finder will use for every field: current contract shape, enum policy, known values, component relations, cross-field constraints, evidence rules, aliases, search hints, priority routing, group membership, and the actual prompt preview.

The generated per-key reports are not the final deliverable. The human or LLM returns per-key Field Studio change files. The category-level summary is the compact final signoff matrix that overwrites only itself inside .workspace/reports/<category>/ and does not delete returned JSON patch files.

2 Why The EG-HBS Site Matters
Downstream Consumer Surface Contract

The EG-HBS hub and snapshot pages are the downstream reason this work has to be precise. The site uses category data to power hub filters, list columns, product cards, snapshots, comparison state, metric tooltips, and recommendation links.

The current EG-HBS code path centers on window.sharedData from views/hub.handlebars. Important keys include displayData, preRenderedCards, listColumns, listToggles, toggleItems, sliderItems, filterOrder, sortingMap, filteringData, filteringSchema, filteringPacked, statsOptions, egmMetrics, egLStats, and versusList.

Site surfaceWhy field contracts matterSource observed
Hub filters String/list enum values become toggle chips; numeric/date fields become sliders; bad enum values become bad filter UX. views/partials/data-filter-menu.handlebars, public/js/data_filter.js
Hub list view List columns sort and display based on field keys, sort type, date/numeric normalization, and missing-value behavior. src/hub/ListView.jsx
Product cards Cards use field values for stats, metrics, carousel/image variants, badges, colors, editions, and snapshot navigation. src/hub/Card.jsx, public/js/hbs___hubHelpers.js
Snapshot detail Snapshot pages expose photos, specs, recommended/similar blocks, and comparison navigation. Wrong shape means wrong display semantics. routes/hubs/01.snapshot.routes.js, views/snapshots.handlebars
Compare/search/SEO Stable field values allow reliable compare rows, query links, structured recommendations, and category pages without manual cleanup. views/hub.handlebars, src/hub/CardView.jsx, public/js/hbs___hubHelpers.js
Practical implication The site can render many shapes, but it cannot infer meaning. If Spec Factory says a field is an open string enum, every accidental value can become a filter chip. Never add unk to enum values or data lists; it is an LLM sentinel that should become status/unknown_reason with no submitted value. Use n/a as data only when not-applicable is intentionally stored or public. Applicability is not the same as value: battery_hours should be numeric when hours are proven and blank/omitted when not applicable or unproven, unless the category explicitly wants a public n/a state. The audit must decide the correct stored shape before writing better prose.
3 Field Studio Edit Order
Order Apply Mapping Studio Before Key Navigator

The generated change file must mirror the actual Studio workflow. Enum values are in Mapping Studio. Key Navigator's Enum Policy only chooses the policy/source/format for a specific field.

OrderPanelWhat belongs here
1 Mapping Studio - Component Source Mapping Component source/type, identity role, maker role, aliases/name variants, reference links, property mappings, variance policy, component-only, product override permission, tolerance, and component-level constraints.
2 Mapping Studio - Enum Data list field/name, normalization, final canonical values, add/remove/rename operations, source phrases that must stay out of chips, and enum-specific guidance.
3 Key Navigator - Contract variant_dependent, type, shape, unit, range min/max, list rules, rounding, and variance policy.
4 Key Navigator - Extraction Priority & Guidance required_level, availability, difficulty, search/routing reason, paste-ready ai_assist.reasoning_note, and the one on/off Variant Inventory Context / PIF Priority Images checkboxes.
5 Key Navigator - Enum Policy policy, source, and format pattern. Do not put actual enum value edits here.
6 Key Navigator - Components Per-key component relation such as component.type or component source cascade. Property/variance setup belongs in Mapping Studio; concrete component entity rows are out of scope.
7 Key Navigator - Cross-Field Constraints Relational rules between fields, plus any dependency that affects extraction or validation.
8 Key Navigator - Evidence min_evidence_refs and tier preference.
9 Key Navigator - Tooltip / Guidance Display tooltip only. This is UI help, not ai_assist.reasoning_note and not keyFinder extraction guidance.
10 Key Navigator - Search Hints & Aliases Aliases, domain hints, content types, and query terms.
4 Per-Key Audit Standard
Rule Guidance Comes Last

For each key, validate the full field contract before writing guidance. A beautiful reasoning_note cannot fix a wrong type, wrong enum, wrong component mapping, wrong evidence gate, or wrong priority route.

Audit itemQuestions to answer
Consumer surfaceShould this key power filters, hub list columns, product snapshot rows, comparison rows, metric cards, search/SEO, recommendations, or none?
Type and shapeIs it string, number, integer, boolean, or date? Is it scalar or list? Does the site need n/a/unk states?
Enum policyIs the set finite and filter-facing enough to be closed? Is it open but should prefer known values? Is it truly open?
Enum listAre canonical values clean, stable, user-facing, and not aliases/source phrases? Are there too many chips?
Priority routeShould it be mandatory? Is availability always/sometimes/rare? Is difficulty calibrated to benchmark-depth extraction?
EvidenceHow many evidence refs are needed? Are manufacturer, datasheet, lab, review, retailer tiers ordered correctly?
ComponentsIs the Field Studio setup treating the value as product-authored, component-derived, or product-overridable?
ConstraintsDoes it depend on another field? Are the relationship and target field correct?
Variant inventory contextShould Key Finder receive the active CEF variant table as an evidence filter for this key? Enable only when variant identity facts such as edition, SKU, release date, colorway, or PIF image status add value without ambiguity.
PIF priority imagesShould Key Finder receive default/base PIF priority-view images for this key? Enable only for visual keys where those views add value; missing/unattachable images are not negative evidence.
Unknown statesWhen should the extractor produce no submitted value plus unknown_reason, intentional n/a data, false/no, or blank/omitted?
Example bankAre there 5-10 calibration products covering common, rare, unknown, conflict, and filter-risk cases?
GuidanceAfter all above decisions, what final paste-ready instruction helps the LLM make the correct judgment?
Tier A Direct Visual

Visible from a single product photo. Guidance should be short and name the view and feature.

Examples: visible openings, visible controls, visible zones, basic left/right or front/back symmetry.

Tier B Subtle Visual

Requires judgment and repeatable thresholds. Guidance must teach the decision.

Examples: proportional location, subtle contour/taper, intended-use/fit class, or symmetry where control layout matters.

Tier C Spec/Source

Not visual. Do not mention photos. Guidance should focus on source tiering, aliases, conflicts, and component identity.

Examples: component model, supported rates, release dates, internal controllers, firmware/support feature names, and lab-measured metrics.

5 Enum Discipline
High Leverage Every Enum Value Can Become A Filter Chip

Enums are the highest-risk part of the audit because non-numeric values become site-facing options. A messy enum does not just hurt extraction. It creates a bad hub filter, confusing compare options, and inconsistent data.

Enum ruleStandard
Canonical valuesStored/user-facing options only. Do not store aliases, retailer prose, SKU suffixes, marketing phrases, colors, or theme names unless users should filter by that exact chip.
Value countUp to 10 is healthy, 11-15 fine, 16-20 tolerable, 21-30 filter fatigue, 30+ broken unless values follow a strong pattern.
Closed policyUse when the set is small, stable, finite, and new values should require a human decision.
Open preferredUse when canonical values exist but real products may introduce legitimate new values. Example: material.
Open policyUse only when the field is genuinely freeform and not a fragile filter chip set.
PatternsIf many values share a structural pattern such as <N> zone (rgb), keep the pattern explicit in guidance and format.
Example: design is edition status, not aesthetics
Recommended design enum:
- standard
- collaboration
- limited edition
- multiple

Do not store:
- ergonomic
- minimalist
- raw visual adjectives
- black
- white
- rgb
- anime
- esports

Reason: those are shape, color, material, lighting, theme, or marketing/aesthetic terms. They do not belong in an edition-status filter.
Example: material should be open preferred
Recommended material policy:
- policy: open_prefer_known
- source: data_lists.material

Reason: common materials can be known, but real products may introduce legitimate physical materials or coatings that should not be blocked. Finish words like matte, glossy, premium, textured, and grippy are not materials unless the source explicitly names a coating.
6 Components And Cross-Field Constraints
Components Product Value vs Component Property

Some values are not really product-authored values. They are properties of a component or sub-entity. The audit must decide how the Field Studio component source mapping should be configured for the key. Concrete component entity row edits are out of scope for the Field Studio change file.

Component mapping fieldMeaning
component source/typeThe component database being mapped, such as sensor or switch.
primary identifier roleThe field or attribute that uniquely identifies the component entity.
field_keyThe product field that receives or references the component property.
variance_policyWhether the component property is authoritative, range/upper/lower bound, majority vote, or override-allowed.
component_onlyTrue when the value should live on the component and not be copied as a product guess.
allow product overrideTrue only when a product-specific implementation can legitimately differ from the component default.
property constraintsEntity-level constraints that protect impossible values.
Cross-field invariant Component or feature dates cannot contradict the product timeline. If credible evidence implies an impossible relationship, the extractor should return no submitted value with unknown_reason or flag the conflict instead of forcing a value.
7 Priority, Availability, Difficulty
Routing These Are Search And Model-Strength Controls

The priority triple is not just labeling. Requiredness decides whether the site should try to publish a field for most products. Difficulty decides the remaining model/search effort after variant inventory, PIF images, aliases, and source hints are available.

FieldValuesAudit meaning
required_level mandatory, non_mandatory Mandatory means the value is buyer/site/benchmark useful and usually distinguishable from public/spec/visual/identity evidence: visible, identifiable from variant identity, listed in specs/docs, or generally exposed by credible sources. It is not restricted to lab-only measurements. Missing proof still becomes explicit unknown status with no submitted value. Non-mandatory means useful enrichment that should not block publish.
availability always, sometimes, rare How often credible public evidence exists and how early/often Key Finder should look for it.
difficulty easy, medium, hard, very_hard Model/search strength after context is injected. Easy is direct spec/photo/PIF/variant lookup or straightforward canonical mapping. Medium needs normalization or light source comparison. Hard needs technical component reasoning, meaningful conflict handling, aliases that change meaning, or source credibility calls. Very_hard is reserved for hidden/lab-grade fields such as proprietary internal component identities, instrumented latency/accuracy measurements, unresolved datasheet links, or lab-only metrics.

Do not mark a key hard just because the wording is subtle. If the answer is generally available and the injected variant/PIF context makes the value direct, easy or medium is the right route.

8 Variant Inventory Context
One Checkbox Evidence Filter, Not A Second Guidance Field

Variant Inventory Context is a single Key Navigator checkbox. When enabled, Key Finder may receive the active CEF variant table joined to SKU/RDF/PIF identity facts so it can reject wrong-color, wrong-edition, wrong-SKU, wrong-release, or sibling-variant evidence. When disabled, none of that variant table context is injected for the key.

Enable it only when those variant identity facts add evidence-filter value without making the answer more ambiguous. Most model-level keys are the same across every variant in a family and should not need the table. List-valued or genuinely variant-varying keys need an explicit rule: product-wide union, exact-variant answer, base/default variant answer, or no submitted value with unknown_reason when sources do not separate variants.

Keep this separate from ai_assist.reasoning_note. The checkbox controls whether the variant table is injected. The reasoning note teaches field-specific judgment. For example, a design reasoning note may say to treat colorways, collaboration graphics, and edition artwork as non-design changes unless the physical shell, layout, construction, or included hardware differs.

QuestionUse
Should the LLM see edition/SKU/release/colorway identity facts?Set Variant Inventory Context to enabled only when they help filter evidence for this key.
How should this key interpret variant evidence?Write that in ai_assist.reasoning_note.
Should the reviewer author custom variant-table prose?No. The generated prompt provides deterministic table-usage guidance when the checkbox is enabled.
9 PIF Priority Images
One Checkbox Default/Base Visual Support

PIF Priority Images is a single Key Navigator checkbox. When enabled, Key Finder uses the category's existing PIF priority-view settings and already-evaluated default/base variant images. It does not add modes, profiles, or a second guidance box.

Enable it only for visually answerable keys where default/base priority views help. Do not enable it for non-visual spec-sheet fields.

The images are supporting context only. If images are missing or cannot be attached, the prompt should say that explicitly. Absence of a visible trait in default/base images is not proof the trait is absent, especially for edition-specific visual changes.

Key Finder remains product-scoped. For yes/no fields, the reviewer must decide in ai_assist.reasoning_note whether one official edition with the visible trait makes the product-level answer yes. For list-like visual fields, guidance may need to say to include the distinct variant/design forms found. Colors usually should not change the answer; editions might when they change shell, layout, material, included hardware, or another visible product fact.

SurfaceBehavior
Field Studio badgeThe checkbox path ai_assist.pif_priority_images has an LLM.KF badge because Key Finder reads it.
Prompt locationPIF_PRIORITY_IMAGES renders inside the Key Finder CONTEXT section after VARIANT_INVENTORY / FIELD_IDENTITY_USAGE and before evidence/source rules.
Prompt labelThe block identifies these as PIF default/base variant images, lists the configured priority views, selected variant, image filenames, source URLs when known, and PIF eval notes.

10 Benchmark Standard
Benchmark What Good Means

The long-term scoring question is simple: after Field Studio contracts and Key Finder guidance are corrected, do extracted app values match the category benchmark/example set?

The audit scripts must stay category-generic. For every category, build a 5-10 product example bank first, then create category benchmark rows that represent the depth expected from a site aiming to combine structured specs with review-depth evidence.

Benchmark layerWhat to measure
Exact value matchDoes the stored value equal the benchmark value after normalization?
Shape matchScalar vs list, number/date/string/enum, blank/no submitted value vs intentional n/a data vs unknown_reason diagnostics.
Enum matchCanonical chip values only, no aliases or noisy variants.
Evidence matchAccepted values have enough evidence refs and use credible source tiers.
Conflict handlingConflicting sources produce the right accepted value, no submitted value with unknown_reason, or a flag, not a silent guess.
Consumer renderingValue supports the intended hub filter/list/snapshot/compare/metric surface without downstream interpretation hacks.

12 LLM Review Instructions
Handoff What To Tell The Reviewer

When handing a generated report to an LLM, require strict JSON patch files first, then the normal prose audit. The returned files go in .workspace/reports/<category>/auditors-responses/ and can be imported as a batch.

  1. One downloadable .json file per changed key, named <category>-<sort_order>-<field_key>.field-studio-patch.v1.json.
  2. A normal audit report with verdict, coverage, highest-risk corrections, per-field rationale, enum cleanup, component source-mapping setup, group audit, and open questions.

Do not ask for a text file. JSON files must omit unchanged paths entirely; prose sentinels are rejected by the importer.

13 Grouping Guidance
Navigation Groups Are How Humans Think Through Keys

Groups are not just cosmetic. The per-key docs show every key, its group, constraints, and components so the reviewer can spot misplaced keys and missing dependencies. Cleaner grouping makes Key Navigator usable and helps category authors find related contract decisions.

Grouping patternExample keysReason
Product identity / lifecycle / availabilitysku, colors, editions, release_date, discontinued, price_rangeThese are purchase/listing/variant lifecycle fields. They should be reviewed together.
Design status / edition typedesignEdition status should not be mixed with visual styling or lighting.
Lighting / electronicslighting, rgb, zone countsLighting behavior is not the same as product design/edition status.
Component identity/propertyComponent type, maker role, tracked attributes, variance policyAudit the Field Studio mapping controls only; concrete component entity data belongs to a separate cleanup pass.
Subtle visual traitsProportional location, contour/taper, intended-use class, orientationThese are often visual Tier B calls and need repeatable guidance.
14 Good vs Bad Guidance
Bad Vague Or Contract-Duplicating
Extract the product material accurately.
Use official sources.
If unsure, default to no.
Design means the way the product looks.

This does not teach a repeatable judgment. It also weakens the unknown policy and duplicates generic source-tier instructions.

Good Paste-Ready And Field-Specific
Extract the proven physical shell or contact-surface material. Do not convert finish words such as matte, glossy, textured, soft, grippy, or premium into material values. If the source only describes coating or texture, use rubberized coating or soft-touch coating only when that coating is explicitly named. Return unk when no credible source proves the material.

This tells the LLM what to include, what to exclude, and when to return unk.

16 Pick-Up Checklist
Runbook If You Are The Next Person
  1. Read this document once.
  2. Open docs/implementation/archive/key-finder-audits/HANDOFF.md if you need implementation details for the generator.
  3. Generate all Key Finder Audit reports for the category from Studio.
  4. Use the category Markdown report as the LLM handoff. Use the per-key Markdown/HTML docs when reviewing a specific key.
  5. Require strict downloadable JSON patch files first, one per changed key, then the prose audit.
  6. Review enum and component changes before guidance. Actual enum value changes go in Mapping Studio Enum Data Lists.
  7. If the human says import patches, apply the approved files from .workspace/reports/<category>/auditors-responses/ to category_authority/<category>/_control_plane/field_studio_map.json only. Do not edit _generated/*, SQLite files, component generated output, compiler, runtime, schema, API, UI source code, or prompt code unless explicitly asked.
  8. Apply approved edits in Field Studio or the control-plane map.
  9. Let the human compile the category rules unless they explicitly ask you to run compile.
  10. Regenerate the audit report and diff it against the previous run.
  11. Run extraction against benchmark products and compare output against the benchmark workbook/range.
Definition of done for this loop A category is not "done" because every field has a long guidance note. It is done when the contract shape, enum list, component mapping, priority routing, evidence gate, constraints, grouping, and guidance together let Key Finder reproduce benchmark-depth values with honest unknown_reason handling, intentional n/a only where public, and clean downstream site rendering.
