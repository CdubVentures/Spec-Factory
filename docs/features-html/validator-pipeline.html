5 Repair Adapter (Per-Key LLM Prompts)

Every prompt includes the FIELD CONTRACT block + the universal response schema. No confidence gating — the 13-step re-validation is the only gate. Binary pass/fail.

Universal Response Schema (all prompts)
{
  "status": "repaired" | "rerun_recommended",
  "reason": string | null,
  "decisions": [{
    "value": "the failing value",
    "decision": "map_to_existing" | "keep_new" | "reject" | "set_unk",
    "resolved_to": "canonical value" | number | null,
    "reasoning": "why this decision"
  }]
}
No confidence field
Removed. Re-validation is the only gate.
All prompts use this
Every Return: instruction matches this exact schema.
P1 Enum (Closed) enum_value_not_allowed (closed)

Shows: unknown values, full registered list, policy, match strategy, format_hint. Decisions: map_to_existing or reject. Cannot invent new values.

P2 Enum (Open Prefer Known) unknown_enum_prefer_known

Same as P1 + allows keep_new. Keep-new values added to known set for re-validation.

P3 Type Coercion wrong_type

Full contract context. Extract correct type from malformed value or return "unk".

P4 Format Mismatch format_mismatch

Shows template + format_hint pattern. Repaired value must match pattern.

P5 Component Not Found not_in_component_db

Shows all entity names + allow_new_components flag. If false, prompt says "KEEP_NEW is NOT allowed."

P6 Cross-Field cross-field constraint failures

Full product context. value = field key, resolved_to = new field value. Uses same schema as all other prompts.

P7 Range Violation out_of_range

Shows range, unit, rounding precision. Correct magnitude or reject.

UNIT Unit Conversion wrong_unit / unit_required

Shows conversion factors when available. Fires for wrong unit or bare numbers when strict_unit_required.

9 Exact Prompt Examples (from source)

Every prompt is assembled from 3 layers: System (universal rules + 8 hallucination patterns), Field Contract Block (all parameters for this field), and Rejection-Specific User Message. All prompts use the same response schema: { status, reason, decisions: [{ value, decision, resolved_to, reasoning }] }. No confidence field. Re-validation is the only gate.

P1 Example 1: Closed Enum — colors field, value "midnight-blue"

Scenario: LLM extracted "midnight-blue" for a list-shaped color field. Alias matching failed (no case-insensitive or normalized match). Enum policy is closed → rejected.

What the LLM receives:

FIELD CONTRACT for 'colors':
  Type: string | Shape: list | Unit:
  Enum: closed | 8 known values | Match: alias
  Template: list_of_tokens_delimited

The field 'colors' has enum policy 'closed' (match: alias).
The following values are not in the registered vocabulary:
  Unknown: ["midnight-blue"]

Registered values (8 total):
  ["black","white","silver","gray","navy","green","pink","two-tone"]

For each unknown value:
1. MAP_TO_EXISTING: If it's a synonym/variant of a registered value,
   return the canonical registered name.
2. REJECT: If it looks hallucinated or nonsensical, reject it.
   Reason: hallucination_detected | not_a_real_colors

You MUST NOT invent new values for a closed enum.
Return JSON matching this exact schema:
{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the unknown value>", "decision": "map_to_existing|reject|set_unk", "resolved_to": "<canonical value or null>", "reasoning": "..." }]
}
Parameters used
type, shape, enum.policy, match.strategy, knownValues.values (full list), parse.template
Expected LLM action
map_to_existing → "navy" (midnight-blue is a navy variant) or reject
Re-validation
Repaired value re-runs all 13 steps. Must be in the known values set.
P7 Example 2: Range Violation — weight field, value 5000

Scenario: LLM extracted 5000 for weight (g). Range is 5–500g. Likely hallucinated magnitude or unit confusion (5000mg = 5g?). 

FIELD CONTRACT for 'weight':
  Type: number | Shape: scalar | Unit: g
  Range: 5 to 500 g
  Rounding: 0 decimals (nearest)
  Template: number_with_unit

The field 'weight' value is outside the expected range:
  Value: 5000 g
  Expected range: 5 to 500 g
Rounding: 0 decimals (nearest). Your value must respect this precision.

This may indicate a hallucinated magnitude, a unit mismatch, or a data entry error.

Decide:
1. If you can determine the correct value, return it.
2. If the value is genuinely unknown, return "unk".
3. If this looks hallucinated, reject it.

Return JSON matching this exact schema:
{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the failing value>", "decision": "map_to_existing", "resolved_to": <corrected number or "unk">, "reasoning": "..." }]
}
Use decision "reject" with resolved_to null if hallucinated.
Parameters used
type, shape, unit, range.min, range.max, rounding.decimals, rounding.mode, parse.template
Why rounding matters
LLM must return a whole number (0 decimals). Returning 82.3 would get rounded to 82 by step 8.
Re-validation
Repaired value must pass range check (5–500) AND rounding won't change it.

Decide:
