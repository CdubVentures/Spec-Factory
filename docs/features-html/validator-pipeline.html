Validator Pipeline Reference

Spec Factory Publish Pipeline — 11-Step Deterministic Validation + LLM Repair + Discovery Enums

1 High-Level Flow
LLM Extracts Value
→
11-Step Deterministic Validation
→
Pass?
Yes → Done (valid)
No → Repair Adapter (Attempt 2)
Build Per-Key LLM Prompt
→
LLM Returns Repair
→
Re-Validate (closed loop)
Pass → Repaired
Fail → still_failed / rerun

The pipeline is a pure function — no DB, no LLM, no side effects. Same input always produces the same output. The only gate is binary re-validation. No confidence tiers.

5 Repair Adapter (Per-Key LLM Prompts)

Every prompt includes the FIELD CONTRACT block + the universal response schema. No confidence gating — the 11-step re-validation is the only gate. Binary pass/fail. The PROMPT_REGISTRY dispatches rejection codes to prompt builders via O(1) lookup.

Universal Response Schema (all prompts)
{
  "status": "repaired" | "rerun_recommended",
  "reason": string | null,
  "decisions": [{
    "value": "the failing value",
    "decision": "map_to_existing" | "keep_new" | "reject" | "set_unk",
    "resolved_to": "canonical value" | number | null,
    "reasoning": "why this decision"
  }]
}
No confidence field
Removed. Re-validation is the only gate.
All prompts use this
Every Return: instruction matches this exact schema.
Zod validated
Response parsed by parseRepairResponse() with Zod schema.

P1 Enum (Closed) enum_value_not_allowed (closed)

Shows: unknown values, full registered list, policy, format_hint. Decisions: map_to_existing or reject. Cannot invent new values.

P2 Enum (Open Prefer Known) unknown_enum_prefer_known

Same as P1 + allows keep_new. Keep-new values added to known set for re-validation.

P3 Type Coercion wrong_type

Full contract context. Extract correct type from malformed value or return "unk".

P4 Format Mismatch format_mismatch

Shows template + format_hint pattern. Repaired value must match pattern.

P7 Range Violation out_of_range

Shows range, unit, rounding precision. Correct magnitude or reject.

UNIT Unit Conversion wrong_unit / unit_required

Shows conversion factors when available. Fires for wrong unit or bare numbers when strict_unit_required.

10 Prompt Examples (from source)

Every prompt is assembled from 3 layers: System (universal rules + 8 hallucination patterns), Field Contract Block (all parameters for this field), and Rejection-Specific User Message. All prompts use the same response schema.

P1 Example: Closed Enum — colors field, value "midnight-blue"

Scenario: LLM extracted "midnight-blue" for a list-shaped color field. Alias matching failed. Enum policy is closed → rejected. 

FIELD CONTRACT for 'colors':
  Type: string | Shape: list | Unit:
  Enum: closed | 8 known values | Match: alias
  Template: list_of_tokens_delimited

The field 'colors' has enum policy 'closed' (match: alias).
The following values are not in the registered vocabulary:
  Unknown: ["midnight-blue"]

Registered values (8 total):
  ["black","white","silver","gray","navy","green","pink","two-tone"]

For each unknown value:
1. MAP_TO_EXISTING: If it's a synonym/variant of a registered value,
   return the canonical registered name.
2. REJECT: If it looks hallucinated or nonsensical, reject it.
   Reason: hallucination_detected | not_a_real_colors

You MUST NOT invent new values for a closed enum.
Return JSON matching this exact schema:
{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the unknown value>", "decision": "map_to_existing|reject|set_unk", "resolved_to": "<canonical value or null>", "reasoning": "..." }]
}
P7 Example: Range Violation — weight field, value 5000

Scenario: LLM extracted 5000 for weight (g). Range is 5–500g. Likely hallucinated magnitude or unit confusion. 

FIELD CONTRACT for 'weight':
  Type: number | Shape: scalar | Unit: g
  Range: 5 to 500 g
  Rounding: 0 decimals (nearest)
  Template: number_with_unit

The field 'weight' value is outside the expected range:
  Value: 5000 g
  Expected range: 5 to 500 g
Rounding: 0 decimals (nearest). Your value must respect this precision.

This may indicate a hallucinated magnitude, a unit mismatch, or a data entry error.

Decide:
1. If you can determine the correct value, return it.
2. If the value is genuinely unknown, return "unk".
3. If this looks hallucinated, reject it.

Return JSON matching this exact schema:
{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the failing value>", "decision": "map_to_existing", "resolved_to": <corrected number or "unk">, "reasoning": "..." }]
}
Use decision "reject" with resolved_to null if hallucinated.
UNIT Example: Unit Conversion — weight field, value "2.65 lb"

Scenario: LLM extracted "2.65 lb" for a field expecting grams. unit_conversions has no lb factor, so deterministic conversion couldn't fire. 

FIELD CONTRACT for 'weight':

  Expected unit: g

Convert the value to g and return the numeric result.
If the value cannot be meaningfully converted, return 'unk'.

{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the original value>", "decision": "map_to_existing", "resolved_to": <converted number or "unk">, "reasoning": "..." }]
}

Scenario: LLM extracted "ergonomic-asymmetric". Policy is open_prefer_known. Alias matching tried case-insensitive + normalized — no match.

  Enum: open_prefer_known | 3 known values | Match: alias

  Unknown: ["ergonomic-asymmetric"]

Known values (3 total):

For each unknown value:

What the Prompts Do NOT Include (Honest Gaps)
MissingWhyImpact
list_rules (dedupe, sort) Deterministic auto-repairs. Step 6 fixes ordering/dupes automatically. None — re-validation handles it.
token_map entries Applied deterministically in step 4. LLM doesn't need synonym mappings. None — re-validation normalizes.
strict_unit_required The unit prompt already tells the LLM to convert to the expected unit. Low — re-validation catches bare numbers.
required_level If the LLM returns "unk", step 10 rejects it for identity/required fields. None — prompt says "if unsure, return unk" and re-validation rejects it.
Component resolution Component name validation is a review/curation concern, not a deterministic validation gate. None — by design.
