5 Repair Adapter (Per-Key LLM Prompts)

Every prompt includes the FIELD CONTRACT block + the universal response schema. No confidence gating — the 12-step re-validation is the only gate. Binary pass/fail.

Universal Response Schema (all prompts)
{
  "status": "repaired" | "rerun_recommended",
  "reason": string | null,
  "decisions": [{
    "value": "the failing value",
    "decision": "map_to_existing" | "keep_new" | "reject" | "set_unk",
    "resolved_to": "canonical value" | number | null,
    "reasoning": "why this decision"
  }]
}
No confidence field
Removed. Re-validation is the only gate.
All prompts use this
Every Return: instruction matches this exact schema.
P1 Enum (Closed) enum_value_not_allowed (closed)

Shows: unknown values, full registered list, policy, match strategy, format_hint. Decisions: map_to_existing or reject. Cannot invent new values.

P2 Enum (Open Prefer Known) unknown_enum_prefer_known

Same as P1 + allows keep_new. Keep-new values added to known set for re-validation.

P3 Type Coercion wrong_type

Full contract context. Extract correct type from malformed value or return "unk".

P4 Format Mismatch format_mismatch

Shows template + format_hint pattern. Repaired value must match pattern.

P7 Range Violation out_of_range

Shows range, unit, rounding precision. Correct magnitude or reject.

UNIT Unit Conversion wrong_unit / unit_required

Shows conversion factors when available. Fires for wrong unit or bare numbers when strict_unit_required.

9 Exact Prompt Examples (from source)

Every prompt is assembled from 3 layers: System (universal rules + 8 hallucination patterns), Field Contract Block (all parameters for this field), and Rejection-Specific User Message. All prompts use the same response schema: { status, reason, decisions: [{ value, decision, resolved_to, reasoning }] }. No confidence field. Re-validation is the only gate.

P1 Example 1: Closed Enum — colors field, value "midnight-blue"

Scenario: LLM extracted "midnight-blue" for a list-shaped color field. Alias matching failed (no case-insensitive or normalized match). Enum policy is closed → rejected.

What the LLM receives:

FIELD CONTRACT for 'colors':
  Type: string | Shape: list | Unit:
  Enum: closed | 8 known values | Match: alias
  Template: list_of_tokens_delimited

The field 'colors' has enum policy 'closed' (match: alias).
The following values are not in the registered vocabulary:
  Unknown: ["midnight-blue"]

Registered values (8 total):
  ["black","white","silver","gray","navy","green","pink","two-tone"]

For each unknown value:
1. MAP_TO_EXISTING: If it's a synonym/variant of a registered value,
   return the canonical registered name.
2. REJECT: If it looks hallucinated or nonsensical, reject it.
   Reason: hallucination_detected | not_a_real_colors

You MUST NOT invent new values for a closed enum.
Return JSON matching this exact schema:
{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the unknown value>", "decision": "map_to_existing|reject|set_unk", "resolved_to": "<canonical value or null>", "reasoning": "..." }]
}
Parameters used
type, shape, enum.policy, match.strategy, knownValues.values (full list), parse.template
Expected LLM action
map_to_existing → "navy" (midnight-blue is a navy variant) or reject
Re-validation
Repaired value re-runs all 12 steps. Must be in the known values set.
P7 Example 2: Range Violation — weight field, value 5000

Scenario: LLM extracted 5000 for weight (g). Range is 5–500g. Likely hallucinated magnitude or unit confusion (5000mg = 5g?). 

FIELD CONTRACT for 'weight':
  Type: number | Shape: scalar | Unit: g
  Range: 5 to 500 g
  Rounding: 0 decimals (nearest)
  Template: number_with_unit

The field 'weight' value is outside the expected range:
  Value: 5000 g
  Expected range: 5 to 500 g
Rounding: 0 decimals (nearest). Your value must respect this precision.

This may indicate a hallucinated magnitude, a unit mismatch, or a data entry error.

Decide:
1. If you can determine the correct value, return it.
2. If the value is genuinely unknown, return "unk".
3. If this looks hallucinated, reject it.

Return JSON matching this exact schema:
{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the failing value>", "decision": "map_to_existing", "resolved_to": <corrected number or "unk">, "reasoning": "..." }]
}
Use decision "reject" with resolved_to null if hallucinated.
Parameters used
type, shape, unit, range.min, range.max, rounding.decimals, rounding.mode, parse.template
Why rounding matters
LLM must return a whole number (0 decimals). Returning 82.3 would get rounded to 82 by step 8.
Re-validation
Repaired value must pass range check (5–500) AND rounding won't change it.
UNIT Example 3: Unit Conversion — weight field, value "2.65 lb"

Scenario: LLM extracted "2.65 lb" for a field expecting grams. unit_conversions has no lb factor (empty {} in this field), so deterministic conversion couldn't fire. Falls through to LLM prompt. 

FIELD CONTRACT for 'weight':
  Type: number | Shape: scalar | Unit: g
  Range: 5 to 500 g
  Rounding: 0 decimals (nearest)
  Template: number_with_unit

The field 'weight' requires unit 'g' but the value
was given in 'lb':
  Value: "2.65 lb"
  Expected unit: g
  Detected unit: lb

Convert the value to g and return the numeric result.
If the value cannot be meaningfully converted, return 'unk'.
Return JSON matching this exact schema:
{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the original value>", "decision": "map_to_existing", "resolved_to": <converted number or "unk">, "reasoning": "..." }]
}
Parameters used
type, shape, unit, range, rounding, parse.template, parse.unit_conversions (shown when non-empty)
Why no auto-convert?
parse.unit_conversions: {} — no factor for lb. If it had { lb: 453.592 }, step 3 would have converted deterministically without LLM.
Re-validation
Converted value must pass range (5–500g) and rounding (0 decimals).
P2 Example 4: Open Prefer Known — shape field, value "ergonomic-asymmetric"

Scenario: LLM extracted "ergonomic-asymmetric". Policy is open_prefer_known. Alias matching tried case-insensitive + normalized — no match in known values. Flagged as unknown_enum_prefer_known for LLM confirmation. 

FIELD CONTRACT for 'shape':
  Type: string | Shape: scalar
  Enum: open_prefer_known | 3 known values | Match: alias
  Template: text_field

The field 'shape' has enum policy 'open_prefer_known' (match: alias).
The following values are not in the known vocabulary:
  Unknown: ["ergonomic-asymmetric"]

Known values (3 total):
  ["ambidextrous","right-handed","left-handed"]

For each unknown value:
1. MAP_TO_EXISTING: If it's a synonym/variant of a known value,
   return the canonical known name.
2. KEEP_NEW: If it's a genuinely valid new value for this field,
   confirm it. Explain why it's legitimate.
3. REJECT: If it looks hallucinated, reject it.

Return JSON matching this exact schema:
{
  "status": "repaired",
  "reason": null,
  "decisions": [{ "value": "<the unknown value>", "decision": "map_to_existing|keep_new|reject|set_unk", "resolved_to": "<canonical value or null>", "reasoning": "..." }]
}
Parameters used
type, shape, enum.policy, match.strategy, knownValues.values (full list), parse.template
KEEP_NEW path
If LLM says keep_new, the value is temporarily added to the known set for re-validation so enum check passes.
MAP_TO_EXISTING
LLM might map "ergonomic-asymmetric" → "right-handed" since ergonomic mice are typically right-handed.
What the Prompts Do NOT Include (Honest Gaps)
MissingWhyImpact
list_rules (dedupe, sort, max/min) List rules are deterministic auto-repairs. If the LLM returns a list, step 7 fixes ordering/dupes automatically. None — re-validation handles it.
token_map entries Token map is applied deterministically in step 5. LLM doesn't need to know synonym mappings. None — re-validation normalizes.
strict_unit_required The unit prompt already tells the LLM to convert to the expected unit. Re-validation enforces the suffix requirement. Low — re-validation catches bare numbers.
block_publish_when_unk This is a publish gate, not a repair instruction. If the LLM returns "unk", step 11 rejects it regardless. None — the prompt already says "if unsure, return unk" and re-validation rejects it.
Component resolution Component name validation (sensor, switch, encoder, material) is a review/curation concern, not a deterministic validation gate. No P5 prompt exists. None — by design. Component names are open-ended; curation happens in the review phase.
10 Discovery Enums (Self-Tightening Vocabulary)

34 fields have open_prefer_known policy but no static known-values list. The discovery enum system closes this gap by accumulating vocabulary from indexed products, so validation gets stricter over time without manual curation.

The Problem

validateRecord.js:20 does knownValues?.enums?.[fieldKey] || null. If a field has no entry in known_values.json (e.g., sensor_brand, grip, design), enumValues is undefined and the enum check at Step 9 is silently skipped. Any value passes — including sensor_brand: "Gucci".

CategoryCountBehavior
Fields with own entry in known_values.json14Enum check works correctly
Boolean/yes_no fields (template handles it)18Template dispatch normalizes before enum — harmless
Component reference fields4Review/curation concern, not validation
String fields with no known values12Enum check silently skipped — any value passes
The Solution: Three New Modules

mergeDiscoveredEnums validation/mergeDiscoveredEnums.js Pure function. Merges compiled known_values + DB-discovered values into a single knownValues object. Deduplicates case-insensitively.
buildDiscoveredEnumMap buildDiscoveredEnumMap.js DB reader bridge. Queries list_values for source='pipeline' rows per field. Receives specDb via injection (no direct DB imports).
persistDiscoveredValue persistDiscoveredValues.js Post-validation persistence. Writes new accepted values to list_values with source='pipeline', needs_review=true.

Zero changes. Stays pure. Merge happens at the caller level before invocation.
DB schema
Zero changes. Uses existing enum_lists + list_values tables with source='pipeline'.
