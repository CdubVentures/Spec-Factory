Product Image Finder (PIF) Reference

Carousel-based image discovery with per-view LLM search, hero images, RMBG 2.0, and cumulative learning

In-app explainer: The PIF panel includes a collapsible “How It Works” section (between KPI grid and All Images) that explains the pipeline, carousel loop, learning, quality gates, and settings to end users. 1 Overview
Carousel Loop Architecture

PIF discovers and downloads product images for each colorway defined by the Color & Edition Finder (CEF). A single Loop action autonomously fills the carousel: focused view searches first, then hero images, then done. Each LLM call targets one view at a time, but side-catches from other views are always kept. Discovery learning accumulates across all calls so repeated loops find new sources.

2 Variant Resolution
Search Target Resolution from CEF Data

PIF builds its search list from CEF's colors array via buildVariantList({ colors, colorNames, editions }). Each entry becomes a search target. The search label is resolved using a strict priority chain.

  • Priority 1: Edition display name — If a combo entry matches an edition, search by edition name.
    Example: "Call of Duty Black Ops 6 Edition"
  • Priority 2: Marketing color name — If a color has a color_name, search by that.
    Example: "Frost White" not "ivory".
  • Priority 3: Raw atom (fallback) — If neither edition nor color_name exists, search by atom.
    Example: "black".
Variant key format
color:{atom} for standard colors, edition:{slug} for editions
Variant type
"color" or "edition" — used for chip styling (CLR/ED)
Variant ID
Stable v_<8-hex> hash from CEF variant registry. Enriched onto each variant after buildVariantList(). null if no registry exists.
Variant ID Linkage (Phase 3)

Problem: variant_key is mutable — CEF can rename it across runs (e.g. ocean-blue → deep-ocean-blue). When this happens, PIF images tagged with the old key become orphaned.

Solution: Every image carries a variant_id (stable hash from CEF registry) alongside variant_key. All filtering uses matchVariant() — a centralized predicate that matches by variant_id when both sides have it, falling back to variant_key for legacy data.

5 Carousel Loop
Loop State Machine

The carousel loop is a server-side orchestrator (runCarouselLoop()) that processes one variant at a time, sequentially. For each variant it calls evaluateCarousel() (pure function) to decide what to do next.

Evaluate
carousel state
→
Focus View
highest priority
→
LLM Call
download + gate
→
Re-evaluate
loop or next
Per-view focus
Each LLM call has ONE unsatisfied view as priority. All 8 views remain in the prompt for side-catches.
Satisfaction
A view is "satisfied" when it has ≥ satisfactionThreshold quality-passing images (default: 3)
Budget exhaustion
A view is "exhausted" when it has used ≥ its per-view attempt budget. Budgets are category-aware (e.g. mouse: top=4, left=4, angle=3, sangle=3, front=2, bottom=2). Flat viewAttemptBudget (default: 5) is a fallback for views without category defaults.
View done
Satisfied OR exhausted. Move to next unsatisfied view in budget priority order.
Mode transitions
Unsatisfied views remain → view mode. All views done → hero mode. Hero done → complete.
Variants
Processed sequentially (not concurrent) for clean progress reporting
Estimation
estimatedCallsRemaining computed from remaining attempts × images needed per view/hero

Forced Cycle (Re-loop When Complete)

When the carousel is already complete and the user hits Loop again, PIF runs a forced cycle: one LLM call per budget view + one hero call. This accumulates more image candidates using fresh discovery context (the learning log ensures new sources are searched).

Trigger
Loop starts and evaluateCarousel returns isComplete: true on first check
Behavior
1 call per budget view (focused prompt) + 1 hero call. No satisfaction checking.
Learning
Discovery log still accumulates — each forced cycle finds new sources the LLM hasn't tried
Purpose
Build a richer candidate pool for a future evaluator phase that picks the best images
Concurrent Single Runs

View and Hero buttons fire individual single-run LLM calls independently of any running loop. They are never locked — the user can fire as many as they want.

View button
Single LLM call with view prompt for one variant. Never disabled.
Hero button
Single LLM call with hero prompt for one variant. Never disabled.
Loop interaction
If single runs fill the carousel while a loop is running, the loop's next iteration re-reads disk, sees complete, and exits cleanly.
6 Hero Image Search
Separate Prompt, Separate Intent

Hero search uses a completely different LLM prompt (buildHeroImageFinderPrompt) than view search. It targets promotional, marketing, and official render images — not standard product photography angles.

Hero targets

  • Official brand hero images
  • Marketing renders and press shots
  • Box art and packaging imagery
  • Dramatic lighting / branded backgrounds
  • Press kit and launch imagery

Hero excludes

  • White-background product shots (those are "views")
  • User photos and unboxing
  • Generic banners and logos
  • Small thumbnails
View name
All hero images use view: "hero"
Quality thresholds
Uses viewQualityMap.hero minimums (not .top)
Attempt budget
heroAttemptBudget (default: 3 calls per variant)
Target count
heroCount (default: 3 images per variant)
Disableable
heroEnabled: false skips hero phase entirely
RMBG
Hero images are never background-removed (they have intentional backgrounds)
LLM spec
HERO_IMAGE_FINDER_SPEC — phase: imageFinder, reason: hero_image_finding
7 Discovery Learning
Cumulative Per-Variant Learning

Every LLM call returns a discovery_log with URLs checked, queries run, and notes. These are accumulated per variant across all runs. Subsequent calls receive the full history so the LLM searches new sources instead of repeating prior work.

Accumulation
accumulateVariantDiscoveryLog(previousRuns, variantKey) unions all prior urls_checked and queries_run for the specific variant
Prompt injection
"Previous searches for this variant (do not repeat — find NEW sources)"
Scope
Per variant key (e.g. color:black). Different variants have independent discovery histories.
Persistence
Stored in each run's response.discovery_log in the JSON store

8 RMBG 2.0 Image Processing

Every downloaded image that passes the quality gate is processed through automated background removal. Originals are always preserved.

9 Accumulation Model (Override)
All-Runs Union (Not Latest-Wins)

PIF overrides the default finderJsonStore latest-wins behavior. Instead, recalculateSelected unions images across ALL non-rejected runs per variant.

Rationale
Each variant run adds unique images. Filenames use -N suffix so no manual dedup needed. Carousel strategy uses full image set to determine satisfaction.
Contrast with CEF
CEF uses default latest-wins (most recent non-rejected run's output is selected). PIF accumulates everything.
On run delete
Recalculates from remaining runs — images from the deleted run are removed from selected aggregate.
Store factory
createFinderJsonStore({ filePrefix: 'product_images', emptySelected: () => ({ images: [] }), recalculateSelected: customUnion })
