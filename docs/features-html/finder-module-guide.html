How to Build a New Finder Module

Definitive guide for adding an Nth LLM-driven discovery module to the Spec Factory finder framework

Update — 2026-04-18: Zod-derived FE types + variantDependent signal (Phase 1)

Four registry-hardening fixes landed. Future variantFieldProducer modules (SKU, pricing, discontinued) now auto-light-up in the review drawer and can derive their frontend response types from their backend Zod schema.

  1. Review drawer uses variantDependent, not field-key equality. FieldReviewDrawer.tsx:915 previously read fieldKey === 'release_date' to trigger variant attribution. Now it reads the variantDependent prop sourced from field_rule.variant_dependent. Any new variantFieldProducer field gets correct attribution automatically — no drawer edit needed. Backend helper isVariantDependentField(fieldKey, specDb) in src/core/finder/finderModuleRegistry.js is the authoritative signal.
  2. EG_LOCKED_KEYS imported from SSOT. src/ingest/compileComponentHelpers.js and src/ingest/compileMapNormalization.js used to hardcode new Set(['colors','editions','release_date']). They now import from src/features/studio/contracts/egPresets.js (derived from EG_PRESET_REGISTRY). When you add an EG-locked field, both ingest compilers pick it up automatically.
  3. Frontend types derived from Zod. New pattern: scripts/generateRdfTypes.js reads the backend Zod schema via schema.toJSONSchema() and emits types.generated.ts with exact types. types.ts imports the generated type and composes it with UI-layer wrapper types (variant metadata, timing). Kills silent drift bugs — the pre-fix hand-written type declared fields (source_url, excerpt) the backend never emitted, so the RDF evidence card silently rendered blank URLs. Compile-time errors now.
  4. Two safe patterns for evidence rendering. New finders rendering evidence should pick one: (a) use the review's resolveEvidenceSources() adapter from tools/gui-react/src/features/review/selectors/publishedSourceSelectors.ts which normalizes modern evidence_refs + legacy evidence_sources shapes, or (b) derive types from Zod (RDF pattern). Never hand-write a local evidence interface — TypeScript cannot verify it matches runtime JSON.

Impact on new-finder checklist: step "Define types" below can be partially auto-generated when the backend has a Zod schema. Follow RDF's pattern — see Release Date Finder Reference § 9 (Frontend Files).

CEF and PIF were not touched — their schemas and panels remain custom (legitimately — CEF has identity-check phase, PIF has image/carousel/eval machinery). The four fixes apply only to shared infrastructure used by all modules.

Update — 2026-04-16: Finder Settings Renderer + Route Context Dedup

The settings UI and route context factories changed. Sections below that reference the old shapes are stale in those specifics. New-finder rules:

  • settingsDefaults → settingsSchema: each module declares a typed array of setting entries ({ key, type, default, min?, max?, allowed?, uiLabel?, uiTip?, uiGroup?, uiHero?, secret?, disabledBy?, allowEmpty?, hidden?, widget?, widgetProps? }). Same shape as RUNTIME_SETTINGS_REGISTRY. deriveFinderSettingsDefaults(schema) produces the legacy { key: stringDefault } map for SQL/DDL.
  • No per-finder settings form: CefSettingsForm / PifSettingsForm / MODULE_SETTINGS_FORMS are deleted. FinderSettingsRenderer renders every primitive entry from the generated FINDER_SETTINGS_REGISTRY. Non-primitive controls (budgets, grids, reorderable lists) are named widgets registered via registerSettingWidget(name, Component).
  • No per-finder route context factory: the contextFile / contextExport registry fields are gone. Every module uses the shared createFinderRouteContext(options) from src/core/finder/finderRouteContext.js, which returns only the shared plumbing (jsonRes, readJsonBody, config, appDb, getSpecDb, broadcastWs, logger). Per-finder orchestrator functions are imported locally by the thin route wrapper that uses them.
  • No settingsFormPath / settingsFormExport: those registry fields are gone. The renderer lookup is by module id alone.

New-finder checklist (O(1)):

  1. Add one FINDER_MODULES entry in src/core/finder/finderModuleRegistry.js with settingsSchema (widgets referenced by name; reuse viewBudget/viewQualityGrid/viewConfig/evalThumbSize where they fit).
  2. Write feature files: orchestrator, LLM adapter, JSON store, panel component, thin route registrar.
  3. Run node tools/gui-react/scripts/generateLlmPhaseRegistry.js.
  4. If you need a brand-new widget type, add a widgets/<Name>.tsx file and register it in widgets/index.ts.

Treat mentions below of settingsDefaults, CefSettingsForm/PifSettingsForm, MODULE_SETTINGS_FORMS, contextFile/contextExport, settingsFormPath/settingsFormExport, or the context-factory step as historical context. The bullets above are the current contract.

1 Decision Tree — What Type of Finder Are You Building?
Three Finder Patterns
PatternExamplefieldKeyssubmitCandidate?ValidationReview Grid?
Field Finder
Discovers spec field values		 CEF (colors, editions) ['colors','editions'] Yes — every value passes through candidate gate Full: enum, token repair, palette check Yes — automatic via field_candidates table
Simple Field Finder
Discovers a single field, minimal validation		 Release Date finder ['release_date'] Yes — but field rule defines lighter validation (e.g., date format + range only) Depends on field rule: type coerce, format, range. No enum, no palette. Yes — automatic via field_candidates table
Artifact Finder
Discovers assets, not field values		 PIF (product images) [] No — images are artifacts, not spec fields None (custom evaluation instead) No — custom persistence + evaluation
How to choose

Does your LLM discover a value that belongs in a product’s spec fields (date, weight, price, dimension)? → Field Finder or Simple Field Finder.

Does it discover assets (images, videos, documents)? → Artifact Finder.

Does it discover variants (build the registry, identity-check on Run 2+, assign permanent v_<hex> hashes)? → CEF-specific. No new finder should re-implement this.

Does it discover a value that varies per variant (e.g. release-date per color, SKU per edition)? → You are a variant-scoped Field Finder. Read variants from specDb.variants.listByProduct(productId) — CEF already populated them — then call submitCandidate({ ..., variantId }) once per variant. The candidate gate handles SQL+JSON dual-write; deleteVariant() auto-cascades your candidate rows. See Section 9 for the wiring.

Settings read pattern (backend) Your orchestrator reads per-category settings via the auto-wired SQL store:

const override = store.getSetting('promptOverride'); // returns '' if unset

To add a new setting key later: add it to settingsDefaults in the registry. No migration needed — the store fallback handles missing rows. But remember: you must also add a UI control in your settings form for the user to actually change it.

Frontend Steps
  1. Create feature folder
    tools/gui-react/src/features/my-finder/ with api/ and components/ subdirs
  2. Define types (Zod-derived, not hand-written)
    The LLM-response sub-shape of your GET response must be derived from the backend Zod schema to prevent silent drift. Follow RDF's pattern:
    1. Copy tools/gui-react/scripts/generateRdfTypes.js → generateMyFinderTypes.js. Change the import to your backend schema and the output path to tools/gui-react/src/features/my-finder/types.generated.ts.
    2. Run it: node tools/gui-react/scripts/generateMyFinderTypes.js. Emits {YourSchema}Gen interfaces derived from schema.toJSONSchema().
    3. In types.ts, import type the generated types and compose them with your UI-layer wrapper types (variant metadata, timing, publisher refs, etc.). Hand-written buildGetResponse-shape additions stay in types.ts; the Zod-validated part stays generated.
    4. Rerun codegen whenever the backend Zod schema changes. Commit the .generated.ts output.

    Why: TypeScript has no way to verify hand-written interface fields exist in actual backend JSON. RDF's evidence card silently rendered blank URLs for months because the local EvidenceSource interface declared source_url/excerpt that the backend never emitted. Deriving from Zod turns drift into a compile error.

9 Candidate Integration — Field Finders Only
submitCandidate() Pattern

If your finder populates spec fields (fieldKeys is non-empty), you must validate LLM output through the candidate gate before persisting:

import { submitCandidate } from '../../publisher/index.js';

// In your orchestrator, after LLM response:
const result = await submitCandidate({
  category,
  productId,
  fieldKey: 'my_field',
  value: response.my_field,
  confidence: 100,                    // direct LLM discovery = 100
  sourceMeta: {
    source: 'myf',                    // matches moduleType
    run_number: runNumber,            // deterministic source_id
  },
  fieldRules: compiledRules,
  knownValues,
  specDb,
  productRoot,
  config,                             // optional: enables auto-publish
  variantId: null,                    // null/omit = product-level field
                                      // pass v_<hex> for variant-scoped fields
});

What happens inside submitCandidate():

  1. 12-step deterministic validation pipeline (reads your field rule — skips inapplicable steps)
  2. Builds deterministic source_id: myf-{productId}-{runNumber}
  3. SQL write to field_candidates table (with variant_id if provided)
  4. JSON write to product.json candidates[] (with variant_id key if provided)
  5. Optional auto-publish if confidence meets threshold

On validation failure (hard reject):

if (result.status === 'REJECTED') {
  // Persist rejected run to JSON + SQL (for audit trail)
  // Set status: 'rejected', no cooldown set
  // See CEF's storeFailureAndReturn() for the full pattern
}
What validation your field gets depends entirely on its field rule
  • Date field with type: 'string' + format check → only type coerce + format validation
  • Enum field with enum.policy: 'closed' → full enum + token repair + palette check
  • No field rule defined → submitCandidate() still works (minimal validation)
The validation pipeline is smart enough to skip steps that don’t apply to your field’s rule shape.
Variant-scoped fields (release-date, SKU, per-variant pricing, …) — moduleClass: 'variantFieldProducer'

If your finder produces one value per variant, iterate over specDb.variants.listByProduct(productId) (CEF-built — do not re-derive) and submit one candidate per variant with that variant’s variant_id:

const variants = specDb.variants.listByProduct(productId);
for (const v of variants) {
  await submitCandidate({
    /* … same as above … */
    fieldKey: 'release_date',
    value: response.releases[v.variant_key],
    variantId: v.variant_id,            // FK anchor to variants table
  });
}
  • SQL UNIQUE(category, product_id, field_key, source_id, variant_id_key) lets the same source_id coexist with different variant_ids — one row per variant.
  • deleteVariant() automatically cascades your rows via specDb.deleteFieldCandidatesByVariantId(...) + JSON mirror filter. No per-feature cleanup code needed.
  • Reseed (candidateReseed.js) round-trips variant_id through product.json → field_candidates.
  • Publisher path ③ (variant-scoped) fires automatically when candidateRow.variant_id is set. Published value lands in product.json.variant_fields[variant_id][fieldKey] — the scalar fields[fieldKey] is untouched. Candidate resolution (markFieldCandidateResolved, demoteResolvedCandidates) is scoped to (productId, fieldKey, variantId), so two variants publishing the same field are independent. See Publisher Pipeline § 3b.
  • Delete cascade for variant_fields: cascadeVariantIdFromCandidates() in variantLifecycle.js drops product.json.variant_fields[vid] when the variant is removed. No per-feature code.

12 Common Patterns & Decisions
Critical: Validate Before Persist

If your orchestrator has multi-stage validation (e.g., discovery + identity check), all validation gates must pass before calling merge(). The JSON store’s merge() writes to disk immediately — there is no rollback.

Wrong ordering creates a ghost successful run + polluted selected state:

// BAD: merge writes to disk, then validation rejects → ghost run
const merged = mergeDiscovery({ ... });        // writes JSON!
if (!validateStage2(result)) {
  return storeFailureAndReturn({ ... });       // appends SECOND run
}

// GOOD: validate first, merge only after all gates pass
if (!validateStage2(result)) {
  return storeFailureAndReturn({ ... });       // single rejected run
}
const merged = mergeDiscovery({ ... });        // clean write
Prompt vs Validation — Where Field Rules Apply

Field Studio rules (ai_assist.reasoning_note, enum policies, parse templates) are not automatically injected into finder LLM prompts. Finder prompts are hand-authored per module.

Field rules apply after the LLM responds, at the candidate validation gate (submitCandidate()). This is by design: finders discover broadly, then the gate validates and repairs.

If your finder needs to inform the LLM about field constraints, read compiled rules explicitly in your prompt builder. Do not assume they flow in automatically.

Accumulation Strategy
Latest-wins
Default. selected = most recent non-rejected run. Used by CEF (colors/editions overwrite).
Accumulate
Override recalculateSelected in createFinderJsonStore. Used by PIF (images union across runs).

Choose based on domain: does a new run replace the answer or add to it?

Run Rejection Pattern

When validation fails or LLM output is invalid:

  1. Persist the run to JSON with status: 'rejected'
  2. Insert SQL run row with status: 'rejected'
  3. Do not update selected or cooldown_until
  4. Return result with rejection reason for UI display

See CEF’s storeFailureAndReturn() for the full pattern.

buildGetResponse Hook

The generic route handler calls your buildGetResponse hook to shape the GET response. This is where you add domain-specific fields beyond the standard (cooldown, runs, etc.):

buildGetResponse: (row, selected, runs,
  onCooldown, { specDb, productId }) => ({
  // Standard fields are already included
  // Add your domain-specific shape:
  my_field: row.my_field,
  published: { ... },
  candidates: { ... },
})

If omitted, the default response includes only standard fields. CEF uses this for published.colors, published.editions, candidates, and variant_registry (sourced from specDb.variants.listByProduct() — the variants table is the runtime SSOT; no summary-column blob).

dataChangeEvents (Extra Domains)

By default, three events are emitted: {routePrefix}-run, {routePrefix}-run-deleted, {routePrefix}-deleted. Each invalidates its own routePrefix domain.

Use dataChangeEvents to add extra domains. Example: CEF adds ['color-registry'] to the 'run' event because discovering colors should invalidate the color registry cache:

dataChangeEvents: {
  'run': ['color-registry'],  // extra domain
  'run-deleted': [],
  'deleted': [],
}
