How to Build a New Finder Module

Definitive guide for adding an Nth LLM-driven discovery module to the Spec Factory finder framework

1 Decision Tree — What Type of Finder Are You Building?
Three Finder Patterns
PatternExamplefieldKeyssubmitCandidate?ValidationReview Grid?
Field Finder
Discovers spec field values		 CEF (colors, editions) ['colors','editions'] Yes — every value passes through candidate gate Full: enum, token repair, palette check Yes — automatic via field_candidates table
Simple Field Finder
Discovers a single field, minimal validation		 Release Date finder ['release_date'] Yes — but field rule defines lighter validation (e.g., date format + range only) Depends on field rule: type coerce, format, range. No enum, no palette. Yes — automatic via field_candidates table
Artifact Finder
Discovers assets, not field values		 PIF (product images) [] No — images are artifacts, not spec fields None (custom evaluation instead) No — custom persistence + evaluation
How to choose

Does your LLM discover a value that belongs in a product’s spec fields (date, weight, price, dimension)? → Field Finder or Simple Field Finder.

Does it discover assets (images, videos, documents)? → Artifact Finder.

Does it need variant identity tracking (stable hashes, identity check on Run 2+)? → That is CEF-specific logic. Most new finders will not need this.

3 The Split Control Plane — Registries You Touch
There Is No Single Manifest

Feature wiring is split across multiple registries. Every new finder touches #1 and #2. The others are conditional.

#RegistryFileWhat It DrivesWhen
1 Finder Module Registry src/core/finder/finderModuleRegistry.js DDL, SQL store, routes, reseed, events, settings, panels, cleanup, codegen Always
2 LLM Phase Definitions src/core/config/llmPhaseDefs.js Model resolution, token limits, phase UI in LLM config sidebar Always (one entry, ~8 lines)
3 EG Preset Registry src/features/studio/contracts/egPresets.js Locked field rules, default toggles, backfill, compile injection Only if your field needs an EG-locked preset (like colors/editions)
4 Phase Schema Overlays src/features/indexing/pipeline/shared/phaseSchemaRegistry.js Multi-prompt preview (identity check, per-view eval), editable criteria Only if your finder has sub-prompts beyond the main discovery call
5 Server Boot (temporary) src/app/api/guiServerRuntime.js Static imports + route definitions Until wireFinderRoutes() is integrated (see Section 0)
For a simple field finder (like Release Date) You touch #1, #2, and #5 only. No EG preset, no schema overlay. Three files modified total (plus your new feature code).

Tier 4: Configuration (Settings, Field Rules — NOT Code)
WhatWhere to DeclareAuto?What You Get
Settings storage (SQL table + read/write + fallback defaults) settingsDefaults in your registry entry AUTO DDL creates {tableName}_settings table. getSetting(key) returns DB value or falls back to registry default. No migration needed to add new keys later.
Settings JSON mirror (durable backup) Automatic via moduleSettingsRoutes.js AUTO Every PUT also writes to category_authority/{category}/{filePrefix}_settings.json. Useful for seeding and debugging.
Settings sidebar entry (nav link in Pipeline Settings) settingsLabel / settingsSubtitle / settingsTip in registry AUTO Codegen creates sidebar link from registry metadata.
Settings form UI (actual controls the user interacts with) ModuleSettingsPanel.tsx MANUAL You must write a settings form component and add a rendering branch in ModuleSettingsPanel. Without this, your sidebar link works but renders the wrong form (defaults to CefSettingsForm, which only shows cooldownDays).
Field rule (if standard field) Category field studio (via GUI or seed script) AUTO Validation pipeline reads rule at candidate gate time; no code needed.
EG-locked field rule EG_PRESET_REGISTRY in egPresets.js (add one builder function) AUTO Locked toggle, backfill, compile injection, sanitization — all auto-derived.
The settings UI gap settingsDefaults gives you storage + sidebar + fallback for free, but the form controls (input fields, toggles, text areas) are hand-written per module. Today ModuleSettingsPanel.tsx hard-switches between PifSettingsForm and CefSettingsForm (lines 1136–1153). Even CEF’s form only renders cooldownDays despite having identityCheckPromptOverride in the registry. You must write your own form and add a rendering branch.
Settings read pattern (backend) Your orchestrator reads per-category settings via the auto-wired SQL store: 
const store = specDb.getFinderStore('myFinder');
const cooldown = store.getSetting('cooldownDays'); // returns default if unset
const override = store.getSetting('promptOverride'); // returns '' if unset

To add a new setting key later: add it to settingsDefaults in the registry. No migration needed — the store fallback handles missing rows. But remember: you must also add a UI control in your settings form for the user to actually change it.

Settings JSON mirror path Every settings PUT also writes a durable JSON copy to:
category_authority/{category}/{filePrefix}_settings.json
This is automatic via moduleSettingsRoutes.js. Useful for seeding initial values or debugging: you can inspect or pre-populate this file directly.

How prompt previews work: The codegen script reads promptBuilderExport and responseSchemaExport from each FINDER_MODULES entry. It generates phaseSchemaRegistry.generated.js which:

  1. Dynamically imports your prompt builder function from src/features/{featurePath}/{camelCase}LlmAdapter.js
  2. Calls it with a placeholder: buildMyFinderPrompt({ product: { brand: '{brand}', model: '{model}' } })
  3. Wraps your Zod schema in zodToLlmSchema()
  4. Exports both as FINDER_PHASE_SCHEMAS['my-finder']

The backend serves this via GET /api/indexing/llm-config. The frontend LlmPhaseSection.tsx renders the prompt preview (left column) and schema preview (right column) automatically.

Naming convention (critical) featurePath: 'my-feature' → codegen derives myFeatureLlmAdapter.js and myFeatureSchema.js using kebab-to-camelCase conversion. If your file is named differently, the import succeeds at codegen time but fails at runtime.

8 LLM Config Preview — Prompt & Schema Display
How Your Finder Appears in LLM Config

After codegen, LlmPhaseSection.tsx renders your finder’s prompt and schema preview automatically. The component uses conditional rendering based on the shape of phaseSchema:

phaseSchema ShapeRendered LayoutExample Module
{ system_prompt, response_schema } Single-column: system prompt (left) + response schema (right) Any simple finder — this is what you get by default
{ system_prompt, response_schema, identity_check_prompt, identity_check_response_schema } Two-column: discovery (left) + identity check (right) CEF only (variant identity check on Run 2+)
{ system_prompt, view_prompts, hero_system_prompt, ... } Tabbed: per-view prompt tabs + hero prompt + schemas PIF image evaluator only
{ ..., eval_criteria_defaults, eval_criteria_categories } Editable tabs: per-category, per-view eval criteria with save PIF image evaluator only
For most new finders You get the single-column layout for free. No overlay, no manual phaseSchemaRegistry.js edits needed. Your prompt builder is called with a placeholder product at codegen time, and the result is displayed in the LLM Config page.
If your finder has sub-prompts (identity check, evaluation, etc.) You must manually add an overlay in src/features/indexing/pipeline/shared/phaseSchemaRegistry.js (the non-generated file). Spread your auto-generated entry and add the extra prompt/schema fields. See COLOR_FINDER_IDENTITY_CHECK or CAROUSEL_BUILDER_PHASE as examples.
9 Candidate Integration — Field Finders Only
submitCandidate() Pattern

If your finder populates spec fields (fieldKeys is non-empty), you must validate LLM output through the candidate gate before persisting:

import { submitCandidate } from '../../publisher/index.js';

// In your orchestrator, after LLM response:
const result = await submitCandidate({
  category,
  productId,
  fieldKey: 'my_field',
  value: response.my_field,
  confidence: 100,                    // direct LLM discovery = 100
  sourceMeta: {
    source: 'myf',                    // matches moduleType
    run_number: runNumber,            // deterministic source_id
  },
  fieldRules: compiledRules,
  knownValues,
  specDb,
  productRoot,
  config,                             // optional: enables auto-publish
});

What happens inside submitCandidate():

  1. 12-step deterministic validation pipeline (reads your field rule — skips inapplicable steps)
  2. Builds deterministic source_id: myf-{productId}-{runNumber}
  3. SQL write to field_candidates table
  4. JSON write to product.json candidates[]
  5. Optional auto-publish if confidence meets threshold

On validation failure (hard reject):

if (result.status === 'REJECTED') {
  // Persist rejected run to JSON + SQL (for audit trail)
  // Set status: 'rejected', no cooldown set
  // See CEF's storeFailureAndReturn() for the full pattern
}
What validation your field gets depends entirely on its field rule
  • Date field with type: 'string' + format check → only type coerce + format validation
  • Enum field with enum.policy: 'closed' → full enum + token repair + palette check
  • No field rule defined → submitCandidate() still works (minimal validation)
The validation pipeline is smart enough to skip steps that don’t apply to your field’s rule shape.

12 Common Patterns & Decisions
Critical: Validate Before Persist

If your orchestrator has multi-stage validation (e.g., discovery + identity check), all validation gates must pass before calling merge(). The JSON store’s merge() writes to disk immediately — there is no rollback.

Wrong ordering creates a ghost successful run + polluted selected state:

// BAD: merge writes to disk, then validation rejects → ghost run
const merged = mergeDiscovery({ ... });        // writes JSON!
if (!validateStage2(result)) {
  return storeFailureAndReturn({ ... });       // appends SECOND run
}

// GOOD: validate first, merge only after all gates pass
if (!validateStage2(result)) {
  return storeFailureAndReturn({ ... });       // single rejected run
}
const merged = mergeDiscovery({ ... });        // clean write
Prompt vs Validation — Where Field Rules Apply

Field Studio rules (ai_assist.reasoning_note, enum policies, parse templates) are not automatically injected into finder LLM prompts. Finder prompts are hand-authored per module.

Field rules apply after the LLM responds, at the candidate validation gate (submitCandidate()). This is by design: finders discover broadly, then the gate validates and repairs.

If your finder needs to inform the LLM about field constraints, read compiled rules explicitly in your prompt builder. Do not assume they flow in automatically.

Accumulation Strategy
Latest-wins
Default. selected = most recent non-rejected run. Used by CEF (colors/editions overwrite).
Accumulate
Override recalculateSelected in createFinderJsonStore. Used by PIF (images union across runs).

Choose based on domain: does a new run replace the answer or add to it?

Run Rejection Pattern

When validation fails or LLM output is invalid:

  1. Persist the run to JSON with status: 'rejected'
  2. Insert SQL run row with status: 'rejected'
  3. Do not update selected or cooldown_until
  4. Return result with rejection reason for UI display

See CEF’s storeFailureAndReturn() for the full pattern.

buildGetResponse Hook

The generic route handler calls your buildGetResponse hook to shape the GET response. This is where you add domain-specific fields beyond the standard (cooldown, runs, etc.):

buildGetResponse: (row, selected, runs,
  onCooldown, { specDb, productId }) => ({
  // Standard fields are already included
  // Add your domain-specific shape:
  my_field: row.my_field,
  published: { ... },
  candidates: { ... },
})

If omitted, the default response includes only standard fields. CEF uses this for published.colors, published.editions, candidates, and variant_registry.

dataChangeEvents (Extra Domains)

By default, three events are emitted: {routePrefix}-run, {routePrefix}-run-deleted, {routePrefix}-deleted. Each invalidates its own routePrefix domain.

Use dataChangeEvents to add extra domains. Example: CEF adds ['color-registry'] to the 'run' event because discovering colors should invalidate the color registry cache:

dataChangeEvents: {
  'run': ['color-registry'],  // extra domain
  'run-deleted': [],
  'deleted': [],
}
