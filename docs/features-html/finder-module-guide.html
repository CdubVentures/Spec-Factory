How to Build a New Finder Module

Step-by-step guide for adding a third (or Nth) LLM-driven discovery module to the Spec Factory finder framework

1 What the Framework Gives You for Free
Auto-Wired from One Registry Entry

Adding one object to finderModuleRegistry.js auto-generates or auto-wires all of the following. You write zero boilerplate for these:

ConcernMechanism
SQL DDL (summary + runs + settings tables)finderSqlDdl.js reads registry at boot
SQL store instance (CRUD + runs)specDb.js loops FINDER_MODULES, creates finderSqlStore per module
Data-change events + invalidation domainsFINDER_DATA_CHANGE_EVENTS derived from dataChangeEvents

Indexing Lab panel (lazy-loaded)Codegen: finderPanelRegistry.generated.ts

3 Step-by-Step Checklist
Backend Steps
  1. Add LLM phase definition
    src/core/config/llmPhaseDefs.js — add one entry: 
    { id: 'myFinder', uiId: 'my-finder', label: 'My Finder',
  subtitle: 'Discovery', tip: '...',
  roles: ['triage'], group: 'discovery',
  globalModel: 'llmModelPlan',
  groupToggle: 'llmPlanUseReasoning', ... }
  2. Create feature folder
    src/features/my-feature/ with api/ and tests/ subdirs
  3. Write Zod schema
    myFinderSchema.js — export myFinderResponseSchema
  4. Write LLM adapter
    myFinderLlmAdapter.js — export prompt builder(s), createMyFinderCallLlm(deps), and the SPEC object: 
    export const MY_FINDER_SPEC = {
  phase: 'myFinder',
  reason: 'my_finding',
  role: 'triage',
  jsonSchema: zodToLlmSchema(myFinderResponseSchema),
};
  5. Write JSON store
    myFinderStore.js — thin wrapper: 
    import { createFinderJsonStore } from '../../core/finder/finderJsonStore.js';

const store = createFinderJsonStore({
  filePrefix: 'my_feature',
  emptySelected: () => ({ items: [] }),
  // Optional: override recalculateSelected for accumulation (like PIF)
});

export const readMyFinder = store.read;
export const writeMyFinder = store.write;
export const mergeMyFinderDiscovery = store.merge;
export const deleteMyFinderRun = store.deleteRun;
export const deleteMyFinderAll = store.deleteAll;
  6. Write orchestrator
    myFinder.js — import shared helpers to avoid boilerplate: 
    import { buildLlmCallDeps } from '../../core/llm/buildLlmCallDeps.js';
import {
  computeCooldownUntil,
  resolveModelTracking,
  resolveAmbiguityContext,
  buildFinderLlmCaller,
} from '../../core/finder/finderOrchestrationHelpers.js';
import { resolveIdentityAmbiguitySnapshot }
  from '../indexing/orchestration/shared/identityHelpers.js';

export async function runMyFinder({ product, specDb, config, ... }) {
  // 1. Model tracking (1 line instead of 12)
  const mt = resolveModelTracking({
    config, phaseKey: 'myFinder', onModelResolved,
  });

  // 2. Identity ambiguity (1 line instead of 15)
  const { familyModelCount, ambiguityLevel } =
    await resolveAmbiguityContext({
      config, category: product.category,
      brand: product.brand, baseModel: product.base_model,
      specDb, resolveFn: resolveIdentityAmbiguitySnapshot,
    });

  // 3. LLM caller (1 line instead of 12)
  const callLlm = buildFinderLlmCaller({
    _callLlmOverride,
    wrappedOnModelResolved: mt.wrappedOnModelResolved,
    createCallLlm: createMyFinderCallLlm,
    llmDeps: buildLlmCallDeps({ config, logger,
      onPhaseChange: onStageAdvance ? (phase) => {
        if (phase === 'writer') onStageAdvance('Writer');
      } : undefined,
      onModelResolved: mt.wrappedOnModelResolved,
      onStreamChunk, onQueueWait,
    }),
  });

  // 4. Call LLM
  const response = await callLlm({ product, ... });

  // 5. Cooldown (1 line instead of 3)
  const { cooldownUntil, ranAt } = computeCooldownUntil();

  // 6. Persist: JSON first, then SQL projection
  // WHY: effort_level + access_mode come from resolveModelTracking.
  // They power the FinderRunModelBadge in run history.
  const merged = mergeMyFinderDiscovery({
    ..., run: { model: mt.actualModel, fallback_used: mt.actualFallbackUsed,
                effort_level: mt.actualEffortLevel,
                access_mode: mt.actualAccessMode, ... },
  });
  const store = specDb.getFinderStore('myFinder');
  store.insertRun({ ..., effort_level: mt.actualEffortLevel,
                         access_mode: mt.actualAccessMode });
  store.upsert({ ... });

  return { ... };
}
  7. Write route handler
    api/myFinderRoutes.js — thin config over the generic handler: 
    import { createFinderRouteHandler }
  from '../../../core/finder/finderRoutes.js';

export function registerMyFinderRoutes(ctx) {
  return createFinderRouteHandler({
    routePrefix: 'my-finder',
    moduleType: 'myf',
    phase: 'myFinder',
    fieldKeys: ['my_field'],

    runFinder: ctx.runMyFinder,
    deleteRun: ctx.deleteMyFinderRun,
    deleteAll: ctx.deleteMyFinderAll,

    getOne: (specDb, pid) =>
      specDb.getFinderStore('myFinder').get(pid),
    listByCategory: (specDb, cat) =>
      specDb.getFinderStore('myFinder').listByCategory(cat),
    listRuns: (specDb, pid) =>
      specDb.getFinderStore('myFinder').listRuns(pid),
    upsertSummary: (specDb, row) =>
      specDb.getFinderStore('myFinder').upsert(row),
    deleteOneSql: (specDb, pid) =>
      specDb.getFinderStore('myFinder').remove(pid),
    deleteRunSql: (specDb, pid, rn) =>
      specDb.getFinderStore('myFinder').removeRun(pid, rn),
    deleteAllRunsSql: (specDb, pid) =>
      specDb.getFinderStore('myFinder').removeAllRuns(pid),
  });
}
  8. Write route context factory
    api/myFinderRouteContext.js: 
    import { runMyFinder } from '../myFinder.js';
import { deleteMyFinderRun, deleteMyFinderAll }
  from '../myFinderStore.js';

export function createMyFinderRouteContext(options = {}) {
  const { jsonRes, readJsonBody, config, appDb,
    getSpecDb, broadcastWs, logger } = options;
  return {
    jsonRes, readJsonBody, config, appDb,
    getSpecDb, broadcastWs, logger,
    runMyFinder, deleteMyFinderRun, deleteMyFinderAll,
  };
}

4 Registry Entry Schema (Full Reference)
finderModuleRegistry.js Entry
{
  // ── Identity ──
  id: 'myFinder',                    // camelCase module ID (used as key everywhere)
  routePrefix: 'my-finder',          // kebab-case URL prefix
  moduleType: 'myf',                 // short code (operations tracker, data-change events)
  moduleLabel: 'MYF',                // chip label in UI
  chipStyle: 'sf-chip-warning',      // Tailwind chip class

  // ── DB Schema (auto-generated DDL) ──
  tableName: 'my_finder',
  runsTableName: 'my_finder_runs',
  summaryColumns: [
    { name: 'items', type: 'TEXT', default: "'[]'" },
    { name: 'item_count', type: 'INTEGER', default: '0' },
  ],
  summaryIndexes: [
    { name: 'idx_myf_cooldown', columns: ['cooldown_until'] },
  ],

  // ── Field Studio Integration ──
  fieldKeys: ['my_field'],           // candidate cleanup on delete-all
  requiredFields: ['my_field'],      // field studio gate (POST returns 403 if disabled)

  // ── LLM Phase ──
  phase: 'myFinder',                 // must match llmPhaseDefs entry

  // ── Feature Module Paths (for route auto-wiring) ──

  contextExport: 'createMyFinderRouteContext',

  filePrefix: 'my_feature',          // .workspace/products/{pid}/{filePrefix}.json

  // ── Per-Category Settings ──
  settingsDefaults: {
    mySettingA: '100',
    mySettingB: 'true',

  promptBuilderExport: 'buildMyFinderPrompt',

  panelFeaturePath: 'my-finder',     // tools/gui-react/src/features/{panelFeaturePath}/

  // ── Data-Change Events ──
  dataChangeEvents: {
    'run': [],                       // suffix → extra domains to invalidate
    'run-deleted': [],
    'deleted': [],

  // ── Codegen: Pipeline Settings UI ──

  settingsSubtitle: 'MYF module settings',

6 Data Flow (Every Finder Follows This)
Universal Finder Data Flow

LLM Call

→
SQL Write
frontend projection
→
WS Broadcast
data-change
JSON first
Durable memory written before SQL. Crash-safe: SQL can be rebuilt from JSON.
SQL projection
specDb.getFinderStore(id).insertRun() + .upsert(). Runs table includes effort_level and access_mode columns for badge rendering in run history.
WS invalidation
Data-change event broadcast → frontend invalidationResolver → React Query refetch
Operations
Registered before 202. Stage/model/stream callbacks update it in real-time. Complete/fail on finish.
Cooldown
30 days from successful run. Rejected runs: no cooldown set.
7 Common Patterns & Decisions
Accumulation Strategy
Latest-wins
Default. selected = most recent non-rejected run. Used by CEF (colors/editions overwrite).
Accumulate
Override recalculateSelected in createFinderJsonStore. Used by PIF (images union across runs).

Choose based on domain: does a new run replace the answer or add to it?

Candidate Validation

If your finder populates Field Studio fields (fieldKeys is non-empty), you should validate LLM output through the candidate gate before writing:

Gate
validateField() from publisher/index.js
Submit
submitCandidate() with sourceMeta: { source: moduleType }
Reject
Persist rejected run to JSON + SQL with status: 'rejected'

See CEF’s storeFailureAndReturn() for the full rejection pattern. PIF skips this because images are artifacts, not field candidates.
