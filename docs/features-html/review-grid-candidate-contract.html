Review Grid Candidate Contract

The single shape contract for everything that feeds the review grid. Any finder module that emits candidates — CEF (colors, editions), future Release Date Finder, SKU Finder, Discontinued Finder, Price Finder, Affiliate Links Finder — produces rows that conform to this contract. Anchors the shape before the next wave of variantFieldProducer modules ships.

1. The two surfaces

There are two shapes to understand, on opposite ends of the pipeline:

  • Storage shape — field_candidates SQL row. What gets written when submitCandidate runs.
  • Display shape — the object that ensureCandidateShape() returns. What the review grid UI actually renders.

Storage is the source of truth; display is derived. The review grid hydrates display candidates from storage rows.

2. Storage shape — the field_candidates row

Defined in src/db/specDbSchema.js:

CREATE TABLE field_candidates (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  category TEXT NOT NULL,
  product_id TEXT NOT NULL,
  field_key TEXT NOT NULL,
  value TEXT,
  unit TEXT DEFAULT NULL,
  confidence REAL DEFAULT 0,
  source_id TEXT NOT NULL DEFAULT '',
  source_type TEXT NOT NULL DEFAULT '',
  model TEXT DEFAULT '',
  validation_json TEXT DEFAULT '{}',
  metadata_json TEXT DEFAULT '{}',
  status TEXT DEFAULT 'candidate' CHECK(status IN ('candidate', 'resolved')),
  variant_id TEXT DEFAULT NULL,
  variant_id_key TEXT GENERATED ALWAYS AS (COALESCE(variant_id, '')) STORED,
  submitted_at TEXT DEFAULT (datetime('now')),
  updated_at TEXT DEFAULT (datetime('now')),
  UNIQUE(category, product_id, field_key, source_id, variant_id_key)
);
ColumnRole
field_keyWhich field this candidate is for (colors, editions, release_date, sku, etc.). Matches Field Studio rule keys.
valueThe candidate value. String, number, or stringified JSON depending on the field's type.
confidence0.0–1.0 score assigned by the finder.
source_idStable per-run identifier — see conventions below.
source_typeShort module tag (cef, rdf, pipeline, user). Used by the review grid for icon/color routing.
variant_idNULL = item-default (shown as the primary value in the grid). String = variant-scoped (drill-in / per-variant breakdown).
variant_id_keyGenerated column: COALESCE(variant_id, ''). SQLite treats NULL as distinct in UNIQUE constraints, so this is the real unique key. Do not write to this column directly — it's auto-computed.
metadata_jsonArbitrary per-source metadata. For item-default rows emitted by variantFieldProducer modules, should include per_variant: { variant_id: value, ... } so downstream can access the full variant breakdown without a second query.
Uniqueness: UNIQUE(category, product_id, field_key, source_id, variant_id_key). Two candidates with the same source_id but different variant_id values are distinct rows — that's how one CEF run can produce one candidate per variant.

3. Variant scoping semantics

Three cases:

variant_idMeaningExample
NULLItem-default. Applies to the whole product. Rendered as the primary value in the grid.A pipeline weight measurement, or the chosen "primary" release date a variantFieldProducer picked.
string (e.g. v_abc123)Variant-scoped. Applies only to that specific variant.The release date for the Black color variant, the SKU for the CoD BO6 edition variant.
string + NULL pairBoth exist simultaneously. Item-default carries metadata_json.per_variant with the full map; per-variant rows carry the variant-specific value.RDF emits one row per variant and one item-default row whose per_variant metadata holds every variant's release date.

Pulling everything for one variant — both variant-scoped rows and item-defaults — is the job of the getVariantDossier helper at src/db/helpers/variantDossierHelper.js. Any UI or publisher code that asks "what do we know about variant X?" should go through it.

4. source_id convention

Format: {modulePrefix}-{productId}-{runNumber}.

  • CEF: cef-mouse-001-3 — the 3rd CEF run on product mouse-001.
  • Future RDF: rdf-mouse-001-1.
  • Pipeline extractions: pipeline (no run number; pipeline is a continuous source).
  • User override in review: user.

The run-numbered variants matter because deleting run N of a finder cascades to DELETE FROM field_candidates WHERE source_id = '{prefix}-{productId}-{N}' — a clean, surgical deletion without touching other runs' candidates.

5. Display shape — ensureCandidateShape()

Defined in src/features/review/domain/candidateInfrastructure.js. The review grid calls this to normalize a raw candidate (from field_candidates or elsewhere) into the shape the UI renders:

{
  candidate_id:   string,            // stable identifier within a product/field scope
  value:          any,               // the value itself
  score:          number in [0,1],   // confidence
  source_id:      string,            // normalized source token (e.g. 'cef', 'pipeline', 'user')
  source:         string,            // human-readable label (e.g. 'CEF', 'Pipeline', 'Reference')
  tier:           string | null,     // optional tier ranking
  method:         string,            // e.g. 'pipeline_extraction', 'specdb_lookup'
  evidence: {
    url:           string,
    retrieved_at:  string,
    snippet_id:    string,
    snippet_hash:  string,
    quote:         string,
    quote_span:    [number, number] | null,
    snippet_text:  string,
    source_id:     string,
  },
  is_synthetic_selected: boolean,    // true when derived from the current slot state
}

This is a read-side contract. Writers don't need to produce this shape directly — they produce storage-shape rows via submitCandidate, and the hydration path into the grid applies ensureCandidateShape() automatically.

6. The write API — submitCandidate

Single entry point for every source: src/features/publisher/candidate-gate/submitCandidate.js.

submitCandidate({
  category, productId, fieldKey,
  value, confidence, sourceMeta,
  fieldRules, knownValues, componentDb,
  specDb, productRoot,
  variantId,        // nullable — null for item-default, string for variant-scoped
  metadata,         // for variantFieldProducer item-default rows: { per_variant: { ... } }
  // ... other opts
}) → { status: 'accepted' | 'rejected', candidateId, value, validationResult }

The gate validates via validateField() before any write. Hard rejections (shape, type, range, closed enum) block the write; soft rejections (unknown_enum_prefer_known) are accepted. A successful call dual-writes to field_candidates (SQL projection) and product.json candidates[] (durable SSOT).

7. Contract checklist for new finders

A new variantFieldProducer module (release date, SKU, etc.) is conformant if:

  1. Every per-variant candidate is submitted with a non-null variantId.
  2. Every item-default candidate is submitted with variantId: null and a metadata.per_variant map.
  3. Every call uses a source_id of the form {modulePrefix}-{productId}-{runNumber}.
  4. Values pass the Field Studio rule for the declared fieldKey (the gate will reject otherwise).
  5. Run deletion cascades via source_id prefix — the module doesn't need to emit its own delete logic.

If those hold, the module lights up in the review grid automatically — no grid-side code needed.
