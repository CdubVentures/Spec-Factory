1 Overview
What It Does

The Carousel Builder takes already-downloaded PIF images, groups them by variant + view, sends each group to a vision-capable LLM, and picks the best image per view + hero shots for the product carousel.

It is a sub-feature of PIF — not a separate finder module. It lives inside the PIF panel and writes eval results directly onto existing ProductImageEntry objects.

No loop, no orchestrator. The GUI fires one operation per view group + one for hero — all simultaneously. The LLM queue serializes the actual calls. Each operation shows individually in the tracker. Re-running clears old eval data and produces fresh results.

Key Design Decisions
LLM Phase
imageEvaluator — separate from imageFinder (discovery), own model config in LLM Config GUI
Persistence
No new DB tables or JSON files. Eval fields written directly onto selected.images[] in existing product_images.json
Re-runnable
Full re-evaluation every time. Clear all eval fields for the variant, then apply fresh results. LLM always sees the complete current candidate pool.
Vision Input
512×512 PNG thumbnails via sharp, base64-encoded as data:image/png;base64,... in image_url content parts
Skip Logic
0 candidates = skip. 1+ candidates = always send to LLM for quality evaluation (single candidates can be rejected).

Step-by-Step Trace
  1. GUI groups images by view: User clicks Eval on a variant row. The GUI reads the current images from React Query cache, groups them by view field, and determines which views exist for this variant (e.g., top, left, angle, front).
  2. GUI fires N+1 POSTs simultaneously: One POST /evaluate-view per view group + one POST /evaluate-hero. Each uses useFireAndForget — all N+1 optimistic stubs appear in the operations tracker instantly.
  3. Each route handler registers its own operation: type: 'pif', subType: 'evaluate'. View ops show progress text like "top view". Hero op shows "hero selection". Returns 202 Accepted immediately.
  4. LLM queue serializes: All N+1 operations hit the LLM queue. The queue processes them one at a time. Every candidate — even single-candidate views — goes through the vision LLM for quality evaluation.
  5. Per-view evaluation (runEvalView): Reads images for this variant+view from product_images.json. Creates 512px thumbnails. Calls vision LLM with view-specific prompt. LLM returns rankings with best: true on the winner. Flags: watermark, badge, cropped, wrong_product. Persists via mergeEvaluation (clear-then-write for this view).
  6. Hero selection (runEvalHero): Reads all hero-view candidates for this variant. Sends them to hero selection LLM. The LLM acts as a legal/quality gatekeeper — checking source safety (no copyrighted editorial photos), cleanliness (no watermarks), and image quality. LLM picks heroes with hero_rank or returns empty array if all fail. Persists via mergeEvaluation.
  7. Each operation completes independently: Data-change event product-image-finder-evaluate broadcast per completion. GUI refreshes as each finishes — badges appear progressively.
  8. GUI Refresh: WS event bridge receives data-change → invalidates React Query cache → gallery cards show BEST / H1 / WM badges as each view completes.

6 LLM Evaluation Criteria
View Evaluation (per-view prompt)

Each view gets its own LLM call with view-specific instructions. The prompt tells the LLM what the correct angle should look like (e.g., "Bird's-eye shot looking directly down" for top, "Strict side profile from the left" for left).

CriterionWhat LLM ChecksImpact
WatermarksGetty, Shutterstock, retailer logos, "SAMPLE" textDisqualify — flag: 'watermark'
Badges / overlaysSale stickers, "NEW" badges, retailer brandingDisqualify — flag: 'badge'
CroppingProduct cut off at edges, missing parts, tight framingPenalty — flag: 'cropped'
Wrong productDifferent model, wrong color, accessoryDisqualify — flag: 'wrong_product'
SharpnessBlur, compression artifacts, noiseAffects ranking
CompositionView angle match, product centered, clean bgAffects ranking
BG removal qualityHalo artifacts, missing parts, jagged edgesAffects ranking
Hero Selection — Legal / Quality Gatekeeper (not art director)

The hero eval LLM is a gatekeeper, not an art director. It does NOT judge composition, framing, or artistic merit. It checks three gates:

GateWhat LLM ChecksImpact
1. Source SafetyIs this a manufacturer promotional image (official product page, press kit, retailer CDN)? Or a copyrighted editorial photo from a review site (PC Gamer, Tom's Hardware, RTINGS, The Verge, etc.)?Editorial = disqualify. This is the primary gate — copyright risk.
2. CleanlinessWatermarks, badges, "SAMPLE" text, retailer branding baked into the imageDisqualify
3. QualityResolution, blur, heavy compression, screenshot-qualityDisqualify if unusable

Any image type is acceptable — cutouts on white, lifestyle desk shots, promotional renders, studio multi-color lineups, unboxing kit layouts, dramatic RGB scenes, themed collection shots. The image style does not matter. Only usability matters.

When picking multiple heroes, the LLM prefers different shots over near-duplicates of the same angle — but this is a tiebreaker for ranking, not a gate for acceptance.

How the LLM distinguishes sources: Editorial/review photos typically show lab environments, test equipment, desk clutter, hands holding the product, inconsistent overhead lighting. Manufacturer promo images have polished studio lighting, consistent brand aesthetics, clean backgrounds, or stylized scenes.

MethodPathBodyResponse
POST /product-image-finder/{category}/{productId}/evaluate-view { variant_key: string, view: string } 202 { ok: true, operationId }
POST /product-image-finder/{category}/{productId}/evaluate-hero { variant_key: string } 202 { ok: true, operationId }

Each endpoint handles ONE operation = ONE LLM call. The GUI fires N view calls + 1 hero call simultaneously per variant. The LLM queue serializes actual execution.
