Release Date Finder (RDF) Reference

Per-variant first-availability date discovery with evidence tiers, confidence gating, and the shared publisher pipeline

Module class: variantFieldProducer — RDF is the first of its class. It iterates CEF variants, runs the LLM per-variant, and routes each result through the shared publisher gate with a variant_id-scoped candidate row. The next variant-scoped field finder (e.g. discontinued, MSRP, weight, dimensions) follows this exact pattern and inherits the standardized budget loop for free — see Finder Module Guide § 9.
Two firing modes (both 202 fire-and-forget):
  • Run — single LLM call per variant, no retry. Header Run button fires one per variant; per-row Run fires one for that variant. Always clickable, spammable (mirrors PIF's View/Hero).
  • Loop — budget loop per variant (up to perVariantAttemptBudget calls, default 3) until the candidate reaches the publisher gate or LLM returns definitive unknown. Header Loop fires one loop per variant in parallel; per-row Loop locks while that variant's loop is in flight. All runs emitted by a single loop call share response.loop_id so the UI can group them.
1 Overview
Per-Variant Release Date Discovery

RDF discovers a first-availability release date per CEF variant (each color and each edition). Every candidate carries evidence (URL + excerpt + tier) and a confidence score. Candidates above threshold flow through the shared publisher gate and populate field_candidates scoped to the variant.

2 Why Variant-Scoped (not Product-Scoped)
Release Dates Differ Across Colorways and Editions

A product’s “default” release date is rarely the only one that matters:

  • Limited colorways drop months after launch.
  • Collab editions (SF6 Chun-Li, CoD BO6, Cyberpunk) ship on their own calendars.
  • Regional SKUs can announce first, then roll out globally.

RDF mirrors the PIF model: iterate CEF’s variant_registry and produce one candidate per variant_id. The scalar product-level release_date field rule still applies (same shape), but each candidate row carries a variant_id foreign key so the same field key can have N coexisting rows.

Reconciliation with the pipeline The main indexing pipeline publishes a product-scalar release_date into product.json.fields.release_date (no variant_id). RDF publishes variant-scoped values into product.json.variant_fields[vid].release_date. They coexist in field_candidates via the UNIQUE(category, product_id, field_key, source_id, variant_id_key) index and live in different containers in product.json, so neither stomps the other. Deleting a variant via deleteVariant() cascades that variant’s rows and drops its variant_fields[vid] entry automatically.

Publisher error path
If submitCandidate() throws, the error lands on candidate.publisher_error. The run itself still persists — the publisher is advisory, not a hard dependency.
Published SSOT
product.json.variant_fields[vid].release_date — written by publisher path ③ (variant-scoped). The feature’s own release_date_finder SQL table is a UI-speed cache, not authoritative. See Publisher Pipeline § 3b.
Loop mode satisfaction predicate

runReleaseDateFinderLoop retries each variant up to perVariantAttemptBudget attempts and stops early when either:

  • submitCandidate() was reached (any status other than 'skipped') — the candidate made it past the local confidence + evidence gate. Publisher-side rejection won’t improve with another LLM call on the same evidence.
  • LLM returned release_date = 'unk' with an unknown_reason — a definitive "no release date exists" answer. Retrying won’t conjure a date that doesn’t exist.

It retries on: LLM failure, below-confidence result, missing evidence, empty/transient response. Loop mode re-reads previousRuns fresh before each attempt so retries see the URLs/queries emitted by prior attempts (single-shot runReleaseDateFinder keeps a frozen snapshot — behavior preserved).

4 Variant Resolution
Backend: CEF Registry is SSOT

The orchestrator reads variants via specDb.variants.listByProduct(productId). It does not re-derive from published.colors. The registry is the one-row-per-variant truth.

// src/features/release-date/releaseDateFinder.js
const variants = specDb.variants.listByProduct(productId);
const targets = variantKey
  ? variants.filter((v) => v.variant_key === variantKey)
  : variants;
for (const v of targets) {
  await produceForVariant({ product, variant: v, ... });
}

Label precedence (via buildFinderVariantRows):

  • Edition → edition_display_name (e.g. “Call of Duty®: Black Ops 6 Edition”)

5 Evidence & Confidence
Every Date Must Cite a Source

The LLM is required to return at least one evidence_refs entry. Universal shape (shared with CEF) — owned by src/core/finder/evidencePromptFragment.js:

FieldTypeNotes
urlstringCanonical URL of the source page.
tierstringClassification only (no enforcement). 6 codes: tier1 mfr / press release, tier2 testing lab / review lab, tier3 authorized retailer / marketplace, tier4 community / forum, tier5 specs aggregator, other.
confidenceinteger 0–100Per-source — how sure the LLM is THIS URL supports the claim. Distinct from candidate-level confidence.
Publisher threshold (dual gate) Every publish path runs both gates:
  • Confidence: candidate confidence ≥ publishConfidenceThreshold
  • Evidence: distinct evidence URLs ≥ fieldRule.evidence.min_evidence_refs (default 1 for release_date) — checked via specDb.countFieldCandidateEvidenceByCandidateId against the relational field_candidate_evidence projection
Failure status: below_threshold or below_evidence_refs — persisted to metadata_json.publish_result. RDF dual-writes one source under variant + scalar; the variant-aware lookup (getFieldCandidateBySourceIdAndVariant) ensures each row's evidence projects against its own candidate_id.

7 Settings
Per-Module and Global Knobs
SettingDefaultControls
perVariantAttemptBudget3Max LLM calls per variant in Loop mode (1–5). 1 = single shot (Loop becomes equivalent to Run). Higher values retry on low confidence / missing evidence / LLM failure until the candidate reaches the publisher gate or returns definitive unknown. Does not apply to plain Run / Run All — those are always single-shot.
minConfidence70Confidence gate for publisher submission. Below → candidate persists locally only with below_confidence: true; Loop will retry.
discoveryPromptTemplate“” (uses default)Optional per-category prompt override. Hidden in the Module Settings panel (edited in LLM Config).
urlHistoryEnabledfalseInject prior run URLs into the prompt (variant-scoped). In Loop mode this refreshes per attempt so retries don’t re-crawl.
queryHistoryEnabledfalseInject prior search queries into the prompt (variant-scoped, per attempt in Loop mode).
urlCooldownDays global90Global URL re-check window. Shared with all finders. Pipeline Settings → Cooldowns.
queryCooldownDays global90Global query re-run window. Shared with all finders.

The cooldown knobs are global (not per-finder) because a URL fetched by RDF should not be fetched again by PIF the next day. Stored in runtime_settings and read at run time.

8 Panel (Indexing Lab)
Layout mirrors PIF

The RDF panel follows the same shape as PIF — the user should feel no shift between them:

  • Header: title + model chip + status chip + Run button + Loop button. ("All" is implied — buttons are in the header.) Run is always clickable; Loop disables when every variant is already looping.
  • KPI grid (4 cards): Release Dates • Variants • Runs • Published N/total. Order mirrors PIF’s Images • Variants • Runs • Carousel N/total.
  • How It Works collapsible explainer (content: rdfHowItWorksContent.ts).
  • All Release Dates section — two-column grid of FinderVariantRows. Each row: arrow, ColorSwatch, label, ED/CLR chip, date/unk chip, confidence %, Run button (always clickable — spammable), Loop button (disables while that variant’s loop is in flight). Expanded body lists evidence and publisher candidates.
  • Run History section — uses shared FinderRunHistoryRow (shared/ui/finder/). Header: arrow • #runNumber • date • FinderRunTimestamp • FinderRunModelBadge • variant swatch + label • date chip • confidence • evidence count • Del. Expanded body: FinderDiscoveryDetails (URLs checked / queries run / notes) + FinderRunPromptDetails (system / user / response). Identical shape to PIF and CEF.
  • Footer: last run, run count, model badge.
Run vs Loop button semantics (mirrors PIF)
  • Run fires single-shot — one LLM call per variant (or one for the per-row variant). Always enabled, safe to spam. Backend hits POST /release-date-finder/:cat/:pid.
  • Loop fires the budget loop — up to perVariantAttemptBudget calls per variant with early stop on publisher-reached or definitive unknown. Locks per-variant via useRunningVariantKeys('rdf', productId, 'loop'). Header Loop fires one loop per variant in parallel (not one multi-variant loop) so per-variant UI feedback stays independent. Backend hits POST /release-date-finder/:cat/:pid/loop with subType: 'loop' on the registered operation.
For the next variant-dependent finder Copy the RDF panel shape: header (Run + Loop) → KPI grid (same 4 card order) → HIW → “All <Thing>” grid (per-row Run + Loop) → Run History → footer. Swap only the module-specific chips, expanded body, KPI derivations, and HIW content. The shared scaffolding (FinderVariantRow, useFinderColorHexMap, buildFinderVariantRows, useFireAndForget, useRunningVariantKeys) stays identical. Backend: declare perVariantAttemptBudget in the registry entry, export a runXxxFinderLoop that wraps runVariantFieldLoop, and register one createVariantFieldLoopHandler call in the routes file.
