Deterministic Publish — Value-Bucket Pooling (shipped 2026-04-22)

The publish decision is fully deterministic: given a value bucket, its pooled qualifying evidence count, the publish threshold, and the field rule's min_evidence_refs, the outcome is computable — the LLM's self-declared confidence is a submission-time sanity check, not a decision maker.

  • Value buckets: candidate rows are grouped by value_fingerprint (scalar equality, or list set-equality — order-independent, case-insensitive, NFC-normalized). Multiple submissions agreeing on the same value pool their evidence into one bucket.
  • Per-ref qualifying: Gate 2 counts only refs where ref.confidence / 100 ≥ publishConfidenceThreshold. Evidence with NULL confidence counts as qualifying (legacy tolerance).
  • Bucket-level Gate 1: a bucket publishes only if its top member's LLM confidence ≥ threshold. Buckets composed entirely of low-confidence guesses never publish, regardless of evidence count.
  • set_union publish: for list fields with item_union='set_union', the published value is the union of all qualifying buckets' items. Below-threshold buckets are excluded from both the union and the resolve cascade.
  • evidence_kind is UI-only: the identity_only SQL exclusion is retired. The 10-kind enum remains as display metadata (tooltip, icon, supporting_evidence quote), but does not gate counting.
  • Gate 1 inconsistency purge: if the LLM self-declares confidence below threshold and the submitted candidate's bucket has enough qualifying evidence to publish, the row + its evidence are hard-deleted at submit time — treated as an unreliable-model signal. When the candidate came from a primary finder run (keyFinder primary_field_key === fieldKey, or any RDF/SKU per-variant run), the run's JSON entry + discovery_log.urls_checked + queries_run + SQL runs row are deleted too so a re-run starts with a clean URL/query slate. Passenger keyFinder candidates leave the run intact (its discovery_log belongs to its own primary field).

4 Dual-Write Architecture

Every publish operation updates both SQL and JSON. SQL is the query projection. JSON is the durable SSOT.

SQL: field_candidates table

  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  category     TEXT NOT NULL,
  product_id   TEXT NOT NULL,
  field_key    TEXT NOT NULL,
  source_id    TEXT NOT NULL,
  source_type  TEXT NOT NULL,
  value        TEXT,
  value_fingerprint TEXT DEFAULT '',    -- deterministic bucket key
  unit         TEXT DEFAULT NULL,
  confidence   REAL DEFAULT 0,
  model        TEXT DEFAULT '',

  metadata_json   TEXT DEFAULT '{}',

5 Threshold Reconciliation
reconcileThreshold() — Re-evaluate all candidates against a new confidence threshold
Input
specDb, category, threshold (0–1), productRoot, dryRun (boolean)
Output
{ unpublished, published, locked, unaffected, total_fields }
Confidence normalization
Mixed scales handled: c > 1 ? c / 100 : c
Stages
'Scanning' → 'Evaluating' → 'Complete'

Both tightening and loosening route through evaluateFieldBuckets() with the new threshold — identical semantics to publishCandidate(). The reconcile loop compares the evaluator's publishedValue to what's currently in product.json and takes one of three actions per field:

ActionWhenWhat Happens
Unpublish Field is currently published but no bucket qualifies at the new threshold (bucket top_confidence fell below threshold, or pooled qualifying refs fell below min_evidence_refs). Demote all resolved rows to 'candidate', delete from product.json fields[] (or variant_fields[vid][fieldKey]).
Publish Evaluator returns a publishedValue that differs from (or replaces absent) current published state. Demote any prior resolved rows, mark every publishedMemberIds row resolved, write new value + linked_candidates to product.json, persist publish_result in the row's metadata_json.
Manual override source === 'manual_override' Never touched — locked fields are always skipped.
Dry run
dryRun: true — preview counts without persisting changes. Exposed via GET /publisher/:category/reconcile.
Apply
POST /publisher/:category/reconcile — applies changes with operation tracking.
