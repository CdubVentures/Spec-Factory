Candidate Gate Steps
1
Identity Guard guard
productId + fieldKey required. Rejects if missing.
2
Field Rule Guard guard
Field rule must exist in fieldRules. Rejects if missing.
3
Validate via validateField() 11-step pipeline
Full deterministic validation. See validator-pipeline.html.
4
Soft vs Hard Rejection Filter always
Soft: unknown_enum_prefer_known passes through (accepted with flag).
Hard: all other rejections block acceptance.
5
Source Merge merge
If candidate exists for this (product, field, value): append source to sources_json array, take max confidence. If new: create entry with single source.
6
Dual-Write sql + json
SQL: specDb.upsertFieldCandidate() with validation_json, sources_json, metadata_json.
JSON: append to product.json candidates[fieldKey][].
7
Persist Discovered Enums conditional
If field has enum.policy === 'open_prefer_known': calls persistDiscoveredValue() to learn new vocab.
8
Auto-Publish conditional
If config provided: calls publishCandidate() with accepted value. Result stored in return object.

4 Dual-Write Architecture

Every publish operation updates both SQL and JSON. SQL is the query projection. JSON is the durable SSOT.

SQL: field_candidates table
CREATE TABLE field_candidates (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  category     TEXT NOT NULL,
  product_id   TEXT NOT NULL,
  field_key    TEXT NOT NULL,
  value        TEXT,
  unit         TEXT DEFAULT NULL,
  confidence   REAL DEFAULT 0,
  source_count INTEGER DEFAULT 1,
  sources_json TEXT DEFAULT '[]',
  validation_json TEXT DEFAULT '{}',
  metadata_json   TEXT DEFAULT '{}',
  status       TEXT DEFAULT 'candidate',
  submitted_at TEXT DEFAULT (datetime('now')),
  updated_at   TEXT DEFAULT (datetime('now')),
  UNIQUE(category, product_id, field_key, value)
);

CREATE INDEX idx_fc_product
  ON field_candidates(product_id);
CREATE INDEX idx_fc_field
  ON field_candidates(category, product_id, field_key);
status
'candidate' (pending) or 'resolved' (published)
sources_json
Array of { source, confidence, submitted_at, run_id, model, artifact, url }
validation_json
{ valid, repairs[], rejections[], llmRepair?: {} }
metadata_json
{ source, method, publish_result, reviewer, reason, evidence }

5 Threshold Reconciliation
reconcileThreshold() — Re-evaluate all candidates against a new confidence threshold
Input
specDb, category, threshold (0–1), productRoot, dryRun (boolean)
Output
{ unpublished, published, locked, unaffected, total_fields }
Confidence normalization
Mixed scales handled: c > 1 ? c / 100 : c
Stages
'Scanning' → 'Evaluating' → 'Complete'
ActionWhenWhat Happens
Tightening (threshold raised) Resolved candidate confidence < new threshold Unpublish: demote SQL to 'candidate', delete from product.json fields[]
Loosening (threshold lowered) No resolved candidate but best candidate ≥ new threshold Publish best: mark resolved in SQL, write to product.json fields[] with source='pipeline'
Manual override source === 'manual_override' Never touched — locked fields are always skipped
Dry run
dryRun: true — preview counts without persisting changes. Exposed via GET /publisher/:category/reconcile.
Apply
POST /publisher/:category/reconcile — applies changes with operation tracking.

10 Manual Override Lock Semantics
Auto-Published
source='pipeline'
→
User Overrides
source='manual_override'
→
Field Locked
Auto-publish blocked
Lock condition
product.json fields[fieldKey].source === 'manual_override'
Unlock
User must manually delete/change the override
Reconcile
Manual overrides are never touched by threshold reconciliation
candidate_override
Does NOT lock. Source is 'candidate_override', which auto-publish recognizes as non-locked.
