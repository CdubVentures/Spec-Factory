Data Ownership & Cascade Flows

How sources, variants, evidence, and published state relate to each other. Deletion always cascades downward, never upward. JSON is durable memory; SQL is the runtime read source.

1 Core Principles

Deletion cascades DOWN, never UP

A parent owns its children. Deleting a parent removes all children. A child never deletes or modifies its parent.

Source → Evidence   (cascade) Evidence ↛ Source   (never)

Variant is the SSOT for published colors/editions

Published state for variant-scoped fields is derived from the variants table, not from candidate consensus. Candidates are historical evidence of what was discovered — they don't drive publishing for variant fields.

Variants table → Published state Candidates ↛ Published state   (for variant fields)

Dual-State Architecture (CQRS)

JSON files are the durable memory (rebuild/seed/audit). SQL tables are the runtime projection (UI reads). Nothing running should read JSON for published state — only rebuilding and seeding reads JSON.

JSON (Durable Memory)

color_edition.json, product_images.json, product.json

Read by: rebuild, reseed, CEF prompt context

Never read by: UI, publisher endpoint, review grid, PIF runtime

SQL (Runtime Projection)

variants, field_candidates, color_edition_finder, product_image_finder

Read by: everything at runtime

Rebuilt from JSON on DB delete

2 Three Source Types + The Variant Entity

Source Types Hierarchy

Every piece of data in the system is produced by a source. There are three source types, each with different ownership rules for what happens when the source or its parent variant is deleted.

Variant (Entity) Active

The central entity. Owns published state for variant-scoped fields + all downstream feature data. Not a source itself — it's what sources discover.

  • Published colors / editions (product.json fields[])
  • PIF images, evaluations, carousel slots
  • Candidate values (strips contributed items on delete)
  • Feature Source data (price, SKU, release date via FK)
  • Never touches: CEF runs, pipeline runs, source artifacts

Variant Source (CEF) Active

Discovers variants themselves. Produces evidence candidates for colors/editions. Published state comes from variants table, not from candidates.

  • field_candidates rows (source_id = cef-{pid}-{run})
  • CEF JSON run entry + SQL run row
  • Re-derives published from variants after candidate strip
  • Never touches: variants table, PIF images
  • Never touches: other sources' candidates

Pipeline Source Future

Discovers scalar facts (weight, sensor, dimensions). No variant relationship. Candidates ARE the authority for published state.

  • field_candidates rows (source_id = pipeline-{pid}-{run})
  • Run artifacts (screenshots, traces, crawl data)
  • May unpublish fields if sole evidence removed
  • Never touches: other sources' candidates
  • Never touches: variants (different domain entirely)

Feature Source Future

Discovers variant-scoped data (price, SKU, release date). Has dual ownership — owned by both the source that produced it AND the variant it belongs to. Either parent's deletion cascades down.

  • field_candidates rows (source_id + variant_id)
  • May unpublish if sole evidence for that variant's field
  • Never touches: its source or its variant
Dual ownership: Delete the source → its candidates go (like pipeline). Delete the variant → ALL feature data for that variant goes (like variant cascade). Two parents, either can kill the child.

One-way rule: Deleting a candidate NEVER deletes or modifies its source (CEF run). Deleting a source ALWAYS deletes its candidates.

Delete CEF Run — Cascade

StepTargetAction
1 SQL run row + JSON run Delete from color_edition_finder_runs + filter from color_edition.json runs[]
2 field_candidates Delete rows where source_id = 'cef-{pid}-{runNumber}'
3 Published state derivePublishedFromVariants() — re-derives from variants table (SSOT, unchanged by run delete)
4 Summary bookkeeping Update latest_ran_at, run_count via targeted updateBookkeeping()
Variants survive run deletion. The variants table is independent of run history. Even if all runs are deleted, variants persist. They were discovered by runs but are not owned by them.

Delete Candidate (from Review Grid) — Self-Contained

StepTargetAction
1 field_candidates Delete the specific candidate row
2 product.json candidates[] Filter from JSON candidates array
3 Published state republishField() picks new winner from remaining candidates (for non-variant fields)
Self-contained. Candidate deletion never touches the source that produced it, never touches variants, and never triggers PIF cleanup. For variant-scoped fields (colors/editions), the published state comes from the variants table anyway.

Pipeline Source → Evidence → Published (candidate-driven fields)

Pipeline Runsource_id: pipeline-{pid}-{run}

Candidate: weight"58g" conf: 0.95
publishes →
Published: weightcandidate consensus
Pipeline Run
produces →
Candidate: dimensions"120x60x40mm"
publishes →
Published: dimensionscandidate consensus
Key difference from variant fields: Pipeline fields (weight, dimensions, sensor, etc.) have their published state determined by candidate consensus (highest confidence resolved candidate wins). Unlike variant-scoped fields, there is no SSOT table — candidates ARE the authority.

Delete Pipeline Source — May Unpublish (Future)

StepTargetAction
1 Run artifacts Delete screenshots, traces, crawl data for this run
2 field_candidates Delete rows where source_id = 'pipeline-{pid}-{run}'
3 Published state republishField() picks new winner from remaining candidates. If no candidates remain for a field, that field is unpublished (deleted from product.json fields[]).
4 Linked evidence If this was the sole source for a published value, that value disappears. Other sources' candidates for the same field are unaffected.
The tricky case: If pipeline run 1 discovered weight: 58g and pipeline run 2 discovered weight: 60g, deleting run 1 removes the 58g candidate. If 58g was the published winner, republishField() promotes 60g. If run 1 was the only source for weight, the field is unpublished entirely. This is correct — no evidence means no published value.

7 Feature Source Flow (Future — Dual Ownership)

Feature Source → Variant-Scoped Evidence with Two Parents

A feature source (e.g. a pricing discovery run) produces data that is scoped to a specific variant. The candidate has both a source_id (who discovered it) and a variant_id (which variant it belongs to). Either parent's deletion cascades to the child.

Pricing Runsource_id: pricing-{pid}-1
produces →
Candidate: price"$79.99" for variant black
publishes →
Published: priceper-variant published list
Variant: blackvariant_id: v_aabb1122
owns →
Candidate: pricevariant_id FK on row

Dual Ownership: Two Cascade Triggers

Delete Source (pricing run)
1. Delete candidates with source_id = pricing-{pid}-1
2. Republish from remaining candidates per variant
3. If sole evidence for a variant's price → unpublish that price
4. Variant itself survives (it has other data: images, colors, etc.)
Delete Variant (color:black)
1. DELETE FROM variant_prices WHERE variant_id = ?
2. DELETE FROM variant_skus WHERE variant_id = ?
3. ALL feature data for this variant wiped
4. Source survives (it may have data for other variants)
The child never deletes either parent. Sources and variants are independent entities. A pricing run doesn't own the variant, and a variant doesn't own the pricing run. They both own the intersection: the price data for that specific variant.

Feature Source Delete — Cascade (Future)

StepTargetAction
1 Source artifacts Delete run-specific artifacts (LLM traces, scraped pages)
2 field_candidates Delete rows where source_id = 'pricing-{pid}-{run}'
3 Per-variant published state For each affected variant: republish from remaining candidates. If no candidates remain for that variant's price → unpublish.
Same pattern as Pipeline Source but scoped per-variant. The source produced data for potentially multiple variants. Deleting the source removes its evidence for each. Variants that still have evidence from other sources keep their published values.
