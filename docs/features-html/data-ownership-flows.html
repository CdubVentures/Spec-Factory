Data Ownership & Cascade Flows

How sources, variants, evidence, and published state relate to each other. Deletion always cascades downward, never upward. JSON is durable memory; SQL is the runtime read source.

AUTHORITATIVE DOCUMENT — DO NOT DEVIATE

This document defines the canonical data ownership rules for Spec Factory. All implementations must conform to these flows exactly. Any agent, contributor, or refactor that changes cascade behavior must update this document first and get explicit human approval. Do not add shortcuts, skip steps, or defer cleanup. These rules are not aspirational — they are the contract.

0 Quick Reference — All Delete & Mutation Flows

Delete Operations

Delete Variant Active
Strips candidate values (combo string, CEF sources only) → CEF table & JSON → field_candidates table & JSON → strips published values → deletes all PIF data (images, run history, eval history, carousel slots)
Delete Variant Source (CEF run) Active
Deletes all candidates (evidence) in all fields for that run → CEF table & JSON → field_candidates table & JSON → deletes discovery data (URLs and queries embedded in the run)
Delete Pipeline/Feature Source Planned
Deletes all candidates in all fields for that run → field_candidates table & JSON → removes published value(s) if sole linked evidence → deletes discovery data (URLs and queries for that source)
Delete Candidate Active
Removes it from that key only → field_candidates table & JSON. No cascade to sources, variants, or PIF.
Delete PIF Source (hero/variant/lifestyle run) Active
All images and data associated with that run deleted (discovery URLs and queries go with the run)
Delete Eval Source Active
Clears image selections (eval flags, hero/hero_rank) → carousel auto-resolves from remaining evals
Delete Field Key Active
Deletes candidates only (DB and product.json). Name changes are ID-driven — no cascade implications.
Delete Product Active
Full cascade: finder runs + summaries (all modules) → variants → field_candidates → pipeline history (runs, crawl, telemetry, item links) → product folder (all JSON + images) → products row

Published Value Operations

Published — Clear All Planned
Unresolves (demotes) current values. Does NOT delete candidates or evidence. Use when uncertain — a review pass can re-resolve.
Published — Clear Individual Value Planned
Unresolves that specific value from the published array. Does NOT delete it. Evidence remains for re-review.

Component Operations

Component — Delete Planned
Deletes that component value everywhere published AND all linked published values (authoritative or with constraint) everywhere.
Component — Update Published/Linked Planned
Updates all AUTHORITATIVE linked values everywhere published. May FLAG other linked values. Evidence left alone — a review fixes shape in validate → publisher.

Enum Operations

Enum — Delete Value Planned
Deletes that enum value everywhere published.
Enum — Update Value Planned
Updates it everywhere AUTHORITATIVE. Evidence left alone — a review fixes shape in validate → publisher.

1 Core Principles

Deletion cascades DOWN, never UP

A parent owns its children. Deleting a parent removes all children. A child never deletes or modifies its parent.

Source → Evidence   (cascade) Evidence ↛ Source   (never)

Variant is the SSOT for published colors/editions

Published state for variant-scoped fields is derived from the variants table, not from candidate consensus. Candidates are historical evidence of what was discovered — they don't drive publishing for variant fields.

Variants table → Published state Candidates ↛ Published state   (for variant fields)

Combo colors are atomic units

Multi-atom color variants like white+silver are stored and published as combo strings, not split atoms. The combo string from variant_key (e.g. color:white+silver → "white+silver") is the canonical form used in candidates, published state, and deletion stripping. Individual atoms ("white", "silver") are only for palette validation.

Combo string = canonical   ("white+silver") Split atoms ↛ Published or stripped   ("white", "silver")

Variant deletion scopes to CEF sources only

When stripping candidate values during variant deletion, only source_type = 'cef' rows are modified. If a pipeline or feature source independently discovered the same color value, that evidence is unrelated to the variant entity and must not be touched.

Strip from source_type = 'cef' only Never strip from pipeline / feature sources

Edition combo atoms stay scoped to their edition

Editions have multi-atom colorways (e.g., dark-gray+black+orange for a special edition). The individual atoms describe the edition's color composition — they are not standalone published colors. Published colors come only from variant_type = 'color' variants.

color variants → Published colors edition color_atoms ↛ Published colors

Dual-State Architecture (CQRS)

JSON files are the durable memory (rebuild/seed/audit). SQL tables are the runtime projection (UI reads). Nothing running should read JSON for published state — only rebuilding and seeding reads JSON.

JSON (Durable Memory)

color_edition.json, product_images.json, product.json

Read by: rebuild, reseed, CEF prompt context

Never read by: UI, publisher endpoint, review grid, PIF runtime

SQL (Runtime Projection)

variants, field_candidates, color_edition_finder, product_image_finder

Read by: everything at runtime

Rebuilt from JSON on DB delete

2 Source Types + Variant Entity

Variant (Entity) Active

The central entity. Owns published state for variant-scoped fields + all downstream feature data.

  • Published colors / editions (product.json fields[])
  • PIF images, evaluations, carousel slots
  • Candidate values (strips combo string from CEF sources only)
  • Feature Source data (price, SKU, release date via variant_id FK)
  • Never touches: CEF runs, pipeline runs, source artifacts

Variant Source (CEF) Active

Discovers variants themselves. Evidence candidates for colors/editions. Published from variants table, not candidates.

  • field_candidates rows (source_id = cef-{pid}-{run})
  • CEF JSON run entry + SQL run row (discovery log goes with it)
  • Re-derives published from variants after delete
  • Never touches: variants table, PIF images

Pipeline Source Planned

Discovers scalar facts (weight, sensor, dimensions). No variant relationship. Candidates ARE the authority.

  • field_candidates rows (source_id = pipeline-{pid}-{run})
  • Run artifacts (screenshots, traces, crawl data)
  • May unpublish fields if sole evidence removed
  • Never touches: variants (different domain)

Feature Source Planned

Discovers variant-scoped data (price, SKU, release date, discontinued). Dual ownership — owned by source AND variant.

  • field_candidates rows (source_id + variant_id)
  • May unpublish if sole evidence for that variant's field
  • Never touches: its source or its variant
Dual ownership: Delete the source → its candidates go. Delete the variant → ALL feature data for that variant goes. Two parents, either can kill the child.

3 Delete Variant — Full Cascade

deleteVariant(productId, variantId)

StepTargetActionFile

2 field_candidates (SQL + JSON) Strip variant's combo string (from variant_key) from candidate arrays where source_type = 'cef' only. Delete candidate row if array becomes empty. Non-CEF sources untouched. variantLifecycle.js
3 color_edition.json Filter variant_registry[] + rewrite selected.colors as combo strings (not split atoms), selected.editions, selected.default_color variantLifecycle.js

Combo string rule: A variant color:white+silver stores color_atoms: ["white","silver"] but the canonical published & candidate value is the combo string "white+silver" from variant_key. Stripping uses the combo string, never individual atoms.

4 Delete Variant Source (CEF Run)

Delete CEF Run — Cascade

StepTargetAction
1 SQL run row + JSON run Delete from color_edition_finder_runs + filter from color_edition.json runs[]. Discovery log (URLs checked, queries run) is embedded in the run — deleted with it.
2 field_candidates (SQL + JSON) Delete rows where source_id = 'cef-{pid}-{runNumber}'. Rewrite product.json candidates[].
3 Published state derivePublishedFromVariants() — re-derives from variants table (unchanged by run delete)
4 Summary bookkeeping Update latest_ran_at, run_count
Variants survive run deletion. The variants table is independent of run history. Even if all runs are deleted, variants persist. They were discovered by runs but are not owned by them.

5 Delete Pipeline/Feature Source

Delete Pipeline or Feature Source — May Unpublish Planned

StepTargetAction
1 Run artifacts Delete screenshots, traces, crawl data, discovery URLs and queries for this source
2 field_candidates (SQL + JSON) Delete rows where source_id = '{type}-{pid}-{run}'
3 Published state republishField() picks new winner from remaining candidates. If sole linked evidence for a published value → that value is unpublished.
Key difference from CEF: Pipeline/Feature candidates ARE the authority for publishing. Removing the sole source means the published value disappears. For CEF, the variants table is the authority and survives run deletion.

6 Delete Candidate

Delete Candidate (from Review Grid) — Self-Contained

StepTargetAction
1 field_candidates (SQL) Delete the specific candidate row by source_id
2 product.json candidates[] Filter from JSON candidates array. Delete key if empty.
Self-contained. Candidate deletion removes from that key only. Touches field_candidates table & JSON. Never touches the source that produced it, never touches variants, never triggers PIF cleanup. Published fields are NOT modified (comment in code: "Fields[] untouched").

7 Delete PIF Source & Eval Source

Delete PIF Source (hero / variant / lifestyle run)

StepTargetAction
1 PIF run entry (JSON + SQL) Remove entire run from product_images.json runs[] and product_image_finder_runs. Discovery log (URLs checked, queries run) is embedded in the run — deleted with it.
2 Images Strip run's images from selected.images. Delete orphaned image files from disk (master + original).
3 SQL projection Update product_image_finder summary: images, image_count, bookkeeping.
Single image delete vs. run delete: Deleting one image strips it from the run's arrays but the run itself survives (discovery log remains — by design, since you're not deleting the source). Deleting the entire run removes everything including its discovery log.

Delete Eval Source

StepTargetAction
1 Eval record Remove from evaluations[] array
2 Image eval fields Clear eval_best, eval_flags, eval_reasoning, eval_source from matching images. Clear hero, hero_rank if hero eval.
3 Carousel slots Auto-resolve from remaining eval winners via resolveCarouselSlots(). User overrides preserved.

8 Published Value Operations

Published — Clear All Planned

Unresolves (demotes) all currently published values for a field. Does NOT delete candidates or evidence. Use when uncertain about published state — values return to candidate status for re-review. A review pass in validate → publisher can re-resolve them.

No delete. Unresolve only. Evidence is preserved. The published value becomes a candidate again.

Published — Clear Individual Array Value Planned

Unresolves a specific value from a published array field. The value returns to candidate status. Other published values in the array are unaffected. Evidence remains intact for re-review.

9 Field Key Operations

Field Key — Name Change Active

Field key identity is ID-driven. The field_key slug (e.g. flawless_sensor) is the stable identifier used across all tables (field_candidates, enum_lists, item_component_links). Display names are derived at render time via humanizeField() — changing the display label has no cascade implications. No data moves, no candidates are affected.

Field Key — Delete Active

StepTargetAction
1 field_candidates (SQL) Delete all candidate rows for this field_key
2 product.json candidates[] Delete the candidates[fieldKey] entry
Candidates only. Deleting a field key removes its candidates from both DB and JSON. Published values, sources, and other field keys are not affected.

10 Component Operations

Component — Delete Planned

Deleting a component deletes that component value everywhere it is published AND all linked published values (authoritative or with constraint) everywhere.

StepTargetAction
1 Component identity store Remove the component entry
2 Published values (all products) Delete this component value from every product where it appears as published
3 Linked published values Delete all linked published values (authoritative or with constraint) that reference this component

Component — Update Published / Linked Values Planned

Updating a published component value or its linked values cascades to all AUTHORITATIVE linked values everywhere published. May FLAG other linked values that are not authoritative. Evidence is left alone — a review pass fixes shape in validate → publisher.

StepTargetAction
1 Component identity store Update the component's canonical value
2 Authoritative linked values Update everywhere this component is published as authoritative
3 Non-authoritative linked values FLAG for review (do not auto-update — evidence may differ)
4 Evidence / candidates Left alone. A review pass in validate → publisher reconciles shape.

11 Enum Operations

Enum — Delete Value Planned

Deleting an enum value removes it everywhere it is published. Evidence / candidates are not automatically deleted (they may still be valid references for audit).

Enum — Update Value Planned

Updating an enum value updates it everywhere it is AUTHORITATIVE (where the system published it as the canonical form). Evidence is left alone — a review pass in validate → publisher fixes the shape. Non-authoritative occurrences may be flagged for review but are not auto-updated.

12 variant_id on field_candidates (Cascade Anchor)

Why variant_id exists on field_candidates

The field_candidates table has a variant_id TEXT DEFAULT NULL column. This is the cascade anchor for future feature sources. It enables identity-based deletion instead of fragile value-matching when a variant is deleted.

Source Typevariant_id UsageOn Variant Delete
cef NULL — CEF runs discover many variants per run, candidates are run-scoped Value-matching: strip combo string from arrays where source_type = 'cef'
pipeline NULL — pipeline discovers scalar facts, no variant relationship Not affected by variant deletion
feature:{module} variant_id = 'v_abc' — run targets a specific variant DELETE WHERE variant_id = ? — clean identity-based cascade
UNIQUE constraint: UNIQUE(category, product_id, field_key, source_id, variant_id). This allows the same source to write one candidate per variant per field. SQLite treats NULL as distinct, so CEF rows (variant_id = NULL) still deduplicate by source_id alone.
