Color & Edition Finder (CEF) Reference

Discovers standard colorways, marketing names, and special editions via LLM web search with two-gate validation

1 Overview
What CEF Does

The Color & Edition Finder discovers every color variant and special edition for a given product. It uses an LLM-driven web search to find all standard colorways, multi-color combos, marketing color names, and named special/limited/collaboration editions.

After LLM discovery, results pass through TWO validation gates. Gate 1 checks every color atom exists in the registered palette (tainted batch: one bad atom rejects the whole run). Gate 2 validates the identity check output (no duplicate matches, no slug changes, valid atoms). Only after both gates pass are results published.

Gate: CEF only runs when Field Studio has colors and/or editions enabled for the category. If neither field is enabled, the module is skipped entirely.

3 Color Rules
Required
  • 1 Normalize every color to a registered palette atom (e.g. manufacturer's "Midnight" → atom black).
  • 2 Multi-color combos: join atoms with + in dominant-to-accent order (e.g. "dark-gray+black+orange").
  • 3 Marketing names are required in color_names when the atom does not match the manufacturer's real name.
  • 4 Every entry in colors must be unique — no duplicate atoms or combos.
  • 5 default_color must exactly equal colors[0].
Excluded
  • 1 Edition-exclusive individual atoms that are not available as a standard SKU must not appear as standalone entries in the master colors array.
  • 2 Do not invent atoms. Only use colors from the registered palette.
  • 3 Do not include marketing name as the atom. The atom is always the normalized palette key.
4 Edition Rules
What IS an Edition
  • 1 Named special, limited, or collaboration versions of the product (e.g. "Faker Edition", "Neon Rider").
  • 2 Slug must be kebab-case (e.g. faker-edition).
  • 3 Colors expressed as a single "+"-joined combo entry (e.g. "dark-gray+black+orange").
  • 4 The edition's combo must also appear in the master colors array (enforced by mergeEditionColorsInto).
What is NOT an Edition
  • 1 Plain color variants — a product in "white" is not an edition.
  • 2 Bundles or kits (product + accessories packaged together).
  • 3 Refurbished or renewed units.
  • 4 Aftermarket or third-party modifications.
5 Variant Registry & Two-Gate Validation
Variant Registry — Stable Variant Identity

Each variant (color or edition) gets a permanent hash assigned on first CEF publish. The variant_registry is stored at the top level of color_edition.json (not inside selected).

Hash format
v_<8-hex> from generateVariantId(productId, variantKey) — SHA-256 of "{productId}:{variantKey}", first 8 hex chars, prefixed v_
Hash NEVER changes
Once assigned, the variant_id is permanent. It survives atom updates, label changes, and any mutable field modifications.
Deterministic
Same product + same key always produces the same hash. Product-scoped (no cross-product collisions).
Registry location
variant_registry[] array on color_edition.json (top-level). Projected to SQL color_edition_finder.variant_registry column.
FieldMutabilityDescription
variant_idImmutablePermanent v_<8-hex> hash. Never changes after creation.
variant_typeImmutable"color" or "edition". Set at creation.
created_atImmutableISO timestamp of first creation.
edition_slugImmutableEdition slug. Never changes (slug change = Gate 2 rejection).
variant_keyMutableCurrent key (atom or slug). Can be updated if validated.
variant_labelMutableDisplay label. Can be updated.
color_atomsMutableArray of color atoms. Can be updated if all atoms pass palette check.
Removed variants
Wrong-product contamination is hard-deleted from the registry + PIF images cascade-deleted. Only for variants that were NEVER real for this product. Discontinued real products stay forever.
Builder
buildVariantRegistry({ productId, colors, colorNames, editions }) — creates fresh registry on Run 1.
Backfill
backfillVariantRegistry({ specDb, productRoot }) — one-time scan for products missing a registry. Idempotent.

Gate 2: Identity Judge Validation (validateIdentityMappings)

Runs AFTER LLM 2 (identity judge), BEFORE any writes. Validates the judge's output. Any failure rejects the entire run. The judge uses web access to verify discoveries against official sources before accepting.

RuleCheckRejection Reason Code
match must have non-null match field A match action must reference an existing variant_id identity_check_invalid
new/reject must have null match field New/rejected entries must NOT reference a variant identity_check_invalid
No duplicate match targets No two match actions may reference the same variant_id duplicate_match_target
Slug change blocked Edition slugs must NOT change (e.g. cod-bo6 → cod-bo6-edition is blocked) slug_change_blocked
Palette check on mappings Any color atoms in new/updated entries must exist in palette unknown_atom_in_mapping
Failure behavior
ANY rule failure → ENTIRE RUN REJECTED. No partial writes.
Applies to
Run 2+ only (Run 1 has no identity check, so Gate 2 does not fire).
Identity Judge Actions (LLM 2 Output)
ActionMeaningmatch fieldNew FieldsExample
match Same variant — confirm identity. Judge compares labels and picks the more accurate/official name. Can update mutable fields (atoms, label) if different and valid. Existing variant_id (non-null) verified: true/false
preferred_label: better name (optional)		 ocean-blue → deep-ocean-blue with preferred_label: "Deep Ocean Blue"
new Genuinely new variant — web-verified before accepting. Create entry with new hash. null verified: true (confirmed on official site) New color crimson-red confirmed on manufacturer site + Best Buy
reject Hallucinated/unverifiable discovery — not found on official sources. Skip entirely. null verified: true (searched but not found) Color not found on official product page or any major retailer
Web verification
Judge uses scoped web queries ("{brand} {model} [color name]") against manufacturer site + 1-2 retailers. Not endless searching.
Ambiguity context
Judge receives familyModelCount + ambiguityLevel + explicit sibling model names (from DB) to detect colors belonging to sibling models.
Trust anchor
Existing registry variants are treated as "confirmed unless proven wrong." Burden of proof is on new discoveries.
Product Identity & Ambiguity Scaling
  • 1 base_model + variant: CEF uses both fields together to identify the exact product. A "DeathAdder V3" (wired) and "DeathAdder V3 Pro" (wireless) are separate products with separate color inventories.
  • 2 Sibling exclusion: Sibling model names are resolved from the DB and injected as “This product is NOT: X, Y, Z”. LLM-discovered siblings are also logged in siblings_excluded.
  • 3 Family count / ambiguity: CEF reads familyModelCount, ambiguityLevel, and siblingModels from resolveIdentityAmbiguitySnapshot. High sibling counts trigger stronger IDENTITY WARNING blocks in the prompt, and explicit model names enable precise exclusion.
AmbiguityFamily SizePrompt Behavior
Easy1 modelStandard identity matching (no siblings)
Medium2–5 modelsCAUTION block + “This product is NOT: X, Y” sibling list
Hard+6+ modelsHIGH AMBIGUITY block + explicit sibling list: triple-check, skip if unconfirmed

7 Run Lifecycle & Multi-Run Learning
Cumulative Knowledge Across Runs

Each CEF run builds on prior runs. The buildKnownInputs(previousRuns) function accumulates discovery from prior non-rejected runs and feeds it into the next prompt as “previous findings.”

knownColors
Union of all colors from prior non-rejected runs (growth pattern: once found, stays visible)
knownColorNames
Latest name per atom across all runs
knownEditions
Union of all edition slugs across all runs

URLs and queries are NOT fed forward — the LLM decides its own search strategy fresh each run. Only colors, names, and editions are injected so the LLM knows what to re-verify and expand beyond.

Selected State (Latest-Wins)
Rule
selected always reflects the latest non-rejected run's output (latest-wins semantics)
Recalculation
On run delete, recalculateCumulativeFromRuns() recomputes selected from remaining runs
Empty state
If all runs deleted, selected reverts to { colors: [], editions: {}, default_color: '' }
Run number
Monotonic high-water mark — never reused even after deletes
Cooldown
30 days from most recent successful run
