Data Integrity Caveats

known gaps

Known edge cases as of 2026-04-03. Most JSON files have verified write-back. These are the remaining weak spots.

IssueImpactBlocker For
field_key_order and field_studio_map not in hash signature. Both have JSON write-back and SQL fallback, but editing JSON directly won’t trigger the category reconcile (buildFieldRulesSignature doesn’t include them). Only works on first seed or when SQL is empty. Direct JSON edits to these two files require deleting spec.sqlite or forcing a full reconcile to propagate. JSON-edit propagation
key_review_state AI confirmation columns have no JSON backing. AI review progress (ai_confirm_primary_status, user_accept_*) is runtime-only, not captured in overrides.json. On rebuild, key_review_state structure recreates but AI confirmation progress resets. Must re-run AI confirmation. Lossless AI review rebuild
Checkpoint persistence is best-effort. product.json and run.json are written during pipeline runs (pipelineCommands.js), but crashes or aborts may leave incomplete checkpoints. Runs or products from incomplete pipeline executions may not be recoverable. Full run history rebuild
5 tables have no rebuild path. billing_entries, field_history, bridge_events, source_corpus, source_intel_domains are SQL-only. Documented in the “No Rebuild Source” section above. Data in these tables is lost on full rebuild. Accepted — operational data that repopulates during use. N/A (accepted loss)
Async fire-and-forget write-backs. llm_route_matrix.json, field_key_order.json, and field_studio_map.json use fs.writeFile(...).catch(() => {}). Silent failure on write error (disk full, permissions). If write-back silently fails, JSON drifts from SQL. Next rebuild uses stale JSON. Write-back reliability
