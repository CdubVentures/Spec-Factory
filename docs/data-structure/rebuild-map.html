  .workspace/products/{pid}/

     color_edition.json ↔ round-trip runtime writes JSON and boot reseed restores color_edition_finder
Bottom line: a deleted-DB rebuild is stronger than existing-DB JSON propagation. The main losses today are SQL-first component review edits, SQL-first enum/list review edits, and key-review history tables.

Data Integrity Caveats

Confirmed gaps from the source audit and spot validations.

GapCurrent Result
Component review mutations are SQL-first runtime edits.Deleting spec.sqlite can lose reviewed component aliases, review status, links, and user property overrides unless they were separately mirrored into _overrides/components/*.json.
Enum/list review mutations are SQL-first runtime edits.Accepted enum promotions and manual enum additions can be lost unless they were separately encoded in durable JSON such as field_studio_map.json.
key_review_runs, key_review_run_sources, and key_review_audit are SQL-only.key_review_state can be reconstructed, but run/source/audit history cannot.
latest/normalized.json and latest/provenance.json do not exist in production.No validation stage yet. The seed path reads them for test harnesses only. Not a rebuild gap — no production data to lose.
product.json.sources seeds crawl_sources as baseline only.Closed. url_crawl_ledger is not seeded from product.json (additive counters). run.json remains authoritative for overlapping rows. Accepted limitation.
Direct field_studio_map.json edits can require compile.The stored map reseeds SQL, but generated artifacts may be stale until compile runs.
Existing-DB reconcile for brand_renames and component overrides.Closed. brand_renames now uses clean-slate-per-brand DELETE before INSERT (prevents stale rows and duplicates). Component overrides now have a reconcileComponentOverrideRows before hook that prunes stale override values, resets review_status, and cleans user aliases.
