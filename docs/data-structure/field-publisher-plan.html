Field Publisher — Implementation Plan

A universal, vertical O(1) module that accepts field values from any source, normalizes via Field Studio rules, resolves by confidence, and publishes to product.json + item_field_state. This is the bridge between extraction sources and the product record.

Governing Principles

CQRS Dual-State

JSON = durable memory. product.json owns fields + candidates. SQLite = UI projection. item_field_state + field_candidates are rebuildable read caches. Frontend never reads JSON.

O(1) Source Scaling

Adding a new extraction source = one call to publishFieldValues(). No new tables, no new columns, no new code in the publisher. The source just submits contributions.

O(1) Field Scaling

Adding a new field = one entry in field_studio_map.json → compile → publisher picks it up automatically. No hardcoded field logic anywhere.

Field Studio is the Brain

The publisher has zero field-specific knowledge. All normalization rules, aliases, enums, types, and component bindings come from field_rules.json at runtime.

Source = Where Data Came From

For website extraction: the source is the URL/artifact (content_hash). For LLM-generated: the source is the model. For manual: the source is the user. Never the extraction tool.

Highest Confidence Wins

All candidates stored. Agreeing sources cluster by normalized value. The group with the highest-confidence source above the publish threshold gets published. Full audit trail preserved.

End-to-End Data Flow
Any source (color finder, crawl extraction, manual override, future sources...) | v publishFieldValues() ← universal entry point | v fieldNormalizer ← reads field_rules.json per key | aliases → canonical form | enums → known_values match | numbers → parse + round + unit | components → component_db lookup | arrays → normalize each element | v Group by normalized value ← cluster agreeing sources | v fieldResolver ← pick winner ≥ PUBLISH_THRESHOLD | v Write product.json ← fields (published) + candidates (all) | v Project to SQL ← item_field_state (winners) + field_candidates (all) | v Frontend reads SQL only

Phase 1 Field Studio Wiring & Normalizer

Build the normalizer that reads field_rules.json and collapses raw values to canonical forms. This is the foundation — without correct normalization, the publisher creates garbage candidate groups. Field Studio must be the single source of normalization truth.

Depends on: field_rules.json (exists) known_values.json (exists) component_db/ (exists)
Why This Is First

Every subsequent phase depends on normalization being correct. If "Cherry MX Red", "MX Red", and "cherry mx red" aren't collapsed into one canonical value, the candidate grouping is broken, the resolution is broken, and the published fields are wrong. Field Studio already has all the rules — this phase wires them into a callable pure function.

Normalizer Contract

fieldNormalizer — Pure Function Contract

  • Input: { fieldKey, rawValue, fieldRules, knownValues?, componentDb? }
  • Output: { value: <canonical>, normalized: boolean, matchType: 'exact'|'alias'|'coerced'|'passthrough' }
  • No side effects. No DB calls, no disk reads. All lookup data passed in.
  • Unknown fields: If fieldKey not in fieldRules → return raw value with normalized: false
  • Unknown values: If value doesn't match any alias/enum → return raw value with matchType: 'passthrough'
  • Never throws. Bad input returns passthrough, never crashes the publisher.

Resolver Logic

Resolution Rules

  • Publish threshold: PUBLISH_CONFIDENCE_THRESHOLD = 70 (hardcoded now, setting later)
  • Winner: Candidate group whose top source has the highest confidence ≥ threshold
  • Tie-break: More corroborating sources wins. If still tied, most recent submitted_at wins.
  • Below threshold: Field not published to fields. Candidates still stored.
  • sources array: Published field carries ALL agreeing sources, ordered confidence descending
Deterministic Proof Gate — Accept or Reject

The publisher has zero tolerance for unverifiable data. It does not fix, retry, or call LLMs. It receives data, checks it against field_rules deterministically, and either accepts or rejects. Validation and fix loops are the source's responsibility — sources must clean their own data before calling the publisher.

Publisher Proof Flow
Source calls publishFieldValues(contributions) | v For each field_key in contributions: | v fieldNormalizer(fieldKey, rawValue, fieldRules) | Look up field_rules.fields[fieldKey] | Check contract.type, known_values, aliases, component_db | | enum? → value in known_values or aliases? | number? → parseable? in range? roundable? | array? → EVERY element provable? | component? → exists in component_db? | string? → alias match or exact? | v matchType: 'exact' | 'alias' | 'coerced' → ACCEPT — normalize & store matchType: 'passthrough' → REJECT — entire contribution refused No LLM calls. No retries. No fix loops. If the source sends bad data, it gets rejected. Period.
Concrete Example: Color Edition Finder

The color edition finder validates its own LLM output before calling the publisher. The finder owns the LLM call, the retry logic, and the fix loop. By the time data reaches the publisher, it must be provably correct.

Color Edition Finder — Source-Side Validation (BEFORE publisher) Step 1: LLM returns response { colors: ["black", "white", "chartreuse", "red"], editions: { "super-deluxe-rgb-edition": { colors: ["black"] } }, default_color: "black" } Step 2: Finder validates against known_values.color (source-side) "black" → registered color ✓ "white" → registered color ✓ "chartreuse" → NOT registered ✗ "red" → registered color ✓ Dirty → finder re-calls its own LLM with rejection context (reuses bound callLlm from createColorEditionFinderCallLlm) Step 3: LLM fix response (source handles this, not publisher) { colors: ["black", "white", "red"], editions: {}, default_color: "black" } All clean → finder now calls publisher with proven data ——— boundary: source → publisher ——— Step 4: Publisher receives proven contribution publishFieldValues({ contributions: { color: { value: ["black", "white", "red"], confidence: 100 }, default_color: { value: "black", confidence: 100 }, editions: { value: {}, confidence: 100 } } }) Step 5: Publisher normalizer re-verifies (deterministic, no LLM) "black" → known_values ✓ "white" → ✓ "red" → ✓ ALL PROVEN → accept, group, resolve, publish
What If Source Sends Bad Data Anyway?
Scenario: Source skips validation, sends "chartreuse" to publisher Publisher normalizer checks "chartreuse" against field_rules: NOT in known_values.color NOT an alias of any known color matchType: 'passthrough' → UNVERIFIABLE → REJECT entire contribution. Nothing stored. Nothing published. → Return { accepted: false, rejected: [{ fieldKey: "color", reason: "unverifiable value: chartreuse" }] } Source gets a clear rejection with reasons. Source is responsible for fixing and resubmitting. Publisher never guesses, never retries, never calls LLMs.

Publisher Proof Contract

  • Deterministic only. The publisher makes zero LLM calls, zero network requests. All verification is local field_rules lookups.
  • Prove or reject. Every value must resolve to exact, alias, or coerced. Any passthrough = reject entire contribution.
  • One bad value = all bad. If color array has 9 valid + 1 invalid, ALL 10 are rejected. Don't cherry-pick.
  • Source owns validation. LLM fix loops, retries, and re-evaluation are the source's job. The publisher is a dumb gate.
  • Clear rejection response. On reject, publisher returns which fields failed and why, so the source can fix and resubmit.
  • Double-check on accept. Even if the source pre-validated, the publisher re-verifies. Defense in depth, zero trust.
