Storage Architecture Overview

All persistent and ephemeral data flows through four top-level storage areas. Git-ignored directories hold runtime state; committed directories hold category configuration and fixture seeds.

Runtime Data Directory

Git-Ignored

.workspace/ is the primary runtime data directory. All SQLite databases, crawl output, product checkpoints, and ephemeral scratch data live here. It is fully git-ignored. Additional runtime paths: .server-state/ (server PID + logs, managed by dev-stack/process-manager tooling) and os.tmpdir()/spec-factory-crawl-videos/ (ephemeral video cache during crawl runs).

Category Authority Structure

Committed

The category_authority/ directory holds all category-specific configuration, compiled metadata, learning data, and overrides. All subdirectories are committed to git. The _runtime/ subdirectory is a legacy migration source (NOT the active write path and NOT git-ignored). Active settings write path: .workspace/global/user-settings.json; active snapshots path: .workspace/runtime/snapshots/.

Database File Details

SQL

Two SQLite database surfaces serve distinct purposes (SpecDb + AppDb). All databases use WAL mode for concurrent reads and are managed by dedicated store modules.

app.sqlite

.workspace/db/app.sqlite
Tables: 5
Size: 50 KB – 2 MB
Scope: Global (one per installation)
Purpose: Cross-category state — brands, user settings, global studio_maps table, global preferences, and app-level metadata. Note: per-category field_studio_map data is primary in spec.sqlite; app.sqlite studio_maps is a separate global table. JSON fallback only.
Seeded from: category_authority/_global/brand_registry.json

spec.sqlite

.workspace/db/{category}/spec.sqlite
Tables: 55 base (+ runtime learning tables in production, including 1 FTS virtual table)
Size: 500 KB – 100 MB+
Scope: Per-category (one per active category)
Purpose: All domain data — products, candidates, components, reviews, evidence, billing, telemetry, sources, crawl metadata, field overrides, pipeline state, and full-text search indexes.
Store modules: 17 dedicated stores

For the complete table-by-table schema with columns, types, indexes, and foreign keys, see Schema Reference.

Storage Safety Rules

These rules are enforced by architecture and documented in CLAUDE.md. Violations are considered bugs.

  • .workspace/ is the primary runtime data directory. All SQLite databases, run output, product checkpoints, and ephemeral scratch data live here. Additional runtime paths: .server-state/ (server PID + logs) and os.tmpdir()/spec-factory-crawl-videos/ (ephemeral video cache).
  • category_authority/_runtime/ is a legacy migration source (NOT git-ignored). Active settings write to .workspace/global/user-settings.json; active snapshots write to .workspace/runtime/snapshots/.
  • .server-state/ is git-ignored. Server PID and log files are ephemeral.
  • No other runtime directories are permitted. Do not create ad-hoc data directories outside the established paths.
  • The root tmp/ directory is banned and gitignored. Use OS tmpdir (os.tmpdir()) for throwaway work.
  • JSON files in category_authority/ are configuration only (immutable at runtime, unless explicitly documented as mutable).
  • No local file "databases" for mutable state (.json, .csv, .txt). Mutable domain state belongs in SQLite.
  • _runtime/snapshots/ is capped at 10 files. Oldest snapshots are auto-pruned by the settings persistence layer.
  • Database files use WAL mode for concurrent read access. Do not copy or move database files while the server is running.
  • Never hand-edit SQLite files directly. All mutations go through store modules with proper transaction handling.
