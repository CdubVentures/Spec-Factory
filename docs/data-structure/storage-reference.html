Storage Architecture Overview

Current persistent and scratch data flows through four verified top-level areas: .workspace/ for runtime state, category_authority/ for committed authority/config data, .tmp/ for repo-local scratch, and .server-state/ for dev server process files.

Runtime Data Directory

Git-Ignored

.workspace/ is the sole runtime data root. All live SQLite files, per-run checkpoints, crawl media, product checkpoints, settings fallbacks, and storage-backed outputs resolve under this tree. Repo-local scratch belongs under .tmp/, not under ad hoc runtime folders. Separate runtime-adjacent paths are .server-state/ for dev server PID/log files and os.tmpdir()/spec-factory-crawl-videos/{runId} for transient crawl-video caching.

Storage Safety Rules

These rules are enforced by architecture and documented in CLAUDE.md. Violations are considered bugs.

  • .workspace/ is the sole runtime data directory. Live DBs, checkpoints, media, snapshots, and storage-backed outputs belong here.
  • .workspace/output/ is the default local output root, but it is created lazily. Its absence only means nothing has written output keys yet.
  • .server-state/ is git-ignored. Server PID and log files are ephemeral process state.
  • .tmp/ is the repo-local scratch root for throwaway test/tool artifacts. It is not runtime state.
  • The root tmp/ directory is banned. Use .tmp/ for repo-local scratch, and use os.tmpdir() only when OS temp semantics are intentional, such as crawl-video caching.
  • Mutable domain state belongs in SQLite unless the file surface is explicitly part of the design, such as checkpoints, registries, overrides, or storage-backed output namespaces.
  • category_authority/_generated/ files are generated output. Regenerate them instead of hand-editing them.
  • .workspace/runtime/snapshots/ is capped at 10 files. Oldest snapshots are auto-pruned by the snapshot cleanup layer.
  • Database files use WAL mode with synchronous = NORMAL. Do not copy or move database files while the server is running.
  • Never hand-edit SQLite files directly. All mutations must go through the DB/store and persistence layers.
