Storage Architecture Overview

All persistent and ephemeral data flows through four top-level storage areas. Git-ignored directories hold runtime state; committed directories hold category configuration and fixture seeds.

Runtime Data Directory

Git-Ignored

.workspace/ is the sole runtime data directory (CLAUDE.md mandate). All SQLite databases, crawl output, product checkpoints, settings, snapshots, and ephemeral scratch data live here. It is fully git-ignored. Additional runtime paths: .server-state/ (server PID + logs) and os.tmpdir()/spec-factory-crawl-videos/ (ephemeral video cache during crawl runs).

Database File Details

SQL

Two SQLite database surfaces serve distinct purposes (SpecDb + AppDb). All databases use WAL mode for concurrent reads and are managed by dedicated store modules.

app.sqlite

.workspace/db/app.sqlite
Tables: 6 (brands, brand_categories, brand_renames, settings, studio_maps, color_registry)
Size: 50 KB – 2 MB
Scope: Global (one per installation)
Purpose: Cross-category state — brands + brand associations + rename history, user settings, global studio_maps table, and color registry. Note: per-category field_studio_map data is primary in spec.sqlite; app.sqlite studio_maps is a separate global table.
Seeded from: category_authority/_global/brand_registry.json, color_registry.json

spec.sqlite

.workspace/db/{category}/spec.sqlite
Tables: 36 (36 in specDbSchema.js)
Size: 500 KB – 100 MB+
Scope: Per-category (one per active category)
Purpose: All domain data — products, components, reviews, evidence, billing, telemetry, sources, crawl metadata, field overrides, and pipeline state.
Store modules: 19 dedicated stores

For the complete table-by-table schema with columns, types, indexes, and foreign keys, see Schema Reference.

Published Output Files

JSON 6 per product + 5 category-level

The publish command generates the final deliverables for each product. These files are the end output of the entire pipeline — from crawl through extraction, review, and publication. The publishStorageAdapter.js writes to the modern key pattern (output/{category}/published/{pid}/...). The listOutputKeys() function reads from both modern and legacy paths for backward compatibility with older data. Intermediate pipeline artifacts (normalized.json, provenance.json, summary.json) are written per-run under output/{category}/{pid}/latest/.

Storage Safety Rules

These rules are enforced by architecture and documented in CLAUDE.md. Violations are considered bugs.

  • .workspace/ is the sole runtime data directory (CLAUDE.md mandate). Subdirectories: global/ (user-settings.json), runtime/snapshots/ (per-run settings snapshots, capped at 10), db/{category}/ (per-category spec.sqlite), db/app.sqlite (global), runs/ (IndexLab run output), products/ (one product.json per product), output/ (pipeline output artifacts), crawlee/ (ephemeral).
  • .server-state/ is git-ignored. Server PID and log files (stdout + stderr) are ephemeral.
  • No other runtime directories are permitted. Do not create ad-hoc data directories outside the established paths.
  • The root tmp/ directory is banned and gitignored. Use OS tmpdir (os.tmpdir()) for throwaway work.
  • JSON files in category_authority/ are configuration only (immutable at runtime, unless explicitly documented as mutable: brand_registry.json, color_registry.json).
  • No local file "databases" for mutable state (.json, .csv, .txt). Mutable domain state belongs in SQLite.
  • .workspace/runtime/snapshots/ is capped at 10 files. Oldest snapshots are auto-pruned by the settings persistence layer.
  • Database files use WAL mode for concurrent read access. Do not copy or move database files while the server is running.
  • Never hand-edit SQLite files directly. All mutations go through store modules with proper transaction handling.
