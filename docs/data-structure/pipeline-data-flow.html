Pipeline Overview

8 logical pipeline phases defined in pipelinePhaseRegistry.js. Phase 01 (Discovery) runs NeedSet + Brand Resolver in parallel. Stage-level cursors (stage:crawl, stage:finalize) interleave after pipeline phases complete.

Phase-by-Phase Data Flow

8 pipeline phases from PIPELINE_PHASES in pipelinePhaseRegistry.js, plus stage-level crawl and finalize cursors. Expand each phase to see the complete data contract.

01 Discovery parallel: needSetPhase + brandResolverPhase
Reads SQL Write Bridge Event
Description
  • Runs two sub-phases in parallel via Promise.all: NeedSet and Brand Resolver
  • NeedSet (runNeedSet): Computes field gaps via computeNeedSet, builds search planning context, produces focusGroups, seedStatus, and seedSearchPlan
  • Brand Resolver (runBrandResolver): Resolves the official brand domain via resolveBrandDomain, returns brandResolution with the official domain
  • Stage cursors: stage:needset (NeedSet), stage:brand-resolver → stage:search (Brand Resolver)
Reads
  • category_authority/{category}/_generated/field_rules.json — field definitions, validation, and constraints
  • category_authority/{category}/_generated/known_values.json — canonical known value enums
  • category_authority/{category}/_generated/cross_validation_rules.json — inter-field validation
  • category_authority/{category}/_generated/field_groups.json — field grouping/ordering
  • category_authority/{category}/_generated/parse_templates.json — extraction templates
  • category_authority/{category}/_generated/key_migrations.json — field key migration maps
  • category_authority/{category}/_generated/component_db/*.json — encoders, materials, sensors, switches
  • item_field_state, component_values, list_values — current product state for gap analysis
SQL Writes
  • UPSERT run_artifacts — stores computed needset as needset artifact type
Bridge Events
  • needset_computed — signals NeedSet computation complete, carries field gap summary
  • brand_resolved — brand domain resolution result
02 Bootstrap bootstrapPhase.execute
Reads
Description
  • Convergence phase: runs AFTER Discovery (parallel), BEFORE Search Profile
  • Promotes brand-resolved domain into categoryConfig.sourceHostMap so downstream stages recognise it as an approved source
  • Resolves job identity (brand, model, variant) via resolveJobIdentity
  • Normalizes missingFields from planning hints (missingRequiredFields + missingCriticalFields + focus_fields)
  • Loads learning artifacts and merges learning-store hints into enriched lexicon
  • Assembles the full execution context: categoryConfig, variables, identityLock, missingFields, learning, enrichedLexicon
Reads
  • brandResolution (from Phase 01 Brand Resolver output)
  • categoryConfig — source hosts, field order, source lookups
  • Learning artifacts from storage (category-scoped)
  • learningStoreHints — merged into enriched lexicon
03 Search Profile runSearchProfile (deterministic)
Reads SQL Write Bridge Events
Description
  • Synchronous, deterministic query generation from NeedSet + Brand outputs
  • Translates missing fields into concrete search queries targeting data gaps
  • Uses focusGroups, seedStatus, brand resolution, learning, and enriched lexicon to build an optimized search profile
  • Stage cursor: stage:search-profile. Checkpoint: afterProfile
Reads
  • NeedSet output: missingFields, focusGroups, seedStatus (from Phase 01/02)
  • brandResolution — official domain for site-targeted queries
  • learning + enrichedLexicon — learned query patterns
  • categoryConfig — field order, field rules
SQL Writes
  • UPSERT run_artifacts — stores search profile as search_profile artifact type
Bridge Events
  • search_profile_generated — search profile is ready
04 Search Planner runSearchPlanner (LLM)
Reads Bridge Events
Description
  • Async LLM-based query enhancement phase
  • Reviews the search profile and suggests additional queries for hard-to-find fields that standard templates might miss
  • Receives searchProfileBase and queryExecutionHistory, returns enhancedRows
  • Stage cursor: stage:search-planner. Checkpoint: afterPlanner
Reads
  • searchProfileBase (from Phase 03 output)
  • queryExecutionHistory — previously executed queries
  • identityLock — brand/model/variant constraints
  • missingFields — fields still needed
Bridge Events
  • search_plan_generated — enhanced plan ready for query journey
Worker Pool
  • LLM workers — used for query enhancement calls
05 Query Journey runQueryJourney
Reads Bridge Events
Description
  • Selects and orders queries for execution from the combined search profile + planner output
  • Emits search_queued events for each planned query BEFORE Search Execution starts, so the GUI renders all planned worker slots immediately
  • Returns queries, executionQueryLimit, selectedQueryRowMap, and profileQueryRowsByQuery
  • Stage cursor: stage:query-journey → stage:fetch. Checkpoint: afterJourney
Reads
  • searchProfileBase + enhancedRows (from Phases 03/04)
  • variables, missingFields, planningHints
  • brandResolution, categoryConfig
  • storage — for query cooldown / history checks
Bridge Events
  • search_queued (per query) — emitted for each planned worker slot with worker_id, slot, query, provider
06 Search Execution executeSearchQueries
Reads SQL Write Bridge Events
Description
  • Executes search queries against configured providers using search worker pool
  • Pre-computes discoveryCap (from serpSelectorMaxKeep), providerState (via searchEngineAvailability), and requiredOnlySearch flag
  • Stage cursor: stage:search-results → stage:parse → stage:index. Checkpoint: afterExecution
Reads
  • queries, executionQueryLimit, selectedQueryRowMap (from Phase 05)
  • frontierDb — query cooldown and deduplication state
  • categoryConfig, config — provider configuration, execution limits
SQL Writes
  • INSERT query_cooldowns (SpecDb) — records executed queries with cooldown state
  • INSERT query_index — indexes query results for deduplication
Bridge Events
  • search_started — search worker begins processing a query slot
  • search_finished — search worker completes a query slot
  • search_results_collected — raw SERP results ingested
Worker Pool
  • Search workers — parallel query execution pool
07 Result Processing processDiscoveryResults (SERP Selector)
Reads SQL Write Bridge Events
Description
  • Processes raw search results through SERP selection: LLM-based URL selector that classifies each result as approved/candidate/reject
  • Scores and filters URLs using domain intelligence, source corpus, and brand resolution
  • Produces the discoveryResult with selected URLs for fetch
  • Stage cursor: stage:serp-selector → stage:prime-sources. No checkpoint (feeds directly into Domain Classifier)
Reads
  • searchResults, searchAttempts, searchJournal (from Phase 06)
  • frontierDb — source corpus and crawl history
  • identityLock, brandResolution, missingFields
  • discoveryCap, providerState — execution constraints
SQL Writes
  • UPSERT source_corpus — new/updated source entries from SERP selection
Bridge Events
  • serp_selector_completed — URL selection pass complete
  • domains_classified — domain tier/quality classification results
08 Domain Classifier final result assembly
Reads
Description
  • Assembles the final pipeline result by merging discovery output with seed search plan
  • Deterministic heuristic classifier (no LLM call) — classifies domains by role (manufacturer, review site, retailer) and routes them to fetch queues
  • Returns finalResult with merged discoveryResult + seed_search_plan_output + enqueue_summary
  • Stage cursor: stage:domain-classifier. Checkpoint: final
Reads
  • discoveryResult (from Phase 07)
  • seedSearchPlan (from Phase 01 NeedSet output)
STAGE stage:crawl Fetch Workers + Plugins
Reads SQL Write File Write Bridge Events
Description
  • Stage-level cursor that runs after pipeline phases complete. Not a pipeline phase itself — it is a bridge concern (lifecycle timing)
  • Parallel fetch workers using Crawlee/Playwright browser automation
  • 6 fetch plugins operate on every page: stealth, cookie_consent, overlay_dismissal, auto_scroll, dom_expansion, css_override
  • 2 extraction plugins: screenshot (full-page + targeted), video (WebM recordings)
  • Captures page content, screenshots, and video recordings
File Writes
  • *.html.gz — content-addressed compressed HTML snapshots
  • screenshots/*.jpg — page screenshots for visual verification
  • video/*.webm — crawl session video recordings
SQL Writes
  • INSERT crawl_sources — per-URL crawl results (status, timing, content hash)
  • INSERT source_screenshots — screenshot metadata and file paths
  • INSERT source_videos — video recording metadata
Bridge Events
  • source_fetch_queued — URL added to fetch queue
  • source_fetch_started — fetch worker begins URL
  • fetch_finished — fetch completed successfully
  • source_fetch_failed — fetch failed (timeout, blocked, error)
  • visual_asset_captured — screenshot or video captured
  • parse_finished — document parsing complete
  • source_processed — evidence extraction pass complete
  • index_finished — candidate values produced and reviewed
SQL Writes — Candidate Pipeline
  • INSERT candidates — field value candidates with provenance
  • INSERT candidate_reviews — LLM review verdicts for candidates
SQL Writes — Product State
  • UPSERT item_field_state — accepted field values
  • UPSERT component_values — component-specific field values
  • UPSERT list_values — list-type field values
  • UPSERT item_component_links — product-to-component associations
  • UPSERT item_list_links — product-to-list associations
Worker Pools
  • Fetch workers — parallel browser-based crawl pool (Crawlee/Playwright)
  • LLM workers — parallel extraction review pool (candidate quality assessment)
STAGE stage:finalize Checkpoint + Panels + Cleanup
SQL Write File Write Bridge Events
Description
  • Stage-level cursor that runs after stage:crawl. Handles checkpoint persistence and run finalization
  • Builds crawl and product checkpoints for recovery and persistence
  • Constructs all 11 RuntimeOps panels and serializes to run_summary artifact
  • Marks run as completed and broadcasts final status
File Writes
  • writeCrawlCheckpoint → writes run.json (schema v2/v3) to .workspace/runs/
  • writeProductCheckpoint → writes/updates product.json in .workspace/products/
SQL Writes — Run Finalization
  • UPDATE runs — status=completed, ended_at, final counters
  • UPSERT run_artifacts — run_summary artifact (serialized 11 RuntimeOps panels)
SQL Writes — Product Lifecycle
  • UPSERT product_runs — links run to product for history tracking
  • UPDATE product_queue — status update (queued → completed)
Bridge Events
  • run_completed — pipeline finalization complete
  • WebSocket broadcast: process-status: idle

RuntimeOps Panels

11 Panels

The GUI's RuntimeOps view is built from 11 panels defined in PANEL_KEYS (buildRuntimeOpsPanels.js). During a live run, panels are built by the single-pass processEventsToPanel() engine (eventProcessingEngine.js). After completion, the run_artifacts.run_summary provides cached panel data. Note: run.json is a separate checkpoint file written by writeCrawlCheckpoint, not the source of panel data.

Checkpoint / Recovery Flow

Checkpoint recovery enables resuming from disk-persisted state after restarts. The seeding process is transaction-wrapped and idempotent, using UPSERT semantics to prevent duplication.

Recovery Sequence

Crawl Checkpoint Details run.json schema v2/v3
SQL Upsert File Source
Source File
  • .workspace/runs/{run_id}/run.json — full crawl checkpoint (schema v2/v3)
  • Written by writeCrawlCheckpoint during stage:finalize
SQL Upserts (seedCrawlCheckpoint)
  • UPSERT runs — restores run record (status, timestamps, settings)
  • UPSERT product_runs — restores run-to-product linkage
  • UPSERT run_artifacts — restores all artifact types (needset, search_profile, run_summary, etc.)
  • UPSERT crawl_sources — restores per-URL crawl results
Guarantees
  • Transaction-wrapped: all-or-nothing seeding
  • Idempotent: ON CONFLICT DO UPDATE prevents duplication on re-seed
  • Deterministic: content-addressed artifacts ensure stable IDs
Product Checkpoint Details product.json
SQL Upsert File Source
Source File
  • .workspace/products/{product_id}/product.json — product checkpoint
  • Written/updated by writeProductCheckpoint during stage:finalize
  • Merges accumulated sources from the run into the existing product record
SQL Upserts (seedProductCheckpoint)
  • UPSERT products — restores product master record
  • UPSERT product_queue — restores queue position and status
Guarantees
  • Transaction-wrapped: atomic product state restoration
  • Idempotent: safe to re-seed on every startup
  • Created at product add time, grown after each run
