Category Propagation Chain

FULLY VALIDATED

Category switching is centralized in a Zustand store, written by the sidebar, normalized on bootstrap, and consumed by every tab via the same hook. The all sentinel was removed entirely — category is always a real category string. The backend guards against 'all' as a safety net.

Sidebar.tsx → setCategory(value) → uiStore.ts → localStorage → useCategorySync.ts → coerceCategories() → Every tab reads useUiStore(s => s.category) → API calls include /<category>/ → Backend SpecDb per category

Key Files

Primary files that define and enforce the category-driven architecture.

pageRegistry.ts

tools/gui-react/src/registries/pageRegistry.ts — SSOT for all tabs. Defines path, label, tabGroup, disabledOnTest, and dividerAfter for every page entry. The disabledOnAll flag and PageEntry.disabledOnAll interface field were fully removed.

TabNav.tsx

tools/gui-react/src/pages/layout/TabNav.tsx — Renders the three-section tab bar with SVG icon labels: Globe (global), List (category), Gear (settings). Uses border-r and border-l separators. Filters pages by tabGroup. No disabledOnAll logic remains.

Sidebar.tsx

tools/gui-react/src/pages/layout/Sidebar.tsx — Category dropdown that drives the entire propagation chain. The "All Categories" option was removed; category is always a real category string.

specDbRuntime.js

src/app/api/specDbRuntime.js — Backend guard: returns null when resolvedCategory === 'all', preventing phantom DB creation. This is the safety net in case 'all' ever reaches the backend.

PipelineSettingsPageShell.tsx

tools/gui-react/src/features/pipeline-settings/components/PipelineSettingsPageShell.tsx — Pipeline Settings with category badge indicators on the Source Strategy and Deterministic Strategy sidebar navigation buttons, marking them as per-category sections within a global settings page.

CatalogPage (products-only)

Simplified Catalog page. Brands was promoted to its own /brands route in the Global group. CatalogPage no longer has TabStrip or sub-tabs — it is purely the per-category product manager.

Resolved Findings

6 Resolved

All six findings from the original audit were resolved during the tab reorganization completed 2026-03-31.

Resolved Finding 1: "All Categories" was not safe as a universal working mode

Original issue: Selected Product stayed enabled in all mode, producing under-specified /product/all/{'{'}pid{'}'} API calls.

Resolution: "All Categories" was removed from the sidebar dropdown entirely. category is always a real category string. All isAll / category === 'all' dead code removed from ~15 frontend files.

Resolved Finding 2: Catalog Models was category-scoped code exposed inside an all-enabled page

Original issue: ProductManager queried /catalog/${'{'}category{'}'}/products with category === 'all', and the backend had no all branch.

Resolution: 'all' can never reach the frontend. Brands was promoted to its own /brands route in the Global group. CatalogPage simplified to products-only (no more sub-tabs). Backend 'all' references in dataChangeContract, brandRoutes, and brandRegistry left intact — they serve multi-category event routing, a different purpose.

Resolved Finding 3: Overview billing card was not a true all-category rollup

Original issue: Overview called /billing/${'{'}category{'}'}/monthly with 'all', and billing did unguarded getSpecDb(category).

Resolution: 'all' sentinel eliminated from the frontend. Overview always receives a real category. Backend guard in specDbRuntime.js returns null for resolvedCategory === 'all' as additional safety.

Resolved Finding 4: Backend could accidentally materialize an all SpecDb

Original issue: specDbRuntime.js would open (or create) a DB for any resolved category token, including 'all', producing a phantom .workspace/db/all/spec.sqlite.

Resolution: Backend guard added: specDbRuntime.js returns null when resolvedCategory === 'all'. Combined with frontend removal of 'all', phantom DB creation is impossible.

Resolved Finding 5: Global tabs wrongly disabled on all

Original issue: LLM Config and Pipeline Settings had disabledOnAll: true despite containing global data.

Resolution: The disabledOnAll flag was fully retired — removed from TabDef and PageEntry interfaces, all registry entries, TabNav rendering logic, and ~15 frontend files. Settings tabs are in their own group (far right) and are always accessible. Pipeline Settings has category badge indicators on Source Strategy and Deterministic Strategy sidebar nav buttons to visually distinguish the per-category sections.

Resolved Finding 6: Categories tab mixed production and test categories

Original issue: CategoryManager rendered all categories uniformly while the sidebar separated _test_* into a dedicated optgroup.

Resolution: Categories tab is now in the Global group with proper test-category separation. The sidebar no longer offers "All Categories" as an option.

Backend Evidence

Key backend code confirming the scope of each data surface. All paths validated against source.

SpecDb = Per-Category

specDbRuntime.js — path.join(config.specDbDir, resolvedCategory, 'spec.sqlite'). One SQLite file per category. 55+ tables, all with category TEXT NOT NULL. Unique constraints combine (category, ...).

AppDb Settings = Global

appDbSchema.js — CREATE TABLE settings (section TEXT NOT NULL, key TEXT NOT NULL, ... PRIMARY KEY (section, key)). No category column. Sections: runtime, convergence, storage, ui, studio.

Runtime Settings Handler = Global

configRuntimeSettingsHandler.js — GET /runtime-settings reads from single config object. PUT /runtime-settings writes to same single object. Zero category awareness.

LLM Policy Handler = Global

llmPolicyHandler.js — /llm-policy GET/PUT with no category parameter. Single global policy applied across all categories.

Source Strategy = Per-Category

sourceStrategyAuthority.ts — All API calls use ?category=${'{'}category{'}'}. Reads/writes category_authority/{'{'}cat{'}'}/sources.json. Correctly scoped. Stays in Pipeline Settings with category badge indicator.

LLM Route Matrix = Per-Category

llmSettingsAuthority.ts — /llm-settings/${'{'}category{'}'}/routes. Stored in SpecDb llm_route_matrix table with category column. Now in ops group as Review LLM tab.

Brand Registry = Global + Junction

appDbSchema.js — brands in AppDb (global). brand_categories(identifier, category) junction table maps brands to categories. Now its own /brands route in Global group.

Category Alias = Guarded

specDbRuntime.js — Returns null when resolvedCategory === 'all'. Prevents phantom DB creation. Backend 'all' references in dataChangeContract, brandRoutes, and brandRegistry left intact (multi-category event routing, different purpose).
