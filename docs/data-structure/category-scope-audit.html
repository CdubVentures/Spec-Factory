Category Propagation Chain

STRUCTURALLY CORRECT

Real category switching is structurally wired correctly. The selected category is centralized in a Zustand store, written by the sidebar, normalized on bootstrap, and consumed by every tab via the same hook. The break is not normal category propagation — the break is the sentinel all, plus a few genuinely global surfaces that are currently treated like category-only tabs.

Sidebar.tsx:161 → setCategory(value) → uiStore.ts:221 → localStorage → useCategorySync.ts:12 → coerceCategories() → Every tab reads useUiStore(s => s.category) → API calls include /<category>/ → Backend SpecDb per category

Validated Findings

6 Issues

Each finding validated against source code with exact file paths and line numbers confirmed.

Finding 1 "All Categories" is not safe as a universal working mode

Selected Product stays enabled in pageRegistry.ts:41 (no disabledOnAll), but ProductPage.tsx:40 calls /product/${'{'}category{'}'}/${'{'}productId{'}'}, and the product store at productStore.ts:7 does not carry the owning category. In all mode, that leaves product detail under-specified — the API gets /product/all/{'{'}pid{'}'}.

  • pageRegistry.ts:41 — no disabledOnAll on Selected Product
  • ProductPage.tsx:47 — api.get(`/product/${'{'}category{'}'}/${'{'}productId{'}'}`)
  • productStore.ts:7 — selectedProductId: string with no category field

Finding 2 Catalog > Models is category-scoped code exposed inside an all-enabled page

ProductManager.tsx:83 queries /catalog/${'{'}category{'}'}/products and /brands?category=${'{'}category{'}'}, and only disables bulk paste at line 447. Backend CRUD at catalogRoutes.js:222 is real-category only — listProducts() at productCatalog.js:586 has no all branch. brands?category=all filters to a literal all category in brandRoutes.js:41.

  • ProductManager.tsx:83 — /catalog/${'{'}category{'}'}/products
  • ProductManager.tsx:447 — only Bulk Paste disabled on all
  • productCatalog.js:586 — listProducts(config, category) — no all handling
  • brandRoutes.js:41 — resolveCategoryAlias(params.get('category')) treats all as literal string

Finding 3 Overview billing card is not a true all-category rollup

Overview correctly uses /catalog/${'{'}category{'}'} at line 105 (backend catalogRoutes.js:321 supports all with merge). But it also calls /billing/${'{'}category{'}'}/monthly at line 111. Billing in queueBillingLearningRoutes.js:157 does getSpecDb(category) unguarded — passing 'all' attempts to open a SpecDb for the literal string all.

  • OverviewPage.tsx:105 — catalog query works with all
  • OverviewPage.tsx:111 — /billing/${'{'}category{'}'}/monthly
  • queueBillingLearningRoutes.js:159 — getSpecDb(category) unguarded

Finding 4 Backend can accidentally materialize an all SpecDb

specDbRuntime.js:37 opens (or creates!) a DB for any resolved category token. resolveCategoryAlias('all') returns 'all' unchanged (it doesn't start with _test_ or test_). Any API call to /billing/all/monthly or similar would create .workspace/db/all/spec.sqlite — a phantom database with empty tables.

  • specDbRuntime.js:37 — function getSpecDb(category) with mkdirSync fallback at line 59
  • categoryAlias.js:34 — resolveCategoryAlias passes all through unchanged
  • Verified: .workspace/db/all/ does not currently exist, but the code path is live

Finding 5 Scope labels are inconsistent — global tabs wrongly disabled on all

Three settings tabs are disabledOnAll despite containing global data:

  • LLM Config pageRegistry.ts:55 — disabledOnAll: true but /llm-policy and /indexing/llm-config are entirely global. LlmConfigPage.tsx:47 only reads runtimeAutoSaveEnabled from uiStore, not category.
  • Pipeline Settings pageRegistry.ts:56 — disabledOnAll: true blanket-disables the whole page. Runtime settings at /runtime-settings are global (AppDb settings table has PRIMARY KEY(section, key) — no category column). Source strategy and deterministic seeds already self-guard with their own isAll checks (PipelineSourceStrategySection.tsx:429, PipelineDeterministicStrategySection.tsx:47).
  • Result: Users must pick a category to access global settings, which is misleading. Editing runtime settings under "mouse" applies to all categories silently.

Finding 6 Categories tab mixes production and test categories

The API supports includeTest at categoryRoutes.js:31, and useCategoriesQuery.ts:7 always asks for it. CategoryManager.tsx:21 renders every returned category as a normal active choice. Meanwhile, the sidebar already separates _test_* into a dedicated optgroup at Sidebar.tsx:164. These two surfaces disagree on presentation.

  • useCategoriesQuery.ts:7 — /categories?includeTest=true
  • CategoryManager.tsx:21 — renders all categories uniformly
  • Sidebar.tsx:164-173 — splits real vs test into separate groups

Recommendations

Ordered by impact and implementation complexity.

1. Remove "All Categories" from the sidebar

The all sentinel is not safely handled across the stack. Most tabs either disable themselves, produce under-specified API calls, or risk materializing a phantom SpecDb. The safest near-term fix is to remove all from the sidebar dropdown entirely. Users should always work within a specific category.

Files: Sidebar.tsx:164 (remove "All Categories" option), categoryStoreSync.js (change DEFAULT_CATEGORY), TabNav.tsx / pageRegistry.ts (remove disabledOnAll logic once no longer needed)

2. Add all guard to backend getSpecDb()

Regardless of whether all is removed from the UI, the backend should refuse to open a SpecDb for the literal token all. Add a guard in specDbRuntime.js:37 to return null when resolvedCategory === 'all'.

Files: specDbRuntime.js:37, categoryAlias.js (optionally add 'all' to reserved tokens)

3. Un-gate global settings tabs (if "All" is kept)

If all is retained as a viewing mode, LLM Config and Pipeline Settings should be accessible — their runtime settings are global. Remove disabledOnAll from pageRegistry.ts:55-56. The per-category sub-sections (Source Strategy, Deterministic Seeds) already have their own isAll guards.

Files: pageRegistry.ts:55-56 (remove disabledOnAll from llm-config and pipeline-settings)

4. Add scope badges to Pipeline Settings sections

The Pipeline Settings page mixes global and per-category sections with no visual distinction. Add Global and Per-Category badges to section headers so users understand what they're editing.

Files: PipelineSettingsPageShell.tsx (section headers), CategoryPanel.tsx, PipelineSourceStrategySection.tsx, PipelineDeterministicStrategySection.tsx

5. Separate test categories in CategoryManager

The sidebar already splits _test_* categories into a dedicated optgroup. CategoryManager should match this presentation — render test categories in a separate section with the same formatTestCategory() display format.

Files: CategoryManager.tsx:80 (split rendering), testMode.ts (shared utilities)

6. Add owning category to productStore

If all mode persists in any form, the product store needs a selectedCategory field so that product detail routes always send the real owning category, not the sentinel all.

Files: productStore.ts (add selectedCategory), Sidebar.tsx (pass category on selection), OverviewPage.tsx (pass category on row click)

Honest Assessment

Strong Moves

Remove all — eliminates 4 bugs and the entire disabledOnAll system in one cut.

Categories + Brands → Global — they are factually global (AppDb). Putting them in a category-scoped group was always wrong.

Review LLM as own tab in review cluster — per-category route matrix belongs with review workflow, not Settings. It's too complex (effort bands, sort/filter, per-field overrides) to cram into Review Grid as a sub-panel. Three review tabs (Grid | Components | LLM) grouped together is clean.

Source Strategy + Seeds as panels below run history — natural top-to-bottom flow in Indexing Lab: history → source config → seed config → launch. These are collapsible panels, not sub-tabs competing with the main UI. The components are already lazy-loaded and self-contained.

Watch Points

Billing → Global — billing data IS per-category (SpecDb), but the aggregate view is the primary use. If moved to Global, the component needs to always render the cross-category rollup and optionally drill down when a sidebar category is selected. The AllCategoriesBilling component already does this.

Indexing Lab density — already ~300 lines of complex state. The two new panels sit below run history, so they don't crowd the main launch UI. They need to be clearly collapsible with sensible defaults (collapsed until the user needs them). Don't let the page become a monolith.

Pipeline Settings becomes very simple — after extracting the two per-category sections, it's just 5 global runtime knob panels. Consider whether LLM Config and Pipeline should eventually merge into one “Global Config” page with a section nav, since both are global and autosave-driven. Not urgent — can ship separately.

What This Fixes

• Findings 1–4 eliminated (no more all sentinel)
• Finding 5 eliminated (no mixed-scope pages — Pipeline becomes purely global)
• Finding 6 addressed (Categories in Global group, test categories separable)
• disabledOnAll flag retired from registry
• Every tab has exactly one scope — no ambiguity
• Users always know what they're editing (global vs for-this-category)
• Review LLM grouped with Review Grid + Components — coherent workflow cluster
• Source config lives where you launch runs — no hunting through Settings

Migration Complexity

Small changes: Remove all from Sidebar dropdown, update categoryStoreSync.js DEFAULT_CATEGORY, rewrite pageRegistry.ts, update TabNav.tsx to render 3 groups, add all guard to specDbRuntime.js.

Medium changes: Extract BrandManager from Catalog into own route, extract Source Strategy + Deterministic Seeds from PipelineSettingsPage into IndexingPage, move LLM Settings into ReviewPage.

The components themselves don't change — BrandManager, PipelineSourceStrategySection, PipelineDeterministicStrategySection, and LlmSettingsPage are already self-contained. This is mostly re-wiring the shell, not rewriting features.

Implementation Order

Sequenced for safety. Each step leaves the app in a working state.

Step 1: Backend guard — block all in getSpecDb()

Add if (resolvedCategory === 'all') return null; to specDbRuntime.js:37. Prevents phantom DB creation regardless of frontend changes. Zero UI impact.

Step 2: Remove "All Categories" from sidebar

Remove <option value="all"> from Sidebar.tsx:164. Update categoryStoreSync.js to never resolve to all. Default to first real category.

Step 3: Add 'global' tab group to pageRegistry

Add tabGroup: 'global'. Move Categories there. Update TabNav.tsx to render 3 groups: Global (left), Category (center), Settings (right). Retire disabledOnAll flag.

Step 4: Promote Brands to own route in Global group

Create /brands route pointing to existing BrandManager component. Remove Brands sub-tab from CatalogPage.tsx (Catalog becomes products-only). Add to Global group in registry.

Step 5: Move Billing to Global group

Change tabGroup from 'ops' to 'global'. Billing component already handles rendering without a specific category (its AllCategoriesBilling path).

Step 6: Extract Source Strategy + Deterministic Seeds into Indexing Lab

Move the two <Suspense>-wrapped sections from PipelineSettingsPage.tsx into IndexingPage.tsx as collapsible panels below run history. Remove 'source-strategy' and 'deterministic-strategy' from PipelineSettingsPageShell.tsx. Components are already self-contained lazy imports — this is a re-wire, not a rewrite.

Step 7: Move Review LLM to category group

Remove dead code: disabledOnAll from TabDef and PageEntry interfaces, isAll logic in TabGroup, CATALOG_TABS / OPS_TABS exports (replaced by GLOBAL_TABS / CATEGORY_TABS). Remove Brands sub-tab from CatalogPage.tsx (Catalog becomes products-only). Separate test categories in CategoryManager to match sidebar optgroup.
