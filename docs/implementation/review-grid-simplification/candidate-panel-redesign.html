Core Model: 1 Source = 1 Candidate Card

Each entry in a SQL row's sources_json array becomes its own card in the drawer. A single SQL row with source_count=3 renders as 3 candidate cards. Each card inherits value and status from the parent row, but shows its own source, model, confidence, and submitted_at.

All cards from a resolved row are green — they are all linked to the published value.

Publisher & Settings: Confidence Threshold Logic

The Setting

publishConfidenceThreshold is a live setting in the settings registry (settingsRegistry.js). Default 0.7, range 0–1, configurable via the GUI under Review & Publisher → Publisher.

// settingsRegistry.js
{ key: "publishConfidenceThreshold", type: "float", default: 0.7,
  min: 0, max: 1, group: "validation",
  uiCategory: "review-publisher", uiSection: "publish-gate" }

Publish Gate (per submission)

Every call to submitCandidate() triggers publishCandidate() which gates on the current threshold:

// publishCandidate.js:79-83
const threshold = config?.publishConfidenceThreshold ?? 0.7;
if (normalizeConfidence(confidence) < threshold) {
  return { status: 'below_threshold' };  // not published, stays 'candidate'
}

Only sources with confidence at or above the threshold at submission time get published and marked resolved. A source at 60% with a 0.7 threshold stays candidate — it never triggers the publish path, so it can never be resolved.

Reconciliation (on threshold change)

When the threshold setting is changed, reconcileThreshold() re-evaluates every published field across all products:

DirectionConditionAction
Tightening (raised) Resolved candidate's confidence < new threshold demoteResolvedCandidates() → status back to 'candidate'.
Remove field from product.json fields[].
Loosening (lowered) No resolved candidate, but best candidate ≥ new threshold Publish best candidate. Mark resolved (scalar: exact match, set_union: overlap). Rebuild linked_candidates. Write to product.json.
Manual override source === 'manual_override' Skipped entirely. Never touched by reconciliation.

Reconciliation is available via:

  • GET /publisher/{category}/reconcile — dry-run preview (counts only)
  • POST /publisher/{category}/reconcile — apply (updates SQL + product.json)
  • GUI: Settings → Review & Publisher → Reconcile section (PublisherReconcileSection.tsx)

What This Means for the Drawer

The drawer never reads or evaluates the threshold. It reads status from the candidates endpoint, which reflects the DB state. When the threshold changes, reconciliation immediately updates the DB, so the next drawer load shows the new resolved/candidate status. Green bg is purely driven by status === 'resolved'.

This means:

  • Every green card was above the threshold when published (or after reconciliation promoted it)
  • Threshold changes propagate instantly through DB reconciliation — no UI-side threshold checks needed
  • Manual overrides are locked and never affected by threshold changes
  • The drawer is a pure read-only projection of the publisher's resolved state

Backend: Candidates Endpoint Fan-Out

The critical backend change. fieldReviewHandlers.js:238-259 currently maps 1 SQL row → 1 candidate object. It must fan out sources_json entries into individual candidate objects.

Current (1 row = 1 object)

// fieldReviewHandlers.js:239-258
const allCandidates = fcRows.map((c) => {
  const firstSource = sources[0];           // only uses first source
  return {
    candidate_id: `fc_${c.id}`,
    value: c.value,
    score: ..., source: ..., status: c.status,
    evidence: { ... },
  };
});

Target (1 source entry = 1 object)

const allCandidates = [];
for (const c of fcRows) {
  const sources = Array.isArray(c.sources_json) ? c.sources_json : [];
  const meta = c.metadata_json || {};

  for (let i = 0; i < sources.length; i++) {
    const src = sources[i];
    allCandidates.push({
      // From parent SQL row (shared)
      candidate_id: `fc_${c.id}_${i}`,     // unique per source entry
      value: c.value,
      status: c.status || 'candidate',

      // From individual source entry
      score: Math.max(0, Math.min(1, Number(src.confidence ?? c.confidence) || 0)),
      source: String(src.source || '').trim().toLowerCase(),
      model: String(src.model || '').trim() || null,
      run_id: src.run_id || null,
      submitted_at: src.submitted_at || c.submitted_at,

      // From parent row metadata
      evidence_url: String(meta.evidence?.url || '').trim() || null,
      metadata: Object.keys(meta).length > 0 ? meta : null,
    });
  }

  // If sources_json is empty, still emit one card from the row itself
  if (sources.length === 0) {
    allCandidates.push({
      candidate_id: `fc_${c.id}`,
      value: c.value, status: c.status || 'candidate',
      score: Math.max(0, Math.min(1, Number(c.confidence) || 0)),
      source: '', model: null, run_id: null,
      submitted_at: c.submitted_at,
      evidence_url: null, metadata: null,
    });
  }
}

Sort Order (after fan-out)

Sort all fanned-out cards by score descending, then by submitted_at descending. Resolved cards naturally sort to the top since they passed the confidence threshold.

Fanned-Out Candidate Object Shape

{
  candidate_id: string,      // "fc_42_0" (row id + source index)
  value: string,             // serialized value from parent row
  status: 'candidate' | 'resolved',  // from parent row

  score: number,             // 0-1, from this source entry's confidence
  source: string,            // "cef", "provenance", "manual_override"
  model: string | null,      // "gpt-4o", "gemini-2.5-flash", etc.
  run_id: string | null,     // links to run_artifacts table
  submitted_at: string,      // ISO timestamp of this source submission

  evidence_url: string | null,   // from metadata_json.evidence.url
  metadata: object | null,       // color_names, edition_details, etc.
}

Confidence Color Scale (10-step)

Maps a 0-1 score to one of 10 CSS classes. Decile bucketing:

// Frontend utility: confidenceColorClass(score: number): string
function confidenceColorClass(score: number): string {
  const pct = Math.round(score * 100);
  if (pct >= 95) return 'conf-100';   // deep green
  if (pct >= 85) return 'conf-90';    // green
  if (pct >= 75) return 'conf-80';    // yellow-green
  if (pct >= 65) return 'conf-70';    // olive
  if (pct >= 55) return 'conf-60';    // gold
  if (pct >= 45) return 'conf-50';    // amber
  if (pct >= 35) return 'conf-40';    // orange
  if (pct >= 25) return 'conf-30';    // coral
  if (pct >= 15) return 'conf-20';    // red-tinted
  return 'conf-10';                    // red
}

Published Section: Data Source

The expandable source table in the Published section needs the linked_candidates data. Two options:

OptionSourceProsCons
A: From products-index payload Add linked_candidates to the FieldState returned by products-index Available immediately without extra query. Single source. Heavier payload for products-index (currently lightweight).
B: From candidates query Derive from the fanned-out candidates array — filter where status === 'resolved' No backend change. Data already loaded when drawer opens. Only available after candidates query fires (drawer open).

Recommendation: Option B. The published source table is inside the drawer which already lazy-loads candidates. Filter the fanned-out candidates where status === 'resolved', sort by score descending, and render in the table. No backend changes needed.

Set Union: Chip Logic for Candidate Cards

For set_union list fields, each candidate card shows its array items as chips. The chip style depends on whether the item is in the published set:

// Given: publishedValue = ["black", "white", "pink"]
// Given: candidate.value = ["black", "red"]
const publishedSet = new Set(publishedValue);

for (const item of candidateItems) {
  if (publishedSet.has(item)) {
    // Normal chip — this item is in the published union
  } else {
    // Dashed-border chip — this item was in the candidate
    // but did not make it into the published union
  }
}

For scalar fields and item_union=none list fields, no chips are needed — just show the value as text.
