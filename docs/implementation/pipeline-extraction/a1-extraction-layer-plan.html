1Executive Summary

Problem. Today IndexLab captures HTML, screenshots, and videos from ~80 URLs per run but never extracts field values. 129 historical runs, zero fields_filled, zero indexed_docs. The pipeline is a fetcher pretending to be an indexer.

Goal. Wire a production-grade extraction layer that turns captured pages into structured field candidates, using best-in-class libraries per data type instead of hand-rolling parsers.

Approach. Four complementary extraction plugins, each responsible for a distinct content type:

  • Metascraper — Structured data: JSON-LD, Microdata, RDFa, OpenGraph, Twitter Cards, HTML meta (Node, one lib covers all 6 formats)
  • Crawl4AI — Unstructured HTML: clean markdown, main content, tables, lists, LLM-assisted extraction (Python sidecar)
  • PDF.js — Manufacturer manuals and spec sheets (Node, Mozilla-maintained)
  • Network XHR Interception — Backend JSON APIs the page renders from (Playwright native, no deps)

Non-goals for A1. Field-value LLM scoring, publisher acceptance logic, and evidence-tier routing remain in their existing modules (src/features/publisher/, src/features/indexing/pipeline/fieldCandidates/). A1 only produces candidates — the existing pipeline consumes them.

Prerequisites met. Today's work (2026-04-22) closed snapshot routing, URL canonicalization, page-content retry, resource blocking, screenshot quality gate, category-term injection, orphan run sweep, sources[] metadata, and runs.counters merge. A1 starts on clean plumbing.

Key environment lessons (captured for M2–M4)

  • Python stdout MUST be redirected before importing BS4/crawl4ai. Library banner text corrupts the stdio JSON protocol. Save _RAW_STDOUT = sys.stdout.buffer first, then sys.stdout = sys.stderr.
  • Node subprocess stdio needs 'error' listeners on all three streams. Without them, a pipe EOF crashes the parent with "Unhandled 'error' event" — appears as exit code 1 at Crawl stage.
  • Python -m <package> requires CWD = package's PARENT dir. Resolve via fileURLToPath(import.meta.url), not process.cwd().
  • Pass env: { ...process.env, ... } to spawn. Bare env strips PATH, which Windows needs to find python.exe.

4Target Extraction Stack

LayerToolLanguageEvidence TierCovers
Structured data metascraper + plugins Node Tier 1 JSON-LD, Microdata, RDFa, OpenGraph, Twitter Card, HTML meta, canonical
Main content + tables + lists crawl4ai v0.8+ Python sidecar Tier 2 Clean markdown, Readability main body, preserved tables, bullet lists, LLM-assisted extract
PDF pdfjs-dist (Mozilla) Node Tier 1 Manufacturer PDFs, spec sheets, manuals. Text layer + fallback to tables.
Backend JSON Playwright page.on('response') Node (no new dep) Tier 1 JSON responses fetched during page render. Often cleaner than the rendered DOM.
Why not a single unified tool? Because the extraction problem is heterogeneous. Structured data parsing is a deterministic problem (schema is known) — metascraper is a mature Node lib that does it fast and cheap. Unstructured content (reviews, forums) is heuristic — Crawl4AI's Readability + LLM fallback is the 2026 state of the art. Mixing both into one Python process is technically feasible (via extruct) but doubles IPC cost on every fetch and forces all extraction through a sidecar.
Why metascraper specifically? It's the canonical Node lib for meta/structured extraction. Backed by microlinkhq, ships 5.x (52K weekly downloads, latest release days old). It already bundles JSON-LD + Microdata + RDFa + OpenGraph + Twitter + HTML fallback as a tiered resolver — one lib handles six formats. Alternatives (web-auto-extractor: unmaintained 8 years; open-graph-scraper: OpenGraph only) don't cover the range.

8Layer 2 · Crawl4AI (Python Sidecar) Shipped Phase 1b

Shipped Phase 1b · Pure BS4 + lxml (no AsyncWebCrawler). 200ms cold spawn. Bundle v2 with JSON-LD + microdata + OpenGraph + <dl>.

Why sidecar: crawl4ai is Python-native. Running it as a sidecar keeps our Node Playwright fetch layer untouched and lets crawl4ai focus on what it's best at (heuristic HTML → structured). Python dependency is bounded to one process — no pollution of Node code paths.

Deployment pattern: stdio subprocess. Node spawns python -m pipeline_extraction_sidecar once per run. Each URL extraction sends one JSON line in, reads one JSON line out. If latency becomes a bottleneck (spawn cost, Playwright-Python warm-up, etc.) we graduate to long-running HTTP on localhost.

Open question: do we let crawl4ai use its own Playwright-Python for JS rendering, or do we feed it pre-rendered HTML from our Node Playwright? Recommendation: feed pre-rendered HTML. Our Node layer already did the hard work (session pool, stealth, BrightData unlock, resource blocking) — we don't want a second browser fleet. Crawl4AI supports arun(url=None, html=<raw>) for pre-rendered inputs.

10Layer 4 · Network XHR Interception Queued · M4

The pro-scraper quiet move — grab the JSON the page renders from

Why: Modern product pages on Dell / Corsair / Razer / Logitech hydrate from backend JSON APIs (/api/products/, /_nuxt/, etc). That JSON is cleaner than anything we can scrape from the rendered DOM. ScrapingBee, Bright Data, and ScrapeOps do this — it's not exotic, it's standard practice.

Implementation: pure Playwright. No new dependency. In the capture phase, attach a listener and filter:

// src/features/crawl/plugins/networkCapturePlugin.js (new — capture-phase plugin)
page.on('response', async (response) => {
  const ct = response.headers()['content-type'] || '';
  const url = response.url();
  if (!ct.includes('json')) return;
  if (response.status() < 200 || response.status() >= 400) return;
  if (url.includes('/analytics') || url.includes('/gtag')) return;   // tracker noise
  if (captures.xhrResponses.length >= settings.networkCaptureMax) return;
  try {
    const body = await response.text();
    if (body.length > settings.networkCaptureMaxBytes) return;
    captures.xhrResponses.push({ url, status: response.status(), body });
  } catch { /* ignore aborted */ }
});

Transform: plugin in extraction phase that scans captures.xhrResponses for JSON shapes matching known product APIs. Heuristic: any object with sku, productId, partNumber, or a nested product/item key with spec-looking fields.

Files:

  • src/features/crawl/plugins/networkCapturePlugin.js (new — capture-phase, attaches listener)
  • src/features/extraction/plugins/xhrIntercept/xhrInterceptPlugin.js (new — transform-phase, processes captured JSON)
  • src/features/extraction/plugins/xhrIntercept/xhrShapeDetector.js (new — schema fingerprinting)
  • src/features/extraction/plugins/xhrIntercept/tests/*.test.js (new)

15Testing Strategy

Per-plugin tests

Every new plugin gets three categories of tests, following the existing screenshotPlugin pattern:

  1. Contract tests (auto-validated by extractionPluginRegistryContract.test.js): plugin has name, phase, onExtract, summarize; stage def exists with matching key.
  2. Unit tests (hand-written per plugin): fixture HTML / PDF / XHR blobs → assert candidate output shape, counts, evidence_tier, confidence bounds. Disabled-setting returns empty. Timeout path emits right event.
  3. Integration tests (shared): run the full pipeline on one fixture URL with all plugins enabled, assert expected candidates appear in specDb.field_candidates with correct plugin attribution.

Fixture strategy

  • metascraper: 5 HTML fixtures — JSON-LD Product, Microdata, OpenGraph-only, empty, malformed.
  • crawl4ai: 3 HTML fixtures with spec tables + 2 without. Python sidecar stubbed via JSON-stream mock in Node tests.
  • pdfExtract: 3 PDFs — Logitech QSG, Razer spec sheet, corrupted. Committed to src/features/extraction/plugins/pdfExtract/fixtures/.
  • xhrIntercept: 4 JSON blobs — Corsair product API shape, Dell config API, Razer catalog, random noise (should be rejected by shape detector).

No live external calls in tests

All plugin tests stub external IO (metascraper resolvers consume strings; crawl4ai sidecar is mocked; PDFs are bundled fixtures; XHR is pre-captured blobs). Live-run verification happens in M5 bake-off, not in unit tests.

16Risks & Open Questions

R1 · Python install friction (crawl4ai)

Crawl4AI's install is ~400MB (Playwright-Python browsers). Mitigation: skip browser install via env var (we only feed pre-rendered HTML). Document in sidecar README. Add a one-shot npm run setup:extraction that installs + verifies.

Blocker risk: low. If crawl4ai install breaks on Windows, sidecar can be skipped (crawl4aiEnabled: false); metascraper + PDF + XHR still produce plenty.

R2 · Field-key normalization across plugins

Metascraper emits description; crawl4ai table detects "Weight" column; XHR intercept sees weightGrams. They all need to map to the same canonical field_key before landing in field_candidates. The existing key registry (src/features/key/) is the SSOT — each plugin's toCandidates converter consults it. If a source field doesn't match any canonical key, the candidate is dropped with a telemetry event.

Decision needed: should unknown fields be logged somewhere for later canonical-key addition? Recommend: yes, a new unknown_field_observations run_artifact that the key-finder team can mine.

R3 · Duplicate candidate flood

Four plugins on 9 URLs = potentially 36+ candidates per field. The publisher deduplicates eventually, but the raw candidate table could explode in size. Mitigation: each plugin dedups its own output against content_hash before emit. The writer also enforces a per-(product_id, field_key, source_url) uniqueness with last-write-wins on confidence.

R4 · Sidecar process death mid-run

If the crawl4ai Python subprocess crashes halfway through a run, remaining URLs lose Layer 2 extraction. Mitigation: client auto-restarts subprocess on death (up to 3 times per run), logs crawl4ai_sidecar_error, continues with remaining plugins for affected URLs (graceful degradation).

R5 · XHR intercept leaking auth tokens

If we're logged into a site (we shouldn't be — no credentials configured), intercepted JSON could include session cookies or tokens in response.headers. Mitigation: networkCapturePlugin only captures response bodies, never headers. Additionally, strip any authorization/cookie/x-csrf-token values before storing in xhrResponses. Covered in unit tests.

R6 · Tests depend on Python

Node tests for the crawl4ai Node-side plugin stub the subprocess. Python-side tests live in pipeline-extraction-sidecar/tests/ and only run when Python is available. CI separation needed. Mitigation: document which suite runs in which environment; mark the Python tests as optional in the repo-wide test script.
