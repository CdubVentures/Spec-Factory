1. Wave Summary

Four migration waves moved all hot-path mutable state from JSON/NDJSON files into SQLite tables. ~3,000-5,000 file I/O operations eliminated per run.

Wave 1 — Complete

Bridge Events

  • bridge_events table (8 cols)
  • Store: sourceIntelStore.js
  • NDJSON bridge writes killed
  • Chokidar killed for events
Wave 2 — Complete

Run Metadata

  • runs table (20+ cols)
  • Store: runMetaStore.js
  • Mid-run run.json writes killed
  • JSON at start + end only
Wave 3 — Complete

Run Artifacts

  • run_artifacts table (6 cols)
  • Store: runArtifactStore.js
  • Mid-run needset/profile JSON killed
  • brand_resolution SQL-only
Wave 4 — Complete

Billing + Metrics

  • billing_entries table (20 cols)
  • metrics table (6 cols)
  • NDJSON billing writes killed
  • Metrics: SQL primary, NDJSON fallback

3. Actual Write Paths During a Run

This is where data goes right now, after all 4 waves. SQL is the primary write target for all mutable state. JSON writes are restricted to run boundaries (start + end).

Key achievement: During a run, all mutable state writes go to SQL only. JSON files are written exclusively at run boundaries (start and end) via the writeJson: true flag. This eliminates ~3,000-5,000 file I/O operations per run.

7. File I/O Impact Summary

Metrics note: The metrics NDJSON path (_runtime/metrics.jsonl) is preserved as a fallback for callers that don't have a specDb reference (e.g., CLI healthcheck). All production runtime paths pass specDb and write to SQL.
Legacy read paths: The billing module still has ledgerKey() and monthlyRollupKey() functions for read-only legacy fallback. No new writes use these paths. They exist so older archived data can still be read.
