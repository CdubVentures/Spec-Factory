## Executive Summary

- The SQL migration is materially real and far along.
- The live schema has `56` tables, not `53`.
- Core Wave 5.5 work is real: `run_summary` is stored as a SQL `run_artifacts` payload, `runs` is slimmed, and `bridge_events` are purged at finalize.
- The docs currently overstate how complete the migration is in three places:
  - `billing_entries` is not fully SQL-only at the consumer layer
  - `metrics` is not production-wired
  - parts of the docs still describe `run-summary.json` as a file even though tests and write-path code now treat it as SQL-only
- The largest remaining file-based surface is not run telemetry anymore. It is final published artifacts and their downstream readers: `summary.json`, `provenance.json`, `traffic_light.json`, and `evidence/sources.jsonl`.
- There is also stale compatibility code that now looks stronger than necessary and should be simplified.

## Doc Drift and Inaccuracies

### 1. Table count is wrong

Doc claim:

- `MIGRATION-STATUS.mmd` and `spec-factory-ideal-architecture.html` say `53 tables`

Live code:

- `src/db/specDbSchema.js` contains `56` `CREATE TABLE` statements

Impact:

- The migration status doc is not an exact representation of the live schema.

### 2. `billing_entries - SQL only` is only partially true

Accurate:

- `src/billing/costLedger.js` writes billing rows to SQL in `appendCostLedgerEntry()`

Not accurate:

- `src/publish/publishAnalytics.js` still reads `_billing/ledger.jsonl`
- `src/billing/costLedger.js` still has JSON fallback readers for rollups and month snapshots

Impact:

- Write-path migration is done.
- Consumer cleanup is not done.
- The doc phrasing should be changed from "SQL only" to "SQL write path complete; legacy read fallbacks remain" unless the read path is finished.

### 3. `metrics - SQL only (B4)` is overstated

Accurate:

- `src/runtime/metricsWriter.js` is SQL-only when a `specDb` is provided
- `metrics` table exists in `src/db/specDbSchema.js`

Not accurate:

- repo-wide search found no production `new MetricsWriter(...)` instantiation outside tests
- `src/pipeline/runProduct.js` wires query/url/knob telemetry, but not runtime metrics writes
- `scripts/verify-wave4.js` still expects `0` metrics rows because the writer is not wired to production

Impact:

- Schema and helper exist.
- Production wiring does not.
- Calling B4 complete is not currently defensible.

### 4. The docs mix two incompatible `run_summary` models

Current implementation:

- `run_summary` is written as a SQL `run_artifacts` payload
- tests verify no `run-summary.json` file is written

Stale doc language still present:

- multiple sections of `spec-factory-ideal-architecture.html` still describe `run-summary.json` as the GUI file
- `RUN-ARTIFACT-STRUCTURE-AUDIT.html` mixes file-era and SQL-artifact-era language in the same document

Impact:

- The docs now describe both the prior design and the current implementation without clearly separating them.

## Remaining Work

### 1. Clean up stale run-reader compatibility paths

Files:

- `src/features/indexing/api/builders/indexlabDataBuilders.js`
- `src/features/indexing/api/builders/runArtifactReaders.js`
- `src/features/indexing/api/builders/runListBuilder.js`

Current state:

- `indexlabDataBuilders.js` still tries `run-summary.json` and `run.json` disk reads
- `runArtifactReaders.js` still reads `needset.json`, `search_profile.json`, `_discovery/*.json`, and `runs/.../logs/summary.json`
- `runListBuilder.js` still stats and reads `run-summary.json` and `run.json`

Why this matters:

- These compatibility layers are now broader than the actual live write path.
- They make the code harder to reason about and can hide migration regressions.

Recommended direction:

- collapse all post-Wave-5.5 past-run readers to SQL-first logic
- keep one explicit pre-migration fallback path instead of many overlapping file probes

### 2. Finish billing consumer migration

Files:

- `src/publish/publishAnalytics.js`
- `src/billing/costLedger.js`

Current state:

- billing writes are SQL
- at least one live analytics reader still consumes `_billing/ledger.jsonl`
- report/snapshot helpers still fall back to JSON artifacts

Recommended direction:

- switch `publishAnalytics` to SQL reads
- retain legacy read fallback only if pre-migration data still has to be supported

### 3. Decide whether `metrics` is actually part of the completed migration

Files:

- `src/runtime/metricsWriter.js`
- `src/pipeline/runProduct.js`
- `scripts/verify-wave4.js`

Current state:

- writer exists
- writer is SQL-only
- production code does not appear to instantiate it

Recommended direction:

- either wire it into production and verify rows appear in `metrics`
- or mark B4 as incomplete in the docs

### 5. Stale runtime event file surface should be removed or re-justified

Files:

- `src/logger.js`
- `src/features/catalog/api/catalogRoutes.js`
- `src/shared/settingsRegistry.js`

Observed behavior:

- `EventLogger` writes to SQL `runtime_events` and in-memory hook paths
- it keeps a `runtimeEventsKey` option but does not append to `_runtime/events.jsonl`
- `catalogRoutes.js` still exposes an `/events` endpoint that reads `OUTPUT_ROOT/_runtime/events.jsonl`
- settings still expose `runtimeEventsKey` as if the file path were live

Conclusion:

- The file-backed runtime event log is now a stale API/config surface.

Recommended direction:

- either remove the endpoint + setting
- or reintroduce a real file writer if the file is still intentionally part of the product

### 6. Schema hygiene work is still pending

Files:

- `src/db/specDbSchema.js`
- `src/features/catalog/api/brandRoutes.js`
- `src/features/catalog/identity/brandRegistry.js`
- `src/db/stores/artifactStore.js`

Observed state:

- `brands` table exists in SQL
- live brand management uses the JSON registry flow, not the SQL table
- `artifacts` table exists in SQL
- live binary metadata flow uses `crawl_sources`, `source_screenshots`, `source_pdfs`, and `source_videos`, not the old `artifacts` table

Conclusion:

- `brands` and `artifacts` look like genuine dead-schema candidates
- they should still be verified one more time before removal, but the migration doc is directionally right here

## Optimization and Simplification Targets

### Highest-value simplifications

1. Replace multi-hop run reader fallbacks with one explicit legacy branch.
2. Stop probing disk for `run-summary.json` where tests already prove SQL-only behavior.
3. Move billing analytics fully onto SQL and delete `_billing/ledger.jsonl` read dependence.
4. Remove dead runtime event file settings/endpoints.
5. Decide whether final published artifacts stay file-based by design or move into SQL-backed retrieval.
6. Drop dead schema after one final verification pass.

### Concrete code hotspots

- `src/features/indexing/api/builders/runArtifactReaders.js`
  - highest concentration of overlapping old-path fallbacks
- `src/features/indexing/api/builders/indexlabDataBuilders.js`
  - still assumes a `run-summary.json` disk era in parts of the logic
- `src/features/indexing/api/builders/runListBuilder.js`
  - still does file stats and file reads that the SQL fast path should make unnecessary
- `src/publish/publishAnalytics.js`
  - direct blocker for a truthful "billing is SQL-only" claim
- `src/features/catalog/api/catalogRoutes.js`
  - has both a live file-based final-output contract and a stale runtime-events file contract in the same module

## Bottom Line

The migration is real, but the docs are now ahead of themselves in some places and behind the code in others.

Most important truths right now:

- run telemetry migration is substantially complete
