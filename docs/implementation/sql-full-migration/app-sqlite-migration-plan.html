2. Current brand distribution

Every brand is currently assigned to only "mouse". But brands like Razer, Corsair, Logitech G make mice, keyboards, headsets, and monitors. The registry already has a categories[] array — it's just not populated.

3. Current settings distribution

The user-settings.json file is a flat JSON document with 4 sections. The RUNTIME_SETTINGS_REGISTRY in src/shared/settingsRegistry.js (238 lines) is the SSOT — every key has type, default, min/max, env var, UI metadata. This is already registry-driven, which makes the SQL migration straightforward.

4. Target schema: app.sqlite tables

-- ================================================================
-- app.sqlite schema (5 tables)
-- Location: .specfactory_tmp/app.sqlite
-- Opened once at bootstrap, shared by all domain runtimes
-- ================================================================

CREATE TABLE IF NOT EXISTS brands (
  identifier     TEXT PRIMARY KEY,           -- 8-hex, immutable (e.g. "84a009b9")
  canonical_name TEXT NOT NULL,              -- display name, mutable (e.g. "Logitech G")
  slug           TEXT NOT NULL,              -- derived from canonical_name, mutable
  aliases        TEXT NOT NULL DEFAULT '[]', -- JSON array of alias strings
  website        TEXT NOT NULL DEFAULT '',
  added_by       TEXT NOT NULL DEFAULT 'seed',
  created_at     TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at     TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE UNIQUE INDEX idx_brands_slug ON brands(slug);

CREATE TABLE IF NOT EXISTS brand_categories (
  identifier TEXT NOT NULL REFERENCES brands(identifier) ON DELETE CASCADE,
  category   TEXT NOT NULL,
  PRIMARY KEY (identifier, category)
);
CREATE INDEX idx_bc_category ON brand_categories(category);

CREATE TABLE IF NOT EXISTS brand_renames (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  identifier  TEXT NOT NULL REFERENCES brands(identifier),
  old_slug    TEXT NOT NULL,
  new_slug    TEXT NOT NULL,
  old_name    TEXT NOT NULL,
  new_name    TEXT NOT NULL,
  renamed_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_br_identifier ON brand_renames(identifier);

CREATE TABLE IF NOT EXISTS settings (
  section    TEXT NOT NULL,   -- 'runtime' | 'storage' | 'ui' | 'convergence'
  key        TEXT NOT NULL,
  value      TEXT,            -- JSON-encoded value (string, int, bool, JSON blob)
  type       TEXT NOT NULL DEFAULT 'string',  -- mirrors registry type
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (section, key)
);

CREATE TABLE IF NOT EXISTS studio_maps (
  category   TEXT NOT NULL,
  map_json   TEXT NOT NULL DEFAULT '{}',  -- the full map object for this category
  file_path  TEXT NOT NULL DEFAULT '',
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (category)
);
Why brand_categories is a join table: A brand belongs to many categories (Razer → mouse, keyboard, headset). A category has many brands. This is a classic many-to-many relationship. The join table enables SELECT b.* FROM brands b JOIN brand_categories bc ON b.identifier = bc.identifier WHERE bc.category = 'mouse' — get all brands for a category with one query.
Why studio_maps is separate from settings: Studio maps are per-category nested objects (key_list, field_mapping arrays) — they're structurally different from flat key-value settings. They deserve their own table with category as PK.
