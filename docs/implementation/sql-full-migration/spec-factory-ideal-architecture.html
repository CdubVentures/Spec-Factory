Spec Factory — Ideal Data Architecture

SQLite as durable working store. JSONs as rebuild safety net. Milestone-only checkpoint writes.

SQLite (primary store)
JSON files (rebuild source)
Real-time push (IPC / WebSocket)
Eliminated in new flow

1. App startup

On first launch or after a rebuild, JSONs seed the SQLite DB. On normal startup, SQLite already has everything — no re-seeding needed.

2. During a run (the hot path)

Every event writes to SQLite only. The IPC channel pushes events to the GUI server, which broadcasts via WebSocket. No NDJSON files. No Chokidar. One write path, one push path.

What gets removed: events.jsonl, run_events.ndjson, Chokidar watchers, run.json mid-run overwrites, needset.json live rewrites, search_profile.json live rewrites. All of this data already exists in SQLite tables. The NDJSON files were only needed because Chokidar watched them to push to the frontend — but IPC already does that faster and more reliably.

3. GUI interactions (settings, reviews, mutations)

All user actions write to SQLite immediately. JSON checkpoint writes happen only at milestones.

Write frequency: Settings JSON writes on every change (rare — maybe 5x per day). Product checkpoint writes at run completion, review save, and publish (1-3 per product per run). Category JSONs write on compile (rare). During a run, zero JSON writes — DB only handles the thousands of events.

4. Run completion (milestone checkpoint)

When a run finishes, write one fat product checkpoint JSON. This is the only time product data hits disk as JSON. It contains everything needed to rebuild this product's DB state.

5. Full write frequency summary

6. Rebuild path (nuke and restore)

If SQLite gets corrupted or you want a clean slate, delete the DB file and restart. The app seeds from JSONs exactly like it does today — except now those JSONs include the fat product checkpoints that capture everything.

8. Migration plan — 24 micro-steps

Each step is one PR. Each leaves the app fully functional. No big bang. The rule: nothing downstream should notice the change. API responses stay the same, GUI gets the same data, S3 archives contain the same artifacts. Only the internal write path changes.

Ordering principle: Start with the highest-frequency write paths that already have SQL tables waiting. End with the new tables and checkpoint architecture. Every step either (a) promotes an existing SQL path from fallback to primary, or (b) creates a new SQL table and wires it in. The JSON write is always kept as a fallback until a later step explicitly removes it.

Wave 5.5 — Run telemetry / product knowledge split (PREREQUISITE for Wave 6)

Separate ephemeral GUI dashboard data from permanent product learning data. Without this split, the Wave 6 checkpoint serializer cannot be designed correctly — it wouldn't know what belongs in the product checkpoint vs what's throwaway run telemetry.

Why this wave exists: Audit of Waves 1-4 revealed that the runs table (20+ columns), run_artifacts (full debug payloads), and bridge_events (500-3,000 rows/run) mix two fundamentally different data lifetimes. The GUI needs worker timelines, fetch timings, query rejection logs, and boot progress to render a past-run dashboard. The product needs field values, candidates, evidence, frontier history, and learning scores for the next run. These two concerns have different write patterns, different sizes, different lifetimes, and different consumers. Mixing them makes the checkpoint serializer (Wave 6) impossible to scope cleanly.

run-summary.json contract (the GUI file)

One file per run. Written once at completion. Contains everything the GUI needs to render a past run's dashboard. NOT needed for product learning, NOT included in the Wave 6 product checkpoint.
