# Product ID Decoupling Audit — 2026-03-29

> **Problem:** `productId` is built from `slug(brand)-slug(model)-slug(variant)`. Renaming any identity field changes the productId, orphaning all data (SQL rows, storage keys, disk paths). The catalog already has a stable `identifier` (8-char hex) on every product — it just isn't used as the canonical key.

## The existing identifier system

Every catalog product already has:
```json
"mouse-alienware-aw610m": {        ← slug-based key (BREAKS on rename)
  "id": 1,                          ← numeric auto-increment
  "identifier": "121cc826",         ← 8-char hex, stable, immutable
  "brand": "Alienware",
  "model": "AW610M"
}
```

Brands also have identifiers. Generated by `generateIdentifier()` in `src/features/catalog/identity/productIdentity.js` using `crypto.randomBytes(4)`.

## Target state

`productId` becomes `{category}-{identifier}` (e.g. `mouse-121cc826`). Identity fields (brand, model, variant) are mutable metadata. The slug-based key becomes a display label only.

## Migration strategy

### Phase 1: New ID generator (small, safe)
- Create `buildProductId(category)` → `{category}-{8-char-hex}` in `src/shared/primitives.js`
- Uses existing `crypto.randomBytes(4)` pattern from `generateIdentifier()`
- Tests first (TDD)

### Phase 2: Update 7 generation sites
- All sites switch to `buildProductId(category)`
- Explicit `--product-id` override still respected
- `explain-unk` requires explicit `--product-id` (no auto-derive from identity)

### Phase 3: Catalog key migration
- `productCatalog.js` uses `{category}-{identifier}` as the catalog key
- `normalizeProductIdentity` stops generating productId
- `updateProduct` identity changes do NOT change productId

### Phase 4: Fix driftScheduler
- `finalPathPartsFromIdentity` → use productId instead of identity slugs

### Phase 5: One-time data migration
- Iterate all products in catalog
- Generate new key: `{category}-{product.identifier}`
- Use existing `migrateProductArtifacts()` to move all storage keys
- UPDATE all 28 SQL tables: `SET product_id = newId WHERE product_id = oldId`
- Rekey catalog JSON object
- Log all renames

### Phase 6: Test fixture updates
- 57 test files need hardcoded IDs updated to new format
- Or: tests use factory helpers that generate IDs (more resilient)

## Existing infrastructure that helps

- `generateIdentifier()` — already generates 8-char hex IDs
- `identifier` field — already on every product and brand in catalog
- `migrateProductArtifacts()` — already renames storage key prefixes
- `appendRenameLog()` — already logs renames
- `artifactMigration.js` — tested, handles latest/runs/final/published/review paths

## What does NOT exist yet

- `buildProductId(category)` function that generates `{category}-{hex}`
- SQL batch UPDATE for 28 tables
- Catalog object rekeying logic
- `final/` path migration from identity-based to productId-based (driftScheduler data)
- Test fixture update tooling

## Risk assessment

| Risk | Severity | Mitigation |
|------|----------|------------|
| Data loss during migration | HIGH | Run migration with dry-run first. Backup sqlite + catalog JSON before. |
| Test breakage (57 files) | MEDIUM | Phase 6 after all code changes. Factory pattern for future tests. |
| GUI disruption | LOW | GUI displays brand+model, not raw productId. Internal routing adapts. |
| Incomplete SQL migration | HIGH | Transaction-wrap all 28 table UPDATEs. Rollback on failure. |
| Old-format IDs in the wild | MEDIUM | Existing `--product-id` override still accepts old format for backward compat. |
