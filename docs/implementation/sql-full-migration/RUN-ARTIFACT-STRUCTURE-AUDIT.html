Run Artifact & Folder Structure Audit

Current state vs optimal structure for SQL-primary architecture | 2026-03-27

1. Current Folder Layout (3 scattered roots)

Today, a single run scatters files across 3 independent directory roots. The archival bundle then re-copies everything into a 4th location. This creates duplicate data, orphaned files, and makes disaster recovery fragile.

3. Optimal Structure (SQL-primary)

The goal: a single run directory per run, with only files that can't live in SQL. SQL owns all structured metadata. Files exist only for binaries (HTML, screenshots, PDFs) and export products (spec.json, provenance.json). No copies, no duplicates, no scattered roots.

5. Optimal Run Directory (what should be saved)

A completed run should produce exactly this file tree. Everything else is in SQL or derivable from SQL.

runs/{runId}/                          ← single directory per run
├── run-summary.json                   ← GUI telemetry (schema v1, ~150KB-1MB)
│                                         Contains: meta, events[], llm_agg, observability
│                                         GUI reads this for past-run dashboard
│
├── normalized/                        ← extraction work product
│   ├── {category}.normalized.json        All field values + confidence + source
│   └── {category}.row.tsv               TSV for bulk import/export
│
├── provenance/                        ← field-level audit trail
│   ├── fields.provenance.json            Why each field value was chosen
│   └── fields.candidates.json            All candidates ranked per field
│
├── evidence/                          ← aggregated evidence
│   └── evidence_pack.json               Source snippets + quotes per field
│
└── summary.json                       ← compact run outcome
                                          validated, confidence, completeness, cost

── NOT in the run directory (shared, content-addressed) ──

sources/{category}/{product}/{hash12}/ ← deduped binary artifacts
├── page.html.gz                       ← shared across runs (same content = same hash)
├── screenshot.{ext}
├── ldjson.json
└── {filename}.pdf

── NOT saved as files (SQL owns this) ──

spec.sqlite                            ← ALL structured data
├── runs                                  14-col slim run record
├── run_artifacts                         needset, search_profile, brand_resolution, run_summary
├── candidates                            36K+ rows per product (all extraction attempts)
├── item_field_state                      73 rows per product (current winners)
├── source_registry                       every URL visited + status
├── source_assertions + evidence_refs     provenance chain
├── billing_entries                       per-LLM-call costs
├── metrics                               timing + gauges
├── crawl_sources                         content_hash → file_path mapping
├── source_screenshots                    screenshot metadata
├── source_pdfs                           PDF metadata
└── ... (48 tables total)
Key principle: If data is structured and queryable, it belongs in SQL. If it's a binary blob (HTML, images, PDFs), it belongs in content-addressed file storage. If it's a human-readable export product (spec.json, provenance.json), it belongs as a file in the run directory. Everything else (run metadata, events, billing, metrics, candidates, evidence) is SQL-only.

6. Archive Bundle (what should be archived)

The archive bundle should be the minimum set of files needed to (a) understand what happened in a run, (b) review the extraction output, and (c) rebuild SQL state if needed.

ARCHIVE: {category}/{product}/{runId}/
│
├── run-summary.json                   ← what happened (GUI telemetry)
├── summary.json                       ← outcome (validated, confidence, cost)
│
├── normalized/                        ← what was extracted
│   ├── {cat}.normalized.json
│   └── {cat}.row.tsv
│
├── provenance/                        ← why each value was chosen
│   ├── fields.provenance.json
│   └── fields.candidates.json
│
├── evidence/                          ← supporting evidence
│   └── evidence_pack.json
│
└── NOT included:
    latest_snapshot/                   ← redundant (SQL is SSOT)
    shared_logs/                       ← redundant (events in run-summary, billing in SQL)
    runtime_traces/                    ← low value for archive (debug only)
    indexlab/                          ← redundant (only had run-summary.json + screencast)
    manifest.json                      ← can be generated from run-summary.json metadata
Archive size reduction: The current bundle includes 5 subdirectories (run_output, latest_snapshot, indexlab, runtime_traces, shared_logs) plus a manifest. The optimal bundle is just the run directory itself — flat, no copies, no subdirectories named after their source location. The archive IS the run directory.

8. Content-Addressed Sources (keep as-is)

The content-addressed source storage is already optimal. No changes needed.

sources/{category}/{product}/{hash12}/
│
├── page.html.gz          ← gzipped HTML (deduped: same content = same dir)
├── screenshot.{ext}      ← fullpage or selector screenshot
├── ldjson.json           ← JSON-LD structured data
├── embedded_state.json   ← JS/React state objects
├── dom_snippet.html      ← key DOM fragment
└── {filename}.pdf        ← extracted PDFs

SQL metadata (crawl_sources table):
  content_hash → directory key
  source_url → original URL
  file_path → relative path to page.html.gz
  has_screenshot, has_pdf, has_ldjson → boolean flags
  http_status, size_bytes, doc_kind, source_tier

SQL screenshot metadata (source_screenshots table):
  screenshot_id, file_path, format, dimensions

SQL PDF metadata (source_pdfs table):
  pdf_id, file_path, filename, pages_scanned
Why this works: Binary files can't go in SQLite (too large, not queryable). Content-addressing means the same page crawled in 3 different runs only stores 1 copy. SQL has all the metadata to find and serve any artifact by content_hash. This is the correct architecture for binary artifacts.
