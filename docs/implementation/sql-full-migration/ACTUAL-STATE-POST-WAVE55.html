1. The Run/Product Split (Wave 5.5)

All data is now cleanly separated into two lifetimes: run telemetry (ephemeral, GUI-only) captured in run-summary.json, and product knowledge (permanent, grows across runs) in SQL.

2. Write Paths During a Run

SQL is the sole structured write target during execution. Zero JSON files written until finalize.

Wave 5.5 achievement: Zero JSON files written during startup or mid-run. The sole structured file created is run-summary.json at finalize, which captures all GUI telemetry in one shot. Product knowledge (needset, search_profile, brand_resolution, candidates, evidence) stays in SQL forever. bridge_events SQL rows are purged after capture. The runs table was slimmed from 22 to 14 columns.

5. run-summary.json Schema (v1)

run-summary.json (~150KB-1MB)

schema_version: 1

telemetry.meta: {
  run_id, category, product_id, status, started_at, ended_at,
  phase_cursor, boot_step, boot_progress,
  identity_fingerprint, identity_lock_status, dedupe_mode,
  s3key, out_root, counters, stages, startup_ms,
  browser_pool, needset_summary, search_profile_summary, artifacts
}

telemetry.events: [ ...raw bridge_events rows ]
// 500-2000 events, same shape as getBridgeEventsByRunId returns

telemetry.llm_agg: {
  total_calls, completed_calls, failed_calls, active_calls,
  total_prompt_tokens, total_completion_tokens, total_cost,
  calls_by_type, calls_by_model
}

telemetry.observability: {
  search_finish_without_start, search_slot_reuse, search_unique_slots,
  llm_missing_telemetry, llm_orphan_finish,
  bridge_event_errors, bridge_finalize_errors
}
GUI read path: readRunSummaryEvents() tries SQL run_artifacts (artifact_type='run_summary') first, then run-summary.json file, then falls back to bridge_events SQL for pre-migration runs. All 18 builder functions receive the same events[] array regardless of source.

6. What's Eliminated

Killed by Wave 5.5:
• run.json — no longer written at startup or finalize (replaced by run-summary.json)
• needset.json — no longer written at startup (data in run_artifacts SQL)
• search_profile.json — no longer written at startup (data in run_artifacts SQL)
• bridge_events SQL rows for completed runs — purged after capture in run-summary.json
• 9 columns dropped from runs table (boot_step, boot_progress, startup_ms, stages, browser_pool, needset_summary, search_profile_summary, artifacts, extra)
• writeJson: true flag eliminated from ensureRun() and finalize()
Total I/O elimination across all waves: ~3,000-5,000 file I/O operations eliminated per run. The only file written during execution is content-addressed binaries (page.html.gz, screenshots). The only structured file created per run is run-summary.json at finalize.
