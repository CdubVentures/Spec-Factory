Two-Gate Navigation Agent

Post-extraction intelligence for crawled pages. Local Nemotron triages extraction gaps. Local Qwen navigates pages that need deeper interaction via Web MCP.

All models local (vLLM / llama.cpp) Gate 1: Nemotron (fast classifier) Gate 2: Qwen 3.5 27B (visual agent) Web MCP: browser tool interface

1 The Problem

The current crawl pipeline is blind to page structure. Every URL gets the same treatment: load, dismiss overlays, scroll, grab HTML. High-value content hiding behind interactive elements is systematically missed.

🔌 Specs Behind Tabs

Manufacturer sites (Razer, Corsair, ASUS, Logitech) hide technical specifications behind a "Tech Specs" or "Specifications" tab. The current pipeline never clicks it. The extraction gets the marketing overview, not the spec table.

🎨 Image Carousels

Product cutout images, 360-degree views, and detail shots live in JavaScript carousels with 4-12 slides. The pipeline captures only whatever slide is visible on load. Razer's gallery, Corsair's hero carousel, Logitech's product shots are all missed.

🔍 Collapsed Sections

Amazon "See more product details", ASUS "Full Specifications" accordions, SteelSeries nested spec groups. Content exists in the DOM but is hidden. DOM expansion plugin only clicks elements matching hardcoded selectors.

⚠ XHR-Loaded Data

Some sites load spec data via XHR only when a tab is clicked (SteelSeries, Dell). Content is not in the initial HTML at all. No amount of CSS override will reveal it. It requires actual page interaction to trigger the fetch.

2 Two-Gate Architecture

Don't burn LLM inference on every page. Run normal extraction first. Then use a fast local model to triage whether the page has untapped content. Only pages with gaps get the expensive interactive agent.

Input Signal (No Raw HTML)

Nemotron doesn't see the full HTML. It receives a lightweight extraction gap report — a structured summary built by a pure-function DOM scanner running in page.evaluate() (~5ms, zero network).

// Extraction gap report — ~200 tokens to Nemotron { "product": "Razer DeathAdder V3 Pro", "category": "mouse", "extraction": { "fields_total": 18, "fields_filled": 12, "low_confidence": ["dpi_range", "switch_type", "battery_life"], "missing": ["weight", "dimensions", "sensor", "polling_rate", "cable_type", "rgb_zones"] }, "page_structure": { "tabs": ["Overview", "Tech Specs", "Gallery", "Reviews"], "active_tab": "Overview", "carousels": 2, "carousel_slides": [8, 4], "collapsed_sections": 3, "collapsed_labels": ["Full Specifications", "In The Box", "Compatibility"], "hidden_with_content": 14 } }

Output Decision

// Nemotron returns — ~50 tokens { "navigate": true, "reason": "6 missing fields likely behind Tech Specs tab; 8-slide carousel has product images", "targets": [ { "type": "tab", "label": "Tech Specs", "priority": 1 }, { "type": "carousel", "index": 0, "priority": 2 }, { "type": "expand", "label": "Full Specifications", "priority": 3 } ] }

HTML Signal Scanner (Pre-Nemotron, No LLM)

A pure-function page.evaluate() call extracts structural hints from the live DOM. Patterns reuse existing logic from domExpansionPlugin and overlayDismissalPlugin.

Detects:
  • [role="tab"], .tab, [data-tab] — tab elements + active state
  • .swiper-slide, .carousel-item, [data-slide] — carousel slide counts
  • [aria-expanded="false"], details:not([open]) — collapsed sections with labels
  • display:none elements with >100 chars content inside
  • "Show all specs" / "View full specifications" type buttons
Performance:
  • Single page.evaluate() call
  • ~5ms execution time
  • Zero network cost
  • Runs after normal extraction completes
  • Reuses selector patterns from existing plugins
💡
Why Nemotron for this: This is structured-data-in, binary-decision-out. No vision, no multi-step reasoning. A fast quantized model handles this at <100ms. The extraction gap report is the same shape every time — Nemotron learns the pattern instantly.

Agent Loop

Qwen receives the page screenshot + Nemotron's target list. It executes MCP tool calls in a tight loop until targets are exhausted or budget is spent.

// Example: Razer product page — carousel + spec tab // Step 1: Qwen sees page screenshot + targets Context: Product "Razer DeathAdder V3 Pro" (mouse) Targets: [tab: "Tech Specs", carousel: 0 (8 slides)] Budget: 8 actions, 15 seconds // Step 2: Qwen clicks the spec tab mcp.click('[data-tab="tech-specs"]') mcp.wait(500) // let XHR-loaded content render mcp.screenshot() // capture spec table // Step 3: Qwen cycles the carousel mcp.click('.carousel-next') mcp.screenshot() // slide 2 mcp.click('.carousel-next') mcp.screenshot() // slide 3 // ... continues until carousel wraps // Step 4: Qwen signals completion mcp.done()

MCP Tool Interface

Web MCP wraps the existing Playwright page object as tool calls. Qwen calls them like any MCP tool. The MCP server runs locally.

ToolSignatureWhat It Does
screenshot () → jpeg Captures current page state as JPEG. Fed back to Qwen for next decision. Also persisted as extraction artifact.
click (selector) → bool Clicks an element. Navigation-safe — blocks page navigation, only allows in-page actions.
scroll (direction, px) → void Scrolls the page. Used when content is below the fold.
wait (ms) → void Waits for content to render (XHR responses, animations). Capped at 2 seconds.
extract_text (selector?) → string Gets text content of an element or full page. Cheaper than screenshot for text data.
done () → void Signals that all targets are addressed. Returns to normal pipeline.

MCP as Standalone Service — Why

🔄 Independent Update Cycle

Web MCP server updates (new tools, bug fixes, Playwright version bumps) ship independently. No crawl pipeline code changes needed. The harness calls tools/list at connect time and gets whatever tools the current MCP version exposes.

🛡 Process Isolation

MCP server runs its own process with its own Playwright context. If the agent loop crashes, hangs, or OOMs, the main crawl session is unaffected. The plugin catches the error and falls back to the normal extraction result.

⚡ On-Demand Only

~70-85% of pages skip Gate 2 entirely. The MCP server only spins up when Nemotron says navigate: true. No idle resource consumption. Spawn on first Gate 2 trigger per batch, reuse across the batch, tear down after.

🔧 Tool Config is SSOT

The MCP server's tool definitions are the single source of truth. Your harness discovers tools dynamically via tools/list. If Web MCP adds hover, select_option, or drag — Qwen picks them up automatically on next connect. Zero code changes.

⚠️
MCP protocol clarification: MCP is a transport protocol, not an LLM integration layer. The MCP server exposes tool schemas via tools/list (JSON-RPC 2.0). Your app (qwenAgentLoop.js) is responsible for:
  • Calling tools/list to discover available tools at connect time
  • Formatting those tool schemas into Qwen's native tool-use format (OpenAI-compatible tools param via vLLM)
  • Parsing Qwen's tool-call responses and routing them back through tools/call to the MCP server
  • Encoding MCP results (screenshots as base64 images) and feeding them back to Qwen as the next turn
MCP gives you the clean tool interface. Your agent harness wires it to the model.

Local Process Layout

# ── Always-on (started with the app) ────────────────────── # Nemotron — lightweight triage model, handles every URL vLLM serve nemotron-model --port 8001 --quantization awq # ~4-8GB VRAM, sub-100ms inference, always running # Qwen 3.5 27B — visual agent model, handles ~15-30% of URLs vLLM serve qwen3.5-27b-instruct-awq --port 8002 --quantization awq # ~18GB VRAM, 500ms-1s per tool-use step, always running # ── On-demand (spawned when Gate 2 triggers) ───────────── # Web MCP server — standalone browser agent service # Spawned by mcpNavigatorPlugin on first Gate 2 trigger per batch # Owns its own Playwright context (not Crawlee's) # Receives target URL, launches browser, exposes tools via MCP # Torn down after batch completes or idle timeout web-mcp-server --transport stdio # OR: web-mcp-server --port 8003 --transport http (persistent mode)

Tool Discovery → LLM Prompt Flow

// qwenAgentLoop.js — the bridge between MCP and Qwen // 1. Connect to MCP server, discover tools const mcpClient = await connectMcp('stdio', mcpServerProcess); const { tools } = await mcpClient.request('tools/list'); // 2. Convert MCP tool schemas → Qwen's OpenAI-compatible format const qwenTools = tools.map(mcpToolToOpenAiTool); // { name, description, inputSchema } → { type:"function", function: { name, description, parameters } } // 3. Build Qwen prompt with tools + screenshot + mission const messages = [ { role: 'system', content: systemPrompt }, { role: 'user', content: [ { type: 'text', text: `Product: ${product}. Targets: ${targets}` }, { type: 'image_url', image_url: { url: `data:image/jpeg;base64,${screenshot}` } } ]} ]; // 4. Agent loop — Qwen decides, MCP executes while (budget > 0) { const response = await vllmChat({ messages, tools: qwenTools }); if (response.finish_reason === 'tool_calls') { for (const call of response.tool_calls) { // Route tool call through MCP protocol const result = await mcpClient.request('tools/call', { name: call.function.name, arguments: JSON.parse(call.function.arguments) }); // Feed result back to Qwen (screenshots become image messages) messages.push(formatToolResult(call, result)); } budget--; } else { break; // Qwen said "done" or returned text } }

8 Rollout Phases

Incremental rollout. Each phase is independently useful and fully reversible.

Phase 1 — HTML Signal Scanner + Telemetry
Build htmlSignalScanner.js and wire it into the onCapture hook. Log signals to bridge_events but don't call any LLM. Collect data on how many pages have tabs, carousels, collapsed sections. Validate that the scanner produces useful signals before investing in the LLM integration.
Phase 2 — Gate 1 (Nemotron Triage)
Stand up local Nemotron. Wire nemotronClient.js into the plugin. Log navigate decisions but don't act on them yet. Compare Nemotron's "should navigate" predictions against the Phase 1 signal data. Tune the prompt and thresholds.
Phase 3 — Gate 2 (Qwen Agent via Standalone MCP)
Stand up local Qwen + standalone Web MCP server in tools/mcp-web-server/. Build src/features/navigation/ feature boundary: qwenAgentLoop.js, mcpLifecycle.js, toolSchemaAdapter.js. MCP server spawned on-demand via stdio. Enable on a single test category first. Measure: additional fields extracted, screenshots captured, time overhead per URL. Compare extraction quality with and without Gate 2.
Phase 4 — Confidence Feedback Loop
Feed post-Gate-2 extraction confidence back to Nemotron. If Gate 2 consistently improves extraction for certain page patterns, Nemotron learns which signals predict high-value navigation targets. Tighten the gate over time — fewer false positives, more targeted Qwen usage.
