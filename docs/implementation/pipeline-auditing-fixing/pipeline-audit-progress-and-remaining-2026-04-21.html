Session 2 Plan 1 — Tier System Fix (B4 + B8)

Root cause

Live searchProfileQueryCap=10 + 10 seed domains on MM731 → seeds consumed 100% of the shared cap, so buildTier2Queries and buildTier3Queries emitted zero rows. Simultaneously, query_index had two writers per external query: executeSearchQueries.js:212 (no tier) and the bootstrapRunEventIndexing.js event hook on discovery_query_completed (with tier). Result: 50% of query_index rows had tier=NULL.

What shipped

  • Per-tier budget floors — 3 new runtime knobs in src/shared/settingsRegistry.js:
    • tier1SeedCap (default 10)
    • tier2GroupCap (default 10)
    • tier3KeyCap (default 20)
    All under uiCategory: "planner" / uiSection: "search-profile" / uiGroup: "Tier Budgets".
  • Budget allocation rewrite in src/features/indexing/pipeline/searchProfile/queryBuilder.js. Each tier slices to its own cap; searchProfileQueryCap stays as the overall ceiling. New helper applyOverallCapWithFloors() guarantees at least one slot per tier with eligible rows when the overall cap is tight — so tier 2/3 can never be starved.
  • query_index single-writer — removed the direct insertQueryIndexEntry call at executeSearchQueries.js:212-220. The bootstrapRunEventIndexing.js hook on discovery_query_completed is now the sole writer, and it carries tier/hint_source/group_key/normalized_key. Fixes B8.
  • focus_groups run_artifact — new artifact type persisted at runNeedSet.js via focus_groups_computed event + bridge handler handleFocusGroupsComputed in src/indexlab/runtimeBridgeEventHandlers.js. Captures the focusGroups[] array (with group_search_worthy, productivity_score, normalized_key_queue) that was previously in-memory-only.
  • tier_counts telemetry on search_plan_generated: {seed: N, group_search: N, key_search: N}. Computed in runSearchPlanner.js, threaded through bridge passthrough.

Session 2 Plan 2 — Enrichment Progression Wire-Up

Root cause

src/indexlab/buildFieldHistories.js was orphaned (flagged as DEFER in docs/implementation/archive/src-audit/src-audit.html:563). No production code called it, so roundContext.previousFieldHistories arrived at NeedSet as {} every run. At needsetEngine.js:339, deriveFieldHistory({round: 0}) discards whatever's in previousFieldHistories and resets to zeros — so history.query_count stayed 0, buildNormalizedKeyQueue emitted repeat_count: 0, and buildTier3Queries never applied any enricher. Every Tier 3 query stuck at 3a forever.

What shipped

  • Bridge fieldProvenance accumulator — new this.fieldProvenance = {} on the bridge (src/indexlab/runtimeBridge.js constructor). handleSourceProcessed in runtimeBridgeEventHandlers.js now derives host_class and evidence_class via classifyHostClass/classifyEvidenceClass (already exported from buildFieldHistories.js — no duplication) and appends one evidence entry per (source, field) pair.
  • Loader — new src/indexlab/loadPriorFieldHistories.js (35 LOC). Queries runs for the most-recent completed run for a given product_id, fetches run_artifacts.artifact_type='field_histories', returns payload or {}.
  • Finalization helper — new src/indexlab/finalizeFieldHistories.js (70 LOC). Extracts enhancement_rows (preferred over queries_generated, which is strings-only) from the run summary, calls buildFieldHistories({previousFieldHistories, provenance, searchPlanQueries, duplicatesSuppressed}), and upserts the result as run_artifacts.artifact_type='field_histories'.
  • roundContext injection in src/app/cli/commands/pipelineCommands.js. Before runProduct(): load prior histories via the loader, construct roundContext = {previousFieldHistories, round: hasPrior ? 1 : 0}, pass to runProduct. The round=1 rule is required because deriveFieldHistory at round=0 discards prev entirely.
  • Finalization call — after serializeRunSummary() and before buildProductCheckpoint, call finalizeFieldHistories and thread the result into buildProductCheckpoint({..., fieldHistories}).
  • Product checkpoint snapshot — src/pipeline/checkpoint/buildProductCheckpoint.js now accepts fieldHistories param and writes field_histories into product.json (sibling of query_cooldowns).
  • Rebuild Contract — src/pipeline/checkpoint/seedFromCheckpoint.js::seedProductCheckpoint rehydrates the field_histories artifact from product.json on DB deletion. Matches the existing query_cooldowns rebuild path. Deleting spec.sqlite and re-booting restores the artifact.

Remaining Open Bugs (from original audit)

B6 — product.json::sources[] null metadata OPEN

Entries in sources[] carry null for hint_source, tier, and provider. Schema pass-through bug somewhere between crawl output and mapProductSource() (src/pipeline/checkpoint/buildProductCheckpoint.js:12-31). Current mapper doesn't read/forward these fields even if upstream provides them.

Scope: small. Trace from source_processed event → crawlResults → mapProductSource(), identify where these fields drop. Likely a 2-3 line fix + a rebuild test.

B9 — query_index.provider CSV-joined OPEN

Rows show provider: 'bing,google' instead of one row per provider. Cause: discovery_query_completed event payload reports the engine config string rather than per-result provider identity. Audit-only concern (downstream analytics lose provider granularity).

Scope: medium. Either split one logical query into N rows (one per provider that returned results) or add a providers_used[] array column. Needs schema decision.

B12 — runs.counters empty OPEN

runs.counters is JSON-text with run-end counts (pages_checked, fetched_ok, indexed_docs, etc.). Currently the field is written as {} on most runs. Bridge has state.counters populated during the run but the snapshot at finalization isn't landing in the row.

Scope: small. Trace runs row update during finalization. Likely missing a counters: bridge.counters on an upsertRun call in writeRunMeta or bridge.finalize.

B13 — Server crash res.setHeader is not a function OPEN

GUI server crashes mid-long-run with this exact error. Hit in Session 2 between Run A and Run B. Unrelated to pipeline code — lives in the GUI request handler. Needs a stack trace with the current Node version to identify the specific middleware.

Scope: medium. Repro, capture stack, likely a middleware that wraps res without preserving prototype methods. Possibly related to SSE endpoint lifecycle.

B14 — Orphan runs.status='running' OPEN

Rows in runs with status='running' never transition when the process dies uncleanly (crash, kill, timeout). GUI still shows them as in-progress. Needs a boot-time sweep that marks any stale running rows as aborted (or similar) if the run directory has no recent activity.

Scope: small. One SQL statement on boot.

Newly Surfaced Findings (from the E2E verification)

1. source_processed.candidates empty in production NEW

The bridge's fieldProvenance accumulator works correctly (7/7 unit tests pass with synthetic events), but on live MM731 runs the candidates array on source_processed events is []. That means the accumulator populates nothing from real sources. The resulting field_histories artifact has query_count per field (because queries are targeting fields via enhancement_rows.target_fields), but domains_tried/host_classes_tried/evidence_classes_tried stay empty.

What this means: the enrichment works at the query level but not at the evidence level. Tier 3 queries carry alias/domain/content-type enrichment based on repeat_count, but the "don't try domains we've already hit" filter doesn't activate because domains_tried stays empty.

Where to look: the extraction layer that should populate candidates before emitting source_processed. Likely src/features/indexing/pipeline/resultProcessing/ or the crawl result → event payload translator.

2. LLM flattens progressive enrichment NEW

Run F had repeat_count=2 on side buttons (Tier 3c — should include aliases + domain_hints). The deterministic template at queryBuilder.js::buildTier3Queries produced the enriched string, but the LLM output after enhanceQueryRows was simply "side buttons count" — no alias/domain_hint visible. The LLM is deciding the enriched version is "less natural" and simplifying.

Concrete example from Run F:

  • Deterministic (input to LLM): Cooler Master MM731 side buttons thumb-buttons forward-back rtings.com
  • LLM output (executed): Cooler Master MM731 side buttons count

Where to look: queryPlannerLlmAdapter.js prompt. Current HISTORY AWARENESS block overrides TIER 1 seed unchanged rule, but doesn't instruct the LLM to preserve progressive enrichment tokens on TIER 3 rows. Should add: "For TIER 3 rows with repeat_count > 0, PRESERVE the alias/domain_hint/content_type tokens in the deterministic query; only rewrite the framing around them."

3. deterministic_query_rows missing from search_profile artifact NEW

runQueryJourney.js:163 assigns deterministic_query_rows: toArray(searchProfileBase?.query_rows) to searchProfilePlanned, but the written search_profile artifact for Run A (20260421072221-c9ecf4) does NOT contain this key. Some earlier write path in the bridge (writeSearchProfile in runtimeBridgeArtifacts.js) is writing the artifact before the journey phase completes, and the later journey output doesn't overwrite with the full payload.

Scope: diagnostic only. Not blocking, but makes debugging tier emission harder post-run. Either: (a) write a second search_profile_v2 artifact at journey completion, or (b) fix the overwrite so the journey output lands.

4. Lagging groups restart their own ladder NEW

In Run G, these groups still had repeat_count=0 across their normalized_key_queue: sensor_performance, switches, electronics, encoder. They became non-worthy at different rounds than the leading groups, so their 3a → 3d ladder starts later.

Not a bug — that's how group worthiness works (coverageRatio >= 0.80 or unresolvedCount < 3 flips a group to non-worthy, triggering Tier 3 on its keys). But worth documenting so reviewers don't expect uniform progression.

URL-dedup bug — Razer 6×-same-page

Run 5 (Razer DeathAdder) used 6 of its 9 URL slots on razer.com/gaming-mice/razer-deathadder-v3-hyperspeed?srsltid=AfmBOo{different token} — the same page rendered 6 times with different Google Merchant Center tracking tokens.

Root cause: crawlSession.js:765 builds uniqueKey = \`${url}::${Date.now()}::${Math.random()}\` — every URL is guaranteed non-identical. No srsltid / utm_* / locale stripping anywhere upstream or downstream.

Also shows up as: Dell runs fetched /en-us/, /en-au/, /en-tc/, /en-sr/, /en-vg/, /en-ag/, /en-pw/ variants of the same monitor page. Corsair run fetched corsair.com/us/, /ww/, /eu/ mirrors of the same SKU.

srsltid is specifically Google Merchant Center's "Search Result Source Listing ID" tag (SEL) — it's widely acknowledged to break rel=canonical. Always strip.

CSS override + overlay dismissal — collateral damage

cssOverridePlugin.js:10-28 is overbroad.

  • HIDDEN_SELECTOR targets [aria-hidden="true"] — that's the a11y convention for decorative icons, collapsed accordions, off-screen tabs, inactive carousel slides. Forcing all of it visible with display:block !important can render hidden carousel slides on top of active ones and expand every accordion at once.
  • FIXED_STICKY_CSS flips position: fixed/sticky → absolute. Un-pins nav bars and floating CTAs — they drop into DOM-order and shift content.
  • .collapse:not(.show) targets Bootstrap collapses — reveals hidden tabs/panes indiscriminately.

overlayDismissalPlugin miss modes:

  • overlayDismissalPlugin.js:14-38: hand-maintained [class*="..."] fragment-match list. Custom modal classes not in the list → never touched.
  • CSS rule sets z-index: -9999 !important but in a transform/filter stacking context this can still leave the element visible — Razer consent banner still appeared in the screenshot I sampled.
  • MutationObserver (lines 66-98) checks inline style="" only. Overlays positioned via CSS classes never trigger it.
  • Coverage threshold = 30% of viewport. Narrow sticky banners / floating newsletter CTAs don't qualify and slip through.

Cookie plugin: 200ms autoconsent fast-fail then fixed selector list (cookieConsentPlugin.js:12-49). Hit rate on listed CMPs is good; miss rate on custom banners is unbounded.

Screenshot quality — crops broken, pages captured too early

  • Crops: screenshotCapture.js:66 uses page.$(selector) which returns first match only. No scroll-to-element before crop, no multi-match, no stitching. Result: Corsair runs capture just the first table row ("Mouse Warranty / 2 Year") as an 8-18KB strip; 40% of my runs produced zero crop files.
  • Pages captured too early: page.goto uses waitUntil: 'domcontentloaded' (crawlSession.js:691). pageStabilizer.js waits for document.fonts.ready + image decode (1500ms cap) but does not wait for a content landmark. Dell support pages (dell.com/support/product-details/…) render in ~1s in a normal browser but we capture before the JS-driven body content paints — screenshot is mostly whitespace with only the top nav visible.
  • quality_score = 0 on every visual_asset_captured event. Score isn't being computed.
  • Wrong-SKU capture: run 9 requested AW3423DWF (FreeSync), SERP returned and we captured AW3423DW (the older G-Sync variant). SERP-selector LLM doesn't reject obvious SKU mismatches.

How it compares to modern scraping

playwright-extra + stealth was deprecated Feb 2025 and no longer beats current Cloudflare. FlareSolverr still works (~55-70% success) but degrades at scale. Commercial unlock APIs (ScraperAPI, ZenRows, Bright Data) hit 80-95% on mixed-difficulty lists, mobile-proxy tiers 99.95% (Scrapfly 2026, ScrapeOps).

"AI companies scraped the whole web" is a misleading benchmark: Cloudflare's 2025 crawler report confirms GPTBot, ClaudeBot, PerplexityBot don't render JS — they're closer to wget. They achieve "near-perfect" rates on the open web partly because sites whitelist them.

74% on a mixed set including Cloudflare-protected sites is middle-of-the-road. Getting to 90%+ without per-request unlock APIs is hard. Webshare rotating-residential (already wired, pay-per-GB, ~$1-3/GB) solves geo-IP blocks but not JS challenges. That's the ceiling of what's cheap.
