1Overview & Goals

Problem

Today, clicking RUN NOW, LOOP, VIEW, HERO, or EVAL in IndexLab fires the LLM call immediately. The exact system prompt, user message, schema, model, effort, and image attachments are only visible after the fact inside Run History. There is no way to read or diff the compiled prompt before it is sent.

This roadmap adds a Prompt Preview affordance next to every run trigger — letting the user inspect exactly what would be dispatched, without dispatching it.

Scope (per feature)
FinderPrompt Trigger PointsUI Affordance
CEF
Phase 1		 One prompt (product-level discovery) Single Prompt button left of RUN NOW in section header → modal popup
PIF
Phase 2		 4 distinct prompts per variant: View, Hero, Loop (multi-sub-prompt), Eval (view-eval + hero-eval) Chevron › on each variant row inside ALL IMAGES → expands horizontally to the left revealing 4 inline buttons VIEW HERO LOOP EVAL. Clicking any opens the same PromptPreviewModal from Phase 1. Loop modal has sub-tabs for each view + hero sub-prompt.
RDF
Phase 3		 One prompt template; Run and Loop use identical template (Loop iterates) Chevron on each variant row → expands left revealing 2 inline buttons RUN LOOP. Clicking either opens PromptPreviewModal.
SKU
Phase 3		 Same as RDF (single template, Loop re-runs it) Identical chevron + 2-button reveal; same modal.
Non-Goals (explicitly out of scope)
  • No editing prompts in the preview — strictly read-only view of what would be sent.
  • No saving / snapshotting preview output — the preview is computed on demand.
  • No side-by-side diffing against last-run prompt (may be a later phase).
  • No per-prompt-fragment editing — prompt override UX lives elsewhere (global prompt registry).
  • No new LLM calls from the preview endpoint — zero network egress to the model provider.

2Shared Backend Contract

Each of the three phases stands on the same backend primitive: a compile-only preview endpoint that reuses existing prompt builders but stops short of invoking the LLM.

2.2 New endpoint: POST /:prefix/:category/:productId/preview-prompt

Registered generically inside src/core/finder/finderRoutes.js next to the existing /loop handler (lines 292–414). Each finder's register*FinderRoutes.js opts in via a new preview config key on the registerScalarFinder / finder-module registration shape.

Request body (discriminated by finder):

// RDF / SKU / CEF (scalar)
{ variant_key?: "Black", mode?: "run" | "loop" }

// PIF (vision, multi-prompt)
{ variant_key: "Black",
  variant_id?: "uuid",
  mode: "view" | "hero" | "view-eval" | "hero-eval" | "loop",
  view?: "top" | "angle" | "side" | "sangle" | ...
}

Response body:

{
  finder: "cef" | "pif" | "rdf" | "sku",
  mode: "run" | "loop" | "view" | ...,
  compiled_at: 1713623400000,
  prompts: [
    {
      label: "view:top",            // for display (e.g. "Run 1 — top view")
      system: "You are ...",         // full compiled system prompt
      user: "{ \"brand\": ... }",      // JSON.stringified user message
      schema: { ... },                 // llm json schema (zodToLlmSchema output)
      model: {
        id: "gpt-5.4",
        thinking_effort: "high",
        web_search: true,
        json_strict: true
      },
      images?: [                        // only for view-eval / hero-eval
        { url: "...", thumb_base64_size: 8123 }
      ],
      notes: [                         // human-readable context
        "previousDiscovery injected: 12 urls, 4 queries",
        "ambiguity tier: medium (3 sibling models)"
      ]
    }
  ],
  inputs_resolved: {                   // summary of where each arg came from
    product_id: "mouse-1d9...",
    variant_key: "Black",
    variant_id: "...",
    family_model_count: 3,
    ambiguity_level: "medium",
    sibling_models: ["M65", "Scimitar"]
  }
}

Key invariants:

  • No LLM dispatch. Preview endpoint never calls callRoutedLlmFn. No operations are registered, no WS events fire, no run is persisted.
  • No mutations. Zero writes to product.json, SQL, snapshots, or logs.
  • Same arg-resolution as real run. Uses the same resolvePhaseModel, buildKnownInputs, buildFamilyContext, evidence fragments, previous-discovery accumulation. Preview output must be byte-identical to what a real run would dispatch right now.
  • Byte-identical guarantee is testable. New integration test asserts preview response matches the prompt_snapshot captured from a real run fired moments later.

2.3 Loop semantics per finder

FinderLoop = N prompts?Preview response shape
CEF Single prompt (no Loop mode in UI) prompts[] has length 1
RDF Loop re-fires the same template, accumulating previousDiscovery Loop preview = prompts[] length 1, labeled "run-loop-iter-1". Preview shows iteration #1 only; subsequent iterations depend on runtime response and cannot be pre-compiled.
SKU Same as RDF Same as RDF
PIF view Single view-targeted prompt prompts[] length 1
PIF hero Single hero prompt prompts[] length 1
PIF Loop N prompts: one per unfilled view + hero (if enabled), per runCarouselLoop prompts[] contains the first attempt for every view in the view budget + hero, labeled "view:top", "view:angle", ..., "hero". Retries are not previewed (they depend on response).
PIF view-eval / hero-eval Vision call against current carousel candidates prompts[] length 1, includes images[] metadata (URL + thumbnail byte count, NOT raw base64, to keep response small)
Design note — Loop determinism Loop preview shows only the first iteration's prompt for scalar finders, and only the initial attempt set for PIF Loop. Later iterations are response-dependent. The preview UI will label this clearly: "Iteration 1 of N — subsequent prompts depend on LLM responses." This is acceptable because the user is validating the starting prompt state, not the full search trajectory.

2.4 CEF gap — full prompt is not captured today

One prerequisite fix CEF's run record stores prompt: {} (empty) today (colorEditionFinder.js:168), while RDF/SKU capture { system, user }. Phase 1 includes a one-line fix to make CEF capture the same shape. Two benefits: (a) the preview/replay parity test has something to compare against, (b) existing Run History prompt detail panel becomes useful for CEF.

2.5 Registry-driven wiring (O(1) feature scaling)

Per the O(1) mandate in CLAUDE.md, the preview endpoint must not require per-finder boilerplate. Each finder's existing registration in src/core/finder/finderModuleRegistry.js already exports buildPrompt and createCallLlm. We add two new optional exports to that shape:

{
  id: 'releaseDateFinder',
  phase: 'releaseDateFinder',
  buildPrompt,
  createCallLlm,
  previewModes: ['run', 'loop'],          // NEW
  compilePreview: async(ctx) => Prompt[],   // NEW (default: uses buildPrompt)
  store
}

A generic registerPreviewRoute(router, moduleId) helper in src/core/finder/previewRoutes.js reads the registry, resolves inputs via the same helpers used by the real run, calls compilePreview, and returns the response envelope. Adding a new finder means populating previewModes and (optionally) overriding compilePreview — no new route files.

5.3 Loop sub-tab generation

Backend: compilePreview for PIF Loop walks the same logic as runCarouselLoop (src/features/product-image/productImageFinder.js:909) but stops after building prompts:

  1. Resolve variant's unfilled views from current image store state.
  2. For each unfilled view in viewBudget: call buildProductImageFinderPrompt with that view focused + ADDITIONAL_VIEWS hints. Push to prompts[].
  3. If heroEnabled: call buildHeroImageFinderPrompt. Push to prompts[].
  4. Return array. Do not call LLM. Do not update image store.

Each prompt is labeled (view:top, view:angle, ..., hero) and rendered as a sub-tab inside the Loop tab.

5.4 Eval preview — candidate image handling

Carousel state dependency Eval prompts inject candidate images (createThumbnailBase64). Preview must NOT re-download or re-thumbnail — that would be a real side effect. Instead, compilePreview reads the current carousel candidate list from productImageStore, builds the prompt with placeholder text for image positions ("[image 1: <1024x1024, 8.1KB> <url>]"), and returns images[] metadata separately. The UI renders an image-list summary card next to the prompt block, not the raw base64. If the carousel has 0 candidates, preview returns the prompt with an empty image list and a note: "Eval cannot run — no candidates in carousel."

7.2 Byte-identical preview-vs-run parity test (the critical one)

Per finder, per mode, the integration test:

  1. Fixes product state to a deterministic fixture (brand, model, previous runs, palette, carousel candidates, etc.).
  2. Calls the preview endpoint and stores the response's prompts[0].
  3. Calls the real finder orchestrator with an LLM mock that captures the dispatched system, user, schema, model.
  4. Asserts deepEqual on all 4 fields. Any drift fails the test.

This is the single most important regression safety net. It ensures preview never misleads the user.

7.3 GUI proof required for phase completion

Per CLAUDE.md Each phase ends with a light-theme GUI smoke run against a real product. Screenshot placed in .workspace/runs/ referenced in the PR. Phase 1 smoke covers CEF modal. Phase 2 covers PIF across all 4 tabs on at least one variant with hero + 2+ views. Phase 3 covers RDF and SKU per variant.

8Risks & Open Questions

Known risks
RiskMitigation
Preview drift — builder functions change but preview path diverges from real-run path over time. Byte-identical parity test per finder runs in CI. Test failure is the only allowed signal; no "tolerable" differences.
CEF empty prompt capture — today colorEditionFinder.js stores prompt: {}. Fixing this changes run record shape for existing downstream consumers. Characterization test for current CEF run record shape before the fix. Audit: grep for run.prompt access in the GUI and run-history viewers. If a consumer assumed empty, migrate it.
PIF Loop preview response size — a variant with 8 views + hero is 9 prompts × ~2KB system = ~18KB JSON. Acceptable. React Query caches; no streaming needed. Response capped at 200KB via response middleware (error if exceeded, signals budget inflation).
Eval preview thumbnail handling — base64 inflation risk. Never return raw base64. Return { url, thumb_base64_size } metadata only. Prompt body has placeholders where images would insert.
Identity / family resolution cost — preview duplicates the buildFamilyContext work. Family context is already cached per-run within the request scope. Preview reuses the same cache key (productId + version). Cost is one DB round-trip per preview click.
Secret leakage in preview — system prompts never contain secrets today, but let's not regress. Existing global prompt registry rules: no API keys, no model-credentials. Preview response passes through the same redact path used by Run History prompt display (if any exists). Add a test asserting no known secret pattern in any preview response body.
Open questions (resolve before Phase 1 MACRO-RED)
  1. Placement of the CEF Prompt button — user spec says "to the left of Run Now." Does FinderPanelHeader's current actionSlot already allow that, or do we need a new leadingActionSlot? Confirm in Phase 1 kickoff.
  2. Drawer width when expanded — PIF reveals 4 buttons, RDF/SKU reveal 2. Estimate w-72 and w-44 respectively (matching sf-shell-header-drawer's w-36 as reference scale). Confirm during Phase 2 visual review.
  3. Drawer close behavior — does opening a modal from a revealed action button auto-close the drawer, or leave it open so the user can click another mode after dismissing the modal? (Plan assumes: leave open, mirroring normal action-button behavior.)
