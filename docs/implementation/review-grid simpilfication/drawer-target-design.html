Drawer Panel — Target Design

What the CellDrawer should look like after frontend cleanup.

Core principle: No accept, no confirm. The pipeline auto-publishes. The resolved candidate IS the published value. The only user actions are override (click candidate or type value) and AI review (per-source or all-sources).

Interaction Model

  1. Click a candidate card → overrides the published value (flows through submitCandidate() with confidence 1.0)
  2. Type in Manual Override → same flow, manual entry
  3. AI Review (per source) → sends that source's data to LLM for validation. Result flows back through submitCandidate(). Future — not connected yet.
  4. AI Review (all sources) → sends all candidates for cross-reference. Future — not connected yet.

That's it. No accept step. No confirm step. No flags. No lanes.

Sections

  1. Published Value — the resolved candidate. Confidence dot + value + source. OVR badge if manual override.
  2. Manual Override — text input + Apply.
  3. Candidates — clickable cards. The published source is marked. Click any other to override.
  4. AI Review — per-source button on each candidate + "Review All Sources" button.
  5. Flags — only if empty (missing_value). No inference flags.

Props (target: ~10)

  • title, subtitle, onClose
  • currentValue (value, confidence, color, source, sourceTimestamp, overridden)
  • onOverride(candidateId, candidate) — click candidate to publish
  • onManualOverride(value) — type to publish
  • isPending
  • candidates, candidatesLoading
  • onRunAIReview(source?) — per-source or all
  • aiReviewPending
  • extraSections (flags)

Dead props to remove (~20)

  • isCurrentAccepted, onAcceptCurrent
  • onAcceptCandidate (rename to onOverride)
  • sharedAcceptedCandidateId

  • pendingPrimaryCandidateId/Ids
  • pendingSharedCandidateId/Ids

  • onAcceptPrimary/Shared
  • onAcceptPrimaryCandidate/SharedCandidate
  • candidateUiContext
  • showCandidateDebugIds
  • extraActions
  • badges (flags are dead in new model)
