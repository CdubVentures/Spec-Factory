Component Review Convergence

Unifying field review and component review under a single Override + Auto-Propagation model. No accept. No confirm. No manual intervention.

Core principle: Both field review and component review follow the same model — the pipeline auto-publishes, the resolved candidate IS the published value. The only user actions are override (click candidate or type value) and AI review (future). Constraint violations trigger auto-review, not manual flags.

1. Unified Interaction Model

Field Review Grid (already done)

  • Click cell → opens drawer
  • Click candidate → overrides published value
  • Type manual value → override
  • AI Review button (future placeholder)
  • Accept / Confirm / Flags / Keyboards

Component Review (target — to converge)

  • Click cell → opens drawer
  • Click candidate → overrides component value
  • Type manual value → override
  • AI Review button (future placeholder)
  • Accept / Confirm Shared / Pending AI badges
  • Auto-propagation on override (variance policies)
  • Auto-review on constraint violation

2. Override → Propagation Flow

When a user overrides a component property value, the system auto-propagates based on the field rule's variance_policy. No human review step.

User Action
Override component property value
Click candidate card or type manual value in component review drawer. Single mutation: POST /review-components/{category}/component-override
↓
System
Write to component_values table
SpecDb upserts the property value. Source = 'user' for manual, preserved source for candidate accept. needs_review set to false.
↓
System
Discover linked products
findProductsReferencingComponent() queries item_component_links table. Returns all product IDs using this component.
↓
Policy Gate
Evaluate variance_policy
Branch based on the property's variance policy from field rules:

Authoritative

Component value IS the product value. No ambiguity.

  1. pushAuthoritativeValueToLinkedProducts()
  2. All linked products receive the new value directly
  3. Products' field_candidates updated with resolved status
  4. No review needed — authoritative is SSOT

Example: Sensor → max_dpi. If PixArt says PAW3950 does 26000 DPI, all products using PAW3950 get max_dpi=26000.

Bounds / Range

Component value constrains product value. Violations are possible.

  1. evaluateAndFlagLinkedProducts()
  2. Each product checked: upper_bound (≤), lower_bound (≥), range (±10%)
  3. Compliant products → no action
  4. Violations → auto-flag for AI review

Example: Sensor max_dpi=26000, product claims dpi=30000. Violation → auto-review.

Constraint Gate
Evaluate cross-field constraints
If the property has constraints (e.g., "dpi <= max_dpi"), evaluate against all linked products. Constraint violations are merged with variance violations.
↓
Auto
Flag violations for auto-review
Products with variance or constraint violations are marked for AI review. No manual intervention. The AI review system (future) will re-evaluate these products and either confirm the value or suggest corrections.
↓
System
Broadcast data-change event
WebSocket data-change event sent with affected product IDs + propagation summary. UI refreshes automatically. Component review grid + field review grid both update.

3. Variance Policy Reference

Policy Behavior On Override Product Impact
authoritative Component value = product value (SSOT) Push value to all linked products Values overwritten. No review.
upper_bound Product value must be ≤ component value Evaluate all linked products Violations flagged for auto-review
lower_bound Product value must be ≥ component value Evaluate all linked products Violations flagged for auto-review
range Product value within ±10% of component value Evaluate all linked products Violations flagged for auto-review
override_allowed No constraint — any value permitted No evaluation Products marked stale (cache invalidation only)
null (none) No variance policy set No evaluation Products marked stale (cache invalidation only)

5. How Review Grid & Component Review Talk

Data flows one direction: Component → Products. Never the reverse. The field review grid shows product-level data. The component review grid shows component-level data. When a component value changes, the cascade updates products. Products never update components.

5a. Component Override → Product Update

Component Review
User overrides sensor max_dpi = 26000
Sensor PAW3950, property max_dpi, variance_policy: authoritative
↓
Cascade
Auto-push to 12 linked products
pushAuthoritativeValueToLinkedProducts() writes max_dpi=26000 to all products using PAW3950
↓
Broadcast
data-change event → both grids refresh
Field review grid sees updated max_dpi values. Component review grid sees updated propagation status.

5b. Constraint Violation → Auto-Review

Component Review
User overrides sensor max_dpi = 20000
But product "Razer Viper V3 Pro" has dpi=35000. Constraint: dpi ≤ max_dpi
↓
Constraint
Violation: 35000 > 20000
evaluateConstraintsForLinkedProducts() flags the Viper V3 Pro
↓
Auto-Review
Product flagged for AI review
No manual intervention. Future AI review system re-evaluates the product's dpi candidates and either confirms or suggests a correction.
↓
Broadcast
Both grids update
Field review grid shows the product with a constraint-violation indicator. Component review shows propagation summary.

6. Enum Review Convergence

Enum review follows the same pattern. Enum values are component-level data. The user can:

  • Override an enum value (rename, select a different candidate)
  • Add a new enum value
  • Dismiss a suggested enum value

No confirm step. No shared-lane. Changes cascade to products via cascadeEnumChange().

7. Implementation Scope

Backend (mutation routes)

  • Retire component-key-review-confirm endpoint
  • Simplify component-override — always set needs_review: false on override
  • Ensure cascadeComponentChange() sets needs_review on violated products (system-only flag)

CellDrawer (shared component)

  • Remove: isCurrentAccepted, pendingAIShared, pendingSharedCandidateIds, onConfirmShared, onConfirmSharedCandidate, candidateUiContext, showCandidateDebugIds
  • All consumers use the same simplified interface: override + AI review (future)
  • Labels: "Published" (not "Accepted"), "Override" (not "Accept")

ComponentReviewDrawer / ComponentSubTab / EnumSubTab

  • Remove confirmSharedLaneMut mutation
  • Remove pendingAIShared / pendingSharedCandidateIds props
  • Remove purple "AI Pending" dot/badge logic
  • Override mutation = the only write action
  • Propagation summary shown after override (toast or inline status)

ReviewValueCell

  • Remove pendingAIShared prop entirely
  • Standard traffic-light colors only (green/yellow/red/gray)
End state: CellDrawer becomes a ~250 LOC component with 12 props. One interaction model across all review surfaces. Variance policies and constraints are system-managed, not user-managed. AI review is the only future addition point.
