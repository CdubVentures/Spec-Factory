1 Context & design choices

Constraint: 47–73+ eligible keys per product. Flat table gets unwieldy; keys need to be navigable by group. Field Studio already organizes keys into a user-curated drag-order; the panel mirrors that shape exactly and picks up drag reorders live via the same WS chain Review Grid uses.

Three rules
  • Groups are the primary navigation. Always visible, collapsible, resolved-count in the header.
  • Action affordances at three scopes: product (top bar) · group (group header) · key (row). Each scope carries the same 4 verbs.
  • Zero drift from Field Studio: group order + key order + variant_dependent flag come from the same reviewLayout query Review Grid already uses — a drag in Studio shows up on the next WS tick.

2 Action model — 4 verbs × 3 scopes

Every scope (product / group / key) surfaces the same four actions. Scope changes what they target, not what they mean. Phases gate which ones are live.

VerbPer-keyPer-groupPer-product (all groups)
▶ Run 1 LLM call for that key. LIVE 3a Walk the group's keys easy→very_hard, 1 call each. PHASE 5 Walk all groups in order, each group walked easy→very_hard, 1 call per key. PHASE 5
∞ Loop Retry this key until resolved OR per-key budget exhausted. PHASE 3b Group walk but each key loops until resolved / budget. PHASE 5 All groups, each key loops until resolved / budget. PHASE 5
History Opens right-side drawer scoped to this key's runs + URL/query history. LIVE NOW Drawer scoped to this group — aggregates all runs across its keys. LIVE NOW Drawer scoped to the whole product — all runs, all keys, newest first. LIVE NOW
Prompt Horizontal slide-over showing system prompt + user message + latest response. LIVE NOW N/A — per-group call doesn't exist in 3a (bundle-planner is Phase 5) N/A — no product-scope prompt
What you can do today (with just 3a shipped) Per-key ▶ Run · Per-key History · Per-key Prompt · Group History · Product History. That's enough to prove the whole 3a pipeline — fire a run, inspect the prompt, watch it publish, scroll the history.

8 What's filtered out (and why)

Out of 73 authored keys on the mouse category, ~26 are not valid keyFinder targets. They're filtered out of the list before you see it. The panel doesn't include them because they're handled by a different path — but you might wonder "hey, why isn't colors here?" Two filters:

1. Variant-dependent keys Anything with field_rule.variant_dependent === true. These vary per color/edition (e.g. the blue variant is wireless, the black is wired). They're handled by the manual override drawer on variant-scoped fields, not by keyFinder. The flag comes free on every ReviewLayout row.
2. Reserved keys Owned by another finder — colors / editions (CEF), release_date (RDF), sku (SKF). Denylist lives in src/core/finder/finderExclusions.js and is already enforced at the backend. We fetch it via a tiny GET /key-finder/:cat/reserved-keys endpoint so the UI can filter them out client-side too.
Decision — do you want a way to see what was filtered?

Default: hide filtered keys entirely. You just see the 47 eligible ones. Clean.

Optional: a tiny "26 keys filtered out · show" link at the bottom of the list. Clicking it expands a small section listing the filtered keys with a reason (variant_dependent / reserved by RDF / etc.). Purely a debug aid.

My recommendation: skip it for now — if you're curious why colors isn't there, the answer is obvious (it's CEF-owned). Add it later only if users actually ask.

Five queries power the panel:

Query keyEndpointPurposeInvalidated by
['reviewLayout', cat] GET /review/:cat/layout EXISTS Group + key order + variant_dependent flag field-studio-map-saved
['kf:summary', cat, pid] GET /key-finder/:cat/:pid/summary NEW Per-key rollup (last_run, status, published, candidate_count) key-finder-run, key-finder-run-deleted, key-finder-deleted
['kf:history', cat, pid, scope, id?] GET /key-finder/:cat/:pid/history?scope=key|group|product&id=… NEW Drawer contents — runs scoped to key / group / product same as summary
['kf:prompt', cat, pid, fk, runId] GET /key-finder/:cat/:pid?field_key=X EXISTS Prompt slide-over — system prompt + user + response for one run same as summary
['kf:reserved', cat] GET /key-finder/:cat/reserved-keys NEW · ~10 LoC Client-side denylist filter long-cached

New backend work: 3 small endpoints (/summary, /history?scope=…, /reserved-keys). Everything else is wiring.

12 Scope staging — decision needed

Per feedback_stage_scope, I don't bundle downstream phases without explicit approval. Three staging options for Phase 4:

OptionWhat ships nowButtons liveEffort
A. Minimal Per-key Run + History + Prompt. Full panel structure, groups, filters, drawer, slide-over. No Loop / Group / Product action buttons rendered. Per-key Run · History (all 3 scopes) · Prompt ~2 days
B. UI-forward recommended Everything in A, plus: Loop / Run-group / Loop-group / Run-all-groups / Loop-all-groups buttons rendered disabled with phase-naming tooltips. Zero layout change when phases flip live. Per-key Run · all 3 Historys · Prompt ~2.5 days
C. Full All 4 verbs live at all scopes. Requires Phase 3b Loop + Phase 5 Smart Loop orchestrators first. Everything ~2 days panel + 3b (~2 days) + Phase 5 (~3–5 days) upstream
Why I recommend B
  1. You get a usable 3a verification dashboard immediately — Run / History / Prompt are the three things you need to prove the pipeline.
  2. UX shape is established now. When 3b + 5 land, disabled buttons become live — no redesign.
  3. Phase 3a Part 2 already 400s unsupported modes — error path is tested.
  4. Low extra cost: ~20 lines of tooltip wiring vs option A.
Ready to plan Phase 4 implementation when you approve:
  1. Pick staging option (A / B / C).
  2. Keep or drop the "show filtered-out keys" debug link? (Recommend drop for now.)
  3. Confirm 3 new backend endpoints (/summary, /history?scope=…, /reserved-keys) are acceptable Phase 4 scope.
  4. Confirm hierarchical layout (this doc) over the roadmap's flat mockup A.
