Indexing Lab — Product Picker Redesign

Audit of the current panel, design goals, and a professional redesign of the hierarchical product selector that gates every indexing run. Matches the unified .sf-panel language used by sibling panels.

1Audit — Current Product Picker src/features/indexing/panels/PickerPanel.tsx

The current picker is the gate for every indexing run, yet it reads as a rough placeholder rather than a primary surface. It uses native selects with numbered labels, hides all of the rich CatalogRow metadata the backend already exposes, and visually disconnects from the unified panel system used by Pipeline, CEF, PIF, RDF, SKU, and Key.

cosmetic1Off-system header chrome.

A bare +/− button and a plain title. Every sibling panel (Pipeline, CEF, PIF, RDF, SKU, Key) uses .sf-panel with a rail, icon tile, tip, and action cluster. The picker is the only panel that doesn't.

functional2Three native selects with "1) 2) 3)" labels.

No search, no type-ahead, no keyboard-friendly filtering. With even a few dozen products per brand the variant select becomes a scrolling nightmare. The numbered prefixes are a hack that leaks the UX wizard into the option labels.

functional3Rich product metadata is thrown away.

CatalogRow already carries confidence, coverage, fieldsFilled/fieldsTotal, lastRun, validated, hasFinal, inActive. None of it is shown. The operator has no way to know the state of the product they're about to run without navigating elsewhere first.

cosmetic4Selection echo is a muted caption.

The selected productId is buried in a grey bar below the selects. Typography, spacing, and hierarchy suggest "debug output" rather than "the thing you're about to operate on."

info5Ambiguity meter is visually orphaned.

The backend supports a 5-tier ambiguity ramp (easy → extra_hard) but the current UI reduces it to success/warning/danger chip classes. The bar is decorative; the tiers have no visible scale legend; very-hard and extra-hard collapse into the same red.

cosmeticCollapsed state tells the operator nothing.

When collapsed, the header shows Product Picker and nothing else — you have to expand to confirm what's selected. This forces an extra interaction on every pipeline visit.

functionalNo recent / pinned selections.

Operators work on the same few products repeatedly during a run session. The picker does not remember recent selections, so each visit starts from the full catalog.

2Design goals principles that drive the redesign

Native to the system

Use .sf-panel, the 3px left rail, the chevron collapse, and the icon-title-tip-actions pattern. The picker is a sibling of Pipeline/CEF/PIF — it should look like one.

Search-first, browse-second

A single fuzzy search covers brand · model · variant · product_id. The three-column drill-down is always visible for browsing but the keyboard path is / → type → Enter.

Selection is rich, not cryptic

Surface every CatalogRow field that matters: confidence, coverage, fields filled/total, last run, validated / hasFinal / active flags. The operator should trust the picker without clicking away.

Ambiguity is visible, graduated

Show the full 5-tier ambiguity ramp with a scale, a marker position, and tier-specific colors — no more collapsing vhard/xhard into the same red.

Collapsed still informs

The collapsed header is a breadcrumb: Razer › Viper V2 Pro › White. No expansion needed to verify selection.

Remembered, not retyped

Recent selections pin as chips at the bottom. One click re-selects a product the operator was working on earlier this session.

6Interaction & keyboard behavior contract

Keyboard

  • /Focus search — from anywhere inside the picker.
  • ↑ ↓Navigate results — moves within the active column.
  • →Drill in — selects highlighted row, advances to next column.
  • ←Step back — unselects current column, focuses previous.
  • EnterSelect & commit — when a full brand/model/variant row is highlighted in search.
  • EscClear search — keeps selection, clears the query.
  • ⇧ EscClear selection — full reset to State A.

Search semantics

  • Fields:matches brand · base_model · variant · product_id · identifier.
  • Ranking:exact id > prefix brand > token overlap > substring.
  • Highlighting:matched tokens get a subtle yellow wrap inside the drill-down rows.
  • Filtering:each column filters to only nodes that contain matches — counts update live.
  • Empty query:drill shows the full catalog.

Data handling

  • Source:uses existing brandOptions, modelOptions, variantOptions, selectedCatalogProduct — no new props.
  • Preview KPIs:read from CatalogRow fields already on selectedCatalogProduct.
  • Last run relative time:formatted client-side from lastRun (ISO).
  • Recents:session-scoped (in-memory store), no persistence to disk.
  • Auto-select:when variantOptions.length === 1 the panel advances automatically, preserving current behavior.

Accessibility

  • Roles:search gets role="combobox"; columns are role="listbox"; rows are role="option".
  • Live count:match count is announced via aria-live="polite".
  • Focus ring:visible 3px --pick-bg ring on search; row focus uses border-left accent.
  • Color independence:ambiguity tier is always paired with a text label, never color alone.
  • Busy state:when busy, controls are disabled + aria-busy="true"; no mid-run swaps.

7Harmony with sibling panels how the picker looks as the root panel of the page

The redesign keeps the picker visually authoritative — it's the top-most panel and the gate for everything below — while matching the rail, icon-tile, and action-cluster grammar of Pipeline, CEF, PIF, RDF, SKU, and Key. The panel accent (indigo) is close to the Key panel but distinct from every data panel.

Why indigo?The sibling panels own the finder colors: pipe=blue, cef=orange, pif=teal, rdf=purple, sku=green, key=indigo-bright. The picker uses a deeper, more neutral indigo (#364fc7, rail #4263eb) so it reads as the root / selector rather than a data producer. The shared accent family keeps the page feeling unified without stepping on any finder's identity.
Integration notes. This mockup doesn't require new props on PickerPanel beyond what IndexingPage.tsx already passes — it re-uses brandOptions, modelOptions, variantOptions, selectedCatalogProduct, selectedAmbiguityMeter. New state is search query (local useState), keyboard focus index (local), and recent-selections (session-scoped store, derived from selection mutations). The ambiguity meter's 5-tier gradient is a CSS gradient — no new data required; the existing selectedAmbiguityMeter.level maps to the marker position.
