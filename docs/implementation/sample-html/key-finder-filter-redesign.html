1 Audit — current filter bartools/gui-react/src/features/key-finder/components/KeyFinderToolbar.tsx

The existing filter bar is a single flex-wrapped row: search input, four native <select> dropdowns, a conditional Clear button, Expand-all and Collapse-all, and a Tip (?) icon.

What works

Not broken — just unpolished
  • All five filter axes are present (search · difficulty · availability · required · status) and functional.
  • Counts inline in option labels (easy (14)) tell you the filter-universe without opening dropdowns.
  • Session-persisted state per (category, productId) — filters survive tab switches.

Issues

Polish gaps
  • Flat visual hierarchy. Search, filters, and view controls (Expand/Collapse) sit in one row, competing for attention. Nothing signals "these 4 controls are one axis-group" vs. "these 2 are view actions."
  • Native selects are generic. The OS widget breaks the rest of the panel's design language (chips, cards, tags). They also hide active state — you can't skim the row and see what's filtered.
  • No result meter. When filters are active you get a dropped-row list, but the toolbar never says "Showing 12 of 47 keys." The user scrolls to find out.
  • No quick queries. 90% of real usage is "show me unresolved mandatory keys" or "show me running." That's 3 clicks across 3 selects today — should be 1 preset.
  • Clear filters appears mid-row only when active, shifting the layout. Expand/Collapse can jump a line on narrow widths.
  • Hidden semantics. Tip icon at far-right hides the "variant-dependent + reserved keys are filtered out" note. New users don't see why 26 keys are missing.
  • Keyboard affordance missing. No / focus hint, no Esc clear.
  • Difficulty color system unused. The panel elsewhere uses easy=green, medium=blue, hard=orange, very_hard=red — the filter row doesn't reflect that scale.

2 Design principles

PrincipleWhy it matters here
Hierarchy firstSearch ≠ facet filters ≠ view actions. Each role gets its own visual band so the eye groups them correctly.
Show state at a glanceActive filters should be visible without clicking any dropdown — segmented chips reveal the whole universe and what's selected.
Optimize for the 90% query"Unresolved", "Mandatory", "Running now" are the recurring questions. Promote them to 1-click presets.
Reinforce the panel design systemReuse the difficulty color scale, chip affordance, and indigo accent already used in rows and headers.
Counts everywhereEvery chip/pill carries its count. You learn the shape of the product at a glance.
Result meter is permanent"X of Y keys shown" lives in the bar, so you always know how filtered the view is.
Keyboard-first/ focuses search, Esc clears active search, click-x on a pill removes that facet.

6 Anatomy of Variant C

Three stacked .filter-row bands inside a single .filter-surface container. Bands are separated by hairline border-top, not by gaps, so the whole unit reads as one component.

Band 1 — Search & meter

  • Search input with magnifier icon + / keyboard hint; swaps to × clear when focused+dirty.
  • Result meter: N / total keys shown with mini progress bar. Number turns orange when filtered down.
  • Tools cluster (right-aligned): Clear filters (conditional, red-tinted pill), Expand/Collapse all, info tip.

Band 2 — Quick presets

  • Left label: QUICK (uppercase, muted).
  • Pill-shaped buttons with a leading colored dot + label + count badge.
  • Exactly one preset can be on at a time. Selecting one clears all axis filters and applies the preset's filter set.
  • Preset set (initial): All · Unresolved · Mandatory unresolved · Running · Below threshold · Resolved.

Band 3 — Segmented axes

  • Two axes per row on desktop; each axis is label + segmented group.
  • Label right-aligned at fixed min-width: 92px so segments start at a consistent column.
  • First segment is always All; toggling it clears the axis.
  • Difficulty + Status chips tint to their token color when .on.

State machine

  • Filters state stays exactly as today: { search, difficulty, availability, required, status }.
  • preset is a derived selector, not persisted state. A preset is "on" iff its filter set exactly matches the current state.
  • Clicking an axis chip clears the matching preset (preset → custom).
  • Clicking a preset overwrites the relevant axes and clears others — preset is authoritative for the moment.
