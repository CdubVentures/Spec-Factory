In-Process Compile/Validate

Last remaining CLI architecture issue — requires dedicated design session

1. Problem

The API server spawns compile-rules and validate-rules as child processes via startProcess() in studioRoutes.js. These are pure JSON→JSON transforms completing in <5s. The spawn adds PID tracking, orphan detection, completion handler re-sync, and stdout parsing — all unnecessary for sub-5-second pure functions.

User-facing impact: Compile is blocked while IndexLab runs. They share a single global process slot. A user cannot compile field rules while a crawl is active.

Current flow (spawned)

POST /studio/:category/compile
  → startProcess('src/app/cli/spec.js', ['compile-rules', '--category', cat])
  → child writes JSON artifacts to disk
  → child exits
  → handleCompileProcessCompletion() re-reads JSON, syncs to SQL

Target flow (in-process)

POST /studio/:category/compile
  → const result = await compileRules({ category, config })
  → handleCompileProcessCompletion() syncs to SQL (using result directly)

2. Blocking Architecture

Backend: single process slot

FileConstraint
processLifecycleState.js Two-state machine: idle ↔ running. No concept of multiple concurrent processes.
processRuntime.js:124 startProcess() throws 'process_already_running' if state.phase === 'running'. Single childProc variable.
studioRoutes.js:180,191 Compile and validate both call startProcess(). Catch blocks return 409.

Completion handlers

FileWhat it does
compileProcessCompletion.js Runs on exit code 0. Invalidates caches twice (before and after DB sync to prevent stale re-caching). Calls syncSpecDbForCategory. Broadcasts process-completed via emitDataChange.
indexLabProcessCompletion.js Runs on every exit. Only acts if command is indexlab. Reconciles interrupted runs in SQL. Not relevant to compile/validate.

Frontend: single Zustand slot

FileConstraint
runtimeStore.ts Single processStatus: ProcessStatus in Zustand state.
StudioPageShell.tsx:209 disabled={compileMut.isPending || processStatus.running} — blocks compile when any process runs.
CompileReportsTab.tsx:70 Same pattern: disabled={compilePending || anyProcessRunning}.
useCategorySync.ts Polls GET /process/status every 5s, pushes into Zustand store.
useWsEventBridge.ts WS process-status channel also pushes into same Zustand slot + React Query cache.

3. What Needs to Change

LayerCurrentRequired
Backend state Single phase: 'idle' | 'running' Per-command state slots or separate lightweight-process tracking that doesn't touch the global slot
API contract GET /process/status returns single object Multiple slots or separate endpoint for compile status
Frontend store processStatus: ProcessStatus Multi-slot Zustand state or separate compile status atom
UI disable gates processStatus.running blocks all commands Per-command running check
Completion handlers Triggered by child process exit event Must work inline (no child process lifecycle event)
WebSocket Broadcasts single global process-status Per-command broadcast or scoped events

5. Prerequisite Decisions

Before implementation, these design choices must be made:

  • Separate compile status vs multi-slot process status? A dedicated compileStatus atom is simpler but less general. A multi-slot processStatusByCommand is more extensible but touches more files.
  • In-process only for compile/validate, or also billing-report/llm-health? The latter are also thin wrappers over importable functions.
  • Per-category concurrency lock? Compile writes to _generated/, validate reads from it. Concurrent compile+validate on the same category would conflict.
  • API response contract? Does POST /studio/:cat/compile return the compile result directly, or still return a process-status-shaped response for backwards compatibility?
