Publisher Tab — GUI Reference

Live audit log for the publish pipeline. Every submitCandidate() call visible in one table. Filterable by date, product, brand, field, source, and status. Sits between Runtime Ops and Review Grid in the main navigation.

1. What It Shows

One row per candidate submission. The table is a direct read of the field_candidates table + product.json candidates[]. Every call to submitCandidate() produces a row. Failed runs (rejected by the gate) also appear — they have status: 'rejected' in the source’s failure history (e.g. color_edition_finder_runs).

Table Columns

ColumnSourceDescription
Submitted submitted_at Timestamp. Sortable, filterable by date range.
Product product_id Product ID. Clickable → links to product detail / review grid.
Brand JOIN products Brand name (from products table). Filterable.
Field field_key Field key (e.g. colors, weight, sensor). Filterable.
Value value Post-validation value. For lists/records: truncated preview, expandable.
Status status candidate or resolved. Filterable.
Confidence confidence 0–100. Color-coded: green ≥80, yellow 50–79, red <50.
Sources source_count Number of independent sources. Badge with count.
Source sources_json[0].source Primary source type: cef, pipeline, override, cli. Chip colored by type.
Repairs validation_json.repairs Repair count badge. Expandable: shows before → after for each repair step.

2. Filters

Filter bar at top of table. All filters are combinable (AND logic). URL-synced for shareability.
FilterTypeOptions
Date range Date picker Last 24h, last 7d, last 30d, custom range. Default: last 7d.
Product Search / dropdown Type-ahead search by product ID, brand, or model name.
Brand Multi-select dropdown All brands in category. Select one or more.
Field Multi-select dropdown All field keys. Select one or more.
Source Chips cef | pipeline | override | cli | all
Status Chips candidate | resolved | all
Has repairs Toggle Show only rows where repairs were applied.

3. Row Expansion (Detail View)

Clicking a row expands an inline detail panel showing the full validation history.

Validation Detail

Repairs applied:
Table of { step, before, after, rule } — e.g. normalize: "Black" → "black"

Rejections:
If any soft rejections (e.g. unknown_enum_prefer_known), show them with detail.

Full value:
For lists/records — formatted JSON view of the complete value.

Source Detail

All sources:
Table of sources with confidence, run_id, submitted_at, source-specific metadata.

CEF sources: model name, run_id
Pipeline sources: artifact hash, URL, run_id
Override sources: overridden_by, reason

4. Data Source

The table reads from the field_candidates SQL table (category-scoped specDb). This is the primary DB table for the publisher pipeline.
-- Query for publisher table (with brand join) SELECT fc.id, fc.product_id, fc.field_key, fc.value, fc.confidence, fc.source_count, fc.sources_json, fc.validation_json, fc.status, fc.submitted_at, p.brand, p.model, p.variant FROM field_candidates fc LEFT JOIN products p ON fc.product_id = p.product_id AND fc.category = p.category WHERE fc.category = ? ORDER BY fc.submitted_at DESC LIMIT 100 OFFSET ?
API endpoint needed
GET /api/v1/publisher/:category/candidates
Query params: page, limit, product_id, brand, field_key, source, status, since, until, has_repairs
Returns: { rows: [...], total: number, page: number }

5. Navigation Placement

App Shell top nav (left to right): Dashboard | Studio | Runtime Ops | Publisher | Review Grid | Settings
The Publisher tab sits between Runtime Ops and Review Grid. It shows the pipeline’s validation activity — what went in, what came out, what was repaired or rejected. The Review Grid shows the resolved product state for human review.
