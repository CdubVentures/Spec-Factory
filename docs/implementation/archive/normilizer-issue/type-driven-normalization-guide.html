1. Problem Statement

Symptom: Edition candidates in product.json are stored as ["[object object]"] instead of the actual edition Record. The list_of_tokens_delimited template calls String(object) before the shape check can catch the mismatch.

But this bug is a symptom of a deeper design problem: three overlapping knobs (contract.type, contract.shape, parse.template) all competing to describe how a value should be cleaned. The template system is almost entirely redundant with the type system.

2. Root Cause: The Triple Redundancy

An audit of all 297 fields across keyboard (103), mouse (81), and monitor (113) reveals:

type + shape + template comboCount%Is template redundant?
string + scalar + text_field13746%YES — type=string already means "trim text"
number + scalar + number_with_unit7425%YES — type=number already means "coerce to number"
string + scalar + boolean_yes_no_unk3512%WRONG TYPE — should be type=boolean
string + list + list_of_tokens_delimited217%YES — type=string + shape=list is sufficient. Extraction submits arrays.
number + scalar + integer_field72%YES — and type should be integer
string + scalar + component_reference62%YES — stays type=string. Component lookup is a review-stage constraint, not a data type.
integer + scalar + number_with_unit41%YES — type=integer already covers it
string + scalar + url_field41%WRONG TYPE — should be type=url
string + scalar + date_field31%WRONG TYPE — should be type=date
object + list + latency_list_modes_ms20.7%RETIRE — split into 3 scalar number keys per mode (wired/wireless/bluetooth). Delete the object fields.
number + list + list_of_numbers_with_unit10.3%YES — type=number + shape=list

integer + scalar + integer_field10.3%YES — type=integer
Bottom line: parse.template is redundant for every single field. In all 297 cases, contract.type + contract.shape is sufficient. The template layer exists because the types were never set correctly.

Pipeline ordering bug

Current pipeline order:
  1. Absence normalization
  2. Template dispatch    ← coerces value based on template name
  3. Shape check          ← too late, value already destroyed
  4. Type check
  ...

Objects hit template dispatch BEFORE shape check can reject them.
Template calls String(object) → "[object Object]" → data corruption.

3. The Solution: Type + Shape (That's It)

Principle: contract.type determines per-value cleaning. contract.shape determines cardinality. parse.template is eliminated entirely. No replacement flags. The pipeline validates, extraction supplies clean data.

3.3 What Was Removed

ConceptWhy Removed
parse.templateRedundant with contract.type in every case.
tokens_delimited flagExtraction submits arrays directly. No string-splitting in the pipeline.
normalization_fnExtraction outputs clean data. Pipeline validates, doesn't transform freeform text.
component_ref typeComponent lookup is a review-stage constraint, not a data type. These fields stay type=string.
key_value / structured shapesNothing needs them. Editions is a slug list. Latency is a number list with metadata stored separately.
object typeLatency fields are retired and split into 3 scalar number keys per mode (e.g., click_latency_wired, click_latency_wireless, click_latency_bluetooth). No object type needed anywhere.

3.5 New Pipeline Order

New pipeline order:
  1. Absence normalization        (unchanged)
  2. Shape check                  ← MOVED UP: rejects mismatched shapes immediately
  3. Type coercion                ← REPLACES template dispatch: driven by contract.type
  4. Unit verification            (unchanged)
  5. String normalization         (unchanged)
  6. Format check                 (unchanged)
  7. List rules                   (unchanged)
  8. Rounding                     (unchanged)
  9. Enum check                   (unchanged)
  10. Range check                 (unchanged)
  11. Publish gate                (unchanged)

Shape check before type coercion = no more data corruption.
