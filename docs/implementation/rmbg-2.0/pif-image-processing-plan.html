PIF Image Processing Pipeline

RMBG 2.0 integration for automated background removal, trimming, and master file generation

PIF RMBG 2.0 Automated Plan

1 Overview

Extend the Product Image Finder (PIF) post-download pipeline with automated background removal and transparent-edge trimming. Every downloaded product image is processed into a clean, full-resolution transparent PNG master file — ready for any downstream export or publishing pipeline.

Key principle: The PIF produces master files. Export/publishing (WebP families, Astro optimization, Photoshop batch processing) is a separate concern handled later when the app is complete and product data is ready for the website.

Stage Details

Stage Input Action Output
Download LLM-provided URL HTTP GET with normalized URL + headers. CDN sizing params stripped. Accept header prefers original format (JPEG/PNG). Raw image file (any format: JPEG, PNG, WebP, AVIF)
Quality Gate Raw downloaded file Read dimensions from file header. Check against minWidth, minHeight, minFileSize. Reject and delete if below thresholds. Verified raw image (passes quality)
Remove BG Verified raw image Feed to RMBG 2.0 model. Produces 8-bit alpha matte with 256 transparency levels. Composite mask onto original to create transparent image. Transparent image (alpha channel applied)
Trim Transparent image Detect bounding box of non-transparent pixels. Crop canvas to content bounds. No padding added (padding is an export concern). Tight-cropped transparent image

3 File Storage

Both original downloads and processed masters are kept. Originals serve as historical reference and can be reprocessed if the model improves. The processed PNG is the display artifact.

Original Preservation

  • Raw downloads saved to originals/ subdirectory with the same filename but original extension
  • Originals are never modified or reprocessed in place
  • If RMBG 2.0 model improves, originals can be batch-reprocessed to regenerate masters
  • Delete operations remove both the master and its corresponding original

Runtime Approach

  • Model loaded once at server boot or on first PIF run, then kept in memory
  • Per-image processing is stateless — feed image in, get mask out
  • No Python dependency required if using ONNX Runtime for Node.js (onnxruntime-node), or call a lightweight Python sidecar
  • Fallback: if model is unavailable, save the raw download without processing (degrade gracefully, don't block the run)
Resource impact: This is a ~176 MB segmentation model, not an LLM. It processes one image at a time using standard CPU inference. No GPU required. No tokens consumed. Runs alongside the existing server without competing for LLM resources.

5 Processing Logic

Per-Image Processing (inside runSingleVariant)

After a successful download passes the quality gate:

  1. Move the raw download to originals/{filename}.{original_ext}
  2. Load the raw image into the RMBG 2.0 model
  3. Receive the alpha matte (grayscale mask)
  4. Composite: apply the matte as the alpha channel on the original image
  5. Trim: detect bounding box of non-transparent pixels, crop to content
  6. Save as {filename}.png (lossless, full alpha precision)
  7. Record both filename (master) and original_filename in the image entry

Error Handling

  • Model not loaded: save raw download as the master file (skip bg removal). Log warning.
  • BG removal fails on specific image: keep the raw download as master. Record bg_removed: false on the image entry.
  • Trim produces empty canvas: model masked out everything. Keep the pre-trim transparent image. Record trim_failed: true.

6 GUI — Before / After Viewer

The PIF lightbox (full-screen image viewer) gains a scrollable before/after mode. The gallery always shows the processed master. Clicking into the lightbox reveals both.

Lightbox Behavior

  • Default view: processed master PNG (bg removed + trimmed) on a checkerboard transparency background
  • Scroll or swipe: transitions to the original raw download (with background intact)
  • Label overlay: "Processed" / "Original" badge in the corner so it's clear which you're viewing
  • Both images accessible: source URL link and download link available for each version

Gallery Card

  • Thumbnail always shows the processed master
  • If bg_removed: false, show a small indicator that processing was skipped
  • Delete button removes both master and original from disk

8 Out of Scope (Future)

These concerns are intentionally deferred:

  • WebP responsive family export — handled by the Photoshop webp-all-options.jsx script or a future Node-based exporter when the app is complete
  • Astro image optimization — Astro's <Image> component handles responsive sizing at build time from the master PNG
  • Padding / canvas extension — an export concern per view preset, not a PIF concern
  • Batch reprocessing — re-running RMBG 2.0 on all originals when the model improves (simple to add later since originals are preserved)
  • CDN upload — publishing processed images to a production CDN

9 Implementation Order
