PIF Vision Evaluator

LLM-powered image quality evaluation — pick the best candidate per view, flag watermarks, select heroes

PIF Vision LLM LLM Lab Queue Plan

1 Problem

PIF downloads 2–3 image candidates per view per variant across multiple runs. Currently the user must manually inspect every image to find the best one for each angle and decide which shots deserve "hero" status. Many downloaded images also contain watermarks, retailer badges, or promotional overlays that disqualify them — but these are only visible on manual inspection.

Goal: Use an LLM with vision capability to automatically evaluate candidates, flag problems (watermarks, badges, crops), rank by quality, and pick the best image per view — plus select hero shots for the product carousel.

2 Architecture

The Vision Evaluator is a new LLM phase (imageEvaluator) with its own prompt configuration. It reads images from disk, creates lightweight thumbnails, and sends them through the existing LLM Lab queue for bounded, serialized inference.

Key Constraints

  • Own LLM phase: imageEvaluator — separate from imageFinder (discovery) and independent prompt config
  • No HTTP serving needed: images read from disk → sharp resize → buffer → base64 → data:image/png;base64,... in image_url content part
  • Queue integration: all calls go through LLM Lab's BoundedConcurrencyGate — fire them all, let the queue serialize
  • No new dependencies: uses sharp (already installed) + existing LLM call infrastructure

Why 512×512 is enough: The LLM is judging composition, watermarks, badges, and sharpness — not pixel-level detail. 512×512 is plenty to spot a Getty watermark, a "SAMPLE" badge, a cropped product, or a blurry photo. Original full-res files stay on disk untouched.

4 Two Evaluation Modes

View Evaluation

Pick the best candidate for a single view of a single variant.

  • Input: all candidates for one view (e.g., top-black.png, top-black-2.png, top-black-3.png)
  • LLM sees them side-by-side as numbered thumbnails
  • Judges: composition, watermarks/badges, sharpness, product visibility, background removal quality
  • Returns: ranked order + per-image flags
  • Winner gets eval_rank: 1 on its image entry

Hero Selection

Pick the best overall promotional shots for the product carousel.

  • Input: the eval_rank: 1 winner from each evaluated view
  • LLM sees all best-per-view images together
  • Judges: which angles show the product most attractively for a product page
  • Returns: ordered list of hero candidates + reasoning
  • Top picks get hero: true flag on their image entry

Evaluation Criteria (prompted)

CriterionWhat LLM ChecksImpact
Watermarks Getty, Shutterstock, retailer logos, "SAMPLE" text, copyright overlays Immediate disqualification — eval_flag: 'watermark'
Badges / overlays Sale stickers, "NEW" badges, retailer branding, promotional text Disqualification — eval_flag: 'badge'
Cropping Product cut off at edges, missing parts, too tight framing Penalty in ranking — eval_flag: 'cropped'
Wrong product Different model, wrong color, accessory instead of product Disqualification — eval_flag: 'wrong_product'
Sharpness Blur, compression artifacts, noise, low resolution appearance Penalty in ranking

BG removal quality Halo artifacts, missing parts, jagged edges (if bg_removed: true) Affects ranking order

Image numbering: Images are labeled Image 1, Image 2, Image 3 in the prompt text, matching the order of image_url content parts. The LLM references these numbers in its response. Backend maps numbers back to filenames via the same ordered list.

6 Queue Strategy

All evaluation calls go through LLM Lab's BoundedConcurrencyGate. Fire them all at once — the queue serializes them in priority order.

// User clicks "Loop Views" for variant "black" (6 budgeted views)

queue.enqueue( evaluate("top", "black", [img1, img2, img3]) ) // priority 1
queue.enqueue( evaluate("left", "black", [img1, img2]) ) // priority 2
queue.enqueue( evaluate("angle", "black", [img1]) ) // priority 3
queue.enqueue( evaluate("sangle", "black", [img1, img2]) ) // priority 4
queue.enqueue( evaluate("front", "black", [img1]) ) // priority 5
queue.enqueue( evaluate("bottom", "black", [img1]) ) // priority 6

// LLM Lab processes them one at a time (or N at a time per MAX_CONCURRENT)
// Each completes independently — results persisted immediately

result ← { best: "top-black.png", flags: [], rank: 1 } // persisted to JSON + SQL
result ← { best: "left-black-2.png", flags: [], rank: 1 }
result ← { best: null, flags: ["watermark"] } // only candidate had watermark
// ...

Priority Order

Views are queued in the category's priority order (from view budget). This means the most important angles get evaluated first:

CategoryQueue Order
Mousetop → left → angle → sangle → front → bottom
Keyboardtop → left → angle → sangle
Monitorfront → angle → rear → left

Skip Logic

  • Views with only 1 candidate: auto-elect (no LLM call needed). Still check for watermarks if eval explicitly requested.
  • Views with 0 candidates: skip entirely
  • Views already evaluated (same images, same eval run): skip (idempotent)

7 Thumbnail Pipeline

No HTTP serving needed. Images are read from disk, resized, and encoded in-process:

// Per image: disk → sharp → base64 → content part
async function imageToContentPart(imagePath) {
  const thumb = await sharp(imagePath)
    .resize(512, 512, { fit: 'inside', withoutEnlargement: false })
    .png()
    .toBuffer();

  const b64 = thumb.toString('base64');
  return {
    type: 'image_url',
    image_url: { url: `data:image/png;base64,${b64}` }
  };
}
PropertyValueRationale
Target size 512×512 Plenty for judging composition + detecting watermarks. Small token cost.
Fit mode inside Preserve aspect ratio, fit within 512×512 box. No distortion.
Format PNG Lossless — no compression artifacts that might confuse watermark detection.
Source Master PNG (processed) Send the bg-removed version so LLM can judge BG removal quality too.

9 GUI — 4-Button UI

Top-Level Buttons (All Variants)

🔍 Eval Views ⭐ Pick Heroes 🔁 Loop Views 🔁 Loop Heroes
ButtonActionScope
Eval Views Evaluate all views for the selected variant. Queues one call per view in priority order. Single variant, all budgeted views
Pick Heroes After views are evaluated, send best-per-view to LLM for hero selection. Single variant, cross-view
Loop Views Loop through all variants, evaluating all views for each. All calls queued at once. All variants, all budgeted views
Loop Heroes Loop through all variants, picking heroes for each. Requires views to be evaluated first. All variants, cross-view

Per-Variant Row Buttons

🔍 View ⭐ Hero 🔁 All Views 🔁 All Heroes

Same 4 actions, scoped to one variant. "View" evaluates the next unevaluated view in priority order. "Hero" runs hero selection for just that variant.

Gallery Card Enhancements

  • Flag chips: watermark badge cropped on flagged images

// PIF has already downloaded images across 3 runs:
// top-black.png (run 1), top-black-2.png (run 2), top-black-3.png (run 3)
// left-black.png (run 1), left-black-2.png (run 3)
// angle-black.png (run 2)

1. User clicks "Eval Views" for variant black

// Queue 3 calls (views with candidates), in category priority order:
→ Call 1: evaluate top (3 candidates) // ~90K tokens
→ Call 2: evaluate left (2 candidates) // ~65K tokens
→ Call 3: evaluate angle (1 candidate) // auto-elect, check for watermarks

// Results come back as queue drains:
← top: best = top-black.png, top-black-2.png flagged [watermark]
← left: best = left-black-2.png, left-black.png [cropped]
← angle: best = angle-black.png (only candidate, no flags)

2. User clicks "Pick Heroes" for variant black

// Send the 3 winners to hero selection:
→ Hero call: top-black.png + left-black-2.png + angle-black.png

← Heroes: #1 angle-black.png, #2 top-black.png

// Gallery now shows: rank badges, watermark flag, hero stars
// Watermarked image dimmed at 40% opacity

13 Out of Scope (Future)

  • Automatic re-download: If best candidate has watermark, auto-trigger a new PIF run for that view. (Requires progressive carousel integration.)
  • Cross-variant comparison: Comparing the same view across different color variants (e.g., is black's top shot better than blue's?).
  • Inpainting/removal: Using LaMa or similar to remove detected watermarks rather than just flagging them.
  • Automation page integration: "Run full pipeline" button that chains: PIF search → RMBG → eval → hero → export.
  • A/B comparison mode: Side-by-side lightbox for comparing two candidates with the LLM's reasoning overlay.
