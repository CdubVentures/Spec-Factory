2 Locked design decisions — do not re-debate

DecisionWhat it means
4 tiers + 1 fallbackeasy / medium / hard / very_hard + fallback. Empty tier inherits the whole fallback bundle (not per-field merge). Fallback default is gpt-5.4 + xhigh + webSearch on.
Run is per-keyOne button click = one fieldKey = one LLM call. The "run all keys by group" dashboard is Phase 4 — do not build it now.
Loop is per-keyRetries a single key up to calcKeyBudget(...).attempts. Smart Loop (group/header) is Phase 5+.
Budget in Pipeline Settings, NOT LLM ConfigThe 12 settingsSchema knobs (3 axis JSON tables + floor + variant weight + bundling toggle + 2 bundling tables + policy enum + 2 history toggles + hidden prompt template) live on the keyFinder finder module. LLM Config only carries the 5 tier bundles.
Discovery history is per-keyScoped by (product_id, field_key). If the URL+query history toggles are on and you click Run 11 times, the 11th call sees all 10 prior URL+query lists. No table change — every run record tags itself with response.field_key; the orchestrator passes a runMatcher to accumulateDiscoveryLog.
CEF/PIF/RDF/SKU-owned keys are EXCLUDEDcolor/edition, hero_image/product_image, release_date, sku — each has its own finder. Field-rule eg_defaults keys are also excluded. Keyfinder runs only on everything else.
productFieldProducer, not variantFieldProducerkeyFinder runs once per (product, field_key) — NOT per (product, variant_id, field_key). Per-variant divergence for connectivity-type fields is handled later via the manual override drawer (see memory feedback_variant_dependent_override_shape).
Backend lives at src/features/key/NOT per-key-finder/. deriveFinderPaths strips the Finder suffix from id: 'keyFinder'. Frontend panel is at tools/gui-react/src/features/key-finder/ (panel path uses the full id).
Billing is per-tier4 reasons under "Key Finder" group: key_finding_easy / _medium / _hard / _very_hard. Cyan palette lightest→darkest. Phase 3a Part 2 orchestrator tags each call with the right tier reason.
Field-rule difficulty is 3-level today, 4-level targetStudio still emits easy|medium|hard; very_hard is the Phase 1 target. Orchestrator must cascade unknown difficulty → fallback tier. Do NOT wait on Phase 1 — the cascade handles it.

4.3   runKeyFinder step list

Canonical order
  1. FieldRulesEngine.create(category, { specDb }) → engine.rules[fieldKey]. Throw if missing.
  2. Resolve policy via existing assembler, then resolvePhaseModelByTier(policy, fieldRule.difficulty) → tier bundle.
  3. calcKeyBudget({ fieldRule, variantCount, settings }) — Run mode ignores attempts but the breakdown goes in run metadata for display + Phase 3b consumption.
  4. Read previous runs via keyStore.read(productId). Build discovery log via accumulateDiscoveryLog(runs, { runMatcher: (r) => r.response?.field_key === fieldKey, includeUrls: keyFinderUrlHistoryEnabled, includeQueries: keyFinderQueryHistoryEnabled }).
  5. buildKeyFinderPrompt({ product, fieldKey, fieldRule, previousDiscovery, ambiguityLevel, familyModelCount, siblingsExcluded, templateOverride?: settings.discoveryPromptTemplate }).
  6. createKeyFinderCallLlm(deps, tier)({ product, fieldKey, variantCount }). The SPEC emits reason: key_finding_${tier}, so billing rolls up automatically.
  7. keyFinderResponseSchema.parse(result) — reject malformed responses.
  8. If value is defensible (not "unk"), call submitCandidate with the publisher metadata (mirror RDF at releaseDateFinder.js).
  9. Append run record to JSON store + SQL via keyStore.write/merge. Tag: { field_key, tier, run_number, ran_at, response: { field_key, ..., discovery_log }, model, duration_ms }.
  10. Emit WS broadcast for live UI refresh.

4.4   Route contract

// POST /api/v1/key-finder/:category/:productId
{ field_key: "polling_rate", mode: "run" }    // mode defaults to "run"

// GET /api/v1/key-finder/:category/:productId?field_key=polling_rate
// Returns: { summary, runs[], candidates[], published? } — scoped by field_key

// DELETE /api/v1/key-finder/:category/:productId/runs/:runNumber?field_key=polling_rate
Why a custom handler instead of createFinderRouteHandler The generic handler expects static fieldKeys: ['release_date']. keyFinder's fieldKeys are dynamic per request (from POST body). Either extend the generic handler to accept a dynamic field-key resolver (risky — touches RDF + SKU) or build a custom wrapper for keyFinder and refactor later. Do the custom wrapper.

6 Pitfalls that already broke the build

PITFALL #1 Stub route handler signature

Symptom: Billing tab shows nothing. Server log spams [gui-server] API error: res.setHeader is not a function on every request.

Root cause: the Phase 2 stub registerKeyFinderRoutes had signature (req, res) and never returned false. The API dispatcher calls handlers with (parts, params, method, req, res) — so the stub received parts as _req and params (URLSearchParams) as res. It then tried to call setHeader on URLSearchParams and threw on every request, taking down billing, products, everything.

Fix: finder route handlers MUST be:

export function registerFooRoutes(ctx) {
  const { jsonRes } = ctx;
  return async function handleFooRoutes(parts, _params, _method, _req, res) {
    if (parts[0] !== 'foo') return false;  // critical — let other handlers run
    jsonRes(res, 200, { ... });
  };
}
PITFALL #2 llmPolicyAdapter.ts is partially hand-wired

Codegen at tools/gui-react/scripts/generateLlmPolicyAdapter.js auto-emits fields for policyGroup: '_json' entries. But the hand-written llmPolicyAdapter.ts has explicit hardcoded cases in flattenLlmPolicy and routeFlatKeyUpdate for each JSON blob (llmPhaseOverridesJson, llmProviderRegistryJson, keyFinderTierSettingsJson).

If you add a new JSON-blob policyGroup entry, you must:

  1. Add the explicit flatten case: flat.newKeyJson = JSON.stringify(policy.newKey ?? {});
  2. Add the explicit route case: parses JSON back into topLevel: { newKey: parsed }.
  3. Run all 3 codegen scripts.

Tech debt — refactor to be fully codegen-driven when a 4th JSON blob lands.

PITFALL #3 Difficulty vocab mismatch

Studio emits difficulty ∈ {easy, medium, hard}. The tier bundles + billing + Zod schema support {easy, medium, hard, very_hard}. When Phase 1 hasn't migrated data to 4 levels, field_rule.difficulty = 'very_hard' will never appear — but the orchestrator must still handle unknown tokens gracefully.

Fix: resolvePhaseModelByTier already cascades unknown → fallback. Don't add branching; trust the cascade.

PITFALL #4 Register the field_key matcher on every run record

Per-key discovery history depends on run.response.field_key being present on every recorded run. If the orchestrator forgets to stamp field_key into the response payload, the runMatcher silently matches zero runs and discovery history is a no-op — with no error.

Fix: Schema already requires field_key: z.string(). Orchestrator must echo it into the saved run record, not just the LLM response.

PITFALL #5 Do NOT revive the old 14-entry registry design

Phase 2 went through one false start where 14 scalar registry entries were proposed (5 tier model scalars + 3 budget axes + floor + variant weight + bundling enable + 2 bundling tables + policy). User correction: tier bundles are ONE JSON blob, budget/bundling live in the FINDER_MODULES settingsSchema (Pipeline Settings), and LLM Config has exactly one "Key Finder" tab.

If a future refactor proposal has 14 registry entries for keyFinder, it's wrong. See memory project_keyfinder_phase2_shipped + roadmap §1.

PITFALL #6 CEF / PIF / RDF / SKU-owned keys must be excluded

Don't let the orchestrator attempt release_date (RDF owns), sku (SKF owns), colors/editions (CEF owns), or hero/product images (PIF owns). Keys with eg_defaults in field_rules are also excluded — they come from a different path.

Fix: Denylist, not allowlist. Pull the list of reserved keys from the registered finders' FINDER_MODULES[*].fieldKeys arrays + a static eg_defaults filter.
