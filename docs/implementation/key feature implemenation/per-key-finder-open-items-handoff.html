3 Open Items (priority-ordered)

Test discipline per CLAUDE.md: BEHAVIORAL change = TDD (red → green → refactor). COSMETIC/CONFIG = no tests. Each open item below declares its [CLASS].
P4 · BEHAVIORAL Variant-aware bundling cost OR pool Needs PO decision first

Spec line: "Variants per Point Should be Fractional ... (this should increase the cost of passengers or reduce the primary pool cost , thus making it so runs are more focused , either one works and be in the Bundling options)"

Current behavior: variant points flow into the primary's retry budget, not into bundling cost or pool. The auditor flagged this: passenger cost + primary pool are both intMap in finderSettingsRegistry.generated.ts:129-130, and packBundle floors values via toIntOrZero in keyBundler.js:31. A sandbox probe with passenger_cost=0.5 floors to 0 (3 passengers fit in a pool of 1).

Decision gate (ask PO before writing code):

  • (A) Passenger-cost scaling: cost[d] × (1 + extras × factor). More variants = more expensive passengers = fewer pack.
  • (B) Pool-reduction scaling: pool[d] − (extras × factor). More variants = smaller primary pool = fewer pack.
  • (C) Leave as-is: variants only affect retry budget. Spec phrase "either one works" suggests they'll accept this; document the choice.

Implementation sketch (A or B):

  • New knob bundlingVariantScaling (enum: none | cost | pool).
  • New knob bundlingVariantFactorPerExtra (float). 0 disables.
  • Register in src/shared/settingsRegistry.js (keyFinder section, group: Bundling).
  • packBundle applies the scaling when reading pool / cost[d]. Drop toIntOrZero's floor on these two reads; floor only at the comparison (or keep fractional all the way).
  • BundlePreviewText (KeyRow.tsx:109) shows the scaled cost (not raw) when variant scaling is active — e.g. passenger_key (1.50).
  • KeyBundlingStrip.tsx reflects the new knobs in its summary line.

Test budget (after PO picks A or B):

  • New keyBundler.test.js case: variant scaling changes the number packed.
  • New KeyBundlingStrip snapshot or render test confirming the scaled cost displays.

Do NOT implement until PO confirms A vs B. This is the single largest behavioral change remaining.

P5 · BEHAVIORAL Published-but-"could do better" peer re-bundling Parked — low value at defaults

Spec line: "re calculates the additional keys based on unresolved or lower confidence (need a knob in pipeline keys for this sometiing could be resolved but we could do better is the point)"

Current behavior: keyPassengerBuilder.js:54-61 drops every published peer unconditionally. The passengerExcludeAtConfidence + passengerExcludeMinEvidence knobs only gate unpublished peers (they read getTopFieldCandidate, not getResolvedFieldCandidate).

Why parked: default publishConfidenceThreshold = 95 and publisher requires ≥2 evidence. Any published peer ALREADY satisfies ≥95 conf + ≥2 evidence by definition. Applying a "drop if conf≥95 AND evidence≥2" re-filter would produce the same pool as "drop if published". Zero practical value at default thresholds.

When to revive: if the PO lowers publishConfidenceThreshold to something like 80 and wants "published but below X / below Y" peers re-packable for refinement. Otherwise, leave alone.

If revived: merge the two exclusion blocks in keyPassengerBuilder.js:54-80. Read confidence + evidence_count off the resolved row; drop only when both knob thresholds cleared. 2 new tests in keyPassengerBuilder.test.js: below-threshold published peer packs; above-threshold published peer drops.

P6 · BEHAVIORAL Global chain reservation ledger (cross-group planning) Deferred — tier caps cover today

Spec line: "you will need to track whats in a loop and indicate on the loop button so as to not send 5 of the same set of keys as we will still fire one key per group but out reach keys and not overlap what additional keys are sent"

Current behavior: no planning ledger. Each chain fires independently; overlap is prevented only by the active in-flight registry (keyFinderRegistry) + tier caps. Under groupBundlingOnly=false with multiple group chains running, two chains can plan to pack the same peer before either registers.

Severity: low. The 2/4/6 tier caps are self-healing: by the N-th overlap the peer is blocked. Easy peers especially — at cap=2, the 3rd concurrent attempt is dropped. Hard peers have cap=6 so overlap is explicitly allowed.

Possible future fix (if the no-cap scenarios bite):

  • Client-side: a shared plannedPrimaries: Set<string> across all active chains (KeyFinderPanel.tsx). Passed to the server via a new reserved_keys body field; passenger builder excludes them.
  • Server-side: an ephemeral "planned" ledger (similar to keyFinderRegistry) with TTL. Clients push planned keys, drop them after firing.
  • Neither is worth doing until we see an actual overlap collision in a real run. Log it, don't build it.

Signal to revive: add a dev log when a peer is skipped by cap. If that log fires frequently in production Loop-All-Groups runs, promote P6.
