1 Executive summary

What we're building: a single src/features/key/ module (id keyFinder) that replaces per-feature finders (release-date, product-image non-image keys, sensor, etc.) with one universal engine. Each Field Studio key drives its own prompt via ai_assist.reasoning_note; the engine picks the right model from an editable tier bundle, computes an attempt budget from 4-axis scoring (required × availability × difficulty × variant count), and (optionally) packs same-group passengers via a point-pool bundler.

Why: accuracy and quality at scale demand a consistent prompt scaffold (identity lock, evidence tiers, confidence rubric, URL+query history). Today each finder reimplements most of it. The per-key architecture makes adding a new field a one-file operation — set axes in Field Studio, write extraction guidance, done.

Success criteria:

  • Adding a new finder-backed key requires zero new feature code.
  • Model choice, attempt budget, bundle cap — all editable in llm config without code changes.
  • Publisher integration is unchanged (submitCandidate with evidence refs); finders never self-gate.
  • Prompt scaffold is universal + editable; per-key text flows from ai_assist.reasoning_note.
  • Run controls let us fire a single key, a group, a whole difficulty tier, or everything — from one dashboard.

3 Simplified axes + three decision tables

Three tables. Each axis owns exactly one decision. All cells live in LLM config, fully editable.

3.2   Per-key budget — scoring formula (four axes, floor 3)

Each key earns a budget (LLM calls before giving up) from a sum of four axis-point tables, clamped below by a floor. Every table is editable in LLM config — no hardcoded matrices.

AxisRegistry keyDefault
required_levelbudgetRequiredPoints (intMap){ mandatory: 2, non_mandatory: 1 }
availabilitybudgetAvailabilityPoints (intMap){ always: 1, sometimes: 2, rare: 3 }
difficultybudgetDifficultyPoints (intMap){ easy: 1, medium: 2, hard: 3, very_hard: 4 }
variant countkeyFinderBudgetVariantPointsPerExtraint; 1 per variant beyond the first
Formula: budget = max(floor, requiredPts + availabilityPts + difficultyPts + max(0, variantCount − 1) × variantPointsPerExtra). Floor stored as keyFinderBudgetFloor (default 3).

Example: sensor_model (mandatory, rare, very_hard) on a 4-variant product → 2 + 3 + 4 + 3×1 = 12 attempts. form_factor (non_mandatory, always, easy) on a 1-variant product → 1 + 1 + 1 + 0 = 3 (at floor).

3.3   Bundling — point pool per primary (opt-in)

Bundling is off by default. When on, only Group Smart Loop and Header Smart Loop use it — per-key Run and per-key Loop always run solo. Each primary's difficulty sets its point pool; each passenger costs points based on its own difficulty; variant count scales passenger cost via the same per-extra-variant knob as §3.2.

Primary pool (bundlingPoolPerPrimary, intMap)
easy primary6 points
medium primary4 points
hard primary2 points
very_hard primary1 point

Pool = max passenger-points a primary can carry. Lower pool = tighter focus; higher pool = more free rides.

Passenger cost (bundlingPassengerCost, intMap)
easy passenger1 point
medium passenger2 points
hard passenger4 points
very_hard passenger8 points

Variant scaling: cost = basePassengerCost + max(0, variantCount − 1) × variantPointsPerExtra.

Planner rule: passengers must be same group, same variant scope, difficulty ≤ primary (keyFinderPassengerDifficultyPolicy = less_or_equal). Pack greedily until pool exhausted. Example: easy primary on a 1-variant product with pool=6 → "4 easy (4pt) + 1 medium (2pt) = 6pt" or "6 easy = 6pt". Same primary on a 4-variant product → each easy passenger now costs 1 + 3 = 4pt, so only 1 easy passenger fits the pool.

4.3   Group run with bundling ON

// Processing order inside a group: mandatory first, then difficulty DESC
// (hard primaries go first so they can surface easier secondaries for free)

resolvedKeys = new Set()

for key in groupKeys.sort(mandatoryFirst, difficultyDesc):
  if key in resolvedKeys: continue          // knocked out by an earlier bundle

  plan = keyCallPlanner(key, bundlingEnabled=true)
  // plan.primary = key; plan.secondaries = [..] from same group, diff ≤ primary.diff, not already resolved

  response = keyCallExecutor.run(plan)

  for result in response.results:
    submitCandidate(result)
    if publisher.published(result.fieldKey, result.variantId):
      resolvedKeys.add(result.fieldKey)   // skip its own later call

5 Universal prompt composition

The skeleton is fixed; the key-specific pieces plug into slots

A single editable template owns the scaffolding (identity, sibling exclusion, evidence tiers, confidence rubric, output schema). Key-specific text flows through slots.

{{IDENTITY_HEADER}}          // "You are looking for the EXACT product X — not siblings."
{{SIBLINGS_EXCLUDED}}

// ── PRIMARY KEY (single call target) ──────────────────────────────
PRIMARY KEY: {{PRIMARY_KEY_NAME}}
Extraction guidance:
{{PRIMARY_EXTRACTION_GUIDANCE}}         // ← rule.ai_assist.reasoning_note

Previous searches for this key:
{{PRIMARY_URL_HISTORY}}                 // ← accumulateDiscoveryLog(primaryKey)
{{PRIMARY_QUERY_HISTORY}}

// ── OPTIONAL: opportunistic secondaries (bundling knob) ───────────
{{IF_SECONDARIES}}
ADDITIONAL (grab if you find values while searching the primary):
{{#each secondaries}}
  - {{this.name}} — {{this.extraction_guidance}}
{{/each}}
{{/IF_SECONDARIES}}

// ── Shared scaffolding ────────────────────────────────────────────
{{EVIDENCE_BLOCK}}                      // buildEvidencePromptBlock({ minEvidenceRefs })
{{CONFIDENCE_RUBRIC}}                   // buildValueConfidencePromptBlock()

// ── Output schema ─────────────────────────────────────────────────
{{OUTPUT_JSON_SCHEMA}}                  // composed: primary required, secondaries optional
URL/query history is PRIMARY-only. Secondaries inherit the primary's search session; they don't add their own history blocks. Keeps prompt weight linear with primary count, not with secondary count.

    {{#each secondaries}}
    {{this.name}}: {{this.schema}}.optional().nullable(),
    {{/each}}

6 Bundling — the opt-in knob

Default: OFF. One call per key. Zero risk of LLM drift from overloaded prompts. Per-key Run and per-key Loop always run solo — bundling affects only Group Smart Loop and Header Smart Loop.

ON: when a Smart Loop fires, each call has a primary key; passengers from the same group are packed via the point-pool model in §3.3 until the pool is exhausted.

eligible as passenger
  • Same group as primary
  • Same variant scope
  • Difficulty ≤ primary (less_or_equal policy)
  • Not already resolved
  • Has ai_assist.reasoning_note or schema
pool rules
  • Primary pool = pool[primary.difficulty] (§3.3)
  • Passenger cost = cost[passenger.difficulty] + variant scaling
  • Planner packs greedily, easiest passengers first
  • Stops when next candidate would exceed remaining pool
never
  • Cross-group bundling
  • Harder passengers than primary
  • Cross-variant bundling
  • Passengers with their own URL/query history (primary owns discovery)
Deferred: cross-key satisfaction skip. The PIF-style heuristic of "if a passenger publishes, skip its own primary call later" is future work. Initial ship prioritizes explicit user control (Run / Loop / Smart Loop buttons) over autonomous skip logic.

7 GUI dashboard mockups

The dashboard is a top-level panel for running the finder. The user needs: (a) see every key in the product with its axes, (b) filter/select, (c) fire runs in flexible groupings, (d) inspect per-key run + URL + query history.

Four run modes:
  • Per-key Run — 1 call, 0 passengers, no iteration. One key, one shot.
  • Per-key Loop — iterates one key (no passengers) until its own budget is exhausted or it resolves.
  • Group Smart Loop — runs a full group with bundling on; fires keys in difficulty order easy→medium→hard→very_hard; each key iterates until its budget is exhausted or it resolves.
  • Header Smart Loop — applies Group Smart Loop to every group in sequence. Top-level "run everything" button.
Smart Loop modes are wired in a separate phase after Phase 2. Phase 2 registers only the per-tier LLM, per-key budget scoring, and bundling knobs.

10.3   Glossary

TermMeaning here
Per-key finderThe universal finder feature this doc describes. Replaces per-field finders for any field that wants LLM-driven discovery.
Primary keyThe key a given LLM call is for. Carries the URL/query history and the bulk of the prompt attention.
Secondary keyOpportunistic passenger in a bundled call — "if you find it, grab it." Doesn't carry its own history.
GroupThe group string on field rules (sensor_performance, connectivity, switches, …). Bundling cohort.
BundleOne LLM call containing one primary + 0..N secondaries. Bundle size is capped by availability × difficulty × global.
AttemptOne LLM call for a given primary (+ any secondaries). Budget matrix bounds attempts per key.
Satisfaction skipDuring a group run, skip a key's own primary call if it already published via a prior bundle (lifted from PIF's per-view satisfactionThreshold).
ResolvedA field where a candidate has published (passed confidence threshold AND evidence gate).
Extraction guidancePer-key instruction text stored at rule.ai_assist.reasoning_note. Injected into the primary key's prompt slot.
Fallback modelStronger/different model used when the difficulty-cell model hits a failure condition (see §9.1).
