Color Registry (Global Tab)

A global color atom registry — the SSOT for every valid color name and its hex code. Lives in its own tab after Brands in the global tab group. The LLM must return only registered atoms. Multi-color variants join atoms with + in dominant color order.

⚠

This must be built FIRST — before the Color & Edition Finder runs. The finder’s LLM prompt references the registry as a closed enum. Without it, there’s no enforcement.

Data Model

// appDb table (global, cross-category)
CREATE TABLE color_registry (
  name       TEXT PRIMARY KEY,   -- "black", "light-blue", "dark-green"
  hex        TEXT NOT NULL,       -- "#3A3F41", "#60a5fa"
  css_var    TEXT NOT NULL,       -- "--color-black", "--color-light-blue"
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

Multi-Color Variant Format

Dominant Color Ordering Critical

"+" joins atoms in dominant color order: 1st dominant + 2nd dominant + 3rd dominant + …

  • "black+red" = mostly black, red accents
  • "white+orange+blue" = white dominant, orange secondary, blue tertiary
  • "white+black+red" = white dominant, black secondary, red tertiary

Infinite valid combinations. The registry validates each atom, not the combo.

Naming Convention: Modifier-First

Correct (modifier-first)Wrong (modifier-last)CSS VariableHex
light-blueblue-light--color-light-blue#60a5fa
dark-greengreen-dark--color-dark-green#15803d
light-graygray-light--color-light-gray#6b7280
dark-blueblue-dark--color-dark-blue#1d4ed8

GUI: Color Registry Tab

  • Location: Global tab group, after Brands
  • One entry in pageRegistry.ts (O(1) tab registration)
  • Table view: Name, hex swatch, CSS variable, edit/delete actions
  • Add color form: Name + hex picker
  • Pre-seeded: ~30 canonical EG colors on first load
  • Used by: Color & Edition Finder LLM prompt (closed enum), review grid validation, swatch rendering

LLM Integration

The Color & Edition Finder’s prompt includes the full registry as a closed enum. The model must return only registered atom names. Multi-color variants split on +, each atom validated against the registry. Unknown atoms are rejected or flagged.

Parse-Time Validation (Future)

A custom parse hook splits each color token on + and validates each atom against the registry. Invalid atoms are flagged in the review grid. This replaces the current open enum policy with registry-backed validation.

A new panel in the IndexLab UI, positioned between the product picker and run history. Four feature toggles with smart cooldowns and completion checkmarks. When the user clicks "Start", the crawl pipeline launches AND whichever features are enabled (and not on cooldown) also fire off concurrently.

Launch Orchestration

When the user clicks "Start" in IndexLab:

  1. Pipeline starts as normal (runProduct)
  2. For each enabled feature, check its cooldown/completion state
  3. Skip features that are satisfied (checked off) unless user forces re-run
  4. Spawn remaining features as parallel async tasks
  5. Each receives the identity anchor: { brand, model, variant, category, product_id }
  6. Features run independently — no dependencies between them
  7. Results stream back to GUI via existing WebSocket/event system
  8. If URL Validation is enabled, it waits for pipeline + Visual Verification to complete first

Feature: Color & Edition Variants

Uses GPT-5.4 xhigh with web search to discover every color variant and special edition of a product. Output must match the EG site format exactly — "+" delimited color strings, kebab-case edition slugs, and hex-to-CSS-variable mappings.

What It Discovers

Color Variants Data

Every available color using "+" notation.

Examples: "black", "white+blue", "black+red", "white+orange+blue"

Special Editions Data

Limited/special editions as kebab-case slugs.

Examples: "cyberpunk-2077-edition", "sf6-chun-li"

Per-Variant Metadata Data

SKU, availability, product URL, and which editions map to which colors.

EG Site Output Format

The enrichment output must produce data that integrates directly into the EG site's product JSON format. This is the target schema the LLM response must map to.

⚠

Format is non-negotiable. The EG site uses this exact structure. Any deviation means manual cleanup before integration.

EG Product JSON — Color & Media Fields

// Target output format (matches EG site product JSON)
{
  "colors": ["black", "black+red", "white+orange+blue"],

  "media": {
    "defaultColor": "black",
    "colors": ["black", "black+red", "white+orange+blue"],
    "editions": ["cyberpunk-2077-edition"],
    "images": [
      { "stem": "top", "view": "top" },
      { "stem": "top---black", "view": "top", "color": "black" },
      { "stem": "top---black+red", "view": "top", "color": "black+red" },
      { "stem": "left___cyberpunk-2077-edition---black+red",
        "view": "left", "color": "black+red", "edition": "cyberpunk-2077-edition" }
    ]
  }
}

EG Image Stem Convention

Filename Encoding EG Format
// Pattern: {view}[___edition][---color]

"top"                                      // base view, no color/edition
"top---black"                               // view + color
"top---white+blue"                          // view + multi-color
"top___cyberpunk-2077-edition---black+red"  // view + edition + color
"left___sf6-chun-li---white+blue"           // view + edition + multi-color

Color "+" Notation Rules

RuleExampleNotes
Single color"black"Lowercase, no spaces
Two colors"black+red"Rendered as 45° linear gradient swatch
Three+ colors"white+orange+blue"Rendered as conic gradient swatch
Light/dark variants"gray-light", "blue-dark"Hyphenated modifier
Edition-specific colors"black+red" on "cyberpunk-2077-edition"Color+edition pair in images array

Visual: How Swatches Render

"black" → solid --color-black: #3A3F41
"black+red" → 45° gradient
"white+orange+blue" → conic gradient
"white+blue" → 45° gradient

Color/Edition LLM Response Schema

The structured output schema the LLM must return. This is the contract between the GPT-5.4 call and the storage layer.

// colorEditionSchema.js — Zod schema for LLM structured output
{
  "colors": [                             // EG-format "+" strings
    "black",
    "white",
    "black+red",
    "white+orange+blue"
  ],
  "default_color": "black",                // most common / marketing default

  "editions": [                            // kebab-case slugs
    {
      "slug": "cyberpunk-2077-edition",
      "display_name": "Cyberpunk 2077 Edition",
      "colors": ["black+red"],          // which colors this edition comes in
      "url": "https://..."                // product page for this edition
    }
  ],

  "variants": [                            // full variant manifest
    {
      "color": "black",                   // EG "+" format
      "edition": null,                   // null = standard, or edition slug
      "sku": "CH-931A1B1-NA",
      "availability": "in_stock",         // in_stock | discontinued | announced | unknown
      "product_url": "https://corsair.com/..."
    },
    {
      "color": "black+red",
      "edition": "cyberpunk-2077-edition",
      "sku": "CH-931A1B1-CP",
      "availability": "discontinued",
      "product_url": "https://corsair.com/..."
    }
  ],

  "hex_map": {                             // individual color → hex (for CSS vars)
    "black": "#3A3F41",
    "red": "#ef4444",
    "white": "#ffffff",
    "orange": "#f97316",
    "blue": "#3b82f6"
  },

  "search_sources": ["corsair.com", "amazon.com", "bestbuy.com"],
  "confidence": 0.95
}
ℹ

hex_map keys are individual color atoms, not "+" strings. The "+" string "black+red" is rendered by splitting on "+" and looking up each atom in the hex_map. This matches how the EG site's getCircleStyle() works — it splits on "+" and maps each segment to var(--color-{name}).

LLM Configuration

Discovery: GPT-5.4 LLM

xhigh thinking + web search. Finds studio cutout image URLs from manufacturer sites and major retailers. Returns direct image URLs (not page URLs).

Validation: Gemini Pro LLM

Vision model confirms: (a) correct product, (b) correct angle, (c) studio cutout quality (clean background, no watermarks). Rejects failures.

Image Quality Gate

  • White/transparent background only (studio cutout)
  • No watermarks, no lifestyle photography, no box art
  • Minimum resolution: 400px on shortest side
  • Max file size: 10MB
  • Format: PNG preferred, JPEG/WebP accepted (converted to PNG)

12

Feature: URL Validation (Separate Feature & Tab)

ℹ

Separate full feature. URL Validation has its own dedicated ops tab and panel — it is NOT part of the Identity Validation panel in IndexLab. This is the feature that finds and verifies all field key/values per source URL. It uses reference images from Visual Verification as identity anchors.

Per-URL accuracy verification using LLM vision. For each crawled URL, the model checks whether the page data actually belongs to the correct product, and whether extracted field values are consistent with what the page shows.

What Validation Answers

Identity Match

Does this page contain information about this exact product? Catches comparison pages, wrong variants, bundled products.

Field Consistency

For each known field value visible on this page, does it match what we have? Catches extraction errors, outdated data.

New Discoveries

Are there values visible that we don't have yet? Catches missed extractions.

Source Quality

Combined identity + field accuracy = quality score per URL. Feeds back into source ranking.

13

The Identity Problem

⚠

Core invariant: We never validate data for the wrong item. Every validation call includes sufficient identity evidence to prevent misidentification — even on comparison pages, accessory listings, or multi-product pages.

Identity Evidence Package (Sent Per Validation Call)

LayerContentTokensPurpose
Identity Anchor { brand, model, variant, sku } ~50 Text-level product identity
Reference Image Studio cutout PNG from Visual Verification (is_reference: true) ~1,000 Visual anchor — the LLM sees what the product looks like
Known Values Current accepted field values from item_field_state ~200–500 Context for consistency checking
Source Screenshot Full page screenshot from crawl artifacts ~2,000 The actual page being validated

Total per call: ~3,500–4,000 tokens. Acceptable for accuracy-critical work.

Why the Reference Image Matters

Failure Modes Without Visual Anchor Risk
  • Comparison pages (rtings.com) — specs for 2+ products side-by-side. LLM may pick the wrong column.
  • Variant confusion — "M75 Air" vs "M75 Air Wireless" differ by one word but look different.
  • Accessory listings — product page also shows compatible accessories with their own specs.
  • Bundled products — retailer bundles keyboard + mouse. Page has specs for both.

What This Gets You Automatically

  • GUI settings tabs: Run generateLlmPhaseRegistry.js → new tabs appear in LLM Config for each phase
  • Per-phase model override: User can set a different model for Color & Edition vs Visual Verification vs Release Date
  • Per-phase token budget: Each phase can have its own max output tokens
  • Per-phase timeout: Web search phases can have longer timeouts (120s) without affecting the crawl pipeline
  • Per-phase reasoning toggle: Enable/disable thinking mode per phase
  • Cost tracking: Each phase's LLM calls are tagged with the phase ID for billing breakdown
  • Category-scoped overrides: Via llm_route_matrix table, different categories can use different models per phase
