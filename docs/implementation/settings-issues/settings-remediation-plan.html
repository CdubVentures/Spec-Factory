1. Current Architecture

All global settings (LLM config + pipeline settings + UI toggles) persist to the same settings table in app.sqlite, sectioned by runtime / ui. Two independent HTTP handlers write to the runtime section via a DELETE-all + INSERT-all pattern. The live server reads from an in-memory config object, not SQL directly.

Frontend (two pages, one Zustand store) Pipeline Settings ──PUT /runtime-settings──┐ ├──> SAME SQL section='runtime' (136 keys) LLM Config ────────PUT /llm-policy─────────┘ DELETE ALL + INSERT ALL on every save │ ▼ app.sqlite ──mirror──> user-settings.json │ applyRuntimeSettingsToConfig(config, ...) │ ▼ In-memory config object (read by all backend consumers via configValue / settingsAccessor)

How Each Handler Builds Its Snapshot

PUT /runtime-settings (pipeline panel) base = snapshotRuntimeSettings(config) ← reads from live config (has boot defaults) overlay incoming body keys on top persist entire 136-key snapshot PUT /llm-policy (LLM panel) base = getUserSettingsState().runtime ← reads from last persisted SQL snapshot overlay only LLM_POLICY_FLAT_KEYS (25 keys) persist entire 136-key snapshot Problem: Different bases. Both handlers build their full snapshot BEFORE calling persist. Even if persist is serialized, the snapshot was already constructed from stale state. Last writer wins.

The Core Fix (v2)

Target architecture: Handlers submit patches only (just the keys they changed). The persistence layer does the canonical merge: Handler A ──patch {llmModelPlan: 'gemini-2.5-flash'}──┐ │ serialized queue Handler B ──patch {maxRunSeconds: 86400}───────────────┤ (one at a time) │ ▼ Inside critical section: 1. read current state from SQL 2. merge patch on top 3. UPSERT only changed keys 4. apply to live config 5. update userSettingsState 6. release lock

2. Issue Registry

SET-001 Two-Writer Race Condition Blocks Runs

Both /runtime-settings and /llm-policy write the full section='runtime' (136 keys) via DELETE-all + INSERT-all. Both build their snapshot before calling persist. Concurrent saves cause last-writer-wins data loss.

Symptom: User saves a pipeline timeout, then quickly adjusts an LLM model. One write is lost on reload.

Why a write lock alone is insufficient: Serializing persistCanonicalSections() does not stop stale payloads that were already constructed by the handler before entering the queue.

userSettingsService.js:312-338 (writeSettingsToAppDb — DELETE-all + INSERT-all) • configRuntimeSettingsHandler.js:56-131 (builds snapshot before persist) • llmPolicyHandler.js:48-58 (builds snapshot before persist)
SET-002 Stale Base Reads Data Loss

The LLM handler reads its base from getUserSettingsState().runtime (the cached SQL load), not from live config. If SQL had "[]" for the registry, that stale value becomes the base for the merge. The runtime handler reads from config which is better but still mutable mid-request.

Symptom: API keys or registry revert to empty after saving LLM settings because the base snapshot was stale.

llmPolicyHandler.js:48-51 (stale read) • configPersistenceContext.js:17,49 (userSettingsState updated only after persist returns)
SET-003 API Key Desync (Registry vs Flat) Blocks Runs

Entering an API key in the Provider Registry panel stores it in providerRegistry[idx].apiKey (JSON blob) but does NOT sync to the flat geminiApiKey config key. The routing snapshot and resolved_api_keys read from the flat key — both report "missing."

Symptom: "LLM Keys Missing" lockout persists even after entering valid keys. Pipeline runs blocked.

LlmConfigPage.tsx:184-199 (onRegistryChange — fixed in Phase 0) • routing.js:95-121 (baseRouteForRole fallback chain) • configIndexingMetricsHandler.js:64-69 (resolved_api_keys)
SET-004 Empty Registry Persists to SQL Blocks Runs

llmProviderRegistryJson = "[]" gets written to SQL. On next boot, shouldSkipBootstrapRuntimeOverride skips it (preserving defaults), but the post-persist live-mode apply does NOT skip it — overwriting the good default with "[]".

Symptom: Provider registry is empty. All model routing fails. All roles show api_key_present: false.

userSettingsService.js:611-630 (shouldSkipBootstrapRuntimeOverride) • configPersistenceContext.js:21-28,36 (live-mode apply after persist) • llmPolicyHandler.js:65-71 • configRuntimeSettingsHandler.js:132-138 (guards — added in Phase 0)
SET-005 Shared Store Refresh Blocking Across Pages Stale UI

The shared runtimeSettingsValueStore blocks hydrate() when dirty=true. This is the /runtime-settings hydration path used by the pipeline settings authority. The LLM page uses hydrateKeys() which is not blocked. The real conflict: when the user navigates from pipeline (dirty) to another page, the store stays dirty until the unmount flush (sendBeacon) completes. During that window, /runtime-settings re-hydration is blocked, leaving stale pipeline values in the shared store.

Symptom: After navigating between settings pages with pending edits, the new page may show stale values that don't match the server.

runtimeSettingsValueStore.ts:45-56 (hydrate guard — blocks when dirty) • useLlmPolicyAuthority.ts:94 (uses hydrateKeys, NOT the blocked path) • useSettingsAutoSaveEffect.ts:88,117-147 (unmount flush via sendBeacon)
SET-006 Spec Seeds Missing From Propagation Cosmetic

Backend broadcasts spec-seeds-updated with domains: ['spec-seeds'], but the frontend invalidation resolver has no template for it. Falls back to the broad FALLBACK_QUERY_TEMPLATES which invalidates many (not all) queries — more expensive than necessary.

Symptom: Editing spec seeds triggers unnecessarily broad cache invalidation instead of targeted invalidation.

specSeedsRoutes.js:43-49 • invalidationResolver.js:9-134,205-230 (no 'spec-seeds' template; falls back to FALLBACK_QUERY_TEMPLATES) • settingsPropagationContract.ts:3-8 (no 'spec-seeds' domain)
SET-007 Persistence Context Stale After Concurrent Writes Data Loss

getUserSettingsState() returns the last successfully persisted state. config and userSettingsState are only updated after persistence returns (configPersistenceContext.js:49), and persistUserSettingsSections() still awaits the JSON mirror after the SQL write (userSettingsService.js:505). There is a stale window between SQL commit and state update.

Symptom: Same as SET-001/SET-002 — intermediate writes are lost.

configPersistenceContext.js:17,49,72

Patch-Based Persistence With Queued Merge SET-001 SET-002 SET-004 SET-007

This is the single backend change that eliminates the write-loss problem. Handlers stop building full 136-key snapshots. Instead, they submit patches (only the keys they changed). The persistence layer owns the canonical merge inside a serialized critical section.

Key insight (from peer review): Serializing persistCanonicalSections() alone is not sufficient. Both handlers build their snapshot before persistence is called. The merge must happen inside the critical section, reading current state at that point, not before.

New Persistence API

// configPersistenceContext.js
async function mergeRuntimePatch(patch) {
  // Enqueue — only one merge runs at a time
  return runtimeWriteQueue.enqueue(async () => {
    // 1. Read current canonical state (from SQL, inside the lock)
    const current = readSettingsFromAppDb(appDb).runtime;
    // 2. Merge patch on top
    const merged = { ...current, ...sanitizeRuntimeSettings(patch) };
    // 3. Write ONLY changed keys (UPSERT, no DELETE-all)
    upsertRuntimeKeys(appDb, patch);
    // 4. Update in-memory state
    userSettingsState = { ...userSettingsState, runtime: merged };
    applyRuntimeSettingsToConfig(config, patch);
    // 5. Best-effort JSON mirror (full snapshot for seed cache)
    await mirrorToJson(merged);
    return merged;
  });
}

Handler Changes

  • configRuntimeSettingsHandler.js — Stop building nextRuntimeSnapshot from snapshotRuntimeSettings(config). Instead, collect only the keys from the incoming body that passed validation into a runtimePatch object. Call persistenceCtx.mergeRuntimePatch(runtimePatch).
  • llmPolicyHandler.js — Stop reading getUserSettingsState().runtime. Instead, pass flatKeys (the 25 LLM keys from disassembleLlmPolicy) directly to persistenceCtx.mergeRuntimePatch(flatKeys).
  • userSettingsService.js — New function upsertRuntimeKeys(appDb, patch) that does INSERT OR REPLACE per key. No DELETE FROM settings WHERE section='runtime'. Existing keys not in the patch are untouched.

What This Fixes

  • SET-001 (race): Writes are serialized. Each merge reads current SQL state inside the lock. No stale snapshots.
  • SET-002 (stale base): No handler builds a base anymore. The persistence layer reads current state itself.
  • SET-004 (empty registry): Handlers never send "[]" unless the frontend explicitly cleared it. Unrelated saves don't touch llmProviderRegistryJson at all.
  • SET-007 (context stale): userSettingsState is updated inside the lock, before the next queued write reads it.

Scope

  • 3-4 files: configPersistenceContext.js, userSettingsService.js, configRuntimeSettingsHandler.js, llmPolicyHandler.js
  • Tests: characterization tests for sequential execution, partial write preservation, concurrent merge correctness
After Phase 1: The empty-registry guards from Phase 0 can be removed. The DELETE-all + INSERT-all pattern in writeSettingsToAppDb is no longer used for runtime writes (still used for ui/convergence/storage sections which are small and single-writer). The JSON mirror still writes the full snapshot for seed-cache purposes.

Frontend Store Transition Safety SET-005

Fix the shared Zustand store's refresh blocking so page transitions don't leave stale data.

Corrected Diagnosis

The LLM page uses hydrateKeys() (useLlmPolicyAuthority.ts:94) which merges without checking the dirty flag — it is never blocked. The blocked path is hydrate() (runtimeSettingsValueStore.ts:50), which is the /runtime-settings hydration used by the pipeline settings authority. When the user navigates away from pipeline settings with dirty edits, the store stays dirty until the unmount flush completes. During that window, any hydrate() call (from React Query refetch, WebSocket invalidation, etc.) is silently dropped.

Changes

  • runtimeSettingsValueStore.ts — Scope the dirty flag per save-domain. hydrate() should only block if the domain being hydrated has pending edits, not if any domain anywhere is dirty.
  • useSettingsAutoSaveEffect.ts — On unmount, clear the domain's dirty flag after flush. If the flush is fire-and-forget (sendBeacon), mark the domain as "flush-pending" so hydration knows to wait briefly rather than block indefinitely.
  • Alternative (simpler): force a synchronous save on unmount (via flushIfDirty()) instead of fire-and-forget sendBeacon, so dirty is cleared before the new page mounts.

Scope

  • 2-3 files: runtimeSettingsValueStore.ts, useSettingsAutoSaveEffect.ts, authority hooks
  • Test: navigate pipeline → LLM with pending edits, verify fresh data loads on new page
  • Independent of Phase 1 — can run in parallel
