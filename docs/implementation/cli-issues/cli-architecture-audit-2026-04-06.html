1. Current Architecture

How it works today

The CLI is a process-boundary layer. The GUI/API server spawns node src/app/cli/spec.js <command> as a child process for all work — crawls, compilation, seeding, review, exports. The child writes JSON to stdout, then process.exit(). Completion handlers in the API server then sync results into SQLite.

GUI Button Click
  → POST /process/start  (processRoutes.js:95)
  → spawn('node', ['src/app/cli/spec.js', 'indexlab', ...args])
  → stdout/stderr streamed to GUI via WebSocket
  → child.on('exit')
    → compileProcessCompletion.js   (JSON → SQL sync)
    → indexLabProcessCompletion.js   (run status reconciliation)
  → broadcastWs('process-status')

Command dispatch

spec.js (457 LOC) is the entry point. It parses args, loads .env, builds config, creates storage, then dispatches via a 26-case switch statement to lazy-loaded command handlers.

Lazy loading pattern

const loadFieldRulesCommands = createLazyLoader(async () => {
  const [{ createFieldRulesCommands }, ...] = await Promise.all([...]);
  return createFieldRulesCommands({ ...injectedDeps });
});

// Dispatch:
case 'compile-rules':
  return (await loadFieldRulesCommands()).commandCompileRules(config, storage, args);

2. CLAUDE.md Compliance Findings

Compliance checklist

  • CQRS Dual-State: JSON = durable, SQLite = projection (review workflow is textbook)
  • O(1) Feature Scaling: Pipeline phase registry is add-one-entry
  • Dependency Injection: All commands use factory pattern
  • No circular dependencies between CLI and features
  • Lazy loading prevents startup penalty
  • No hardcoded secrets
  • Runtime dirs use .workspace/ correctly
  • Feature boundary violations: direct internal imports (not through index.js)
  • DB path SSOT: 4 competing resolution patterns
  • Test coverage: 7/26 commands tested (domain logic untested)
  • Dead commands: export-overrides / migrate-overrides not wired in dispatcher
  • Rebuild contract violation: migrate-to-sqlite phases 2 & 4 one-way
  • Unused _storage parameter in 15+ command signatures
  • pipelineCommands.js at 495 LOC (exceeds soft limit)
  • reviewCommand.js at 362 LOC with 11 subcommands in one function
  • Process isolation used for pure functions (compile, validate) that don't need it

4. All Issues (Categorized)

Critical — CLAUDE.md violations

F1. Feature boundary violations — direct internal imports

pipelineCommands.js bypasses public barrel exports:

  • :11 — import { buildJobFromDb } from '../../../features/indexing/orchestration/bootstrap/buildJobFromDb.js'
  • :13 — import { buildRuntimeOpsPanels } from '../../../features/indexing/api/builders/buildRuntimeOpsPanels.js'
  • :16 — import { deriveFullModel } from '../../../features/catalog/identity/identityDedup.js'
  • :6 — import { SpecDb } from '../../../db/specDb.js' (direct, not injected)
  • :7-10,12,14-15 — 6 more direct imports from pipeline/checkpoint, indexlab internals

CLAUDE.md: "Features must not import other features' internals" / "Use explicit public contracts"

F2. DB path resolution — 4 competing patterns (O(1) SSOT violation)
  • Pattern A: openSpecDbForCategory() in cliHelpers.js — reads SPEC_DB_DIR env, config.specDbDir, fallback .workspace/db
  • Pattern B: PROJECT_ROOT/.workspace/db/ hardcoded in pipelineCommands.js:18-20 (used 3 times: lines 107, 150, 392)
  • Pattern C: config.specDbDir || '.workspace/db' inline in dataUtilityCommands.js:19,62
  • Pattern D: Dynamic import('../../../db/specDb.js') in pipelineCommands.js:149 and dataUtilityCommands.js:16,57

CLAUDE.md: "O(1) Feature Scaling — STRICTLY FORBIDDEN from writing code that requires a human to manually touch [multiple files] to add a simple setting"

F3. Test coverage gap — 19 commands untested

CLAUDE.md requires tests for "domain logic, state transitions, parsers/mappers/transforms, boundary contracts, error handling." Untested commands with domain logic:

  • batchCommand.js — scoring algorithms (explore/exploit/bandit), concurrency pool, brand reward index
  • fieldRulesCommands.js — 9 commands that parse args and delegate to compiler
  • dataUtilityCommands.js — seed-db, seed-checkpoint (DB operations)
  • pipelineCommands.js — run-one, run-ad-hoc, run-until-complete (only indexlab has characterization test)
  • publishingCommands.js — product-reconcile
F4. Rebuild contract violation — migrate-to-sqlite phases 2 & 4

Phase 2 imports billing ledger JSONL → SQL. Phase 4 imports learning profiles JSON → SQL. No reverse path exists. If spec.sqlite is deleted, billing entries and learning profiles are lost.

CLAUDE.md: "Any new SQL table must support a 'Deleted-DB Rebuild'"

Medium — maintainability issues

F5. Dead commands — export-overrides / migrate-overrides not in dispatcher

export-overrides is in the switch (spec.js:360-361), but migrate-overrides is exported from exportOverridesCommand.js:76 and never wired into the dispatcher or usage text. The lazy loader only creates the export command, not the migrate command.

F6. Process isolation overkill for pure functions

The API server spawns a child process for compile-rules and validate-rules (studioRoutes.js:176,187). These are pure JSON→JSON transforms completing in <5s. The spawn adds: PID tracking, orphan detection, completion handler re-sync, stdout parsing. All unnecessary — these functions are directly importable.

CommandNeeds process isolation?Why / why not
indexlab, run-*, run-batchYESBrowser processes, hours-long, crash isolation
compile-rules, validate-rules, category-compileNOPure functions, <5s, no spawned children
discoverBORDERLINEFast but calls pipeline internals
billing-report, llm-health, list-fields, etc.NOThin wrappers over importable functions
F7. pipelineCommands.js — 495 LOC, 11 direct imports, mixed concerns

commandIndexLab alone is 325 lines handling: seed parsing, identity resolution, product ID dedup, fixture creation, SpecDb opening, IndexLabRuntimeBridge setup, IPC event forwarding, config overrides, runProduct orchestration, checkpoint writing, summary serialization, and bridge finalization. This violates the 700 LOC soft limit and single-responsibility.

F8. SpecDb open/close ceremony duplicated

The same pattern appears 15+ times across commands:

const specDb = await openSpecDbForCategory(config, category);
try {
  // ... work
} finally {
  try { specDb?.close(); } catch { /* no-op */ }
}

Should be a single withSpecDb(config, category, fn) helper.

F9. Unused _storage parameter in 15+ command signatures

Commands like commandSeedDb(config, _storage, args), all fieldRulesCommands, generateTypes, exportOverrides — receive storage but never use it. The uniform (config, storage, args) signature is enforced by the dispatcher but not needed by these handlers.

Low — nice-to-have

F10. reviewCommand.js — 11 subcommands in one function

362 LOC, 11 if (action === '...') blocks. Each block is self-contained. Could decompose into per-action handlers for readability.

F11. run_profile hardcoded to 'standard'

spec.js:414 — output.run_profile = 'standard' ignores the --profile / --thorough CLI flags. These flags are parsed in args but never applied to the output.

F12. filterKeysByBrand reads JSON for brand lookup

cliHelpers.js:39-57 — When filtering by brand, reads every product JSON file via storage.readJsonOrNull(key). The DB getAllProducts() already returns brand. Could use DB brand directly and skip the JSON reads.

5. Phased Remediation Plan

Phase 1 — Unify DB access & fix boundary violations

effort: small — ~2-3 hours

1a. Single DB path resolver

Create one function that all commands use. Kill the 4 competing patterns.

// cliHelpers.js (replace current openSpecDbForCategory)
export async function openSpecDbForCategory(config, category) {
  const { SpecDb } = await import('../../db/specDb.js');
  const root = process.env.SPEC_DB_DIR || config.specDbDir || '.workspace/db';
  const dbDir = pathNode.join(root, String(category).trim());
  await fsNode.mkdir(dbDir, { recursive: true });
  return new SpecDb({ dbPath: pathNode.join(dbDir, 'spec.sqlite'), category });
}

1b. Add withSpecDb helper

export async function withSpecDb(config, category, fn) {
  const specDb = await openSpecDbForCategory(config, category);
  if (!specDb) return fn(null);
  try { return await fn(specDb); }
  finally { try { specDb.close(); } catch { /* */ } }
}

1c. Fix pipelineCommands.js imports

  • Delete PROJECT_ROOT constant (lines 18-20)
  • Delete direct import { SpecDb } (line 6)
  • Inject openSpecDbForCategory via the factory (already done for other commands)
  • Route all feature imports through public barrels:
    • buildJobFromDb → through features/indexing/orchestration/index.js
    • buildRuntimeOpsPanels → through features/indexing/api/index.js
    • deriveFullModel → through features/catalog/index.js
  • Inject checkpoint builders via factory instead of direct imports

1d. Wire or delete migrate-overrides

Either add it to the lazy loader + dispatcher + usage text, or remove the export entirely.

Extract shared identity resolution

Both commandIndexLab (lines 104-117) and commandRunAdHoc (lines 389-401) contain identical brand+model+variant lookup logic. Extract to a shared helper:

// shared/resolveProductId.js
export function resolveProductIdFromDb(specDb, { brand, model, variant, category }) {
  const rows = specDb?.getAllProducts?.() || [];
  const match = rows.find(r =>
    eq(r.brand, brand) && eq(r.base_model, model) && eq(r.variant, variant)
  );
  return match?.product_id || buildProductId(category);
}

Phase 3 — In-process compile/validate (optional)

effort: medium — ~3-5 hours

Instead of spawning node spec.js compile-rules as a child process, call the function directly from the API server. This eliminates process overhead for sub-5-second operations.

Current flow (spawned)

POST /studio/:category/compile
  → startProcess('src/app/cli/spec.js', ['compile-rules', '--category', cat])
  → child writes JSON artifacts to disk
  → child exits
  → handleCompileProcessCompletion() re-reads JSON, syncs to SQL

Proposed flow (in-process)

POST /studio/:category/compile
  → const result = await compileRules({ category, config })
  → handleCompileProcessCompletion() syncs to SQL (using result directly)

What this eliminates

  • Child process spawn + PID tracking
  • Stdout/stderr parsing
  • Orphan detection for compile processes
  • ~50-100ms latency per invocation

What stays spawned

  • indexlab, run-one, run-ad-hoc, run-batch, run-until-complete — browser processes, long-running
  • discover — calls into pipeline
  • intel-graph-api — starts a server
