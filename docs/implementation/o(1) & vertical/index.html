Recommended Refactor Order

The sequence below is optimized for the fastest drop in change cost and the biggest reduction in cross-slice coupling.

Priority Move Expected Payoff
1 Collapse src/pipeline into feature-owned indexing orchestration. Removes the biggest split-runtime authority and cuts the highest-value non-vertical drift.
2 Unify src/api and src/app/api around one mounted route SSOT. Route changes become safer, more local, and easier to reason about.
3 Move category/source authority fully into one indexing-owned registry chain. Tier and source work gets much closer to true one-file change cost.
4 Delete src/utils as a category by relocating each helper to core, shared, or the owning feature. Restores ownership clarity and removes generic-helper gravity.
5 Put IndexLab behind a stable exported indexing contract. Indexing internals can change without breaking the run-lab surface.
6 Merge src/cli and src/app/cli into one command assembly boundary. CLI changes stop crossing two orchestration stacks.
7 Then absorb queue, publish, retrieve, and intel into owning slices or clear core contracts. This cleans up the second wave after the highest-friction boundary splits are gone.
8 Split the giant testing harness files after runtime boundaries are stabilized. Prevents test-surface refactors from being invalidated by continuing runtime architecture churn.

Bottom Line

Strictly speaking, the repo has 28 non-canonical top-level directories. Practically, the number that should drive planning is 13 major offender areas. Of those, the most expensive blockers are the split pipeline, split API assembly, split source authority, src/utils, IndexLab coupling, and the two-headed CLI surface.

Method: live scan of the checked-out repo on April 2, 2026. "Major offender" means the area is both structurally non-vertical and likely to increase normal change cost beyond O(1) by splitting authority, bypassing feature public APIs, or centralizing too much cross-slice logic.
