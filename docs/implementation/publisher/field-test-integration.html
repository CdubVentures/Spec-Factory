Field Test — Universal Contract Compliance Proof

The field test takes any category's contract and deterministically proves the validator returns values in the shape the contract declares. The contract IS the test. Zero hardcoded data. Zero category-specific logic. Add a category → field test auto-generates proof.

1. What the Field Test Is

You design a category's field rules in Field Studio: types, shapes, units, enums, ranges, components, constraints, list rules. That contract defines the exact shape every value must have when it reaches the product record.

The field test answers one question: "Does my contract actually enforce what I designed?"

It reads your contract, generates synthetic data that exercises every dimension the contract declares, runs that data through the validator, and checks that every value came back in the declared shape. If you say dpi is a number with range 100–30000, the field test generates an in-range value and an out-of-range value, feeds both through the validator, and proves the in-range value passes and the out-of-range value is rejected.
Universal by design
Add a new category tomorrow with 200 fields. The field test auto-generates scenarios from those 200 fields and proves the validator enforces every one. No new test code. No hardcoded values. The contract drives everything.

2. How It Works

1 Read the Contract
analyzeContract() reads your category's compiled rules from DB (field_studio_map.compiled_rules). Extracts every field's type, shape, unit, enum policy, range, component references, cross-validation rules, list rules. Builds three matrices: field rules, components, enums. Auto-generates scenario definitions.
↓ derive scenarios
2 Generate Deterministic Test Data
buildDeterministicSourceResults() creates synthetic source data for each scenario. Every value is derived from the contract — correct values, wrong-type values, out-of-range values, unknown components, invalid enums, missing required fields. Five fake sources per product, each with field candidates and evidence.

The inputs are deterministic and reproducible. Same contract → same test data every time.
↓ validate
3 Run Through the Validator
runTestProduct() resolves the best value per field from the synthetic sources, then calls validateRecord() which runs every value through the 12-step deterministic pipeline: absence → template dispatch → shape → unit → type → normalize → format → list rules → rounding → enum → range → publish gate. Cross-field constraints run separately in validateRecord. Component resolution is a review-phase concern.

Produces: validated fields, per-field rejections, per-field repairs, cross-field constraint results, curation suggestions.
↓ assert
4 Assert Contract Compliance
buildValidationChecks() asserts the exact output for each scenario:
• happy_path: every field populated, correct type, correct shape, coverage ≥ 70%
• closed_enum_reject: specific fields rejected with enum_value_not_allowed
• range_violations: specific fields rejected with out_of_range
• new_{component}: curation suggestion emitted for unknown component
• Every scenario maps to a contract dimension and asserts the validator enforced it.

3. What Each Scenario Proves About Your Contract

Every scenario is auto-generated from your contract. If your contract doesn't declare component references, the component scenarios aren't generated. If it has no closed enums, the closed_enum scenario is skipped. The test is shaped by what you designed.
ScenarioContract DimensionWhat the Assertion Proves
happy_path All fields, all rules Every field passes validation. Output type matches contract.type. Output shape matches contract.shape. No rejections. Coverage ≥ 70%.
missing_required priority.required_level Required fields that weren't provided are reported in missing_required_fields[]. The validator doesn't invent values.
closed_enum_reject enum.policy: closed A value not in known_values for a closed-enum field is rejected with enum_value_not_allowed. The validator doesn't let invalid enum values through.
new_enum_values enum.policy: open_prefer_known A new value for an open_prefer_known field passes but produces a curation suggestion. The validator accepts it while flagging it for human review.
similar_enum_values parse.token_map + enum matching A near-match value (e.g. "grey" for a catalog with "gray") is normalized to the canonical form. The validator applies token maps before rejecting.
range_violations contract.range A value outside min/max bounds is rejected with out_of_range. The validator enforces the range constraint.
new_{component} parse.template: component_reference An unknown component name produces a new_component curation suggestion. The validator flags it for human verification.
similar_{component} Component alias matching A near-match component name resolves to the canonical identity via alias lookup. The validator finds the right component.
component_constraints Component property constraints A component property that violates its constraints (e.g. sensor date after release date) is flagged. The validator checks component-level rules.
variance_policies Component variance policies Component properties with non-authoritative variance policies (upper_bound, override_allowed) have scoring applied. The validator respects variance rules.
cross_validation cross_validation_rules Field combinations that violate cross-field constraints are flagged. The validator checks inter-field consistency.
list_fields_dedup contract.list_rules Duplicate values in list fields are removed. Sorting applied per rule. Min/max items enforced. The validator maintains list integrity.
multi_source_consensus Candidate resolution When 4 sources disagree, the highest-scored candidate wins. Fields are resolved, not left ambiguous.
min_evidence_refs evidence.min_evidence_refs Fields requiring 2+ evidence sources but receiving only 1 are flagged as below pass target.
tier_preference_override Source tier ranking When a tier-2 source is preferred over tier-1 for a field, the correct value is selected.

6. Tightening the Assertion Layer (After Validator)

Today's buildValidationChecks() uses loose assertions ("has fields", "has confidence"). After the validator is wired in, assertions should be tightened to exact shape proofs. The contract declares the expected output — the assertion verifies it precisely.
ScenarioToday (loose)After Validator (exact proof)
happy_path At least 1 field present. Has confidence value. For every field in field_order: output type === contract.type. Output shape === contract.shape. Zero rejections. If contract.range exists, value is within bounds. If contract.unit exists, value is unitless number.
closed_enum_reject Failures array has entries with "enum" reason. For each closed-enum field that received an invalid value: rejections contains { reason_code: 'enum_value_not_allowed', field_key: X }. The output value is 'unk'.
range_violations Failures exist for out-of-range fields. For each field with contract.range that received an out-of-bounds value: rejections contains { reason_code: 'out_of_range' }. The output value is 'unk'. The rejection detail shows expected min/max vs actual.
new_{component} Curation suggestion exists. A new_component suggestion exists for the specific component type. The suggestion's value matches the fabricated unknown name. The component field's output is the raw value (not resolved to a canonical).
missing_required Missing required fields flagged. missing_required_fields contains exactly the fields with required_level: 'required' or 'critical' that were omitted from the source data. Count matches the contract's required field count minus the fields deliberately provided.
The contract drives the assertion
The assertion doesn't hardcode expected values. It reads the contract and derives what the output must look like. "This field is type number with range 100–30000" → assert typeof output === 'number' && output >= 100 && output <= 30000. The assertion is as universal as the test data generation.
