Candidate Gate & Publish — Implementation Plan

Three phases, thirteen sub-steps. Each sub-step has a clear deliverable and a completion proof. Phases build on each other — no phase starts until the previous phase’s proof is green.

Build Order & Rationale

Phase A builds the gate. Phase B wires a real source to prove candidates land correctly. Phase C builds resolution and publish — it can only be proven with real candidates from Phase B.
#PhaseDeliverableProof
A Candidate Gate submitCandidate() — validate, dual-write to field_candidates table + product.json candidates[] Unit tests green. Candidate rows appear in DB + JSON for valid input. Invalid input rejected, never persisted.
B Wire CEF Color & Edition Finder calls submitCandidate() after each run. First real source proves the gate. CEF run → candidates appear in field_candidates + product.json candidates[]. Review grid can read them.
C Publish Resolved publishResolved() — resolve winners, cross-field validate, dual-write to item_field_state + product.json fields[] Full E2E: CEF → candidate → resolve → item_field_state populated. Component fields → item_component_links.
A

Candidate Gate — submitCandidate()

5 sub-steps. Pure orchestration: validate → dual-write. No LLM, no component resolution.

A1 Schema — field_candidates table

What: Add field_candidates DDL to specDbSchema.js. Add store methods to specDb: upsertFieldCandidate(), getFieldCandidates(productId, fieldKey), getAllFieldCandidates(productId), deleteFieldCandidates(productId).

Deliverable: Table exists. Store methods have contract tests (insert, upsert/merge, query, delete).
CREATE TABLE IF NOT EXISTS field_candidates ( id INTEGER PRIMARY KEY AUTOINCREMENT, category TEXT NOT NULL, product_id TEXT NOT NULL, field_key TEXT NOT NULL, value TEXT, confidence REAL DEFAULT 0, source_count INTEGER DEFAULT 1, sources_json TEXT DEFAULT '[]', validation_json TEXT DEFAULT '{}', submitted_at TEXT DEFAULT (datetime('now')), updated_at TEXT DEFAULT (datetime('now')), UNIQUE(category, product_id, field_key, value) ); CREATE INDEX IF NOT EXISTS idx_fc_product ON field_candidates(product_id); CREATE INDEX IF NOT EXISTS idx_fc_field ON field_candidates(category, product_id, field_key);
Key design decision
UNIQUE(category, product_id, field_key, value) — same value from multiple sources merges into one row (sources accumulate in sources_json, source_count increments, confidence takes the max). Different values for the same field = different candidate rows.

A2 Contract — submitCandidate() boundary

What: Define the public function signature, all inputs, outputs, error cases, and invariants. No implementation yet — just the contract document.
// src/features/publisher/candidate-gate/submitCandidate.js submitCandidate({ // Identity category, // 'mouse' productId, // 'mouse-00345848' fieldKey, // 'colors', 'weight', 'sensor' // Payload value, // any shape: scalar, list, record confidence, // 0-100 sourceMeta, // opaque: { model, run_id } or { artifact, url, run_id } etc. // Injected dependencies fieldRules, // field_rules.json (full rule set) knownValues, // compiled + discovered enum values componentDb, // component DB accessors (for checkConstraints) specDb, // for field_candidates write + persistDiscoveredValue productRoot, // path to product directory for product.json write }) // Returns: → { status: 'accepted' | 'rejected', candidateId: number | null, // field_candidates.id (null if rejected) value, // post-validation value (may differ from input due to repairs) validationResult, // full validateField() output }
Invariants:
• No value reaches field_candidates without passing validateField().
• Invalid values are never persisted — not to DB, not to JSON.
• Same (product, field, value) from a new source → source appended, not duplicated.
• open_prefer_known fields trigger persistDiscoveredValue() on accept.
• Field rule missing for fieldKey → reject with reason: 'no_field_rule'.

A3 RED — test matrix for submitCandidate()

What: Write the full boundary test matrix. All tests fail (no implementation yet).

Test categories:
CategoryCases
Happy path Scalar value accepted → DB row + JSON candidate entry. List value accepted. Record value accepted.
Rejection Shape mismatch rejected. Type mismatch rejected. Closed enum unknown rejected. Range violation rejected. No field rule → rejected.
Repairs Input repaired (normalization, rounding, dedupe) → repaired value persisted, original not.
Source merge Same value submitted twice with different sources → one candidate row, two entries in sources_json, source_count = 2, confidence = max.
Discovery enums open_prefer_known field with new value → accepted + persistDiscoveredValue() called.
Dual-write On accept: DB row exists AND product.json candidates[fieldKey][] entry exists. On reject: neither written.
Edge cases value = null → absence-normalized to unk. Empty string → absence-normalized. Confidence = 0. Missing sourceMeta fields.

A4 GREEN — implement submitCandidate()

What: Minimum code to pass the full A3 test matrix.

Implementation steps (inside the function):
1. Look up fieldRules[fieldKey] — reject if missing.
2. Call validateField({ fieldKey, value, fieldRule, knownValues, componentDb }).
3. If validationResult.valid === false → return { status: 'rejected' }. No writes.
4. Build source entry: { ...sourceMeta, confidence, submitted_at: now() }.
5. DB write: specDb.upsertFieldCandidate() — insert or merge source into existing row.
6. JSON write: Read product.json, append/merge into candidates[fieldKey][], write back.
7. If open_prefer_known → call persistDiscoveredValue().
8. Return { status: 'accepted', candidateId, value, validationResult }.

A5 Rebuild contract — rebuildFieldCandidatesFromJson()

What: If the DB is deleted, rebuild field_candidates from product.json candidates[] for every product.

Deliverable: Delete DB → run rebuild → all candidate rows restored. Test proves round-trip.

Audit status: rebuild yes, source edit yes (per CLAUDE.md Dual-State mandate).

What: Write integration test: mock CEF output → call submitCandidate for each field → assert candidates in DB + JSON.

Test cases:
• CEF returns colors: ["black", "white"] → field_candidates row for colors exists, product.json candidates.colors[] exists.

What: Add submitCandidate() calls after mergeColorEditionDiscovery().

Implementation:
1. After CEF’s own dual-write (its JSON + SQL tables), loop over output fields.
2. For each field (colors, editions), call submitCandidate() with:

C

Publish Resolved — publishResolved()

5 sub-steps. Reads candidates, resolves winners, cross-validates, dual-writes to DB + JSON.

C1 Contract — publishResolved() boundary

What: Define the public function signature, resolution algorithm, and write targets.
// src/features/publisher/publisher/publishResolved.js publishResolved({ // Identity category, productId, // Injected dependencies fieldRules, knownValues, componentDb, crossRules, // cross_validation_rules.json specDb, // reads field_candidates, writes item_field_state productRoot, // reads/writes product.json }) // Returns: → { status: 'published' | 'blocked', resolved: Record<fieldKey, { value, candidateId, // winning field_candidates.id confidence, sourceCount, }>, crossFieldFailures: [], // from validateRecord() fieldsWritten: number, }
Invariants:
• Only reads from field_candidates — never from raw source data.
• Every resolved value passes validateRecord() (cross-field) before publish.
• item_field_state.accepted_candidate_id points to the winning field_candidates.id.
• This is the only path to item_field_state. No caller writes directly.

C2 Contract — resolution algorithm

What: Define how a winner is chosen when multiple candidate values exist for the same field.

Resolution order (tiebreakers):
1. Source count — more independent sources agreeing = higher weight.
2. Max confidence — highest confidence among sources for that value.
3. Recency — most recent submitted_at wins ties.

Single candidate: If only one candidate value exists for a field, it wins automatically (no scoring needed).

No candidates: Field not resolved. No write to item_field_state. Existing row (if any) remains.

C3 RED → GREEN — resolution + publish tests

What: Full boundary test matrix. Tests fail first, then implement.
CategoryCases
Single candidate One value for field → auto-wins. Written to item_field_state + product.json fields[].
Multi-candidate resolution Two values, different source counts → higher count wins. Same count, different confidence → higher confidence wins. Same both → most recent wins.
Cross-field validation Resolved set passes validateRecord() → published. Cross-field failure → status = 'blocked', failures returned.
Dual-write targets item_field_state upserted with value + confidence + accepted_candidate_id. product.json fields[fieldKey] written with value + confidence + resolved_at + sources.
Component linking Component field (e.g. sensor) resolved → item_component_links row written. Non-component field → no link row.
List linking List field (e.g. colors) resolved → item_list_links rows written (one per list value).
No candidates Field with zero candidates → not resolved. Existing item_field_state row unchanged.
Rebuild contract Delete DB → rebuild from product.json fields[] → item_field_state restored.

C4 GREEN — implement publishResolved()

Implementation steps (inside the function):
1. specDb.getAllFieldCandidates(productId) — read all candidates.
2. Group by fieldKey. For each field, run resolution algorithm (C2).
3. Build resolved field set: { fieldKey: winnerValue, ... }.
4. validateRecord({ fields, fieldRules, knownValues, crossRules, componentDbs }) — cross-field checks.
5. If cross-field failures with action: 'reject' → return { status: 'blocked' }.
6. DB writes (per resolved field):
   a. specDb.upsertItemFieldState() with accepted_candidate_id.
   b. If component field → specDb.upsertItemComponentLink().
   c. If list field → specDb.upsertItemListLinks() (one per list value).
7. JSON write: Update product.json fields[fieldKey] for each resolved field.
8. Return { status: 'published', resolved, crossFieldFailures, fieldsWritten }.

C5 E2E proof — full pipeline

What: End-to-end integration test proving the full flow on at least one real product.

Flow:
1. CEF runs on a product → submitCandidate() called for each field.
2. field_candidates rows exist (from Phase B).
3. publishResolved() called → reads candidates, resolves, cross-validates.
4. item_field_state rows exist with correct values + accepted_candidate_id.
5. product.json fields[] entries exist with value + confidence + resolved_at + sources.
6. For component fields: item_component_links row exists.
7. For list fields: item_list_links rows exist.

Rebuild proof: Delete DB → rebuild from product.json → all tables restored.
