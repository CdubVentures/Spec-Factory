Publish Pipeline — Feature Overview

One feature, three sub-modules. Every field value in the system — from any source — flows through this pipeline before it can reach the product record. Field Studio is the contract. The pipeline enforces it with 100% certainty.

Next: Build candidate-gate/ and publisher/ sub-modules, then re-wire review domain through the publish pipeline.

After

One pipeline, every source. No field value reaches storage without passing through the publish pipeline.

Field Studio defines the contract (type, shape, unit, enum, range, format, list rules, rounding, constraints). The pipeline enforces it deterministically. 12 checks, <1ms, zero LLM cost for valid data.

Two-phase write: validated candidates first, resolved winners second. Cross-field checks run on the full resolved set before final publish.

1 validation/

Pure functions. Zero dependencies.
No DB, no LLM, no side effects.
Only imports src/shared/.
Owns all logic internally — no src/engine/ imports.

Exports: validateField(), validateRecord()

Spec: universal-validator-reference.html

2 candidate-gate/

Source-side gate.
Called by any source after a run.
Calls validateField() per field.
Dual-writes valid candidates to product.json candidates[] + DB.
Rejects invalid values before they enter the candidate pool.

Exports: submitCandidate()

3 publisher/

Resolution + publish.
Takes validated candidate pool.
Resolves winner per field (consensus, confidence).
Calls validateRecord() for cross-field checks.
Dual-writes to product.json fields[] + item_field_state.

Exports: publishResolved()
Dependency chain (one-way, no cycles)
candidate-gate/ → validation/ (imports validateField)
publisher/ → validation/ (imports validateRecord)
validation/ → src/shared/ only. Zero engine imports. Standalone.

3. End-to-End Data Flow

Every field value in the system flows through this pipeline. Two phases: candidate submission (per-source) and resolution + publish (per-product).
SOURCE Any data source completes a run
Color & edition finder, pipeline extraction, review override, CLI manual entry, future sources.
Each produces raw field values: { fieldKey, value, confidence, run_id, source_meta }
↓ per field
PHASE 1 Candidate Gate — submitCandidate()
For each field the source produced:
1. Load the field rule from Field Studio (field_rules.json)
2. Call validateField({ fieldKey, value, fieldRule, knownValues, componentDb })
3. The 12 deterministic checks run: absence → template dispatch → shape → unit → type → normalize → format → list rules → rounding → enum → range → publish gate. (Component resolution is a review-phase concern, not a validation step.)
↓
VALID Persist as candidate
Dual-write:
• product.json → candidates[fieldKey][] (append)
• DB candidate table (with source, run_id, confidence, validation result)

Includes any repairs applied (normalization, rounding, dedupe, etc.)
INVALID Rejected — never persisted
Value fails Field Studio contract.
Returns rejection detail: { reason_code, dimension, detail }
Optionally returns repairPrompt for LLM repair.

Garbage never enters the candidate pool.
↓ after all sources have submitted
PHASE 2 Resolution — publishResolved()
1. Resolve winner per field from the validated candidate pool.
   Consensus scoring: source count, confidence, source tier ranking.
   Each candidate is already validated — any winner is shape/type/unit correct.

2. Cross-field validation via validateRecord() on the full resolved field set.
   Checks: default_color ∈ colors, edition.colors ⊆ colors, sensor_date ≤ release_date, wireless → battery_hours required, etc.
   Auto-repairs where possible (e.g. set default_color = colors[0]).
   Flags non-repairable inconsistencies for review.
↓ cross-field pass
PUBLISH Write resolved values
Dual-write the resolved, validated, cross-field-checked values:
• product.json → fields[fieldKey] (with value, confidence, resolved_at, sources[])
• item_field_state → upsert row (category, product_id, field_key, value, confidence, source)
• item_component_links → for component_reference fields
• item_list_links → for list-shaped fields

This is the only path to item_field_state. No caller writes directly.

4. Field Studio Is the Contract

The validator doesn't decide what's valid. Field Studio already decided. Every check in the pipeline reads from the rules you designed. The validator just enforces them.
DimensionField Studio RuleWhat the Validator ChecksOn Fail
Shape contract.shape: scalar | list | record Value struct matches declared shape REJECT
Type contract.type: string | number JS type matches. Auto-coerce where safe. REPAIR or REJECT
Unit contract.unit: mm | g | Hz | ... Same unit → strip suffix. Wrong unit → reject. REJECT
Format parse.template → format regex Post-normalization value matches template format UNKNOWN → LLM
Normalization parse.token_map + tooltip rules trim, lowercase, hyphens, token_map applied ALWAYS REPAIRS
Enum enum.policy: closed | open_prefer_known | open Value in known_values for closed. Flag for open_prefer_known. REJECT (closed)
List Rules list_rules: dedupe, sort, min, max Dedupe, sort, enforce item count bounds ALWAYS REPAIRS
Rounding ui.display_decimals: 0 | 1 | 2 Round to declared decimal places ALWAYS REPAIRS
Range contract.range: { min, max } Value within declared bounds REJECT
Component parse.template: component_reference Resolves in component_db (exact → alias) UNKNOWN → LLM
Cross-Field cross_validation_rules[] Inter-field constraints (DSL operators) REPAIR or FLAG

5. Who Calls What

Every source calls submitCandidate(). Resolution calls publishResolved(). No source ever writes directly to item_field_state or product.json fields[].
SourceCallsWhenWhat Happens
Color & Edition Finder submitCandidate() After LLM returns colors/editions/default_color Each field validated → persisted as candidate with source: 'cef', run_id, model
Pipeline Extraction submitCandidate() After LLM extraction per source URL Each field validated → persisted as candidate with artifact, url, confidence
Review Override submitCandidate() When human sets a value via review grid Override validated → persisted as candidate with source: 'override'
CLI Manual Entry submitCandidate() When human sets a value via CLI Value validated → persisted as candidate with source: 'cli'
Resolution Trigger publishResolved() After candidates collected (end of run, on demand, after override) Resolve winners → cross-field validate → dual-write to fields[] + item_field_state

7. Build Order

Each sub-module is built with its own contract + test matrix. RED → GREEN → REFACTOR. The dependency chain dictates the order.
#Sub-ModuleDepends OnNatureStatus
1 validation/ src/shared/ only Pure functions. 18 modules, 12 check dimensions, O(1) dispatch. Zero engine imports. COMPLETE — 458 tests
2 repair-adapter/ validation/ LLM repair orchestration. Takes repairPrompt from validator, calls LLM, re-validates. 3-attempt escalation chain. NEXT
3 candidate-gate/ validation/, repair-adapter/ Source-side gate. Calls validateField(), optionally repairs, dual-writes candidate. PLANNED
4 publisher/ validation/ Resolves candidates, calls validateRecord() for cross-field, dual-writes resolved values. PLANNED

8. Public API

// src/features/publisher/index.js // This is the only thing consumers import. // Phase 1: Any source calls this after producing field data. // Validates each field against Field Studio, persists valid candidates. export { submitCandidate } from './candidate-gate/submitCandidate.js'; // Phase 2: Called after candidates are collected. // Resolves winners, cross-field validates, publishes to product record. export { publishResolved } from './publisher/publishResolved.js'; // Direct access to validation (for testing, field test integration, etc.) export { validateField } from './validation/validateField.js'; export { validateRecord } from './validation/validateRecord.js';
