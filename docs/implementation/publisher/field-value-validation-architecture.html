Validation Feature — src/features/validation/

A new vertical feature module that deterministically validates any LLM-extracted field value against the field studio contract before it reaches the product record.
Independent of src/engine/. Reuses ruleAccessors and normalizationFunctions only. Tested against all 297 fields across 3 live categories.

1. Why This Exists

Today

The pipeline (src/pipeline/runProduct.js) has zero validation. LLM outputs go directly to storage unvalidated.

src/engine/ exists but is only wired to review (2 callers: qaJudge + overrideWorkflow). It has 11 proven gaps and is not universal.

Result: hallucinated values, wrong units, wrong types, and malformed tokens reach the product record.

Goal

A single validation gate that any source calls before publishing.

Deterministic first: shape, type, unit, format, enum, list rules, rounding, range, constraints — all pure functions, zero LLM cost, <1ms.

LLM repair only on mismatch: unknowns get a targeted repair prompt with the field contract + registered values.

O(1) scaling: adding a field or category requires zero code changes. The validator reads parse.template and the field contract from field_rules.json.

4. Validation Pipeline (12 deterministic steps)

Every step is a pure function. Steps run in order. Each receives the output of the previous. The pipeline short-circuits on shape failure (Step 1). All other steps accumulate repairs + rejections.
INPUT Raw LLM value + field rule
validateField({ fieldKey, value, fieldRule, knownValues?, componentDb? })
↓
1 Template Dispatch
Read fieldRule.parse.template (e.g. "latency_list_modes_ms"). Look up in NORMALIZATION_FUNCTIONS map. If found, call the template normalizer FIRST. If it handles the value, skip to Step 6 (enum). If no template match, continue to Step 2.
↓
2 Shape Check REJECT if wrong
Verify contract.shape: scalar → not array/object. list → must be array. record → must be object.
Short-circuits entire pipeline on failure.
↓
3 Type Verification
Verify contract.type: string → typeof === 'string'. number → typeof === 'number' + isFinite. integer → isInteger. Auto-coerce where safe: "42"→42, true→"yes".
↓
4 Unit Verification REJECT wrong unit
If contract.unit exists: parse value for unit suffix. Same unit → strip text ("120 mm"→120). Different unit → REJECT with wrong_unit + include expected/detected for repair prompt.
↓
5 Normalize + Format
Strings: trim → lowercase → hyphens → token_map. Then check format regex per template. "DARK GREEN" → "dark-green". "grey" → "gray" (via token_map).
↓
6 Enum Check
closed: value must be in known_values. Apply token_map before rejecting. open_prefer_known: pass but flag for curation review. open: pass.
↓
7 List Rules
For shape: list: dedupe → sort → enforce min_items / max_items. All in one pass. sort: 'none' preserves order (critical for colors[0] = default).
↓
8 Rounding
For numbers: round to ui.display_decimals or contract.rounding.decimals. display_decimals: 0 → round to integer. 2 → two decimal places.
↓
9 Range Check
For numbers with contract.range: verify min ≤ value ≤ max. Out of range → REJECT with out_of_range.
↓
PASS Value is valid
Return { valid: true, value, confidence: 1.0, repairs[], rejections: [] }
Caller publishes to product record.
Steps 10-11 (constraints + component resolution) run at record level via validateRecord(), not per-field, because they need access to other field values.

5. Template Dispatch (O(1) Scaling)

Every field has parse.template in its rule. The validator looks it up in a normalizer map. Adding a new template = add one function to the map. Zero changes to the pipeline.

No normalization_fn. That seam is dead (0 fields use it). Template name IS the dispatch key.
// templateDispatch.js import { NORMALIZATION_FUNCTIONS } from '../../engine/normalizationFunctions.js'; // Map template names to normalizer keys in the existing NORMALIZATION_FUNCTIONS map. // If the template name IS already a key in the map, it dispatches directly. // Otherwise, alias it here. const TEMPLATE_NORMALIZERS = { latency_list_modes_ms: 'parse_latency_list', // already in map list_of_numbers_with_unit: 'parse_polling_list', // already in map list_of_tokens_delimited: 'normalize_color_list', // already in map (for colors) boolean_yes_no_unk: 'normalize_boolean', // already in map date_field: 'parseDate', // already in map // Templates handled by the type-branch fallback (no special normalizer needed): // text_field, number_with_unit, integer_field, integer_with_unit, // url_field, component_reference, list_numbers_or_ranges_with_unit }; export function dispatchTemplate(templateName, value, context) { const normalizerKey = TEMPLATE_NORMALIZERS[templateName] || templateName; const fn = NORMALIZATION_FUNCTIONS[normalizerKey]; if (typeof fn !== 'function') return null; // no match → fall through to type branch return fn(value, context); }
