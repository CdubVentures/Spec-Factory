Field Value Validation Module

Architecture for validating LLM-extracted field values against field studio rules before publishing to the product record.
Deterministic normalization first, validate-phase LLM only on mismatch.

1. The Problem

Today

LLM outputs (color finder, pipeline extraction, future sources) go directly to storage with no validation against field studio rules.

The color registry is a closed vocabulary of 77+ atoms, but nothing enforces membership. Token maps exist in egPresets (grey→gray) but are never applied to actual values. The validate LLM phase exists but only handles enum consistency review for the curator UI.

Goal

A single validation gate between any LLM source and the field publisher.

Deterministic first: apply token_map, dedupe, enforce list_rules, check registry membership. Free and fast.

Validate LLM only on mismatch: if deterministic checks fail, call the validate phase to repair or recommend re-run.

Every future source (pipeline extraction, manual, community) validates through the same gate. O(1) scaling.

2. Validation Flow

SOURCE LLM Output (e.g., Color Edition Finder response)
{ colors: ["black", "Grey", "midnight blue"], editions: { "Launch Edition": { colors: ["black"] } }, default_color: "black" }
↓
always runs
STEP 1 Zod Shape Validation DETERMINISTIC
Validates struct: colors is array of strings, editions is record, default_color is string.
Already exists: colorEditionFinderResponseSchema. If shape fails → reject immediately, no repair possible.
↓
shape ok
STEP 2 Token Map Normalization DETERMINISTIC NEW
Apply fieldRule.parse.token_map from egPresets to each value.
"Grey" → "gray" · "gray-light" → "light-gray" · "light grey" → "light-gray"
Editions: lowercase + kebab-case — "Launch Edition" → "launch-edition"
Source: getEgPresetForKey(fieldKey, ctx).parse.token_map
↓
normalized
STEP 3 List Rules Enforcement DETERMINISTIC
Apply fieldRule.contract.list_rules: · dedupe: true — remove duplicates after normalization · max_items: 100 (colors) / 50 (editions) — truncate if over cap · sort: 'none' — preserve order (colors[0] = default)
↓
deduped & capped
STEP 4 Registry Membership Check DETERMINISTIC NEW
Colors (enum_policy: 'open' but registry is closed vocabulary):
Each color atom checked against appDb.listColors() (77+ registered names). Multi-color: split on +, check each atom. e.g. "black+red" → check "black" ✓ and "red" ✓

Editions (enum_policy: 'open'): no registry check needed. Any kebab-case slug is valid.

Cross-field invariants:
· default_color ∈ colors · every edition's colors ⊆ top-level colors
↓
ALL PASS
All values are registered atoms.
Cross-field invariants hold.
↓ Publish to product record
confidence: 1.0
UNKNOWNS EXIST
e.g. "midnight blue" not in registry.
Deterministic repair failed.
↓ Call validate-phase LLM
↓
unknowns only
STEP 5 Validate Phase LLM Call phase: validate
Only called when unknowns exist. Small, cheap prompt — just the failing values + full registry + field rules.

Input: unknown values, full color registry (names + hex), field formatting rules from egPresets
Task: For each unknown: map to registered value, confirm as new (open enum), or flag as hallucination
Model: Uses existing validate phase model deck (base + fallback, configurable per-phase in GUI)
↓
status: "repaired"
LLM mapped unknowns to registered values.
"midnight blue" → "dark-blue"
Apply repairs, re-run deterministic check.
↓ Publish with repair log
status: "rerun_recommended"
LLM says values look hallucinated.
"sparkle-unicorn" → no match
↓ Return to orchestrator
Orchestrator retries with escalation model.
3 strikes max → flag for manual review.

3. Data Contracts

Validator Input

{ fieldKey: 'colors', rawValue: ['black', 'Grey', 'midnight blue'], fieldRule: getEgPresetForKey('colors', ctx), registry: appDb.listColors(), // registry = [{ name: 'black', hex: '#000' }, ...] }

Validator Output

{ valid: true, value: ['black', 'gray', 'dark-blue'], confidence: 1.0, repairs: [ { original: 'Grey', resolved: 'gray', rule: 'token_map' }, { original: 'midnight blue', resolved: 'dark-blue', rule: 'validate_llm', llmConfidence: 0.95 } ], rejections: [], llmCalled: true, llmModel: 'gpt-4.1' }

Validate LLM Prompt (sent only on mismatch)

System: You validate field values against a registered vocabulary. For each unknown value: 1. If it maps to a registered value, return the mapping with confidence. 2. If it appears to be a genuine new value and the field allows open enum, confirm it. 3. If it looks hallucinated or nonsensical, flag it and recommend re-running the source. Field rules: - Registered atoms only (closed color vocabulary) - Modifier-first: "light-blue" not "blue-light" - Normalize "grey" to "gray" - kebab-case for editions User: { "field_key": "colors", "unknown_values": ["midnight blue"], "registered_values": [ "black (#000000)", "white (#ffffff)", "dark-blue (#1e3a5f)", "navy (#1e3a8a)", ... ], "formatting_rules": "lowercase, hyphens, ..." }

Validate LLM Response Schema

{ status: "repaired" | "rerun_recommended", reason: string | null, decisions: [ { value: "midnight blue", decision: "map_to_existing" | "keep_new" | "reject", resolved_to: "dark-blue" | null, confidence: 0.95, reasoning: "hex #191970 closest to dark-blue #1e3a5f" } ] } // status = "repaired" when all decisions // are map_to_existing or keep_new. // // status = "rerun_recommended" when any // decision is "reject" AND the LLM believes // the source data is fundamentally wrong.

7. Registry Wiring Per Field

The validator is generic. Each field key provides its own registry. For colors, the registry is the full color_registry table. For editions (open enum), no registry is needed — any kebab-case slug passes.
Field KeyEnum PolicyRegistry SourceMembership CheckToken Map
colors open (but closed vocab) appDb.listColors() → 77+ atoms with hex Every atom must be in registry. Multi-color: split on +, check each. grey→gray, gray-light→light-gray, +15 rules
editions open None (any slug valid) Format check only: kebab-case, lowercase, no spaces None (slugify only)
default_color derived Must equal colors[0] Cross-field invariant, not standalone registry check Same as colors
future fields... per field rule per field — could be DB table, static list, or none driven by fieldRule.enum.policy driven by fieldRule.parse.token_map

8. Escalation Chain

Per CLAUDE.md escalation mandate: max 3 attempts before flagging for manual review.
ATTEMPT 1
DETERMINISTIC CHECK
Token map + list rules + registry check.
If all pass → publish.
If unknowns → go to attempt 2.
ATTEMPT 2
VALIDATE LLM REPAIR
Send unknowns + registry to validate phase.
If repaired → apply + re-check + publish.
If rerun_recommended → go to attempt 3.
ATTEMPT 3
RE-RUN SOURCE
Orchestrator re-runs the source LLM call (with escalation model if configured).
Feeds back: "These values were invalid: [...]"
If still fails → flag for manual review.
