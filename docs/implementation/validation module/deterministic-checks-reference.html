Deterministic Validation Checks

Every check below is derived from the field studio rules in egPresets.js. Each maps a specific rule parameter to a pure function with concrete pass/fail/repair scenarios.
No LLM needed for any of these — the validate-phase LLM is only called when these checks produce unknowns.
1 Shape Validation DETERMINISTIC EXISTS (Zod)

Rule source

contract.type: 'string' contract.shape: 'list'
Validates the raw LLM response is the correct struct before any value checks. Uses the existing colorEditionFinderResponseSchema (Zod). If this fails, reject immediately — no repair possible for wrong shape.
ScenarioInputResult
Valid shape { colors: ["black"], editions: {}, default_color: "black" } PASS
colors is not an array { colors: "black, white", ... } REJECT string instead of array
editions is an array instead of record { editions: ["launch-edition"], ... } REJECT array instead of Record<slug, {colors}>
colors contains non-string { colors: ["black", 42, null], ... } REJECT non-string in array
edition missing colors sub-array { editions: { "launch": "black" }, ... } REJECT edition value is string, not {colors:[]}
missing colors field entirely { editions: {}, default_color: "black" } REJECT required field missing
editions omitted (has .default({})) { colors: ["black"], default_color: "black" } PASS Zod defaults to {}
default_color omitted (has .default('')) { colors: ["black"], editions: {} } PASS Zod defaults to ''

3 Token Map Application DETERMINISTIC NEW

Rule source

parse.token_map
The token_map in egPresets.js defines exact string replacements. Applied after lowercase normalization. Each value is checked against the map; if found, replaced with the canonical form.
Colors have 19 rules. Editions have 0 (empty token_map — slugification in Check 2 is sufficient).
For multi-color values (black+red), split on +, apply map to each atom, rejoin.

4 List Rules Enforcement DETERMINISTIC NEW

RuleColorsEditionsBehavior

5 Format Validation DETERMINISTIC NEW

Rule source

ui.tooltip_md formatting rules ai_assist.reasoning_note formatting rules
After normalization and token map, validate that values match the expected format. This catches malformed tokens that survived normalization.

Colors: format regex

/^[a-z][a-z0-9-]*(\+[a-z][a-z0-9-]*)*$/ Single atom: lowercase letters/digits/hyphens Multi-color: atoms joined by "+" No spaces, no underscores, no uppercase
"black"PASS
"light-blue"PASS
"black+red"PASS
"dark-green+light-pink"PASS
""FAIL empty string
"black red"FAIL space (should be + or -)
"black+"FAIL trailing +
"+red"FAIL leading +
"123"FAIL starts with digit

Editions: kebab-case regex

/^[a-z0-9]+(-[a-z0-9]+)*$/ Kebab-case slugs only. Lowercase letters, digits, hyphens. No spaces, no underscores, no uppercase. Must start with letter or digit.
"launch-edition"PASS
"cyberpunk-2077-edition"PASS
"sf6-chun-li"PASS
"wilderness"PASS
""FAIL empty string
"Launch Edition"FAIL uppercase + space
"launch_edition"FAIL underscore
"-edition"FAIL leading hyphen
Note: Format failures after Check 2 (normalization) and Check 3 (token map) suggest the LLM produced something genuinely malformed — not just a case/spacing issue. These are UNKNOWN candidates that go to the validate LLM.
6 Registry Membership DETERMINISTIC NEW

Rule source

enum.policy: 'open' (but color_registry is closed vocab) enum.match.strategy: 'exact' Registry: appDb.listColors() → 77+ atoms
Colors only. Every color atom must exist in the color_registry table. The registry is the SSOT — seeded at boot from color_registry.json (77+ entries including light-/dark- variants).

Editions skip this check entirely — enum_policy: 'open' with no closed vocabulary. Any kebab-case slug is valid.

Multi-color values: split on +, check each atom independently. If any atom is unknown → the whole value goes to the unknown list → validate LLM decides.
InputAtoms CheckedResultAction
"black" black ∈ registry? YES KNOWN Keep as-is
"light-blue" light-blue ∈ registry? YES KNOWN Keep as-is
"black+red" black YES, red YES KNOWN Both atoms pass
"midnight-blue" midnight-blue ∈ registry? NO UNKNOWN → validate LLM (maybe dark-blue?)
"arctic" arctic ∈ registry? NO UNKNOWN → validate LLM (maybe white?)
"black+midnight" black YES, midnight NO UNKNOWN → validate LLM (one atom failed)
"sparkle-unicorn" sparkle-unicorn ∈ registry? NO UNKNOWN → validate LLM (likely hallucination)
edition: "launch-edition" skipped (open enum, no registry) SKIP Editions are open vocab

How the registry Set is built

// At validation time: const allColors = appDb.listColors(); // [{ name: 'amber', hex: '#f59e0b' }, { name: 'beige', hex: '#f5f5f4' }, ...] const registrySet = new Set(allColors.map(c => c.name)); // Set { 'amber', 'beige', 'black', 'blue', 'brown', 'coral', 'cyan', // 'dark-blue', 'dark-brown', 'dark-cyan', 'dark-green', 'dark-red', // 'gold', 'gray', 'green', 'indigo', 'light-blue', 'light-gray', // 'light-green', 'light-pink', 'light-red', 'magenta', 'navy', // 'olive', 'orange', 'peach', 'pink', 'purple', 'red', 'silver', // 'teal', 'turquoise', 'violet', 'white', 'yellow', ... } // 77+ entries total // For each color value: function checkAtom(atom) { return registrySet.has(atom); // O(1) exact match }
7 Cross-Field Invariants DETERMINISTIC NEW

Rule source

Domain contract: colors[0] = default_color Domain contract: edition.colors ⊆ top-level colors colorEditionSchema.js + handoff invariants
These check relationships between fields, not individual values. Run after all per-field checks pass. Some can be auto-repaired, others are genuine mismatches.
InvariantCheckOn Fail
default_color === colors[0] The first color in the array IS the default. default_color must match. AUTO-REPAIR Set default_color = colors[0]. Log repair.
default_color ∈ colors The default must appear somewhere in the colors array. AUTO-REPAIR If not colors[0], prepend default_color to colors array.
edition.colors ⊆ colors Every color listed in an edition must also appear in the top-level colors array. AUTO-REPAIR Add missing edition colors to top-level colors array (append, don't change [0]).
No empty edition slugs Edition keys must be non-empty strings. REJECT Remove empty-key editions. Log warning.
No duplicate edition slugs Edition record keys are unique by definition (JS object), but check anyway for sanity. N/A Object keys are inherently unique.

Pipeline Order Summary
Checks run in this exact order. Each step operates on the output of the previous step. The pipeline short-circuits on shape failure (Check 1). All other checks produce a repaired value + audit log.
#CheckCan Repair?On Total FailCost
1 Shape Validation (Zod) NO Reject immediately. Wrong struct = useless data. ~0ms
2 Lowercase + Hyphen Normalization ALWAYS Cannot fail. Always produces output. ~0ms
3 Token Map Application ALWAYS Cannot fail. Passthrough for unknown tokens. ~0ms
4 List Rules (dedupe, max_items) ALWAYS Cannot fail. Dedupes and truncates. ~0ms
5 Format Validation (regex) NO Malformed tokens → unknown list → validate LLM ~0ms
6 Registry Membership (colors only) NO Unregistered atoms → unknown list → validate LLM ~0ms
7 Cross-Field Invariants MOSTLY Auto-repair where possible (default_color, edition colors). Reject empty slugs. ~0ms
? Validate LLM (only if unknowns exist) MAYBE Repair or recommend re-run. Caller decides next step. ~2-5s + tokens
Happy path (most runs): Checks 1→7 all pass → PUBLISH — zero LLM cost, <1ms total

Minor repair: Checks 2-4 fix case/dupes/token_map, 5-7 pass → PUBLISH — zero LLM cost

Unknown values: Check 5 or 6 finds unknowns → VALIDATE LLM — cheap call, just unknowns + registry

Hallucination: Validate LLM says rerun → RERUN SOURCE — orchestrator retries with escalation model
