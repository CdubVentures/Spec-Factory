2 Shape Validation DETERMINISTIC NO REPAIR — REJECT

Rule Sources

ui_field_catalog[].shape: 'scalar' | 'list' parse_templates[].template → implied shape
The very first gate. If the returned value has the wrong structural shape, reject immediately. Shape is non-negotiable — an LLM cannot "repair" a scalar into a list or vice versa without re-running the entire extraction.
// Deterministic check function function checkShape(value, expectedShape) { if (expectedShape === 'scalar') { if (Array.isArray(value)) return { pass: false, reason: 'expected scalar, got array' }; if (typeof value === 'object' && value !== null) return { pass: false, reason: 'expected scalar, got object' }; return { pass: true }; } if (expectedShape === 'list') { if (!Array.isArray(value)) return { pass: false, reason: `expected array, got ${typeof value}` }; return { pass: true }; } if (expectedShape === 'record') { if (typeof value !== 'object' || value === null || Array.isArray(value)) return { pass: false, reason: 'expected object/record' }; return { pass: true }; } }

5 String Normalization DETERMINISTIC ALWAYS REPAIRS

Rule Sources

parse.token_map{} tooltip_md: "lowercase + hyphens" ai_assist.reasoning_note
Normalization runs as a deterministic chain. Each step is a pure function. The chain order matters. Normalization never fails — it always produces output.
// Normalization chain (in order) function normalizeValue(raw, fieldRule) { let v = raw; v = v.trim(); // Step A: trim whitespace v = v.toLowerCase(); // Step B: lowercase v = v.replace(/\s+/g, '-'); // Step C: spaces → hyphens v = v.replace(/_/g, '-'); // Step D: underscores → hyphens v = v.replace(/-{2,}/g, '-'); // Step E: collapse multiple hyphens v = v.replace(/^-|-$/g, ''); // Step F: trim leading/trailing hyphens v = applyTokenMap(v, fieldRule.parse?.token_map); // Step G: token map lookup return v; }

13. Full Pipeline Order & Short-Circuit Rules
Checks run in this exact order. Each step operates on the output of the previous step. The pipeline short-circuits on shape failure (Step 1). All other steps accumulate repairs + unknowns.

Common System Prompt (prepended to all repair prompts)

You are a field value validator for the Spec Factory product database. Your job is to repair or reject LLM-extracted field values that failed deterministic validation. RULES: - You MUST return valid JSON matching the response schema. - Confidence 0.0-1.0: only values ≥ 0.8 will be auto-applied. Values 0.5-0.8 are flagged for human review. Values < 0.5 are treated as rejection. - You MUST NOT hallucinate new values. If you're unsure, return "unk". - For closed enums, you MUST map to an existing registered value or reject. You cannot invent new entries for a closed vocabulary. - For open_prefer_known enums, you may confirm genuinely new values, but you must provide reasoning for why it's legitimate. - Check for common LLM hallucination patterns: • Values that sound plausible but don't exist (e.g., "phantom-blue") • Overly specific values the source couldn't have known • Mix-ups between similar products • Round numbers that feel "guessed" rather than measured

Response Schema (all repair prompts)

{ status: "repaired" | "rerun_recommended", reason: string | null, decisions: [ { value: string, // the original unknown value decision: "map_to_existing" | "keep_new" | "reject" | "set_unk", resolved_to: string | number | null, confidence: number, // 0.0 - 1.0 reasoning: string // explain the decision } ] } // status = "repaired" when ALL decisions are map_to_existing, keep_new, or set_unk // status = "rerun_recommended" when ANY decision is "reject" AND // the LLM believes the source data is fundamentally wrong

LLM Hallucination Patterns (injected into prompts)

HALLUCINATION RED FLAGS — reject if you see these: 1. COLOR NAMES that don't correspond to real colors: "phantom-blue", "sparkle-unicorn", "galaxy-fade", "rgb-wave" → These are made-up marketing terms, not color tokens. 2. SENSOR/SWITCH NAMES with impossible specs: "PAW9999" (doesn't exist), "PixArt 100K DPI" (no such sensor) → Cross-check against the component database provided. 3. MEASUREMENTS that violate physics: Mouse weighing 1g or 5000g. Mouse that's 500mm long. → Check sanity bounds provided. 4. DATES in the far future or distant past: A mouse released in 2030, or a sensor from 1990. → Products in this DB are typically 2015-present. 5. EDITION NAMES that are generic descriptions, not real editions: "regular", "standard", "basic", "default", "normal" → These are not real edition names. The base product has no edition. 6. FIELD VALUE = FIELD NAME or GENERIC LABEL: sensor: "sensor", switch: "switch", weight: "weight" → LLM echoed the prompt template. 7. VALUES COPIED FROM A DIFFERENT PRODUCT: If values seem internally inconsistent (wireless mouse with 0 battery hours, wired mouse with 500h battery), suspect cross- contamination. 8. PERFECT ROUND NUMBERS for measured values: weight: exactly 100g, height: exactly 40mm → Real measurements have some precision. Flag low confidence.
