Normalizer & Parse Template Refactor

SUPERSEDED — See Type-Driven Normalization — Implementation Guide

Original issue brief for the Spec Factory dev team — April 2026. Kept for historical reference only.

1. The Bug

Symptom: Editions candidates in product.json are stored as ["[object object]"] instead of the actual edition Record.

The Color & Edition Finder (CEF) submits editions as a Record object:

{
  "call-of-duty-black-ops-6-edition": { "display_name": "Call of Duty: Black Ops 6 Edition", "colors": ["black+orange"] },
  "witcher-3-edition":                { "display_name": "The Witcher 3 Edition", "colors": ["black+beige"] }
}

The validation pipeline runs list_of_tokens_delimited template dispatch on this value before checking its shape. The normalizer calls String(object) → "[object Object]" → splits on delimiters → ["[object object]"].

Data flow (broken path)

CEF submits Record  →  validateField()
                          ↓
                    Step 1: dispatchTemplate("list_of_tokens_delimited", value)
                          ↓
                    normalizeColorList(value)  →  parseList(value)
                          ↓
                    String({...})  →  "[object Object]"  →  split  →  ["[object object]"]
                          ↓
                    Step 2: checkShape(["[object object]"], "list")  →  PASS (it's an array now)
                          ↓
                    Corrupted value written to DB + product.json

2. Root Cause: Templates Are Shape-Coupled

The current parse templates encode both the cardinality (scalar vs list) and the value-cleaning logic in a single name. This creates redundancy and shape-coupling:

Current TemplateWhat It Actually DoesProblem
number_with_unit Strip unit, coerce to number (scalar) Same logic, duplicated for scalar vs list
list_of_numbers_with_unit Split list, then strip unit + coerce each (list)
list_of_tokens_delimited Split string on delimiters, lowercase each Name implies list-only; crashes on objects
token_list Nothing — has no normalizer, falls through Dead template. Was only used as ui.input_control (now retired/derived), never as parse.template on any field
text_field / text_block Both do nothing (fall through to type-branch) Two names for the same no-op

The shape dropdown (scalar, list, structured, key_value) already tells the system the cardinality. The parse template should only describe how to clean each individual value, not re-encode cardinality.

3. Proposed Template Taxonomy

Principle: Shape determines cardinality. Template determines per-value cleaning. They are independent axes.

New Template Per-Value Behavior Replaces
text Trim whitespace text_field, text_block
number Strip unit suffix, coerce to number number_with_unit, list_of_numbers_with_unit
range Parse range expression (e.g. "1-5" → {min:1, max:5}) (new, extracted from mixed)
mixed_number_range Accept numbers and/or ranges in same list list_numbers_or_ranges_with_unit
boolean Normalize yes/no/true/false/1/0 → "yes"/"no"/"unk" boolean_yes_no_unk
date Parse to YYYY-MM-DD date_field
url Validate/passthrough URL url_field
component_ref Component identity lookup component_reference
tokens_delimited Split string on , ; | /, lowercase, trim each token list_of_tokens_delimited, token_list (dead)

How templates work with each shape

ShapeTemplate behaviorExample
scalar Apply template to the single value "Black" + tokens_delimited → "black"
list If string: split first, then apply template to each element.
If array: apply template to each element.		 "Black, White" → ["black", "white"]
["Black", "White"] → ["black", "white"]
key_value Pass through. Keys are domain identifiers, values are structured.
Template does not coerce objects.		 { "cod-edition": { colors: [...] } } → unchanged
structured Pass through. Nested structures are domain-specific. unchanged

4. Interim Fix (Applied)

Status: A minimal fix has been applied to prevent data corruption while the full refactor is planned.

In normalizers.js, parseList() now checks type before coercing:

export function parseList(value) {
  if (Array.isArray(value)) return value;
  if (value == null) return [];
  if (typeof value === 'object') return value;   // ← NEW: objects pass through
  const token = String(value ?? '').trim();
  if (!token) return [];
  return token.split(/[,;|/]+/).map(part => String(part || '').trim()).filter(Boolean);
}

normalizeColorList() guards against non-array returns:

export function normalizeColorList(value) {
  const parsed = parseList(value);
  if (!Array.isArray(parsed)) return parsed;    // ← NEW: objects pass through
  return parsed.map(entry => toStringSafe(entry).toLowerCase()).filter(Boolean);
}

Effect: Objects pass through the normalizer untouched. Shape check (checkShape) handles the mismatch with a clean rejection — no data corruption.

6. Migration Strategy

  1. Alias map: Add a TEMPLATE_ALIASES map in templateDispatch.js that maps old names → new names. This makes the dispatch backward-compatible during rollout.
  2. Update SSOT: Change egPresets.js to use new template names.
  3. Update registry: Change parseTemplateRegistry.ts to new names + remove dead entries.
  4. Recompile: Trigger Field Studio recompile for all categories (reseed from egPresets).
  5. Remove aliases: After all compiled rules are updated, remove the alias map.

7. CEF-Specific Context

The CEF is currently the only source that submits key_value-shaped editions data. The crawl pipeline submits editions as flat slug lists (matching shape: list). Once the normalizer refactor lands:

  • The editions field rule can use shape: key_value + template: tokens_delimited
  • The dispatch will see key_value shape → pass through without coercion
  • The crawl pipeline can continue submitting flat lists for the same field — the list shape path handles those
  • Or: editions can stay shape: list and the CEF extracts slug keys before submitting, storing the full Record in the new metadata slot on the candidate

Both approaches work. The key decision is whether editions in product.json should be a flat slug list or a keyed Record. The v2 schema shows a Record, suggesting shape: key_value is the intended target.
