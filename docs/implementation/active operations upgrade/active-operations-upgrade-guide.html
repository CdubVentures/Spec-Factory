1 Why this exists

Five finders (RDF, SKU, keyFinder, PIF, CEF) each manually wire 6–8 telemetry callbacks from their route to the operations registry, and each orchestrator manually emits the pending-before / completed-after LLM-call shape by hand. Nothing enforces the shape. When I added keyFinder telemetry I used stage: instead of label: and the modal rendered blank rows until caught.

This is a textbook O(1) Feature Scaling violation per CLAUDE.md. Adding a new finder today requires touching:

  • The orchestrator (declare 6 callback params, emit pending + completed per LLM call)
  • The route handler (map 6 callbacks to updateStage / updateModelInfo / updateQueueDelay / appendLlmCall / markPassengersRegistered / updateLoopProgress)
  • Every LLM emission point (get the 13-field payload exactly right)

Goal: drop this to "call one helper, pass one config object."

2 Hard constraints

PIF does not change. productImageFinder.js, its carousel strategy, LoopProgressGrid in tools/gui-react/src/features/operations/components/OperationsTracker.tsx, and PIF's updateLoopProgress shape all stay exactly as they are. Verified working. The refactor adds new helpers; it does not migrate PIF onto them.
PIF keeps the grid; RDF / SKU / keyFinder get a pill. A grid only makes sense when a loop has multiple parallel sub-units (views + hero in PIF). Single-field loops (keyFinder) or single-variant-per-iteration loops (RDF / SKU) render as a two-budget pill — see Stage 3.
Subtractive engineering. After each stage lands, delete the now-dead inline callback wiring / emission code from the migrated finder. No dead fallbacks, no "just in case" branches.

3 Current debt inventory

3.1 Route-level callback wiring (Stage 1 target)

Every custom route rewrites the same 6-line block. Sample from src/features/key/api/keyFinderRoutes.js:

onPassengersRegistered: (fks) => markPassengersRegistered({ id: op.id, passengerFieldKeys: fks }),

Near-identical copies live in finderRoutes.js (RDF/SKU), productImageFinderRoutes.js (PIF — leave alone), and colorEditionFinderRoutes.js (CEF).

3.2 LLM-call emission shape (Stage 2 target)

The modal's LlmCallRow reads 13 fields: label, prompt.{system,user}, response, model, isFallback, thinking, webSearch, effortLevel, accessMode, usage, variant, mode, discovery_log (via response.discovery_log).

Every orchestrator emits these manually twice per call (pending → completed). One wrong key (e.g. stage vs label) and the row renders blank. No schema enforcement.

3.3 Loop progress shape (Stage 3 target)

PIF emits a rich { views[], hero, carouselProgress } that LoopProgressGrid renders as a cell grid. keyFinder emits { iteration, attemptsRemaining, loopId } — too thin for the grid, so it renders nothing. RDF/SKU emit minimal progress too. Goal: keyFinder/RDF/SKU get a pill shape; PIF keeps the grid.

4 Stage 1 · buildOperationTelemetry

Scope

What: a single helper that returns the full callback bundle pre-wired to op.id + batcher. Routes spread it into their opts object. Kills ~30 lines of duplicated wiring per route.

Risk: Low. Mechanical. Existing tests catch regressions.

Plan mode: Not needed.

4.1 New file

src/core/operations/buildOperationTelemetry.js

import {
  updateStage, updateModelInfo, updateQueueDelay,
  updateLoopProgress, appendLlmCall, markPassengersRegistered,
} from './index.js';

// WHY: Canonical callback bundle that all finder routes map to the operations
// registry. Any finder that accepts telemetry callbacks spreads this into its
// opts. Adding a new standard callback = add it here once, not in every route.
export function buildOperationTelemetry({ op, batcher, mode = 'run' }) {
  const bundle = {
    onStageAdvance: (name) => updateStage({ id: op.id, stageName: name }),
    onModelResolved: (info) => updateModelInfo({ id: op.id, ...info }),
    onStreamChunk: (delta) => {
      if (delta?.reasoning) batcher.push(delta.reasoning);
      if (delta?.content) batcher.push(delta.content);
    },
    onQueueWait: (ms) => updateQueueDelay({ id: op.id, queueDelayMs: ms }),
    onLlmCallComplete: (call) => appendLlmCall({ id: op.id, call }),
    onPassengersRegistered: (fks) =>
      markPassengersRegistered({ id: op.id, passengerFieldKeys: fks }),
  };
  if (mode === 'loop') {
    bundle.onLoopProgress = (loopProgress) =>
      updateLoopProgress({ id: op.id, loopProgress });
  }
  return bundle;
}

5 Stage 2 · withLlmCallTracking

Scope

What: a wrapper that handles the pending-before / completed-after onLlmCallComplete dance. Enforces the canonical shape. Caller supplies the prompt + tier capabilities + callLlm function; wrapper takes care of both emissions + resolveModelTracking wiring.

Risk: Moderate. CEF emits three call rows per run (Discovery + Identity + Confirm). PIF carousel emits per-view. The abstraction must support N emissions per run, one per conceptual LLM call.

Plan mode: Yes — align on the signature before touching five orchestrators.

5.1 Proposed signature

src/core/llm/withLlmCallTracking.js

// WHY: Canonical LLM-call tracking wrapper. Emits onLlmCallComplete twice
// (pending + completed) with the exact shape OperationDetailModal reads.
// Use once per conceptual LLM call — CEF calls this 3x per run (Discovery,
// Identity, Confirm), PIF calls it once per view/hero iteration, keyFinder
// calls it once per iteration.
export async function withLlmCallTracking({
  label,                    // 'Discovery' | 'Identity' | 'Confirm' | ...
  prompt,                   // { system, user }
  initialModel,             // displayed in pending row (best-known pre-call)
  tierCapabilities,         // { thinking, webSearch, effortLevel } snapshot
  modelTracking,            // from resolveModelTracking(...)
  onLlmCallComplete,        // downstream callback (plug in buildOperationTelemetry's)
  callFn,                   // async () => ({ result, usage })
  extras = {},              // tier, reason, variant, mode (passed through to both emissions)
}) {
  // Pending emit — makes row appear immediately with prompt visible
  onLlmCallComplete?.({
    label, prompt, response: null,
    model: initialModel, isFallback: false,
    thinking: tierCapabilities.thinking,
    webSearch: tierCapabilities.webSearch,
    effortLevel: tierCapabilities.effortLevel,
    accessMode: '',
    ...extras,
  });

  const startedAt = new Date().toISOString();
  const startMs = Date.now();
  const { result, usage } = await callFn();
  const durationMs = Date.now() - startMs;

  // Completed emit — upserts pending row via appendLlmCall's merge rule
  onLlmCallComplete?.({
    label, prompt, response: result,
    model: modelTracking.actualModel || initialModel,
    isFallback: modelTracking.actualFallbackUsed,
    thinking: modelTracking.actualThinking,
    webSearch: modelTracking.actualWebSearch,
    effortLevel: modelTracking.actualEffortLevel,
    accessMode: modelTracking.actualAccessMode,
    usage, started_at: startedAt, duration_ms: durationMs,
    ...extras,
  });

  return { result, usage, durationMs };
}

5.3 Multi-call orchestrators (CEF example)

CEF calls withLlmCallTracking three times per run with different labels: 'Discovery', 'Identity', 'Confirm'. Each call is independently emitted as pending + completed. appendLlmCall's upsert rule only merges when the immediately previous call has response: null and the new call has non-null response, so consecutive label blocks stay separate. No change needed to the registry.

5.4 Migration order

Same as Stage 1, excluding PIF. Migrate one orchestrator per commit. After each commit, open the modal on a real run for that finder and confirm the LLM-call rows render identically to before.

6 Stage 3 · Two-budget pill (no grid)

Scope

What: A new LoopProgressPill frontend component for RDF / SKU / keyFinder. Shows two budgets side by side: (a) publish progress (count resolved vs target), (b) LLM call budget (attempts used vs budget). User sees at a glance whether a loop stopped because the target was met or because the budget was exhausted.

PIF stays on LoopProgressGrid — no change. Frontend branches on op.loopProgress shape: if views[] or hero present → grid; else → pill.

Risk: Moderate. Three loop orchestrators to migrate. Frontend gets one new component + a shape-based router.

Plan mode: Yes — finalize the pill visual + edge cases (target met with 0 budget used; budget exhausted with partial target; final_status display) before committing to a shape.

6.1 New loopProgress pill shape

Added to the existing op.loopProgress field. Frontend uses shape detection: if views or hero is present, render grid (PIF). Else render pill.

{
  // Publish-target budget — "count satisfied vs target"
  publish: {
    count: <number>,          // e.g. 0 or 1 for keyFinder; up to N for RDF/SKU
    target: <number>,         // usually 1 (keyFinder) or field_count (RDF/SKU)
    satisfied: <boolean>,     // count >= target
    confidence: <number|null>, // 0-100, latest publish confidence (keyFinder); null for RDF/SKU
  },
  // LLM-call budget — "attempts used vs attemptBudget"
  callBudget: {
    used: <number>,           // iterations completed
    budget: <number>,         // max attempts
    exhausted: <boolean>,     // used >= budget
  },
  // Terminal signal — populated on final iteration only
  final_status: 'published' | 'definitive_unk' | 'budget_exhausted' | 'skipped_resolved' | null,
  loop_id: <string>,
}

6.2 Pill visual

Two horizontal bars stacked in a single pill card. Cell grid is NOT used for these finders.

  • Row 1 (publish): "✓ 1/1 published · conf 88" or "· 0/1 not yet" or "✗ 0/1 definitive unk"
  • Row 2 (callBudget): "3/8 attempts" with a fill bar. Gray if not exhausted, red if exhausted before publish.
  • Header: loop_id (short), final_status chip if non-null.

Early-stop readability: publish.satisfied === true AND callBudget.used < callBudget.budget → "stopped pre-budget because target met" is visually obvious (green checkmark on row 1, unfilled bar on row 2).

6.4 Frontend router

tools/gui-react/src/features/operations/components/LoopProgressRouter.tsx (new):

export function LoopProgressRouter({ lp }: { lp: LoopProgress }) {
  // PIF path — grid renderer, unchanged
  if (lp.views || lp.hero) return <LoopProgressGrid lp={lp} />;
  // RDF / SKU / keyFinder path — new pill renderer
  if (lp.publish && lp.callBudget) return <LoopProgressPill lp={lp} />;
  return null;
}

8 What NOT to touch

Do not modify any of these in this refactor:
  • src/features/product-image/productImageFinder.js — PIF orchestrator
  • src/features/product-image/api/productImageFinderRoutes.js — PIF routes
  • src/features/product-image/carouselStrategy.js — carousel logic

Shape-detection branching in the frontend is intentional. PIF's shape and the pill's shape share the same op.loopProgress field, distinguished by field presence (views/hero → grid, publish/callBudget → pill). No enum, no type tag. If PIF's shape ever needs to evolve, the detection still holds.

9 Test strategy

9.1 Stage 1 (buildOperationTelemetry)

  • Unit: verify the bundle contains exactly the expected keys for run mode and loop mode. Each callback, when invoked, maps to the right registry function with the expected payload shape.
  • Regression: existing route tests (keyFinderRoutes.*.test.js, finderRoutes.test.js, etc.) must stay green.

9.2 Stage 2 (withLlmCallTracking)

  • Unit: given a fake callFn, verify two onLlmCallComplete emissions — first with response: null, second with response + usage + modelTracking values.
  • Multi-call test: three sequential withLlmCallTracking calls with labels 'A', 'B', 'C' produce 6 emissions (2 per call) that the registry upserts into 3 separate rows.
  • Regression: keyFinder's existing "emits onLlmCallComplete twice per LLM call" test must stay green after orchestrator migration.

9.3 Stage 3 (pill)

  • Shape contract test: keyFinderLoop emits publish + callBudget shape each iteration, and emits final_status on terminal iteration for each of the four exit conditions (published / definitive_unk / budget_exhausted / skipped_resolved).
  • Frontend component tests:
    • LoopProgressPill renders "0/1 not yet" when publish.count === 0.
    • Renders "1/1 published" with green check when publish.satisfied.
    • Renders "X/Y attempts" with red fill when callBudget.exhausted.
    • Renders early-stop state when publish.satisfied && callBudget.used < callBudget.budget.
  • PIF regression: snapshot the PIF grid with a representative views[] + hero payload and assert unchanged. Characterization-grade — this is the contract PIF stays on.
