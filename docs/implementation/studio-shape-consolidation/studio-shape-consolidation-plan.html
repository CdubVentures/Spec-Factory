2. Scope boundary (what this refactor does NOT touch)

OUT OF SCOPE — compile layer stays as-is

The compileMapNormalization.js normalizer (437 lines) and validator (192 lines) define the StudioConfig shape implicitly for the compiler pipeline. This is a separate, deeper concern touched by 17 files across the ingest layer. It is explicitly excluded from this refactor.

  • src/ingest/compileMapNormalization.js — normalizer + validator, 700 lines
  • src/ingest/compileFileIo.js — load/save, uses normalizer
  • src/ingest/categoryCompile.js — re-exports validate/load/save
  • src/ingest/compileContextLoader.js — compile pipeline context
  • src/ingest/compileAssembler.js — compile assembly
  • src/ingest/compileOutputWriter.js — compile output
  • src/features/studio/api/studioRouteContext.js — DI wiring only, no shape changes
  • 6 ingest test files — compile harness + validation + persistence tests
  • 2 db contract tests — monitor + keyboard

WHY: The compile layer defines how raw workbook data normalizes into the StudioConfig shape. That's a compiler concern, not an API contract concern. Touching it would require characterizing 700+ lines of normalization logic, affecting 17 files across an active migration surface. The risk/reward ratio is wrong for this refactor.

OUT OF SCOPE — frontend panel contracts stay as-is

studioPagePanelContracts.ts defines UI prop interfaces (7 importers). These are React component prop contracts, not data shapes. They import from studio.ts and will automatically benefit from the type derivation in Phase 4 without direct changes.
