CEF Variant Identity & Validation

Stable variant hashes, LLM-validated publishing, and PIF decoupling

Implementation Plan — 3 Phases

! The Problem

Fragile Current Variant Identity

Today, variant identity is a computed string derived on-the-fly from CEF data:

  • Colors: color:black, color:light-blue+blue+silver
  • Editions: edition:cod-bo6

There is no stable ID. The variant_key is recomputed every time from buildVariantList() using CEF's live selected.colors and selected.editions. This creates three failure modes:

Drift Color Name Refinement

A user refines "Ocean Blue" to "Deep Ocean Blue" in a new CEF run. The color atom changes from ocean-blue to deep-ocean-blue.

Result: color:ocean-blue → color:deep-ocean-blue. All PIF images tagged with the old key become orphans.

Drift Edition Color Atom Changes

An edition's color combo grows from ["black+orange"] to ["black+orange+gold"] as more detail is discovered.

Result: The edition no longer maps to the old combo entry in the colors array. The variant_key may shift or the variant may appear as both a color and an edition.

Coupling PIF Live Dependency on CEF

The PIF panel derives its variant list from the CEF query on every render. When a new CEF run fires (even without publishing), the WS data-change event invalidates the CEF cache. During refetch or if the new run changes selected, the PIF variant list shifts — breaking eval buttons, carousel slots, and hero assignments.

→ Solution Overview

Each variant gets a permanent hash ID (v_<8-hex>) assigned once at publish time. The hash never changes, even if the variant's name, color atoms, or display label are updated later. PIF stores this hash on every image, decoupling it from CEF's live state.

The Golden Rule: CEF Run 1 auto-publishes freely and creates initial variant hashes. CEF Run 2+ MUST be validated against the existing variant registry by an LLM call before publishing. No exceptions.

1 Variant Registry Foundation

Create the variant registry, hash generation, and storage. First CEF run auto-publishes and assigns hashes. No validation yet — just establishing the identity layer.

Task Hash Generation Function

Create generateVariantId() — produces a v_<8-hex> hash from the variant's original key at creation time.

import crypto from 'node:crypto';

// WHY: 8 hex chars = 4 billion combinations. Combined with
// product_id scoping, collision is effectively impossible.
export function generateVariantId(productId, variantKey) {
  const input = `${productId}:${variantKey}`;
  return 'v_' + crypto.createHash('sha256')
    .update(input).digest('hex').slice(0, 8);
}

The hash is deterministic — same product + same original key always produces the same ID. This enables idempotent first-run publishing.

2 LLM Validation Gate (Run 2+)

Every CEF run after the first MUST pass through an LLM validation call that resolves variant identity against the existing registry. The LLM decides: same variant (update metadata) or genuinely new (create new hash).

No Exception Rule: CEF Run 2+ cannot publish without LLM validation. The system must never silently create a duplicate variant or orphan existing PIF images by changing a hash.
Task LLM Validation Prompt

The LLM receives the existing variant registry and the new CEF discoveries. For each new discovery, it must classify:

ClassificationActionExample
Same Variant Keep existing variant_id. Update label, atoms, slug if changed. "Ocean Blue" → "Deep Ocean Blue" — same product, better name.
New Variant Generate new variant_id. Add to registry. A brand new color "Crimson Red" that never existed before.
Retired Keep in registry with retired: true. Never delete. Old limited edition no longer sold. PIF images still linked.
Task Prompt Structure
System prompt:
  You are validating a Color Edition Finder update for: {brand} {model}

  EXISTING VARIANT REGISTRY (published):
  {variant_id} | {type} | {variant_key} | {label} | atoms: {color_atoms}
  v_a1b2c3d4   | color  | color:black   | Black   | atoms: [black]
  v_e5f6a7b8   | color  | color:ocean-blue | Ocean Blue | atoms: [ocean-blue]
  v_f3a4b5c6   | edition | edition:cod-bo6 | COD BO6 | atoms: [black+orange]

  NEW CEF DISCOVERIES:
  color:black (label: "Jet Black", atoms: [black])
  color:deep-ocean-blue (label: "Deep Ocean Blue", atoms: [deep-ocean-blue])
  color:crimson-red (label: "Crimson Red", atoms: [crimson-red])
  edition:cod-bo6 (label: "COD Black Ops 6", atoms: [black+orange+gold])

  For EACH new discovery, respond:
  {
    "mappings": [
      {"new_key": "color:black", "match": "v_a1b2c3d4", "action": "update", "reason": "same color, name refined"},
      {"new_key": "color:deep-ocean-blue", "match": "v_e5f6a7b8", "action": "update", "reason": "same blue variant, name refined"},
      {"new_key": "color:crimson-red", "match": null, "action": "create", "reason": "genuinely new color"},
      {"new_key": "edition:cod-bo6", "match": "v_f3a4b5c6", "action": "update", "reason": "same edition, color atoms expanded"}
    ],
    "retired": []
  }

Rules:
- A color rename (Ocean Blue → Deep Ocean Blue) is the SAME variant.
- Color atoms gaining/losing modifiers for the same base color is the SAME variant.
- An edition gaining extra color detail in its combo is the SAME variant.
- A completely different color or product line is a NEW variant.
- Existing variants not in the new discoveries are NOT deleted — list them in "retired" if they appear discontinued.
Task Validation Response Schema (Zod)
const variantMappingSchema = z.object({
  new_key: z.string(),
  match: z.string().nullable(),   // existing variant_id or null
  action: z.enum(['update', 'create']),
  reason: z.string(),
});

const variantValidationResponseSchema = z.object({
  mappings: z.array(variantMappingSchema),
  retired: z.array(z.string()),   // variant_ids no longer in discoveries
});
Task Registry Update Logic

After the LLM returns mappings, apply them to the registry:

  1. Update matches: For each action: "update", find the entry by variant_id. Update variant_key, variant_label, color_atoms, edition_slug, edition_display_name, updated_at. Hash stays the same.
  2. Create new: For each action: "create", generate a new variant_id via generateVariantId(), create a full registry entry.
  3. Mark retired: For each variant_id in retired, set retired: true on the entry. Never remove from the array.
  4. Write: Save updated variant_registry to JSON and SQL.

3 PIF Decoupling & Migration

PIF stores variant_id on every image. The GUI reads the variant registry directly, decoupled from CEF's live query state. Eval, hero selection, and carousel slots all use the stable hash.

Task PIF Image Model Update

Add variant_id to every PIF image entry in product_images.json:

{
  "filename": "top-black.png",
  "view": "top",
  "variant_id": "v_a1b2c3d4",
  "variant_key": "color:black",       // kept for display, but not used for linkage
  "variant_label": "Black",
  "variant_type": "color",
  ...
}

At image download time, PIF looks up the variant_id from the variant registry. All downstream operations (eval, hero, carousel slots) use variant_id for grouping and linkage.

Task PIF Variant List from Registry

The PIF panel currently derives its variant list from CEF's live query:

// BEFORE (fragile — breaks when CEF re-runs)
const variants = buildVariantList(cefData.selected);

After Phase 3:

// AFTER (stable — reads from variant registry, independent of CEF state)
const variants = pifData.variantRegistry
  .filter(v => !v.retired)
  .map(v => ({ key: v.variant_id, label: v.variant_label, type: v.variant_type }));

The PIF GET endpoint includes the variant registry in its response (read from color_edition.json). CEF can re-run freely without affecting PIF's variant display or eval functionality.

Task Eval & Hero Linkage by variant_id

All eval and hero operations switch from variant_key to variant_id:

  • POST /evaluate-view: body uses variant_id instead of variant_key
  • POST /evaluate-hero: body uses variant_id
  • mergeEvaluation(): matches images by variant_id
  • carousel_slots: keyed by variant_id instead of variant_key
Task Migration Script

One-time migration for existing data:

  1. Backfill variant_registry: For every product with a color_edition.json but no variant_registry, generate the registry from current selected data (same as first-publish logic).
  2. Backfill PIF images: For every image in product_images.json, look up the variant_key in the registry and stamp the variant_id. If no match, create a new registry entry.
  3. Backfill carousel_slots: Re-key carousel_slots from variant_key to variant_id.
  4. SQL rebuild: Trigger a full rebuild of SQL projections from the updated JSON.
