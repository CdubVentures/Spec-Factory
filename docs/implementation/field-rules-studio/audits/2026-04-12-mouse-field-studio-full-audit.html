Critical Findings

C1Zero numeric range validation on ALL 22 numeric fieldsPERSIST
dpiweightheightwidthlngthbattery_hoursclick_forceaccelerationipspolling_rate+12 more

No field defines contract.range. The cross_validation_rules.json has only 3 rules (wireless_battery, sensor_dpi_consistency, dimensions_consistency) — zero range plausibility checks. The keyboard category has range checks on every numeric field.

Consequence: DPI of 999,999,999, weight of −50g, battery life of 0h all pass publisher validation.

Fix: Add contract.range: { min, max } to every numeric field in field_overrides. See Recommended Ranges table.
C2Garbage test data in production enum listsPERSIST
switch_typefeet_materialcoating
FieldGarbage ValuesImpact
switch_type"eqwe", "yay"Pollutes open_prefer_known matching; LLM may try to map real switches to these
feet_material"l"Single-char incomplete entry pollutes normalization
coating"A", "help"Placeholder text treated as valid coating options
Fix: Remove garbage values from data_lists in field_studio_map.json, then recompile.
C3Unit mismatch: click_force (gf) vs releasing_force (g)PERSIST
click_forcereleasing_force

Both measure switch actuation/release force but use different unit tokens. click_force correctly uses "gf" (gram-force). releasing_force uses "g" (ambiguous: mass vs force). The switch component_source also declares releasing_force with "unit": "g".

Fix: Standardize to "gf" in both component_sources and field_overrides.
C4sensor_date: string type with date-ordering constraintPERSIST
sensor_date

Contract: type: "string". Constraint: "sensor_date <= release_date". But release_date is properly typed as "date". String-vs-date comparison is undefined behavior.

Sample values confirm date data: "2018-01-01", "2022-04-01".

Fix: Change to contract.type: "date".
C5switch_life_span: string type but numeric dataPERSIST
switch_life_span

Compiled as type: "string" (line 9058). Component_source switch also declares "type": "string". But actual data values are lifespan numbers ("20", "50", "70", "100" million clicks). No unit defined.

Fix: Change to type: "number", add unit: "million clicks", add range: { min: 1, max: 200 }.
C6encoder_life_span and encoder_steps: numbers without unitsPERSIST
encoder_life_spanencoder_steps

Compile report explicitly warns: "field encoder_life_span: numeric type 'number' without unit declared" and same for encoder_steps. Both have unit: "".

Fix: encoder_life_span → unit "million rotations". encoder_steps → unit "steps".
C7connectivity enum: meaningless valuesPERSIST
connectivity

known_values contains: "1", "2.5", "n/a", "unk" alongside valid values. "1" and "2.5" are data-entry errors. "n/a"/"unk" should be handled by unknown_reason, not as enum values.

Fix: Remove "1", "2.5", "n/a", "unk". Add "USB-A dongle", "USB-C dongle" if needed.
C8sensor_date: nonsensical rounding on a string fieldNEW
sensor_date

The compiled contract has rounding: { decimals: 0, mode: "nearest" } on a field with type: "string" (line 7380-7385). You cannot round a string. This was injected by the compiler because the component_source declares sensor_date as a number-like property, but the type override is string. The rounding block is dead config that could confuse downstream consumers.

Fix: Either remove the rounding block or (better) change the type to "date" per C4.

High Findings

H1rgb: string type with yes_no boolean enum sourcePERSIST
rgb

contract.type: "string" but enum.source: "yes_no". Similar boolean-indicator fields (bluetooth, hot_swappable, silent_clicks) are correctly typed as "boolean". Sample values are "yes"/"no" (line 7052-7056).

Fix: Change to type: "boolean".
H2nvidia_reflex: invalid availability value "rare"PERSIST
nvidia_reflex

Confirmed at line 5941: availability: "rare". Valid enum: always | expected | sometimes | never. "rare" is not recognized by the schema.

Fix: Change to "sometimes".
H3form_factor conflates handedness with form factorPERSIST
form_factor

Values: ambidextrous, left, right. These describe handedness, not form factor. Mouse form factor typically means: standard, ergonomic, vertical, trackball, mini/travel.

Fix: Rename to handedness, or redefine values.
H4hump enum: contradictory / redundant valuesPERSIST
hump

Contains "back - rear" (redundant: back = rear) and "back - front" (confusing span notation). Standard positions: rear, center, front.

Fix: Normalize to: rear, center, front, rear-center, front-center.
H5encoder_steps: closed enum on numeric field with suspect valuePERSIST
encoder_steps

Closed policy with string values ["5","16","18","20","24","25","26","30","59"]. Any new encoder with a different step count (e.g., 12, 36) is rejected. The value "59" is highly unusual (standard: 12-24). Values are strings but field is numeric.

Fix: Change to "open_prefer_known". Remove "59" if unverified. Add range: { min: 5, max: 48 }.
H6click_latency & sensor_latency: selection_policy references nonexistent fieldsPERSIST
click_latencysensor_latency

click_latency.selection_policy.source_field: "click_latency_list" — no such field exists. Same for sensor_latency referencing "sensor_latency_list". The mode-specific variants are individual scalar fields, not a list.

Fix: Create the aggregate list fields, or rewrite selection_policy to reference the 3 variant fields.
H7polling_rate: no known_values enum for discrete standard ratesPERSIST
polling_rate

The only critical field. Has enum.policy: "open" with no known_values source and no data_list (manual_values is empty). Polling rates are discrete: 125, 250, 500, 1000, 2000, 4000, 8000 Hz.

Fix: Add known_values or data_list with standard polling rate values.
