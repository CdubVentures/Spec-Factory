1. Critical Issues (Must Fix)

Critical Switch actuation_force / bottom_out_force

Unit mismatch between component_sources and field contract.

The component_sources definition declares actuation_force with unit "gf" (gram-force), while the field contract and UI suffix use "g" (grams). The same applies to bottom_out_force.

component_sources
unit: "gf" (lines 47-50 of field_studio_map.json)
field contract
unit: "g" (field_rules.json, line 320)
UI suffix
"g"

While "g" and "gf" are used interchangeably in the keyboard industry, this creates an internal inconsistency. The component_db switch properties store values that were defined with "gf" but the validation pipeline processes them against "g" rules. If a unit-conversion phase ever distinguishes between mass and force, this will silently produce wrong values.

Recommendation: Standardize on one unit token across all layers. Use "gf" everywhere (technically correct) or "g" everywhere (industry convention). Update component_sources to match the field contract, or vice versa.
Critical Switch actuation_adjustment_step vs adjustable_input_granularity

Duplicate fields capturing the same real-world specification.

  • actuation_adjustment_step — type: number, unit: mm, range: 0.01–0.5, group: Switch
  • adjustable_input_granularity — type: string, enum: ["No", "0.1 mm", "0.2 mm", "0.4 mm"], group: Performance

The alias for adjustable_input_granularity includes "adjustable actuation step", which is semantically identical to actuation_adjustment_step. Both tooltips describe the same concept: the minimum adjustable actuation step size.

This violates the Single Source of Truth mandate. During ingestion, the LLM could populate one, both, or neither with conflicting values. There is no cross-validation rule to reconcile them.

Recommendation: Deprecate one field. If you want a boolean "does this keyboard support adjustable granularity?" use the numeric field with null for absence. Alternatively, keep adjustable_input_granularity as the boolean gate (No/Yes) and actuation_adjustment_step for the numeric precision, but add a cross-validation rule to ensure consistency.
Critical Switch hot_swappable (variance_policy)

Incorrect variance_policy: "authoritative" on a keyboard-level property.

In component_sources, hot_swappable is marked with variance_policy: "authoritative" as a switch property. However, hot-swappability is a property of the keyboard PCB/socket, not the switch itself. A Cherry MX Red switch is physically compatible with hot-swap sockets, but whether a specific keyboard supports hot-swap depends on its PCB design.

With "authoritative" policy, the component_db switch property would override the keyboard's actual hot-swap capability, producing incorrect data. For example, all keyboards with Cherry MX Red would inherit the switch's hot_swappable value instead of reflecting the actual keyboard design.

Recommendation: Change hot_swappable variance_policy from "authoritative" to "override_allowed" in component_sources. The keyboard-level value should always take precedence over the switch-level default.
Critical Switch adjustable_actuation_min (display_decimals vs rounding)

Precision truncation: rounding allows 2 decimals but display shows only 1.

contract.rounding
decimals: 2 (e.g., 0.05mm)
ui.display_decimals
1 (would show "0.1mm")

A value of 0.05mm (which is within the valid range 0.01–2.5mm) would be stored correctly but displayed as 0.1mm in the UI, hiding meaningful precision. The Wooting 60HE supports 0.1mm minimum actuation, but future products may go lower.

Recommendation: Align ui.display_decimals with contract.rounding.decimals. Set display_decimals to 2 for this field.
Critical Connectivity wireless_technology (overlaps bluetooth_version)

Overlapping data with no reconciliation rule.

  • wireless_technology enum: ["2.4 GHz", "Bluetooth 5.0", "Bluetooth 5.1", "Bluetooth 5.2", "Bluetooth 5.3"]
  • bluetooth_version enum: ["None", "5.0", "5.1", "5.2", "5.3", "5.4"]

If wireless_technology is set to "Bluetooth 5.3", the Bluetooth version is already encoded. But bluetooth_version is a separate field. During ingestion, these two fields could contain conflicting Bluetooth versions (e.g., wireless_technology = "Bluetooth 5.1" but bluetooth_version = "5.3"). There is no cross-validation rule to catch this.

Recommendation: Add a cross-validation rule: if wireless_technology starts with "Bluetooth", the embedded version must match bluetooth_version. Alternatively, remove the version from wireless_technology values (use just "Bluetooth") and rely on bluetooth_version for the specific version.
Critical Features onboard_memory (conflated boolean + count)

Enum values mix boolean semantics with numeric counts despite a dedicated count field.

  • onboard_memory enum: ["No", "Yes", "3 profiles", "5 profiles"]
  • onboard_profile_count — separate numeric field (range 1–100) already exists in the same group

The values "3 profiles" and "5 profiles" in onboard_memory duplicate information that should live in onboard_profile_count. During ingestion, if a keyboard has 5 onboard profiles, should the LLM set onboard_memory = "5 profiles" or onboard_memory = "Yes" + onboard_profile_count = 5? The ambiguity will cause inconsistent data.

Recommendation: Reduce onboard_memory to a strict boolean enum: ["No", "Yes"]. The specific count belongs exclusively in onboard_profile_count. Add a cross-validation rule: if onboard_memory = "No", then onboard_profile_count must be null.

2. Warnings (Should Fix)

Warning General color vs colors

Two color fields with confusing naming and different semantics.

  • color — string, open policy, title-case values ("Black", "White", "Silver") — represents primary color/finish label
  • colors — closed policy, 69 lowercase hyphenated design tokens ("dark-blue", "light-gray") — represents normalized palette tokens

The plural naming is counterintuitive: color (singular) is a descriptive label, while colors (plural) is a design-system token set. During LLM ingestion, the model may confuse which field to populate, or may populate both with conflicting information. The color field uses title-case while colors uses lowercase-hyphenated, adding another confusion vector.

Recommendation: Rename colors to color_tokens or normalized_colors to clearly distinguish it from the user-facing color label. Add a tooltip that explicitly says "Do not confuse with the 'color' field."
Warning Performance scan_rate (range too restrictive)

Max range of 8000 Hz is too low for modern keyboards.

Current range
125 – 8,000 Hz
Real-world max
Wooting 80HE: 24,000 Hz scan rate; some brands claim 30,000+ Hz

Scan rate is distinct from polling rate. While USB polling rate is typically capped at 8,000 Hz, the internal key matrix scan rate on Hall Effect and magnetic switch keyboards can be much higher. Products from Wooting, Razer, and SteelSeries already exceed the 8,000 Hz cap.

Recommendation: Increase scan_rate cross-validation max to at least 40000 Hz. Also update the contract range in field_studio_map.json accordingly.
Warning Switch switch_lifespan (range ceiling)

Max of 200 million keystrokes may be too low.

Current range
10 – 200 million keystrokes
Real-world
Hall Effect/magnetic switches are often marketed with "unlimited" or "100M+" lifespan. Some optical switches claim 150M+.

While 200M covers most switches, the marketing trend is toward higher numbers. More importantly, Hall Effect switches (no physical contact) technically have no mechanical wear limit. Consider whether this field should support a sentinel value for "unlimited."

Recommendation: Either raise max to 500 (million keystrokes) to future-proof, or add an enum value like "Unlimited" to known_values alongside the numeric range. If keeping numeric-only, 500M provides enough headroom.
Warning Layout key_count (range ceiling)

Range 20–120 may exclude edge cases.

  • Some macro pads/numpads have as few as 3–12 keys
  • Some full-size keyboards with extra macro columns have 104–130+ keys (e.g., SteelSeries Apex Pro with macro keys, or Corsair K100 with 110 keys + 6 macro keys = 116)

If the system ever indexes macro pads or numpads as a sub-category, the min of 20 will reject valid products. The max of 120 is tight for keyboards with extra macro rows.

Recommendation: Expand range to 10–130 to cover macro pads at the low end and macro-heavy full-size boards at the high end.
Warning Dimensions typing_angle (range ceiling)

Max of 15 degrees too low for ergonomic keyboards.

Current range
0 – 15 degrees
Ergonomic keyboards
Split keyboards like ZSA Moonlander support tenting up to ~45 degrees. Kinesis Advantage has ~20-degree tenting options.

If "typing_angle" encompasses tenting angle (which the tooltip "Keyboard resting angle" suggests), 15 degrees excludes many ergonomic keyboards. If it only refers to the front-to-back incline of flat keyboards, 15 degrees is borderline but acceptable.

Recommendation: Clarify the tooltip to specify whether this is front-to-back incline only. If tenting is included, raise max to 50. If not, consider adding a separate tenting_angle field for split keyboards, or expand the tooltip to disambiguate.
Warning General warranty (no enum values or structure)

Free-text string field with no enum guidance leads to inconsistent data.

Warranty is typed as an open string with no enum values defined in known_values.json. In practice, warranties cluster around standard durations: "1 year", "2 years", "3 years", "Limited Lifetime". Without even a seed enum, the LLM could produce wildly inconsistent formats: "1yr", "1 Year", "12 months", "1-year limited".

Recommendation: Add seed enum values to the data_lists: ["1 year", "2 years", "3 years", "5 years", "Limited Lifetime", "None"]. Set policy to "open_prefer_known" to allow unusual values while normalizing common ones.
Warning General price_range (no structure)

Free-text string with no parsing guidance for a highly structured value.

Prices follow predictable patterns ("$49.99", "$99-149", "Budget", "Premium") but the field has no format hint, no enum, and no parse template. The LLM will produce inconsistent formats.

Recommendation: Either define tiered enum values (e.g., "Budget ($0-50)", "Mid-range ($50-150)", "Premium ($150-300)", "Enthusiast ($300+)") or change the type to a structured format with currency + amount fields.
Warning Connectivity bluetooth_version (incomplete enum)

Enum starts at 5.0 but older keyboards use Bluetooth 3.0, 4.0, 4.1, 4.2.

Current values
["None", "5.0", "5.1", "5.2", "5.3", "5.4"]
Missing
"3.0", "4.0", "4.1", "4.2"

Budget and older wireless keyboards (Logitech K380, Microsoft Sculpt, older Anne Pro models) use Bluetooth 4.x or even 3.0. Since the policy is "open", the system will accept them, but the seed enum biases the LLM toward thinking only BT 5.x exists.

Recommendation: Add "3.0", "4.0", "4.1", "4.2" to the enum values. These are still common in currently-sold products.
Warning Switch switch_type (policy should be stricter)

Policy is "open_prefer_known" but switch types are a closed taxonomy.

The current values cover the complete taxonomy of keyboard switch technologies: Mechanical, Low-profile Mechanical, Optical, Magnetic, Hall Effect, Membrane, Scissor, Hybrid. There are no realistic "unknown" switch types that should be auto-accepted.

With "open_prefer_known", the LLM could introduce nonsensical values like "Capacitive" (which would be a Topre variant covered by "Mechanical" or should be added explicitly) or typos that silently pass validation.

Recommendation: Change policy to "closed_with_curation" to prevent invalid switch type values from entering the system. If a genuinely new switch technology emerges, it should go through the curation workflow.
Warning General editions (only 1 alias)

Field has very few aliases for a concept with many names.

Current aliases
["editions"]
Suggested additions
"special edition", "limited edition", "collaboration", "colorway", "variant edition"

Product pages use many terms for editions: "Special Edition", "Limited Edition", "x [Brand] Collaboration", "[Character] Edition". With only one alias, the LLM parser may miss this field entirely on many product pages.

Recommendation: Add at least 3–5 common aliases to improve parsing coverage.
Warning Keycaps keycap_thickness (missing unit)

Numeric field with range 0.8–2.5 but no unit specified in the contract.

The range clearly refers to millimeters (keycap wall thickness is typically 1.0–1.5mm), but the contract omits the unit property entirely. The UI suffix also appears to be missing. Without a unit, the validation pipeline skips unit conversion, and the UI displays a raw number without context.

Recommendation: Add "unit": "mm" to the contract and "suffix": "mm" to the UI config.
Warning General warranty (only 1 alias)

Single alias "warranty" misses common phrasings.

Current aliases
["warranty"]
Suggested
"warranty period", "guarantee", "manufacturer warranty", "limited warranty"
Recommendation: Add common warranty-related terms as aliases to improve LLM parsing coverage.

3. Suggestions (Consider)

Suggestion Keycaps keycap_profile (missing popular profiles)

Missing popular enthusiast profiles in enum:

  • MT3 — Drop's signature sculpted profile, very popular in the enthusiast community
  • MDA — Mix of DSA and MA, gaining popularity
  • ASA — Akko's sculpted profile
  • MOA — similar to SA but shorter, used by several Chinese manufacturers
Recommendation: Add "MT3", "MDA", "ASA", and "MOA" to the seed enum. Since policy is "open", the system won't reject them, but having them in the seed list improves LLM normalization accuracy.
Suggestion Features backlighting (missing enum value)

Missing "Single-color LED" value. Some keyboards have red, blue, or green single-color backlighting that doesn't fit "White" or "RGB". Current enum: ["None", "White", "RGB", "Per-key RGB"].

Recommendation: Add "Single-color" or "Single-color LED" to the enum.
Suggestion Layout form_factor (missing "Ergonomic")

No enum value for ergonomic/split form factors. Keyboards like the Kinesis Advantage, ZSA Moonlander, and Dygma Defy don't fit neatly into percentage-based form factors.

Current values: "1800 Compact", "40%", "60%", "65%", "70%", "75%", "96%", "Full-size (100%)", "TKL (80%)"

Recommendation: Add "Ergonomic" and/or "Split" to form_factor enum. These are distinct physical form factors, not just layouts.
Suggestion Switch switch_name (small seed list)

Only 15 switches in the seed enum for an "open_prefer_known" policy. The market has 500+ switch variants. While new values are accepted, a larger seed list would significantly improve normalization accuracy during LLM ingestion (the model would have more examples of the canonical naming format).

Notable missing switches: Cherry MX Silent Black, Gateron Oil King, Kailh Box Jade, Razer Purple, HyperX Aqua, Akko CS switches, Outemu series, Boba U4T, Holy Panda.

Recommendation: Expand the seed list to at least 30–40 common switches covering all major manufacturers.
Suggestion Switch switch_type (missing "Capacitive")

Topre switches use electrostatic capacitive technology, which is distinct from mechanical switches. Currently covered by "Mechanical" in the enum, but enthusiasts strongly distinguish Topre as a separate category.

Current values: Mechanical, Low-profile Mechanical, Optical, Magnetic, Hall Effect, Membrane, Scissor, Hybrid

Recommendation: Add "Electrostatic Capacitive" or "Topre" to the switch_type enum to properly classify Topre/Niz keyboards.
Suggestion Build case_material (missing common values)

Missing materials seen in the keyboard market:

  • Brass — used in premium custom keyboards (weight plates, cases)
  • Carbon Fiber — emerging in high-end gaming keyboards
  • Steel — used in industrial keyboards

Current values: Plastic, ABS, Aluminum, CNC Aluminum, Polycarbonate, Wood

Recommendation: Consider adding "Brass", "Carbon Fiber", and "Steel" to the seed enum.
Suggestion Connectivity connection (missing "Tri-mode")

Many modern wireless keyboards are marketed as "tri-mode" (Wired + 2.4GHz + Bluetooth). The current enum value "Hybrid" covers this, but "Tri-mode" is a more specific and commonly-used industry term.

Current values: Wired, 2.4 GHz Wireless, Bluetooth, Hybrid

Recommendation: Add "Tri-mode" as an alias for "Hybrid" in the aliases list, or add it as a separate enum value. Also consider adding "Dual-mode" for keyboards that support Bluetooth + Wired but lack 2.4GHz.
Suggestion Performance debounce_time (min includes 0)

Cross-validation allows min: 0 ms. While Hall Effect switches genuinely have near-zero debounce, traditional mechanical switches with 0ms debounce would produce key chatter. A minimum of 0 is technically correct for the range, but the system might benefit from a conditional cross-validation rule: if switch_type is "Mechanical", debounce_time should be ≥ 1ms.

Recommendation: No action required if the system doesn't support conditional cross-validation. Note for future: conditional validation rules would make this more precise.
Suggestion Identity brand (only 1 alias)

The brand field has aliases ["brand", "manufacturer"] but misses common synonyms used in product data:

  • "maker", "company", "OEM", "vendor"
Recommendation: Add "maker" and "vendor" as aliases to improve parsing coverage on diverse data sources.

4. Missing Cross-Validation Rules

The system has 27 range-based plausibility rules but lacks relational rules between fields. These missing rules could allow logically impossible data combinations:

RuleFieldsInvariantSeverity
Force ordering actuation_force + bottom_out_force bottom_out_force > actuation_force (physics invariant: you must pass actuation before bottoming out) Critical
Adjustable actuation ordering adjustable_actuation_min + adjustable_actuation_max min < max Critical
Actuation ≤ travel actuation_distance + total_travel actuation_distance ≤ total_travel (actuation point must be before end of travel) Critical
Wireless technology ↔ bluetooth_version wireless_technology + bluetooth_version If wireless_technology contains "Bluetooth X.Y", bluetooth_version must equal "X.Y" Warning
Polling rate ordering polling_rate_wired + polling_rate_wireless polling_rate_wired ≥ polling_rate_wireless (wired always has equal or higher bandwidth) Warning
Battery life ordering battery_life_off + battery_life_rgb battery_life_off ≥ battery_life_rgb (lighting off always lasts longer) Warning
Height ordering height_front + height_rear height_rear ≥ height_front (keyboards slope upward toward the back) Suggestion
Onboard memory ↔ profile count onboard_memory + onboard_profile_count If onboard_memory = "No", onboard_profile_count must be null Warning
Per-key RGB ↔ backlighting per_key_rgb + backlighting If per_key_rgb = "Yes", backlighting should be "Per-key RGB" or "RGB" Suggestion
