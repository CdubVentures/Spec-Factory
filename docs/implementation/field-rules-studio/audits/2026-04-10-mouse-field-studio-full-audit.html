Critical Findings

C1 Zero numeric range validation on ALL numeric fields
dpi weight height width lngth battery_hours click_force acceleration ips polling_rate + 12 more

No field in the mouse category defines a contract.range object. The keyboard category has plausibility ranges on every numeric field, generating cross-validation rules. The mouse has zero.

This means the publisher pipeline will accept any number: DPI of 999,999,999, weight of -50g, or battery life of 0 hours — all pass validation.

The cross_validation_rules.json only has 3 rules (wireless_battery, sensor_dpi_consistency, dimensions_consistency) but none are range plausibility checks.

Fix: Add contract.range: { min, max } to every numeric field in field_overrides. See the Recommended Ranges table below.
C2 Garbage test data in production enum lists
switch_type feet_material coating

Several data_lists contain obvious test/garbage values that have leaked into compiled known_values.json:

FieldGarbage ValuesImpact
switch_type"eqwe", "yay"LLM will try to match real switch names against these; "eqwe" and "yay" appear in the closed known_values enum
feet_material"l"Single character — incomplete entry. Pollutes open_prefer_known matching.
coating"A", "help""help" is a placeholder. "A" is a single character. Both pollute coating normalization.
Fix: Remove "eqwe", "yay" from switch_type. Remove "l" from feet_material. Remove "A", "help" from coating. Re-compile.
C3 Unit mismatch: click_force (gf) vs releasing_force (g)
click_force releasing_force

Both fields measure switch actuation force but use different units:

  • click_force: contract.unit = "gf" (gram-force) — correct industry standard
  • releasing_force: contract.unit = "g" (ambiguous: grams-mass vs grams-force)

The component_sources switch definition also declares releasing_force with "unit": "g". Since both measure the same physical quantity (force), they should use the same unit token.

Fix: Standardize both to "gf" (gram-force). Update releasing_force in component_sources and field_overrides.
C4 sensor_date declared as string but has date-ordering constraint
sensor_date

sensor_date has contract.type: "string" but carries the constraint "sensor_date <= release_date".

String lexicographic comparison only works if both values are ISO-8601 formatted (YYYY-MM-DD). But the field has no format validation, no parse template, and release_date is properly typed as "date". A string-vs-date comparison is undefined behavior in the publisher pipeline.

Fix: Change sensor_date to contract.type: "date" to match release_date. Add the date_field parse template.
C5 switch_life_span typed as string but contains numeric data
switch_life_span

The component_sources switch definition declares switch_life_span with "type": "string", and the generated field rule also shows data_type: "string".

However, the actual data contains numeric values representing lifespan in millions of clicks (e.g., "20", "50", "70", "100"). Without numeric typing, the publisher cannot do range validation, rounding, or meaningful comparison.

Additionally, the field has no unit declared. In the mouse domain, switch lifespan is universally measured in "million clicks".

Fix: Change type to "number" and add unit: "million clicks". Add range: { min: 1, max: 200 }.
C6 encoder_life_span and encoder_steps: numeric fields without units
encoder_life_span encoder_steps

Both fields are type: "number" but have unit: "" (empty string). The compile report explicitly warns about this:

  • "field encoder_life_span: numeric type 'number' without unit declared"
  • "field encoder_steps: numeric type 'number' without unit declared"

encoder_life_span should have unit "million rotations" or "million clicks". encoder_steps should have unit "steps/revolution" or "notches".

Fix: Set encoder_life_span unit to "million rotations". Set encoder_steps unit to "steps".
C7 connectivity enum contains meaningless values
connectivity

The connectivity data_list and known_values contain:

  • "1" — meaningless numeric string
  • "2.5" — meaningless numeric string
  • "n/a" — should be handled by unknown_reason, not as a valid value
  • "unk" — same, should be unknown_reason not an enum value

Legitimate values exist alongside them: "2.4GHz Dongle", "2.4GHz RF Dongle", "2.4GHz WIFI Dongle". The field describes the wireless dongle/adapter type but "1" and "2.5" are data entry errors.

Fix: Remove "1", "2.5", "n/a", "unk". Add proper values like "USB-A dongle", "USB-C dongle" if needed.

High Findings

H1 rgb field: string type with yes_no boolean enum source
rgb

rgb has contract.type: "string" but enum.source: "yes_no" — a boolean enum. Similar fields (bluetooth, hot_swappable, silent_clicks) are correctly typed as "boolean".

This means the publisher will try string normalization (lowercase, trim) instead of boolean coercion. "yes"/"no" values won't be converted to true/false.

Fix: Change rgb to contract.type: "boolean" and data_type: "boolean".
H2 nvidia_reflex: invalid availability value "rare"
nvidia_reflex

nvidia_reflex has availability: "rare". The valid enum for availability per the schema is: always | expected | sometimes | never. "rare" is not a recognized value and may silently pass or cause unexpected behavior in field priority classification.

Fix: Change to "sometimes" (closest valid equivalent).
H3 form_factor conflates handedness with form factor
form_factor

Known values are: ambidextrous, left, right. In the mouse industry, "form factor" typically means the overall design category (e.g., standard, ergonomic, vertical, trackball, mini/travel). The values listed describe handedness, not form factor.

Meanwhile, there is a separate shape field with values asymmetrical, ergonomic, symmetrical which describes shape geometry. And a design field with no enum values.

This creates semantic confusion: is a right-handed ergonomic mouse "right" form_factor + "ergonomic" shape? The two fields overlap.

Fix: Rename to handedness or redefine form_factor to mean actual form factor with values like standard, ergonomic, vertical, trackball, travel.
H4 hump enum values are inconsistent / contradictory
hump

Values: "back - front", "back - rear", "back mid", "front", "middle".

  • "back - rear" is redundant: "back" and "rear" mean the same thing
  • "back - front" suggests a hump spanning back-to-front but this isn't standard terminology
  • Standard mouse hump positions are: rear, center/mid, front
Fix: Normalize to standard positions: rear, center, front, rear-center, front-center.
H5 encoder_steps has closed enum on a numeric field
encoder_steps

encoder_steps is type: "number" with enum.policy: "closed" and string values ["5","16","18","20","24","25","26","30","59"].

Issues:

  • A closed enum on a numeric field means only these exact values are accepted — any new encoder with a different step count (e.g., 12, 36) will be rejected
  • The value "59" is unusual for encoder steps (standard counts are 12, 16, 18, 24) — likely test data
  • Values are strings but the field is numeric
Fix: Change to "open_prefer_known" policy. Remove "59" if unverified. Consider range validation { min: 5, max: 48 } instead.
H6 click_latency references nonexistent source field
click_latency

click_latency has a selection_policy that references "source_field": "click_latency_list". No field named click_latency_list exists in the 85-field inventory. The mode-specific variants are individual scalar fields (click_latency_wired, click_latency_wireless, click_latency_bluetooth), not a list.

The selection_policy rule says: "If one measurement remains after selecting the best mode OR all remaining are within tolerance_ms, set scalar to median; else unk." This logic cannot execute without the source field.

Fix: Either create the click_latency_list aggregate field, or rewrite selection_policy to reference the three variant fields directly.
H7 polling_rate has no known_values enum despite discrete standard values
polling_rate

polling_rate is the only critical field in the entire category. It has enum.policy: "open_prefer_known" but no enum source is defined — no data_list entry and no known_values bucket.

Polling rates are discrete industry-standard values: 125, 250, 500, 1000, 2000, 4000, 8000 Hz. Without a known_values list, the LLM has no guidance on valid values and non-standard values (like 333 Hz or 750 Hz) would pass unchecked.

Fix: Add a polling_rate data_list or known_values entry with values: [125, 250, 500, 1000, 2000, 4000, 8000].

H9 releasing_force: missing rounding and wrong unit in contract
releasing_force

releasing_force is type: "number" with unit: "g" but has no rounding definition. Compare to click_force which correctly has rounding: { decimals: 0, mode: "nearest" }.

Additionally, the compile report warns: "field releasing_force: enum_policy=open but parse_rules is empty"

Fix: Add rounding: { decimals: 0, mode: "nearest" }. Change unit to "gf". Add parse_rules.
H10 Generic tooltips provide zero LLM guidance

Nearly every field has the identical template tooltip: "Define 'X' according to contract. If not provable with evidence, output 'unk' with unknown_reason."

This wastes tokens and provides no domain-specific guidance. Compare to the keyboard category which has field-specific tooltips explaining exactly what the value means and how to find it.

Fix: Write field-specific tooltip_md for at least all required and critical fields (14 fields minimum).
H11 Empty ai_assist.reasoning_note on all component sources

All 4 component_sources (sensor, switch, encoder, material) have ai_assist.reasoning_note: "". The keyboard category provides substantive reasoning notes that guide the LLM on why each component is important and how to prioritize its data.

Fix: Add reasoning notes to each component source explaining its role in the mouse specification hierarchy.

Medium Findings

M1 Component DB: flawless_sensor coverage at 4.3%
flawless_sensor

Only 5 out of 116 sensors have flawless_sensor populated. This is a key differentiator for gaming mice (indicates whether a sensor has no native smoothing/acceleration). The vast majority of modern PixArt/HERO sensors are flawless — this data is easily available from sensor.fyi.

M2 Component DB: sensor_date coverage at 24.1%
sensor_date

Only 28 of 116 sensors have release dates populated. This undermines the sensor_date <= release_date constraint — it can't fire for 76% of sensors.

M3 Component DB: click_force at 56.6%, releasing_force at 36.8%
click_force releasing_force

Force measurements are missing for the majority of switches. This data is available from manufacturer datasheets (Omron, Kailh, TTC, etc.).

M4 design field: no enum values, unclear purpose
design

design is type: "string", shape: "list" with enum.policy: "open_prefer_known" but no known values defined anywhere. No data_list, no known_values bucket. What values does this field accept? "gaming"? "office"? "ergonomic"? Without any guidance, the LLM has to guess.

M5 price_range: string type with no defined ranges
price_range

Has type: "string" and enum.policy: "open_prefer_known" but no known_values. Should define buckets like budget, mid-range, premium, ultra-premium with dollar thresholds in tooltip_md.

M6 Many fields have zero aliases

41 out of 85 fields have zero aliases. Aliases are critical for search discovery — an LLM searching for "LOD" won't match lift, and "actuation weight" won't match click_force unless aliases exist.

Notable missing aliases: bluetooth (needs "BT", "BLE"), cable_type (needs "cord type", "wire type"), feet_material (needs "mouse feet", "skates"), hot_swappable (needs "hot swap"), silent_clicks (needs "quiet clicks", "silent switches").

M7 click_latency_bluetooth/wired/wireless share same field_studio_hints row
click_latency_bluetooth click_latency_wired click_latency_wireless

All three variants point to row: 70 in dataEntry hints. Same issue with sensor_latency variants (all row 51). These should have distinct rows or the hints should distinguish them.

M8 coating known_values are minimal and contain garbage
coating

After removing garbage ("A", "help"), only 2 real values remain: "matte" and "rubber grips". Missing common coatings: glossy, soft-touch, textured, PBT, UV coating, sandblasted.
