C-04 encoder_steps closed enum with string values but numeric contract type
encoder_steps
encoder_steps has contract.type = "number" and the data_list is typed as number in the component source. However, the known_values enum contains string values: ["5", "16", "18", "20", "24", "25", "26", "30", "59"] with a closed policy. This means:
  1. The publisher will try to coerce to number but then check against string enum values — potential mismatch.
  2. The closed policy prevents accepting valid step counts not in this list (e.g., 8, 10, 12, 15, 32, 36, 48).
Fix: Change enum policy to "open_prefer_known" since encoder step counts vary widely. If keeping as closed, expand the list significantly. Also ensure compile pipeline handles string↔number coercion for enum matching.
C-05 acceleration value_form/shape contract inconsistency
acceleration
The acceleration field has conflicting shape declarations:
  • contract.shape = "list", contract.type = "mixed_number_range"
  • output_shape = "list"
  • But value_form = "single" and value_form_sample = "scalar"
The value_form says "this field holds a single value" while the contract says "this is a list of mixed number ranges." Sample values are simple scalars like "40", "50" — suggesting it should be scalar/number, not list/mixed_number_range.
Fix: Decide on canonical shape. If acceleration is always a single max G value (e.g., "40g"), change to type: "number", shape: "scalar". If it needs to represent ranges (e.g., "40-50g"), use type: "range", shape: "scalar". The current mixed_number_range/list is overkill for this field.

Informational Notes

I-01 lngth field key is a non-standard abbreviation
lngth
The field key lngth is a truncated spelling of "length", likely to avoid a JavaScript reserved word conflict. The compile report notes the migration was reversed: "reversed 'lngth -> length' to 'length -> lngth'". While functional, this is confusing for contributors and LLM agents.
Recommendation: Consider whether length_mm or mouse_length would be clearer. If keeping lngth, ensure the alias list includes "length" for LLM discovery.
I-02 connectivity vs connection semantic overlap
connectivityconnection
Two very similar fields exist:
  • connection: {bluetooth, hybrid, wired, wireless} — connection mode
  • connectivity: {2.4GHz Dongle, 2.4GHz RF Dongle, ...} — wireless technology
The distinction is valid but the naming is confusing. Many data sources will use "connectivity" to mean what connection captures.
Recommendation: Consider renaming connectivity to wireless_technology or wireless_receiver for clarity. Add detailed tooltips explaining the distinction.
I-03 rgb field overlaps with lighting
rgblighting
rgb is typed as string but appears to capture boolean-like info ("yes"/"no"). Meanwhile lighting captures structured zone info ("3 zone (rgb)", "none", etc.). The semantic boundary between these two fields is unclear.
Recommendation: Consider making rgb a boolean ("has RGB? yes/no") and leaving lighting for the detailed zone breakdown. Or merge them into a single structured field.
I-04 design field is undefined
design
The design field has no known_values, no aliases, no meaningful tooltip, and a generic description. It's unclear what it captures that shape, form_factor, and honeycomb_frame don't already cover.
Recommendation: Define a clear contract for what "design" means (e.g., overall aesthetic: minimal, gamer, ergonomic), or consider removing it if it's redundant.

I-06 hump known_values naming inconsistency

The hump enum values use inconsistent naming: "back - front" and "back - rear" use " - " separator, while "back mid" uses a space. Also "back - front" and "back - rear" are confusing — does "back - front" mean "hump centered between back and front" or "hump is in the back, with front being..."? The semantics are unclear.
Recommendation: Normalize to consistent naming: e.g., "rear", "center", "front", "rear-center", "center-front" with a clear tooltip defining each.
I-07 feet_material enum uses mixed casing

The normalize: "lower_trim" is set on the data_list, but the known_values in the compiled output retain original casing. This could cause case-sensitive enum matching failures.
