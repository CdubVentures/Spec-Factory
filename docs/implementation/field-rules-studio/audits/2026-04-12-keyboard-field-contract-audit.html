2. Critical Issues

CriticalSwitchactuation_force / bottom_out_force — unit mismatch

Source map declares "gf" (gram-force); field contract uses "g" (grams).

component_sources
actuation_force.unit = "gf" | bottom_out_force.unit = "gf"
field contract
actuation_force.contract.unit = "g" | bottom_out_force.contract.unit = "g"
UI suffix
"g" for both

While the keyboard industry uses these interchangeably, the internal inconsistency means the component_sources layer and the field_rules layer disagree on the canonical unit token. If a unit-conversion phase ever distinguishes mass from force, values will silently corrupt.

Fix: Standardize on one token everywhere. Either update component_sources to "g" or update field contracts to "gf".
CriticalSwitchactuation_adjustment_step vs adjustable_input_granularity

Duplicate fields capturing the same real-world specification.

  • actuation_adjustment_step — number, mm, range 0.01–0.5, group Switch, order 18
  • adjustable_input_granularity — string, enum ["No","0.1 mm","0.2 mm","0.4 mm"], group Performance, order 50

Alias for adjustable_input_granularity includes "adjustable actuation step" which is semantically identical. Both tooltips describe the same concept. Violates SSOT mandate. During ingestion, the LLM could populate one, both, or neither with conflicting values.

Fix: Deprecate adjustable_input_granularity. Use actuation_adjustment_step as the canonical numeric field (null = not supported, value = step size in mm). This preserves the information "No" would capture (null means no adjustable actuation step).
CriticalSwitchadjustable_actuation_min — display precision loss

Contract rounds to 2 decimals but UI displays only 1.

contract.rounding
decimals: 2 (stores 0.05mm)
ui.display_decimals
1 (shows "0.1mm")

A value of 0.05mm would be validated and stored correctly but displayed as 0.1mm, hiding meaningful precision. Wooting and DrunkDeer products differentiate at sub-0.1mm levels.

Fix: Set ui.display_decimals: 2 to match contract.rounding.decimals.
CriticalConnectivitywireless_technology overlaps bluetooth_version

Overlapping data with no reconciliation rule.

  • wireless_technology enum includes "Bluetooth 5.0" through "Bluetooth 5.3"
  • bluetooth_version enum has "5.0" through "5.4" and "None"

During ingestion, these can contain conflicting Bluetooth versions (e.g., wireless_technology = "Bluetooth 5.1" but bluetooth_version = "5.3"). No cross-validation catches this.

Fix: Add cross-validation: if wireless_technology contains "Bluetooth X.Y", bluetooth_version must equal "X.Y". Alternatively, remove version numbers from wireless_technology values (use just "Bluetooth") and rely on bluetooth_version for specifics.
CriticalFeaturesonboard_memory — conflated boolean + count

Enum mixes boolean semantics with counts despite a dedicated count field.

  • onboard_memory enum: ["No", "Yes", "3 profiles", "5 profiles"]
  • onboard_profile_count — integer [1–100], already exists in same group

"3 profiles" and "5 profiles" duplicate data that belongs in onboard_profile_count. Ambiguity: should a 5-profile keyboard be onboard_memory="5 profiles" or "Yes" + onboard_profile_count=5?

Fix: Reduce onboard_memory to ["No", "Yes"]. Counts belong exclusively in onboard_profile_count. Add cross-validation: if onboard_memory="No", onboard_profile_count must be null.

3. Warnings

WarningSwitchhot_swappable source map inconsistency

Source field_studio_map.json component_sources still declares hot_swappable: variance_policy: "authoritative". The compiler silently overrides to "override_allowed" in the generated output. The source should match intent.

Fix: Update component_sources to variance_policy: "override_allowed" for hot_swappable.
WarningPerformancescan_rate range too low
Current range
125 – 8,000 Hz
Real-world max
Wooting 80HE: 24,000 Hz; Razer claims 30,000+ Hz scan rates

Scan rate is distinct from USB polling rate. Hall Effect keyboards have scan rates far exceeding 8,000 Hz. Valid products would be rejected.

Fix: Increase scan_rate max to at least 40000 Hz in both the contract range and cross_validation_rules.
WarningSwitchswitch_lifespan range ceiling

Max 200 million keystrokes. Hall Effect switches are marketed with lifespan claims of 300M+ or "unlimited." The max should accommodate these values without rejecting valid data.

Fix: Raise max to 500 million keystrokes.
WarningConnectivitybluetooth_version missing BT 4.x

Enum starts at 5.0 but budget/older keyboards (Logitech K380, Microsoft Sculpt, older Anne Pro) use Bluetooth 3.0, 4.0, 4.1, 4.2. Since policy is "open" they won't be rejected, but the sparse seed list biases LLM normalization.

Fix: Add "3.0", "4.0", "4.1", "4.2" to the enum seed values.
WarningSwitchswitch_type policy too loose

Policy is "open_prefer_known" but switch types are a closed taxonomy. The 8 values (Mechanical, Low-profile Mechanical, Optical, Magnetic, Hall Effect, Membrane, Scissor, Hybrid) cover all known switch technologies. Nonsensical values or typos could silently enter the system.

Fix: Change to "closed_with_curation". New technologies should go through curation.
WarningGeneralwarranty — no enum or structure

Free-text string with no seed enum. Known values list is empty. Warranties cluster around standard durations but without normalization guidance, LLM produces inconsistent formats ("1yr", "1 Year", "12 months").

Fix: Add seed enum: ["1 year", "2 years", "3 years", "5 years", "Limited Lifetime", "None"]. Set policy to "open_prefer_known".
WarningGeneralprice_range — unstructured

Type: string, no enum, no format hint. Prices follow patterns ("$49.99", "Budget", "$100-200") but with no guidance the LLM will produce inconsistent formats.

Fix: Either define tiered enum ["Budget ($0-50)", "Mid-range ($50-150)", "Premium ($150-300)", "Enthusiast ($300+)"] or add a format_hint for structured price ranges.
WarningGeneralcolor vs colors naming confusion

color = list of title-case user-facing labels ("Black", "Silver"). colors = list of lowercase hyphenated EG palette tokens ("dark-blue", "light-gray") with closed policy. The singular/plural naming is counterintuitive and risks LLM confusion during ingestion.

The colors tooltip now has excellent EG-compatible guidance, but the field name alone doesn't signal "this is a design-system concept, not a product attribute."

Consider: Rename colors to color_tokens to prevent confusion.
WarningLayoutkey_count range may exclude edge cases

Range 20–120. Macro pads (3–12 keys) and full-size keyboards with extra macro columns (up to 130 keys, e.g., Corsair K100 = 116 keys) may fall outside.

Fix: Expand to 10–135 if macro pads are in scope.
WarningDimensionstyping_angle range ceiling

Max 15 degrees. Ergonomic split keyboards (ZSA Moonlander, Kinesis Advantage) support tenting up to 45 degrees. If typing_angle covers tenting, 15 is too low.

Fix: Clarify tooltip. If tenting is in scope, raise max to 50. If not, consider a separate tenting_angle field.

4. Suggestions

SuggestKeycapskeycap_profile missing popular profiles

Missing: MT3 (Drop), MDA, ASA (Akko), MOA. Since policy is "open" they won't be rejected, but seed values improve LLM normalization.

SuggestFeaturesbacklighting missing "Single-color LED"

Some keyboards have red, blue, or green single-color backlighting. Current: ["None", "White", "RGB", "Per-key RGB"]. Add "Single-color LED".

SuggestLayoutform_factor missing ergonomic forms

No "Ergonomic" or "Split" form factor. Kinesis, ZSA Moonlander, Dygma Defy don't fit percentage-based categories. Add "Ergonomic" and/or "Split".

SuggestSwitchswitch_name — small seed list

Only 15 switches for "open_prefer_known" policy. Missing notable switches: Cherry MX Silent Black, Gateron Oil King, Kailh Box Jade, Akko CS series, Boba U4T, Holy Panda, Outemu series. Expand to 30–40 for better normalization.

SuggestSwitchswitch_type missing "Electrostatic Capacitive"

Topre switches use electrostatic capacitive technology. Currently lumped with "Mechanical". Enthusiasts strongly distinguish these. Add "Electrostatic Capacitive".

SuggestSwitchswitch_brand missing popular brands

Missing: "Akko", "Outemu", "TTC", "JWK/Durock", "Boba (Gazzew)". These are widely used in pre-built and enthusiast keyboards.

SuggestConnectivityconnection — add "Tri-mode" alias

"Tri-mode" (Wired + 2.4GHz + Bluetooth) is a common industry term. Currently mapped to "Hybrid". Add as alias or enum value. Also consider "Dual-mode" (Bluetooth + Wired).

SuggestGeneraleditions — sparse aliases

Only alias: ["editions"]. Missing: "special edition", "limited edition", "collaboration", "colorway", "variant edition". Reduces parser coverage.
