1 Why this upgrade

The M75 verification gap

While auditing RDF results for the Corsair M75 Air Wireless family, a human looking at the stored evidence_refs[] on each release-date candidate could see only {url, tier, confidence}. Several URLs had no visible date signal when opened in a browser — the page simply lists the product. The LLM's reasoning (which did include literal quotes and inference chains) was buried inside discovery_log.notes, not tied to the specific URL it came from.

Verified by fetching the actual URLs on 2026-04-19:

  • Corsair Explorer article → visible on-page text: "As of 04/19/2024, You can now get the M75 AIR in your choice of Black, Grey, or White!" — strong, verifiable
  • TweakTown news → byline "Published Oct 12, 2023" + body "available now for USD 199.99" — strong, verifiable
  • Corsair product page for Black SKU → no date anywhere on the page or in metadata — URL was likely cited for identity-only purposes, but the evidence_refs shape did not distinguish it

The LLM sees the same rendered HTML a browser does (no hidden backend). The fix is to persist what on the page supported the claim alongside each URL so a human can click through and verify.

The core shift Evidence today: "here is a URL with a tier + my confidence." Evidence after this upgrade: "here is a URL, here is the exact text or reasoning on it that supports the claim, and here is the kind of evidence that text represents."

2 Scope — who opts in

Opt-in consumers
ConsumerShapeRationale
RDF (release-date finder) {url, tier, confidence, supporting_evidence, evidence_kind} Answers "what value." Often requires deduction (rebadged sensors, byline-as-proxy, artifact metadata).
variantScalarFieldProducer (every per-variant scalar — dpi, polling_rate, sensor, switch, etc.) {url, tier, confidence, supporting_evidence, evidence_kind} Same as RDF — value-focused with deduction common across 80–113 keys per category.
CEF (color/edition finder) {url, tier, confidence} — unchanged Question is "does this colorway/edition exist on this product." A URL either lists it or it doesn't; a quote would be redundant to the URL page title/h1.
PIF (product-image finder) {url, tier, confidence} — unchanged Image-level validation (HEAD check + content-type + dimensions), not text evidence.

The extended schema is built into the shared evidence prompt fragment at src/core/finder/evidencePromptFragment.js and gated by an includeEvidenceKind option. Finders opt in by passing the flag; the default is off to preserve CEF/PIF behavior.

3 The evidence_kind enum

Derived by surveying every field key across every current category — 80 (mouse) + 103 (keyboard) + 113 (monitor) = 296 keys. Each kind maps to real cases the LLM will encounter.

Enum values
evidence_kind When to use Example field keys
direct_quote common Verbatim on-page text states the claim. Reader can ctrl-F to find it. dpi, polling_rate, actuation_force, actuation_distance, switch_lifespan, refresh_rate, response_time_gtg, screen_size, resolution, weight, cable_length, key_count, layout
structured_metadata common Machine-readable tag in HTML source — JSON-LD, OG tag, meta tag, "Date First Available". sku, mpn, gtin, release_date (Amazon/JSON-LD), any field where the answer is in application/ld+json
byline_timestamp date The URL's own publish/review date is the proxy — "this review dated Oct 2023 means the product existed by then." release_date (bounded-from-below by a dated review), discontinued (bounded-from-above by last dated listing)
artifact_metadata oem PDF manual / datasheet / spec-sheet file metadata — revision footer, filename pattern, copyright. sensor_date (datasheet footer), sensor_brand, switch_brand, mcu, encoder_brand
visual_inspection photo Read from product images/renders. Counting ports, reading finish, identifying layout from photos. honeycomb_frame, thumb_rest, rgb_zones, coating, feet_material, layout (ANSI/ISO), hdmi_ports count, usb_c_ports count, stabilizer_mount, adjustable_feet, pip_pbp
lab_measurement test Third-party test lab value. RTINGS, TFT Central, LinusTechTips, NotebookCheck, etc. click_latency_*, sensor_latency_*, input_lag_*, response_time_mprt, overshoot_*, color_gamut_dci_p3, color_accuracy_delta_e, brightness_hdr_peak, contrast_ratio_dynamic, flawless_sensor
comparative_rebadge oem Component is a known rebadge of an OEM part; OEM-level specs propagate to this product. sensor (PMW3360/3389/3395), switch_type (Omron D2FC, Kailh GM 8.0), encoder, mcu (Nordic nRF52, ARM Cortex)
inferred_reasoning soft Chain-of-reasoning across multiple signals when no single one is sufficient. The supporting_evidence field describes the reasoning. form_factor, hand_size, grip, silent_clicks, hump, front_flare, shape, typing_feel, sound_profile
absence_of_evidence negative The negative case. "No retailer listings exist before X" / "manufacturer page no longer carries this SKU." discontinued, release_date (lower-bound by absence of earlier listings)
identity_only weak URL cited only to establish the product identity — not evidence for the field value itself. Must not count toward the evidence-threshold gate. Any manufacturer product-page URL used just to prove the SKU exists
Guardrails for the publisher gate identity_only evidence refs do not satisfy minEvidenceRefs. Every value-bearing field must have at least one ref whose evidence_kind is something other than identity_only. Rationale: a URL that just proves the product exists is not evidence for the claim.

4 Schema shape

Zod schema (additive — existing shape still valid)

Location: src/features/release-date/releaseDateSchema.js, src/core/finder/variantScalarFieldSchema.js (or equivalent — find at plan-time). Also add to the shared evidencePromptFragment.

const EVIDENCE_KIND = z.enum([
  'direct_quote',
  'structured_metadata',
  'byline_timestamp',
  'artifact_metadata',
  'visual_inspection',
  'lab_measurement',
  'comparative_rebadge',
  'inferred_reasoning',
  'absence_of_evidence',
  'identity_only',
]);

const evidenceRefExtended = z.object({
  url: z.string().url(),
  tier: z.string(),
  confidence: z.number().int().min(0).max(100),
  supporting_evidence: z.string().max(280),   // quote OR reasoning summary
  evidence_kind: EVIDENCE_KIND,
});

// RDF + scalar producers use evidenceRefExtended.
// CEF + PIF keep evidenceRefBase ({url, tier, confidence}) — unchanged.

5 Prompt changes

evidencePromptFragment.js gains an opt-in block

buildEvidencePromptBlock({ minEvidenceRefs, includeEvidenceKind }). When includeEvidenceKind: true, append this block (sourced from globalPromptRegistry under key evidenceKindGuidance):

For EACH evidence_ref, include:
- "supporting_evidence": <=280 chars. Either the EXACT quote from the page
  that supports the claim, or — for inferred/deductive cases — a one-line
  summary of the reasoning tying the URL to the claim.
- "evidence_kind": one of the following. Pick the most specific match.

  direct_quote          — verbatim on-page text states the claim
  structured_metadata   — JSON-LD / OG / meta tags / "Date First Available"
  byline_timestamp      — the URL's own publish/review date IS the proxy
  artifact_metadata     — PDF manual/datasheet file metadata (footer, rev)
  visual_inspection     — read from product images/renders
  lab_measurement       — third-party test lab value (RTINGS, TFT Central)
  comparative_rebadge   — component is a known OEM-part rebadge
  inferred_reasoning    — chain-of-reasoning across multiple signals
  absence_of_evidence   — the negative case (no listings before X)
  identity_only         — URL proves SKU exists, NOT the field value

Rules:
- identity_only refs do NOT satisfy the minimum-evidence threshold — at least
  one ref per field must be a substantive kind.
- supporting_evidence must be empty string "" when evidence_kind is
  identity_only (there is no claim-supporting content to cite).
- For direct_quote: copy the text exactly as it appears; do not paraphrase.
- For inferred_reasoning: name the signals you combined ("review dated
  2023-10 + Corsair press release 2023-10-12 + Amazon listing from Oct 2023").
Global prompt registry additions

Add to src/core/llm/prompts/globalPromptRegistry.js:

  • evidenceKindGuidance — the block above (user-editable from the GUI Global Prompts panel)

No changes to identityWarningEasy/Medium/Hard, siblingsExclusion, evidenceContract, evidenceVerification, valueConfidenceRubric, discoveryHistoryBlock.

Per-finder wiring
  • RDF — releaseDateLlmAdapter.js: pass includeEvidenceKind: true into buildEvidencePromptBlock. Update releaseDateFinderResponseSchema to use the extended evidence ref shape.
  • Scalar producer — variantScalarFieldProducer.js: same flag, extend the response-schema builder. This propagates to every per-key finder module built on the producer.
  • CEF / PIF — no change. Verify the default (includeEvidenceKind: false) preserves current behavior.

6 Publisher gate

Counting rule update

Find the publisher code that checks evidence_refs.length >= minEvidenceRefs (likely in src/features/publisher/). Change the count to ignore refs whose evidence_kind === 'identity_only'.

const substantiveRefs = (evidence_refs || []).filter(
  (r) => r.evidence_kind !== 'identity_only'
);
const meetsMinimum = substantiveRefs.length >= minEvidenceRefs;

Legacy refs without evidence_kind default to "substantive" (count toward the minimum). This keeps pre-migration data valid.

7 GUI evidence drawer

Evidence-refs list rendering

In the variant/candidate evidence drawer (RDF + per-key finder panels), render each ref as:

  • URL (clickable, opens in new tab)
  • Kind badge — color-coded: green for direct_quote/structured_metadata; teal for byline_timestamp; orange for artifact_metadata/comparative_rebadge; purple for visual_inspection; pink for lab_measurement; orange-soft for inferred_reasoning; red for absence_of_evidence/identity_only
  • Supporting evidence — quoted block for direct_quote/structured_metadata; italicized "(reasoning)" prefix for inferred_reasoning/comparative_rebadge; dim "(identity-only — not counted)" for identity_only
  • Tier + confidence on a secondary line

Add a "copy quote" button for direct_quote refs so a user can ctrl-F the source page to verify.

8 Migration + backfill

Existing runs

Old evidence_refs[] without supporting_evidence/evidence_kind remain valid. No backfill required; the GUI falls back to rendering just URL + tier + confidence when the new fields are absent.

Rerunning RDF or any scalar-producer module on a product will produce the new shape going forward. No schema version bump needed — the extended fields are additive-optional.

9 Test strategy (MACRO-RED first)

Tests to write BEFORE any code
  • Prompt fragment: buildEvidencePromptBlock({ includeEvidenceKind: true }) includes the 10 enum values and the "identity_only does not satisfy minimum" rule; includeEvidenceKind: false does not.
  • Zod: extended evidenceRef rejects an unknown evidence_kind, rejects supporting_evidence > 280 chars, accepts empty string when evidence_kind === 'identity_only'.
  • RDF response validator: accepts the new payload shape with all 10 kinds; rejects a payload where every ref is identity_only (no substantive evidence).
  • Publisher gate: a candidate with 2 refs — one identity_only, one direct_quote — passes with minEvidenceRefs: 1; a candidate with 2 identity_only refs fails with minEvidenceRefs: 1.
  • CEF + PIF regression: existing evidence shape still compiles and validates; includeEvidenceKind defaults to false.
  • End-to-end: an RDF run on a 3-sibling-family product produces stored evidence_refs[] with supporting_evidence and evidence_kind populated on every non-identity-only ref.

10 Acceptance criteria

Done when
  • A new RDF run on mouse-10836a9a produces evidence_refs[] with supporting_evidence matching the actual page text (e.g., "As of 04/19/2024, You can now get the M75 AIR in your choice of Black, Grey, or White!") and evidence_kind: "direct_quote".
  • The GUI evidence drawer renders the quote inline under each URL.
  • A candidate whose only evidence is identity_only refs is rejected by the publisher gate.
  • CEF and PIF stored prompts are byte-identical before/after the change (regression safety).
  • Full finder test suite green (1710+ existing tests) plus new tests for this upgrade.
  • One scalar-producer field (pick dpi or polling_rate) has a live run demonstrating a mix of direct_quote and identity_only refs correctly tagged.
