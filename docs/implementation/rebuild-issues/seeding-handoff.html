2. Known Issues

Issue 1: Upsert-only seeders don’t delete stale rows blocking

When a JSON source removes an entry, the SQL row persists. This means editing JSON and expecting SQL to match only works for additions and updates, not deletions.

SurfaceProblemFix Pattern
brand_registry.jsonRemoved brands stay in SQLAfter upsert loop, delete brands NOT in JSON set
user-settings.jsonRemoved setting keys stay in SQLAfter upsert loop, delete keys NOT in JSON sections
product_catalog.jsonRemoved products stay in SQLAfter upsert, delete products NOT in catalog for category
checkpoint reseedDeleted run/product dirs → stale SQL rowsAfter scan, delete rows for run_ids NOT found on disk
color_edition reseedDeleted product dirs → stale color rowsAfter scan, delete rows for product_ids NOT found on disk

Template: The color_registry reconcile (built today) is the reference implementation. Pattern: build Set of source keys → upsert all → delete SQL rows where key NOT IN set.

Issue 2: Override files are stale for pre-T3a products needs backfill

The write-back to .overrides.json was added in T3a (today). Products reviewed before today have stale override files without ai_review sub-objects. A one-time backfill is needed: read SQL state for all reviewed products, write refreshed .overrides.json files.

Implementation: Call readOverrideFile() (which reads from SQL including AI review state) for each product with overrides, then write the result to disk. Can be a CLI command or one-time script.

Issue 3: Old run.json files lack brand_resolution can’t backfill

The brand_resolution field was added to buildCrawlCheckpoint today (T3c). Existing run.json files on disk predate this change and don’t contain it. On rebuild, those runs won’t have brand_resolution artifacts. Only new runs going forward will capture it.

Mitigation: Could write a backfill that reads brand_resolution from current SQL (if DB still exists) and patches old run.json files. Low priority — brand_resolution is re-derivable by re-running the brand resolver.

Issue 4: key_review_runs telemetry not checkpointed needs design

key_review_runs, key_review_run_sources, and key_review_audit are actively read by the review workbench (AI review history, model used, cost, confidence). Base key_review_state rows rebuild from derived data, but the run telemetry (which model ran, token counts, cost) is SQL-only.

Options:

  • Include key_review_run data in .overrides.json per product (alongside ai_review)
  • Write a separate {productId}.review_runs.json checkpoint
  • Accept as re-derivable: next AI review run repopulates the data

Issue 5: Dead tables still in schema cleanup

These tables exist in specDbSchema.js but have no writers, no readers, and no purpose:

  • curation_suggestions — seed path removed
  • component_review_queue — seed path removed
  • source_corpus — sourceCorpus module removed
  • audit_log — zero readers, write-only dead code
  • runtime_events — replaced by bridge_events, zero readers

Remove from: specDbSchema.js, specDbStatements.js, specDb.js, store modules, and any test fixtures.

4. O(1) Vertical Seeding Architecture

The goal: adding a new JSON-backed data surface should require exactly one registry entry. No manual wiring in seed functions, no per-surface if/else chains, no scattered import paths.

Current Architecture (mixed)

App DB:   seedAppDb() — per-source hash check, hardcoded per surface
Spec DB:  seedSpecDb() — 1,875-line function with 10 sequential steps
          reseedPhases[] — O(1) registry (4 phases), each independent

The reseed phase registry is already O(1) and vertical. seedSpecDb() is not — it’s a monolith.

Target Architecture

Every JSON → SQL surface is a registry entry with:
  {
    name: 'component-identity',
    source: (config, category) => loadComponentDbs(config, category),
    seed: (db, data) => reconcileComponents(db, data),
    hash: (config, category) => fieldRulesSignature(config, category),
    scope: 'category',       // 'global' | 'category'
    runOn: 'seed+reconcile', // 'seed' | 'reseed' | 'seed+reconcile'
  }

Adding a new surface = one entry in the registry.
The generic engine handles: hash comparison, seed/skip decision,
logging, error isolation, and stale-row pruning.

Migration Path

  1. Phase A: Convert each seedSpecDb() step into a standalone reconcile function with the same signature: (db, data) → { seeded, removed }
  2. Phase B: Register each as a seed phase entry in a unified registry (merge seedSpecDb steps + reseedPhases into one array)
  3. Phase C: Add full reconcile (delete stale) to each surface that currently only upserts
  4. Phase D: Add hash gating per surface so unchanged sources skip entirely
  5. Phase E: Delete the seedSpecDb() monolith — the registry IS the seed

What Each Reconcile Function Must Do

1. Read source (JSON file, checkpoint dir, derived data)
2. Build expected set (all keys/IDs from source)
3. Upsert all rows from source into SQL
4. Delete SQL rows where key NOT IN expected set
5. Return { seeded: N, removed: N }

This is the color_registry pattern from today — generalized to every surface.

7. SSOT Overlaps — JSON Must Be Sole Authority

Principle: JSON = SSOT. SQL = derived cache. Edit a JSON file, rebuild, SQL matches perfectly. No circular sync, no dual writes, no ambiguity about which copy wins.

Four overlaps violate this principle today. Each must be resolved before the O(1) seed registry can be built, because the registry needs exactly one JSON source per surface — no conflicts.

Overlap 1: Product Identity blocking

Same data (brand, model, variant) lives in 4 places:

FileWritten WhenContains
product_catalog.jsonUser adds product via GUI/CLIbrand, model, variant, category
product.json (per product)Product added + after each runSame identity fields + runs_completed, sources, fields
normalized.json (per product)After extraction pipelineApplied identity values (output snapshot)
SQL productsSeed + reseedMerged from all above

Problem: If you edit product_catalog.json to fix a brand typo, product.json still has the old name. On rebuild, whoever seeds last wins — non-deterministic.

Resolution

FileRole After Fix
product_catalog.jsonSSOT for product identity (brand, model, variant, category). Editable. Rebuild source.
product.jsonRun history only. Stores runs_completed, latest_run_id, sources[], fields[], query_cooldowns. Identity fields removed or marked as “snapshot at creation, not authoritative.”
normalized.jsonDerived output. Never read back during seed. Read-only snapshot of extraction results.
SQL productsDerived cache. Rebuilt from product_catalog.json (identity) + product.json (run history).

Implementation:

  1. Stop writing identity fields to product.json in writeProductIdentity() / writeProductCheckpoint() (or keep as non-authoritative snapshot with clear marker)
  2. Seed path reads identity from product_catalog.json only, run history from product.json
  3. Reconcile: delete SQL products NOT in catalog JSON for that category

Files: src/features/catalog/products/writeProductIdentity.js, src/pipeline/checkpoint/writeProductCheckpoint.js, src/db/seed.js (seedProductCatalog)

Overlap 2: Queue / Run State blocking

Same data (runs_completed, latest_run_id, status) lives in 3 places:

FileWritten WhenContains
_queue/state.jsonQueue adapter writes after mutationsstatus, retry_count, last_run_id per product
product.json (per product)After each run completesruns_completed, latest_run_id, sources[]
SQL product_queueSeed + runtimeFull queue row (all fields)

Problem: Queue adapter switches between SQL and JSON modes depending on whether specDb is available. No reconciliation — if SQL and JSON diverge, reads are non-deterministic. runs_completed in product.json can disagree with state.json.

Resolution

FileRole After Fix
product.jsonSSOT for run accumulation: runs_completed, latest_run_id, sources[]. Updated after each run.
_queue/state.jsonSSOT for operational queue state: status, retry_count, priority. Separate concern from run history.
SQL product_queueDerived cache. Rebuilt by merging product.json (run history) + state.json (queue ops). Single code path, no adapter switching.

Implementation:

  1. Kill the dual-path adapter — remove SQL-direct queue writes. All queue mutations write to state.json.
  2. Seed path reads product.json for run history, state.json for queue ops, merges into SQL
  3. Reconcile: delete SQL queue rows for products NOT found on disk

Files: src/queue/queueState.js, src/pipeline/checkpoint/scanAndSeedCheckpoints.js, src/db/seed.js (seedQueueState)

Overlap 3: Settings Circular Sync high severity

Same data (runtime, studio, ui settings) lives in 2 places:

FileWritten WhenContains
user-settings.jsonSQL mirrors to JSON after API mutationsruntime, convergence, storage, studio, ui sections
SQL settings + studio_mapsBoot (from JSON) + API mutationsSame sections as rows

Problem: Circular sync: Boot: JSON → SQL. Runtime: SQL mutates. Write-back: SQL → JSON. Next boot: JSON → SQL. If crash happens between SQL write and JSON mirror, the mutation is lost on next rebuild. Two concurrent sessions = last-write-wins with no merge.

Resolution

FileRole After Fix
user-settings.jsonSSOT for all settings. API mutations write here directly. Editable by hand.
SQL settings + studio_mapsDerived cache. Rebuilt from JSON on boot via hash-gated reconcile. Never mutated at runtime.

Implementation:

  1. API mutation path (userSettingsService) writes to JSON file directly instead of SQL
  2. Remove SQL → JSON mirror (the write-back). JSON is the only write target.
  3. Boot path unchanged: JSON → SQL (hash-gated). But SQL is never mutated after boot.
  4. Full reconcile: delete SQL setting keys NOT in JSON

Files: src/features/settings-authority/userSettingsService.js, src/db/appDbSeed.js

Overlap 4: Override Values medium severity

Same data (override_value, provenance, ai_review) lives in 3 places:

FileWritten WhenContains
.overrides.json (per product)Legacy: at review time. Phase E3: read-onlyoverride_value, provenance, ai_review per field
SQL item_field_state + candidate_reviewsRuntime (primary since Phase E3)Same + more columns (confidence, timestamps)
latest/normalized.jsonFinalize snapshotApplied field values (derived output)

Problem: Phase E3 made SQL the primary write target and JSON read-only. But for the “JSON = SSOT” goal, this is backwards. The .overrides.json files need to be the authority, and SQL must rebuild from them. Currently, pre-T3a products have stale .overrides.json files missing ai_review data.

Resolution

FileRole After Fix
.overrides.jsonSSOT for human decisions: override_value, provenance, ai_review. Runtime writes here first.
SQL item_field_stateDerived cache. Rebuilt from .overrides.json + candidates.json on seed.
latest/normalized.jsonDerived output. Never read back during seed. Snapshot only.

Implementation:

  1. Backfill first: Export current SQL state to .overrides.json for all existing products (one-time, makes JSON current)
  2. Flip write direction: override workflow writes to JSON first, SQL rebuilds from it
  3. Seed path reads .overrides.json as authority for item_field_state override columns + candidate_reviews
  4. normalized.json is never read during seed (output only)

Files: src/features/review/domain/overrideWorkflow.js, src/features/review/domain/overrideHelpers.js, src/db/seed.js
