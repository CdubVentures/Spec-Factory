Key architectural decision: The accepted_candidate_id columns in item_field_state, component_values, and list_values will be left as nullable TEXT columns after candidates removal. They have no FK constraints and no NOT NULL constraints. All are currently NULL in the live DB. Dropping columns in SQLite requires full table recreation — not worth the risk. They become inert dead weight.
Safe invariant: key_review_* tables survive all 7 phases. They have zero FOREIGN KEY references to source_registry or candidates. The seed function seedSourceAndKeyReview() phases 9b–9e seed key_review_* from item_field_state, component_values, list_values, and candidate_reviews directly — not through source_registry.

7

High risk — largest scope

The pipeline no longer writes candidates.json (writeDiscoveryPayloads() is a no-op). All 27,783 item_field_state rows have accepted_candidate_id = NULL. The entire candidate store, review candidate runtime, and candidate infrastructure code is disconnected. This phase touches ~40 files across DB, API, review domain, review API, CLI, publish, and GUI.

This is the largest phase. Consider splitting into sub-phases: (7a) DB layer, (7b) API + domain, (7c) review routes + handlers, (7d) GUI types + components, (7e) test cleanup. Run tests after each sub-phase.
