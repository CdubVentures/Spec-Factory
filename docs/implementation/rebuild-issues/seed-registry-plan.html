0. The Core Idea

One file. Every surface. Three scopes. — src/db/seedRegistry.js is the single source of truth for “what gets seeded, when, and where.” A new developer reads this file and knows the full system. No archaeology across bootstrap, seed.js, and specDbRuntime.js required.

The registry serves two purposes:

  1. Discoverability — every surface is documented in one place with its scope, tables, dependencies, source file, and data-flow direction. Applies to ALL scopes (global, category, reseed).
  2. Orchestration — the category seed engine and reseed phase runner iterate the registry instead of maintaining inline dispatch. Applies to category + reseed scopes only.

Global surfaces appear in the registry for discoverability but are not engine-dispatched. They have different semantics (different DB, interleaved config rehydration, SQL-primary data flow) that make generic dispatch wrong.

Three scopes

ScopeDBWhenError modelRegistry role
global app.sqlite Once at server bootstrap Each surface hash-gated independently documented — not engine-dispatched
category spec.sqlite (per category) On category open / field-rules change Hard-fail: any throw aborts the entire seed orchestrated — engine iterates registry
reseed spec.sqlite (per category) After auto-seed, on partial rebuild Best-effort: errors caught per-phase, logged orchestrated — runtime reads registry

O(1) — honest boundaries

O(1) is conditional on the field-rules signature boundary. Category seed is gated by buildFieldRulesSignature() which hashes 7 specific inputs (field_rules.json, known_values.json, parse_templates.json, cross_validation_rules.json, ui_field_catalog.json, component_db/*, component_overrides/*). A new surface whose source falls inside this boundary is truly O(1) — one registry entry and done. A new surface with a novel source outside this boundary also requires updating the signature in src/field-rules/loader.js so hash-gated reconcile detects the change. That makes it O(2), not O(1).
ScenarioFiles to touchScaling

New category surface, novel source outside boundary1 (registry) + 1 (signature) + seederO(2)
New reseed surface1 (registry entry) + seederO(1)
New global surface1 (registry metadata) + seeder + bootstrap wiringO(2)

Why global surfaces are documented but not dispatched

// createBootstrapSessionLayer.js — the actual boot order
1. seedAppDb({ appDb, brandRegistryPath, userSettingsPath })   // brands + settings → app.sqlite
2. rehydrateConfig(appDb)                                      // MUST happen between seeders
3. seedColorRegistry(appDb, colorRegistryPath)                  // colors → app.sqlite

Step 2 rehydrates runtime config from SQL after settings are seeded. Color registry depends on that config. Additionally: global surfaces write to appDb (not specDb), are SQL-primary with JSON fallback, and each has its own per-surface hash-gate.

1. Design Constraints (From Code Reality)

Constraint 1: Seeders have different signatures — don’t unify them

SeederActual SignatureI/O Pattern
seedComponents(db, fieldRules)Reads fieldRules.componentDBs in-memory
seedComponentOverrides(db, config, category)Scans _overrides/components/ dir
seedListValues(db, fieldRules, config, category)Reads fieldRules + known_values
seedProducts(db, config, category, fieldRules, fieldMeta)Multi-source directory scan + consolidated overrides
backfillComponentLinks(db, fieldMeta, fieldRules)Derived from SQL state — no JSON
seedSourceAndKeyReview(db, category, fieldMeta)Derived from SQL state — no JSON

Each registry entry uses a one-line closure to adapt its seeder to execute(ctx). Existing seeder functions are never modified.

Constraint 2: Category seed is hard-fail; reseed is best-effort

If any step in seedSpecDb() throws, the entire sync fails and specDbSyncService records failure. Reseed phases catch errors per-phase and log them. The engine preserves both semantics via scope.

Constraint 3: Category-level hash-gating stays outside the registry

buildFieldRulesSignature() produces one hash from 7 inputs. If unchanged, triggerHashGatedReconcile skips the entire seedSpecDb call. This gates whether the engine runs at all — not a per-surface concern.

Constraint 4: Reconcile/cleanup steps are surface-local hooks

  • reconcileComponentDbRows(db, fieldRules) — runs before seedComponents
  • reconcileListSeedRows(db, fieldRules, config, category) — runs before seedListValues
  • db.pruneOrphanCandidateReferences() — runs before AND after seedSourceAndKeyReview

These are before/after hooks on their respective surface entries.

Constraint 5: No circular imports

A naive design creates a cycle: seedRegistry.js imports step functions from seed.js, seedEngine.js imports from seedRegistry.js, seed.js imports from seedEngine.js.

Fix: Category surfaces use a factory pattern. seedRegistry.js exports buildCategorySurfaces(steps) that receives step functions as arguments — no import from seed.js. seedEngine.js receives surfaces as a parameter — no import from seedRegistry.js. seed.js is the composition root that wires everything together.

// Dependency graph (no cycles):
seedRegistry.js  ← (no imports from seed.js or seedEngine.js)
seedEngine.js    ← (no imports from seed.js or seedRegistry.js)
seed.js          → seedRegistry.js (buildCategorySurfaces)
seed.js          → seedEngine.js   (runCategorySeed)

Global entry (documentation only — no execute)

  scope:      'global',
  db:         'app.sqlite',
  tables:     ['brands', 'brand_categories', 'brand_renames'],
  source:     'category_authority/_global/brand_registry.json',

  hashGated:  true,
  hashKey:    'brand_registry',
  calledFrom: 'src/api/bootstrap/createBootstrapSessionLayer.js',

}

No execute, no summarize. Metadata only. New dev reads it and knows: what exists, where it lives, what it writes to.

Category entry (factory-built, engine-dispatched)

Category surfaces are built via buildCategorySurfaces(steps) to avoid circular imports. The factory receives step functions as arguments and returns surface entries:

// seedRegistry.js — factory function (no imports from seed.js)
export function buildCategorySurfaces(steps) {
  return [
    {
      key:       'components',
      label:     'Components',
      scope:     'category',
      dependsOn: [],
      tables:    ['component_identity', 'component_aliases', 'component_values'],

      before:    (ctx) => steps.reconcileComponentDbRows(ctx.db, ctx.fieldRules),
      execute:   (ctx) => steps.seedComponents(ctx.db, ctx.fieldRules),
      after:     null,
      summarize: (result, beforeResult) => ({
        components_seeded: result.identityCount,
        removed_identity_rows: beforeResult?.removed_identity_rows ?? 0,
        removed_value_rows: beforeResult?.removed_value_rows ?? 0,
        removed_alias_rows: beforeResult?.removed_alias_rows ?? 0,
        removed_item_component_link_rows: beforeResult?.removed_item_component_link_rows ?? 0,
        removed_key_review_rows: beforeResult?.removed_key_review_rows ?? 0,
      }),
    },
    // ... remaining surfaces
  ];
}

seed.js is the composition root — it passes the private step functions into the factory:

// seed.js — composition root (imports registry + engine, no cycle)
import { buildCategorySurfaces } from './seedRegistry.js';
import { runCategorySeed } from './seedEngine.js';

const surfaces = buildCategorySurfaces({
  seedComponents, reconcileComponentDbRows,
  seedComponentOverrides,
  seedListValues, reconcileListSeedRows,
  seedProducts,
  backfillComponentLinks,
  seedSourceAndKeyReview,
});

export async function seedSpecDb({ db, config, category, fieldRules, logger }) {
  return runCategorySeed({ db, config, category, fieldRules, logger, surfaces });
}

Reseed entry (static data, runtime-dispatched)

Reseed surfaces import from feature modules (not seed.js), so they are plain static entries — no factory needed.

{
  key:       'checkpoint',
  label:     'Checkpoints',
  scope:     'reseed',
  tables:    ['products', 'product_queue', 'runs', 'product_runs',
              'run_artifacts', 'crawl_sources', 'url_crawl_ledger',
              'query_cooldowns', 'screenshots', 'videos'],
  shouldRun: (ctx) => Boolean(ctx.indexLabRoot),
  execute:   (ctx) => scanAndSeedCheckpoints({
               specDb: ctx.db, indexLabRoot: ctx.indexLabRoot,

  formatLog: (category, result) =>
    result.runs_seeded > 0
      ? `${category}: ${result.runs_seeded} runs re-seeded from checkpoints`
      : '',
}

4. How to Add a New Surface (The Whole Point)

New dev guide: “I need to seed a new table”

  1. Open src/db/seedRegistry.js. Read the inventory. Determine scope:
    • Global (app-wide, runs once at boot, writes to app.sqlite) → add a metadata entry, implement seeder in its domain module, wire into createBootstrapSessionLayer.js.
    • Category (per-category, writes to spec.sqlite) → add a surface to the buildCategorySurfaces factory return array. Pass the step function via the steps arg. Done — engine picks it up.
    • Reseed (per-category, best-effort, runs after auto-seed) → add an entry to RESEED_SURFACES. Done — runtime picks it up.
  2. Write the seeder function. It can have any signature — the closure in the registry entry adapts it.
  3. For category surfaces: check whether your data source is inside the buildFieldRulesSignature() boundary (the 7 inputs in src/field-rules/loader.js:127). If not, add your source to the signature so hash-gated reconcile detects changes on category reopen.

Phase 2: Category Seed Engine contract → red → green

2a — Contract

Module: src/db/seedEngine.js

Exports:

  • runCategorySeed({ db, config, category, fieldRules, logger, surfaces })

The engine receives surfaces as a parameter (dependency injection). No imports from seedRegistry.js or seed.js.

Engine algorithm:

Build ctx → Toposort surfaces → For each: before → execute → after → summarize → Aggregate & return
  1. Project fieldRules via projectFieldRulesForConsumer(fieldRules, 'seed')
  2. Build fieldMeta via buildFieldMeta(projectedFieldRules)
  3. Pack into ctx
  4. Topological sort surfaces by dependsOn
  5. For each surface:
    • before(ctx) if defined → beforeResult
    • execute(ctx) → result
    • after(ctx) if defined
    • summarize(result, beforeResult, afterResult) → merge into aggregate
    • Log via logger
  6. On error: throw immediately (hard-fail)
  7. Collect product errors into errors array
  8. Return { category, duration_ms, errors, counts: db.counts(), ...aggregated }

Appendix A: Why the Original Plan Was Rejected

  1. JSON-centric entry shape. Proposed jsonPath → seeder(db, jsonData, ctx). Only 3 of 12 surfaces read JSON from a path. Corrected: execute(ctx) with closure adapters.
  2. Per-surface hash-gating. Category seed is all-or-nothing. Corrected: hash-gating stays outside the registry.
  3. Missing orchestration hooks. Corrected: before/after hooks on surfaces that own them.
  4. Unified error semantics. Corrected: scope determines error model.
  5. Forced global dispatch. Corrected: global surfaces are metadata, not engine-dispatched.

Appendix B: v2 → v3 Changes

Four issues fixed from v2 review:

  1. Inventory drift. Colors corrected from “53” to 77 (EG_DEFAULT_COLORS, test asserts ≥ 70). Checkpoint tables corrected from 4 to 10 (products, product_queue, runs, product_runs, run_artifacts, crawl_sources, url_crawl_ledger, query_cooldowns, screenshots, videos).
  2. O(1) honesty. Added explicit caveat: O(1) only holds for surfaces whose source falls within the buildFieldRulesSignature() boundary (7 inputs). Novel sources require updating the signature = O(2). Section 0 now has a clear table.
  3. Circular import. Fixed via factory pattern: buildCategorySurfaces(steps) receives step functions as arguments instead of importing them from seed.js. runCategorySeed receives surfaces as a parameter instead of importing them from seedRegistry.js. seed.js is the composition root. Dependency graph is acyclic.
  4. Characterization test permanence. Marked as temporary refactor guard. Phase 5c explicitly retires or relaxes it after wiring is proven. An exact-shape assertion that must be updated for every new surface would undermine the O(1) goal.
