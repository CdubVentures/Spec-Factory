## How It Works

The query journey ranks every query in a run into a single ordered list. Two ranking regimes apply:

1. **Sent queries** — ordered by first send timestamp (immutable once sent)
2. **Unsent queries** — ranked afterward by Pscore (planned priority score)

```
Example first line in the panel:

  First searched: Asus ROG Harpe Ace Extreme specs site:eloshapes.com (Deterministic)
  T+0.0s. First query sent in runtime execution order.
```

The label shows what was actually sent first, its selection method (Deterministic or Planner), and the timestamp offset.

## Sent Query Ordering (T+seconds)

Once a query has been dispatched to SearXNG, its position is locked by its `sent_ts` timestamp.

| Field | Source | Meaning |
|-------|--------|---------|
| `execution_order` | assigned 1-based by send order | immutable rank |
| `sent_ts` | earliest timestamp from search attempt records | when the query first left the system |
| `T+{n}s` | `(sent_ts - firstSentTs) / 1000` | seconds offset from the very first sent query |

**Ordering rule:** All sent queries come first, sorted by `execution_order` ascending. The query with `execution_order = 1` is the baseline (`T+0.0s`).

**Source:** `prefetchQueryJourneyHelpers.js:107-125`
```javascript
if (row.execution_order != null && row.sent_ts) {
  const sentMs = parseTsMs(row.sent_ts);
  const deltaSec = sentMs > 0 && firstSentMs > 0
    ? Math.max(0, (sentMs - firstSentMs) / 1000) : 0;
  row.order_metric = Math.round(deltaSec * 10) / 10;
  row.order_metric_label = `T+${row.order_metric.toFixed(1)}s`;
}
```

## Unsent Query Ranking (Pscore)

Unsent queries are ranked by a deterministic priority score computed from four components.

### Formula

```
Pscore = Pass + Target + Attempts + Constraints
```

**Theoretical range:** 0-118. **Practical range:** 0-60 for most queries.

**Source:** `prefetchQueryJourneyHelpers.js:83-97`
```javascript
const passType = (hasValidate ? 28 : 0) + (hasReason ? 20 : 0)
               + (hasPrimary ? 14 : 0) + (hasFast ? 8 : 0);
const targetCoverage = Math.min(24, row.target_fields.length * 4);
const attempts = Math.min(10, row.attempts * 2);
const constraints = (hasFieldRules ? 8 : 0) + (hasDomainConstraint ? 6 : 0);
const total = Math.max(0, passType + targetCoverage + attempts + constraints);
```

---

### Component 1 — Pass (0-70)

Points based on which LLM planner pass generated the query. A single query can be generated by multiple passes — all matching passes are additive.

| Pass | Points | Reason Code | Model | When |
|------|--------|-------------|-------|------|
| **Validate** | +28 | `discovery_planner_validate` | llmModelValidate (or llmModelReasoning) | Only when `missingCriticalFields.length > 0` |
| **Reason** | +20 | `discovery_planner_reason` | llmModelReasoning | Always (extended thinking mode) |
| **Primary** | +14 | `discovery_planner_primary` | llmModelPlan | Always (default planner pass) |
| **Fast** | +8 | `discovery_planner_fast` | llmModelFast | Always (manufacturer-biased) |
| Deterministic | 0 | search profile rules / base templates | n/a | Fallback when LLM unavailable |

**Why additive:** If a query was generated by both Primary (+14) and Reason (+20), it scores +34 — signaling that two independent planning strategies converged on the same query.

**Pass detection:** `prefetchQueryJourneyHelpers.js:14-22` normalizes reason codes:
- `discovery_planner_primary` -> Primary
- `discovery_planner_fast` -> Fast
- `discovery_planner_reason` -> Reason
- `discovery_planner_validate` -> Validate

**Planner pass sequence** (from `discoveryPlanner.js:162-198`):
1. Primary — "Keep queries compact and practical."
2. Fast — "Bias toward official manufacturer and support documents first."
3. Reason — "Prioritize unresolved critical fields and avoid repeating weak query patterns."
4. Validate — "Return only high-yield queries for critical field closure." (conditional)

---

### Component 2 — Target (0-24)

Points based on how many missing fields the query explicitly targets.

| Target Fields | Points |
|---------------|--------|
| 0 | 0 |
| 1 | 4 |
| 2 | 8 |
| 3 | 12 |
| 4 | 16 |
| 5 | 20 |
| 6+ | 24 (capped) |

**Formula:** `Math.min(24, targetFieldCount * 4)`

**Data source:** `row.target_fields` — populated from LLM planner response or field-rule search hints.

---

### Component 3 — Attempts (0-10)

A **retry boost** for previously searched queries. When coverage is still incomplete, queries that already returned some results can rank higher for re-execution.

| Logged Attempts | Points |
|-----------------|--------|
| 0 (new query) | 0 |
| 1 | 2 |
| 2 | 4 |
| 3 | 6 |
| 4 | 8 |
| 5+ | 10 (capped) |

**Formula:** `Math.min(10, loggedAttempts * 2)`

**Data source:** `query_rows.attempts` from the search profile — aggregated by `buildQueryAttemptStats()` in `discoveryHelpers.js:133-162`. Each search attempt record (one per provider per query execution) increments the counter.

**Why this exists:** A query that was searched once and returned 3 results (but didn't fully cover its target fields) should rank above an untried query with similar characteristics, because it has proven signal potential.

---

### Component 4 — Constraints (0-14)

Points for queries that are scoped to specific domains or enriched with field-rule hints.

| Constraint | Points | Detection |
|-----------|--------|-----------|
| Field-rule hints | +8 | `hint_source` starts with `field_rules.` |
| Site/domain constraint | +6 | Query contains `site:` operator, OR `source_hosts` populated, OR `domain_hints` populated |
| Both | +14 | Both conditions present |

**Detection code:** `prefetchQueryJourneyHelpers.js:89-94`
```javascript
const hasFieldRules = row.hint_sources.some(
  (source) => normalizeToken(source).startsWith('field_rules.')
);
const hasDomainConstraint = row.source_hosts.length > 0
  || row.domain_hints.length > 0
  || /\bsite:\s*[a-z0-9.-]+/i.test(row.query);
```

**Why field-rule hints score highest (+8):** These represent explicit curator knowledge about where to find data for specific fields — more targeted than generic domain constraints.

**Why site: scores +6:** `site:eloshapes.com` narrows the search space to a single known high-value domain, making the query inherently more productive.

## Sort Order

**Source:** `prefetchQueryJourneyHelpers.js:218-237`

```javascript
return rows.sort((a, b) => {
  // Sent queries first, ordered by execution_order
  if (a.execution_order != null && b.execution_order != null)
    return a.execution_order - b.execution_order;
  if (a.execution_order != null) return -1;
  if (b.execution_order != null) return 1;
  // Unsent queries by Pscore descending
  if (a.order_metric !== b.order_metric)
    return b.order_metric - a.order_metric;
  // Tie-breaker: lexicographic
  return a.query.localeCompare(b.query);
});
```

**Result:**
1. Sent queries (ascending by `execution_order`)
2. Unsent queries (descending by Pscore, then alphabetical)

## Pre-Execution Prioritization (Backend)

Before queries reach the panel, the backend applies its own priority sort to decide execution order. This is a separate scoring system from Pscore.

**Source:** `discoveryQueryPlan.js:297-327` (`prioritizeQueryRows`)

| Signal | Points | Notes |
|--------|--------|-------|
| `site:` prefix | +6 | Domain-scoped queries execute first |
| Doc-type keywords (manual, datasheet, spec, pdf) | +5 | Technical content bias |
| Brand token in query | +3 | Brand identity match |
| Brand word in query | +2 | Full brand name match |
| Model word in query | +2 | Model name match |
| Known review sites (rtings, techpowerup) | +1 | Trusted lab sources |
| Target fields covering missing fields | +3 per field (max +6) | Missing field coverage |
| Domain hint present | +2 | Explicit domain guidance |
| Doc hint present | +1 | Document type guidance |

This backend score determines **which queries SearXNG receives first** (and thus which ones get `execution_order = 1, 2, 3...`). The panel's Pscore then ranks whatever is left unsent.

## Identity Guard (Pre-Execution Filter)

Before execution, all queries pass through `enforceIdentityQueryGuard()` (`discoveryQueryPlan.js:332-443`):

- **Must contain brand tokens** (if brand exists)
- **Must contain all required digit groups** (from model numbers like "g502" -> requires "502")
- **Must not contain foreign model tokens** (e.g., a "g305" token when product is "g502")

Rejected queries are logged in `query_reject_log` with reasons and never reach execution.

## Worked Example

Product: **Asus ROG Harpe Ace Extreme** (mouse category)

### Sent Queries (execution order)

| # | Query | Selected By | Justification | Results |
|---|-------|-------------|---------------|---------|
| 1 | `Asus ROG Harpe Ace Extreme specs site:eloshapes.com` | Deterministic | T+0.0s. First query sent. | 4 |
| 2 | `Asus ROG Harpe Ace Extreme sensor specifications` | Planner (Primary) | T+1.2s. Sent later. | 7 |
| 3 | `ROG Harpe Ace Extreme DPI weight dimensions` | Planner (Primary+Reason) | T+2.8s. Sent later. | 3 |

### Unsent Queries (Pscore order)

| Rank | Query | Pscore | Breakdown |
|------|-------|--------|-----------|
| 4 | `ROG Harpe Ace Extreme datasheet PDF site:rog.asus.com` | P50 | pass 14 + target 20 + attempts 2 + constraints 14 |
| 5 | `Asus ROG Harpe Ace Extreme review sensor test` | P42 | pass 28 + target 12 + attempts 2 + constraints 0 |
| 6 | `Harpe Ace Extreme switch type click latency site:rtings.com` | P28 | pass 14 + target 8 + attempts 0 + constraints 6 |

**Reading the breakdown:**
- Query ranked #4 has Pscore 50: Primary pass (+14), targets 5 missing fields (+20), 1 prior attempt (+2), has both field-rule hint and site: constraint (+14)
- Query ranked #5 has Pscore 42: Validate pass (+28), targets 3 fields (+12), 1 prior attempt (+2), no constraints (+0)
